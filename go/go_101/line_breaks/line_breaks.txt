-- Line Break & Semicolon Insertion Rules
--
  In Go, code style is not just a matter of preference; it is enforced by the
  language grammar. Specifically, Go utilizes an automatic semicolon insertion
  mechanism that dictates where you can and cannot break a line of code.





-- The Semicolon Insertion Mechanism
--
  Although Go code rarely contains visible semicolons (`;`), the formal Go
  grammar uses them as statement terminators. To keep the code clean, the
  Go lexer automatically inserts a semicolon at the end of a line if the
  line's final token is:

  1. An identifier (variable name, function name, etc.).
  2. A basic literal (e.g., 123, 3.14, "hi", 'a').
  3. One of the following keywords: break, continue, fallthrough, return.
  4. One of the following operators/delimiters: ++, --, ), ], }.

  Because of this, breaking a line after these tokens effectively "ends" the
  statement, which can lead to syntax errors if the statement was intended
  to continue on the next line.


-- The Curly Brace Rule ({)
--
  A common rule in Go is that the opening brace `{` of a code block (like in
  `if`, `for`, `switch`, or `func`) must stay on the same line as the
  preceding statement.


  example
    ```go
      func braceRules() {
        // --  Example: Incorrect Line Break

        // for i := 5; i > 0; i--
        // { // COMPILE ERROR: unexpected newline, expecting { after for clause
        // }


        // why it fails:
        // The token 'i--' is in the lexer's "insert semicolon" list.
        // The compiler sees: for i := 5; i > 0; i--; {
        // This makes the semicolon separate the for-header from its body.


        // --  Example: Correct Line Break

        for i := 5; i > 0; i-- { // OK: No semicolon inserted between -- and {
          fmt.Println(i)
        }
      }

    ```





-- Exceptions: When Newlines are Allowed
--
  If the token ending a line is not in the semicolon-triggering list,
  a line break is perfectly legal.


  example
    ```go
      func legalBreakExamples() {
        // -- Bare For Loop
        // The keyword 'for' is NOT in the list of tokens that trigger semicolons.
        // Therefore, the opening brace CAN go on a new line here.
        for
        {
          fmt.Println("Infinite loop")
          break
        }


        // -- Trailing Operators
        // If you end a line with an operator that requires a second operand
        // (like +, -, *, /), the lexer will NOT insert a semicolon.
        x := 10 +
          20 +
          30
        fmt.Println(x) // Output: 60
      }

    ```





-- The Comma Rule in Composite Literals
--
  Semicolon insertion also explains why trailing commas are often mandatory
  in Go multi-line definitions.


  example
    ```go
      func commaRuleExample() {
        // -- Single Line
        // Commas separate elements; the closing '}' is on the same line.
        _ = []int{1, 2, 3}


        // -- Multi-Line
        _ = []int{
          1,
          2,
          3, // <--- MANDATORY COMMA
        }

        // WHY:
        // If you omit the comma after '3', the lexer sees the literal '3' at the
        // end of the line and inserts a semicolon: 3;
        // The compiler then sees: []int{ 1, 2, 3; } -> This is a syntax error.
        // The comma prevents the semicolon from being inserted.
      }

    ```





-- Semicolon Insertion Rules (ASI)
--
  Although Go code rarely contains visible semicolons (`;`), the formal
  language grammar uses them as statement terminators. To keep source code
  clean, Go compilers use "Automatic Semicolon Insertion" (ASI).


  -- The "Noisy" vs. "Clean" Go
  --
    Most semicolons in Go are optional for the programmer because the lexer
    inserts them into the token stream automatically during compilation.


  example
    ```go
      func comparison() {
        // -- The "Noisy" Way (Valid but non-idiomatic)
        // You can manually add semicolons, but 'go fmt' will strip them.
        var i int;
        i = 5;
        fmt.Println(i);


        // -- The "Clean" Way (Idiomatic)
        // The compiler sees these as identical to the noisy way above.
        var j int
        j = 5
        fmt.Println(j)
      }

    ```


  -- The Insertion Rules (The Lexer's Logic)
  --
    When the compiler breaks your code into tokens, it automatically inserts a
    semicolon after a line's final token IF that token is:

    1. An IDENTIFIER (variable names, function names, etc.).
    2. A LITERAL (integer, float, imaginary, rune, or string).
    3. A specific KEYWORD: break, continue, fallthrough, or return.
    4. A specific OPERATOR or PUNCTUATION: ++, --, ), ], or }.


    -- Rule 2: Single-line Omission
    --
      A semicolon may be omitted before a closing ')' or '}'. This allows
      complex statements to stay on one line.


    example
      ```go
        func insertionLogic() {
          // The compiler effectively sees:
          // x := 10;
          x := 10


          // The compiler effectively sees:
          // if x > 5 { fmt.Println("Hi"); };
          if x > 5 { fmt.Println("Hi") }
        }

      ```


  -- Code Demonstration: Semicolon "Invisibility"
  --
    This example shows a valid Go program with every possible optional semicolon
    included. Running `go fmt` would transform this into the clean Go we use daily.

    ```go
      package main; // Inserted after identifier 'main'

      import "fmt"; // Inserted after string literal '"fmt"'

      func main() {
        var (
          i   int; // Inserted after identifier 'int'
          sum int;
        );           // Inserted after ')'

        for i < 6 {
          sum += i; // Inserted after identifier 'i'
          i++;      // Inserted after operator '++'
        };            // Inserted after '}'

        fmt.Println(sum);
      }; // Inserted after '}'

    ```



  -- The "Why" Behind Formatting Constraints
  --
    Because Go is so predictable with its semicolon insertion, it forces
    certain styles. For instance, the "Opening Brace Rule":


    exmaple
      ```go
        func example()
        { // ERROR: Lexer inserted a ';' after 'example()', separating the body!
        }

        func example() { // OK: Lexer sees '{' and knows the statement continues.
        }

      ```





  -- Semicolon Insertion - Rule 2 and Its Consequences
  --
    Go's formal grammar relies on semicolons (;) to separate statements.
    While the first rule of Automatic Semicolon Insertion (ASI) handles newlines,
    the "Second Rule" handles the boundaries of blocks and lists.


    -- Rule 2: Omission Before Closing Punctuation
    --
      A semicolon may be omitted immediately before a closing parenthesis ')' or
      a closing curly brace '}'.


    This rule allows us to group multiple declarations or statements on a single
    line without needing a trailing semicolon at the very end of the group.


    example
      ```go
        package main


        import (
          _ "math"; "fmt" // Semicolon required BETWEEN items, optional before ')'
        )


        func ruleTwoExamples() {
          // Single-line variable group
          var ( a int; b string )

          // Single-line constant group
          const ( M = iota; N )

          // Single-line type definitions
          type ( MyInt int; T struct{ x bool; y int32 } )

          // Single-line interface definition
          type I interface{ m1(int) int; m2() string }

          // Single-line function body
          f := func() { fmt.Print("a"); fmt.Println("b") }

          f()
        }

      ```


  -- Under the Hood: What the Compiler Sees
  --
    When the compiler processes the "shorthand" code above, it effectively
    normalizes it by inserting the missing semicolons before the closing
    characters.

    -- How the Compiler Sees Rule 2
    --
      ```go
        var ( a int; b string; );
        const ( M = iota; N; );
        type I interface{ m1(int) int; m2() string; };
        func f() { fmt.Print("a"); fmt.Println("b"); };

      ```


  -- When Semicolons are mandatory
  --
    The compiler only inserts semicolons at newlines (Rule 1) or before ')' and '}'.
    If you put multiple statements on the same line, you must use a semicolon
    to separate them.


    example
      ```go
        func mandatorySemicolons() {
          // Multiple variables on one line
          var x = 1; var y = 2


          // Multiple statements on one line
          x++; y = x * 2


          // Multiple calls on one line
          fmt.Print(x); fmt.Println(y)

          _ = y
        }

      ```





-- Case Study: Why the Bare `for` Loop is Legal
--
  As established, the lexer only inserts a semicolon after specific tokens
  (identifiers, literals, ++, ), }, etc.).

  The keyword 'for' is NOT on that list. Therefore, if you write:
    ```go
      for
      {
        // ..
      }

    ```

  The lexer DOES NOT insert a semicolon after 'for'. If it did (for;), the
  loop would be syntactically broken. This is why the code above is valid Go.





-- Consequences: Increment/Decrement are NOT Expressions
--
  In many languages (C, Java), `a++` is an expression that returns a value.
  In Go, `++` and `--` are strictly statements.


  The semicolon rules are the reason why. If Go allowed `++` in an expression,
  the internal statement termination would become ambiguous.


  ```go
    func incrementStatementDemo() {
      a := 0

      // -- INVALID CODE
      // println(a++)


      // WHY IT FAILS:
      // The lexer sees the token '++', which is an "insert semicolon" trigger.
      // It also sees the closing ')', which triggers Rule 2 (insert semicolon before ')').

      // The compiler effectively sees:
      // println(a++;);

      // This is a syntax error because println() expects a value (expression),
      // not a terminated statement.

    }

  ```





-- Line Breaks in Selectors & Method Chaining
--
  Go's Automatic Semicolon Insertion (ASI) rules create a specific constraint
  when performing method chaining or accessing fields across multiple lines.
  Specifically, the placement of the dot (`.`) relative to the newline is
  syntactically significant.


  -- The Rule: No Leading Dots on Newlines
  --
    In Go, you cannot start a newline with a dot (`.`) for a selector.


    -- Why?
    --
      Remember the ASI rule: If a line ends with an identifier or a closing
      parenthesis `)`, the lexer automatically inserts a semicolon.

      If you write:
         anObject
             .MethodA()

    The lexer sees `anObject` (an identifier) followed by a newline. It inserts
    a semicolon: `anObject;`. The next line `.MethodA()` then becomes a syntax
    error because a statement cannot start with a dot.


  -- The Correct Way: Trailing Dots
  --
    To wrap a method chain across multiple lines, the dot MUST remain at the
    END of the previous line.


    -- Why?
    --
      The dot `.` is NOT in the list of tokens that trigger a semicolon.
      By ending a line with a dot, you tell the compiler: "This statement is
      incomplete; keep looking on the next line."


    example
      ```go
        type Builder struct {
          value string
        }


        func (b Builder) Add(s string) Builder {
          b.value += s
          return b
        }


        func (b Builder) Print() {
          fmt.Println("Result:", b.value)
        }


        func lineBreakDemo() {
          b := Builder{value: "Go"}


          // --  LEGAL: Breaking AFTER the dot

          // The lexer does not insert semicolons after the dots, so the statement
          // continues correctly.
          b.
            Add(" is").
            Add(" Awesome").
            Print()


          // --  ILLEGAL: Breaking BEFORE the dot

          // b
          //  .Add(" fails") // COMPILE ERROR: unexpected .
          //
        }

      ```





-- Semicolon Insertion - Edge Cases & Traps
--
  Go's Automatic Semicolon Insertion (ASI) is generally a convenience that
  leads to cleaner code. However, because it is based on rigid rules rather
  than developer intent, it can occasionally produce valid Go code that
  behaves in unexpected, "weird," or even paradoxical ways.


  -- Syntactic Optical Illusions: Multi-line Control Flow
  --
    Because the keywords `for`, `if`, and `switch` do not trigger a semicolon
    insertion, you can technically spread their headers across multiple lines.


    example
      ```go
        func alwaysFalse() bool { return false }


        func weirdControlFlow() {
          // -- The Multi-line For

          for
          i := 0  // Semicolon inserted here: i := 0;
          i < 3   // Semicolon inserted here: i < 3;
          i++ {   // Semicolon inserted here: i++;
            fmt.Print(i)
          }
          // Result: 012


          // -- The Multi-line If

          if x := alwaysFalse() // Semicolon inserted: x := alwaysFalse();
          !x {                   // Logic: If NOT false (true)
            fmt.Println("\nIf block executed")
          }
        }

      ```


  -- THE SWITCH TRAP: Semantic Logic Shift
  --
    This is a dangerous edge case. Placing the opening brace `{` of a `switch`
    on a new line changes the logic of the program entirely.

    -- The Syntax Rule
    --
      Switch has the form: switch Init; Expression { ... }
      If Expression is omitted, it defaults to 'true'.


    example
      ```go
      func switchTrapDemo() {
        // scenario A: Standard Switch
        // The result of alwaysFalse() is compared against the cases.
        switch alwaysFalse() {
        case true:  fmt.Println("A: true")
        case false: fmt.Println("A: false")
        }


        // SCENARIO B: The Trap (Newline before brace)
        // The lexer sees `alwaysFalse()` followed by a newline and inserts a ';'.
        switch alwaysFalse()
        {
        case true:  fmt.Println("B: true")
        case false: fmt.Println("B: false")
        }


        // WHY SCENARIO B PRINTS "TRUE":
        // The compiler rewrites Scenario B as:
        //    switch alwaysFalse(); true { ... }

        // 1. `alwaysFalse()` becomes the optional "Initialization Statement".
        // 2. Because there is a newline/semicolon before the brace, the
        //    "Comparison Expression" is missing.
        // 3. Go defaults missing comparison expressions to `true`.
        // 4. The switch now checks: "Does true == true?" or "Does true == false?"
        // 5. It executes `case true` regardless of what the function returned.
      }

      ```


  -- The Label Trap: Labels vs. Blank Statements
  --
    Rule: A label declaration (e.g., `MyLabel:`) MUST be followed by a statement.
    Even a "Blank Statement" (a lone semicolon) satisfies this rule.


    example
      ```go
        func labelTrapDemo(x int) {
          switch x {
          case 1:
            {
              goto A
            A: // OK: Followed by '}', which triggers ASI Rule 2 (semicolon insertion)
            }
          case 2:
            goto B
            // B:
            // COMPILE ERROR: missing statement after label
            // WHY: 'case' is a keyword, not a statement. There is no '}' here to
            // trigger a semicolon, so the label is "naked."

            // To fix, we can add a manual blank statement:
            B: ;
          case 0:
            goto C
          C: // OK: Followed by '}', which triggers a hidden semicolon/blank statement.
          }
        }

      ```

      -- Internal View: How the Compiler sees Labels 
      --
        // case 1:
        //   {
        //     goto A
        //     A:
        //   ;} // Semicolon automatically inserted before '}'





-- Mandatory vs. Optional Commas
--
  In Go, commas act as separators in lists (parameters, arguments, composite 
  literals). Unlike some languages that strictly forbid "trailing commas," 
  Go's syntax rules make them either optional or absolutely mandatory 
  depending on the placement of newlines.


  -- The Golden Rule: Newlines and Separators
  --
    Go compilers NEVER insert commas automatically. However, they DO insert 
    semicolons automatically.

    -- The Rule 
    --
      1. Mandatory: If an item in a list is followed by a NEWLINE, a comma MUST be 
         present after that item.
      2. Optional: If the closing delimiter ( `)`, `]`, or `}` ) is on the SAME 
         line as the last item, the trailing comma is optional.


  -- Why it happens: The ASI Interaction
  --
    This rule exists because of Automatic Semicolon Insertion (ASI). 
    Recall that ASI inserts a semicolon if a line ends with a literal, 
    identifier, or closing punctuation.


    ```go
      // -- The Failure Logic 
      vals := []int{
        1,
        2  // <--- No comma here!
      }


      // 1. The lexer sees the integer literal '2' followed by a newline.
      // 2. ASI triggers and inserts a semicolon: `2;`
      // 3. The compiler sees: `[]int{ 1, 2; }`
      // 4. Result: SYNTAX ERROR (unexpected semicolon, expecting comma or }).

    ```


  -- Code Demonstration: Valid Syntax
  --
    ```go
      // Trailing commas in parameter/result lists are valid even on one line.
      func multiply(a int, b int, ) (result int, ) {
        return a * b,
      }


      func validCommaExamples() {
        // --  Composite Literals (Slices/Arrays) 
        _ = []int{2, 3, 5, 7, 9}  // OK: Standard
        _ = []int{2, 3, 5, 7, 9,} // OK: Optional trailing comma on same line


        _ = []int{
          2,
          3,
          5, // OK: Required comma because of newline
        }


        // -- Function Calls
        fmt.Println(1, 2, 3)   // OK: Standard
        fmt.Println(1, 2, 3, ) // OK: Optional trailing comma


        fmt.Println(
          "Multi-line",
          "call",
          "requires",
          "comma", // OK: Mandatory comma
        )


        // -- Explicit Type Conversions 
        // Even single-argument conversions follow the rule if split by lines.
        _ = string(
          65, 
        )
      }

    ```


  -- Code Demonstration: Invalid Syntax (The ASI Trap)
  --
    ```go
      func invalidExamples() {
        // ERROR: Missing comma before newline
        _ = []int{
          1,
          2  // Syntax Error: unexpected newline, expecting comma or }
        }


        // ERROR: Missing comma in function signature
        var _ = func(a int, b string // Syntax Error: unexpected newline
        ) {}


        // ERROR: Missing comma in conversion
        _ = string(65
        ) // Syntax Error: unexpected newline
      }

    ```
