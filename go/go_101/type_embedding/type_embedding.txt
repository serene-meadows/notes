-- Type Embedding (Anonymous Fields)
--
  Go does not support traditional class-based inheritance. Instead, it favors
  composition over inheritance, and its primary tool for this is Type Embedding.


  Type Embedding allows you to declare a struct field using only a type name,
  without an explicit field name. This provides a powerful way to "absorb" the
  properties and methods of the embedded type.





-- The Syntax: Declaring an Anonymous Field
--
  In a struct definition, you can simply list a type name on its own line.
  This creates an "embedded field," which is also commonly called an
  "anonymous field".


  example
    ```go
      func embeddingSyntax() {
        // Let's create some type aliases for clarity
        type P = *bool
        type M = map[int]int


        // `x` is a variable of an anonymous struct type that embeds six other types.
        var x struct {
          // -- The Embedded Fields
          string      // A named non-pointer basic type
          error       // A named interface type
          *int        // An unnamed pointer type
          P           // An alias to an unnamed pointer type
          M           // An alias to an unnamed map type
          http.Header // A named type from another package
        }
        _ = x
      }

    ```





-- The Implicit Name Rule
--
  While these fields are called "anonymous," they are not truly nameless.
  The Go compiler automatically assigns an implicit name to each embedded field.


  The Rule: The implicit name of an embedded field is the unqualified
  type name of the embedded type.


  example
    ```go
      func implicitNaming() {
        type P = *bool
        type M = map[int]int


        var x struct {
          string
          error
          *int
          P
          M
          http.Header
        }


        // We can access each field using its implicit name.
        // The implicit name of `string` is "string".
        x.string = "Go"


        // The implicit name of `error` is "error".
        x.error = nil


        // The implicit name of `*int` is "int".
        x.int = new(int)


        // The implicit name of `P` (which is `*bool`) is "P".
        x.P = new(bool)


        // The implicit name of `M` (which is `map[int]int`) is "M".
        x.M = make(M)


        // The implicit name of `http.Header` is "Header".
        x.Header = http.Header{}
      }

    ```

  CRITICAL NOTE: Because the implicit name is the type name, you can only
  embed a specific type (or a pointer to it) ONCE within a struct. You
  cannot embed `string` twice, for example, as this would create a field name
  collision.





-- Type Embedding Rules
--
  Go has strict rules about which types can be used as "anonymous" embedded
  fields in a struct. While the official specification can be dense, the rules
  can be simplified into a few key principles.


  -- The Core Rules of Embedding
  --
    You can embed a type `T` if it is a named type or a pointer to a named type.
    However, there are some critical restrictions on what `T` can be.


    -- Rule A: You CAN Embed...
      1. A named non-pointer type (`T`). This includes structs, interfaces, and
         custom defined basic types.
      2. A pointer to a named non-pointer type (`*T`). This applies to structs and
         custom defined basic types.


    -- Rule B: You CANNOT Embed...
      1. A named pointer type itself.
      2. A pointer to an interface type (`*MyInterface`).
      3. A pointer-to-a-pointer type (`**MyType`).
      4. Any unnamed type (except for pointers to named types, e.g., `*int`).


  -- Examples of Legal and Illegal Embeddings
  --
    ```go
      // Let's define some types to test the rules.
      type Encoder interface { Encode([]byte) []byte }
      type Person struct{ name string; age int }
      type Alias = struct{ name string; age int } // Alias to an unnamed struct
      type AliasPtr = *struct{ name string; age int } // Alias to a pointer
      type IntPtr *int // A defined (named) pointer type
      type AliasPP = *IntPtr // Alias to a pointer-to-a-pointer


      func embeddingExamples() {

        // -- LEGAL Embeddings
        var _ struct {
          Encoder     // Named interface type
          Person      // Named struct type
          *Person     // Pointer to a named struct type
          Alias       // Alias to an unnamed struct (acts like a named type here)
          AliasPtr    // Alias to a pointer to a struct
          int         // Pre-declared named basic type (acts as an embedded field)
          *int        // Unnamed pointer to a named basic type
        }


        // -- ILLEGAL Embeddings (All cause COMPILE ERRORS)
        // var _ struct {
        //   // Rule B1: Cannot embed a named pointer type itself.
        //   IntPtr

        //   // Rule B2: Cannot embed a pointer to an interface.
        //   *Encoder

        //   // Rule B3: Cannot embed a pointer-to-a-pointer.
        //   *IntPtr
        //   AliasPP

        //   // Rule B4: Cannot embed unnamed non-pointer types.
        //   struct{ age int }
        //   map[string]int
        //   []int64
        //   func()
        //
        //   // Unnamed pointer to an unnamed type
        //   *chan int
        // }
      }

    ```





-- The "No Duplicate Field Names" Rule
--
  A struct cannot have two fields with the same name. This rule extends to the
  implicit names generated by type embedding.


  Because the implicit name of an embedded type `*T` is `T`, you CANNOT embed
  both `T` and `*T` in the same struct.

  example
    ```go
    // COMPILE ERROR: duplicate field 'MyType'

    // type MyContainer struct {
    //     MyType
    //     *MyType
    // }

    ```





-- The "No Recursive Embedding" Rule
--
  A struct type cannot embed itself, either directly or indirectly through a
  chain of other structs. If it could, the compiler would not be able to
  determine the size of the struct, as it would be infinitely large.


  example
    ```go
      // COMPILE ERROR: invalid recursive type Node
      // type Node struct {
      //     Node // Direct recursive embedding
      // }

      // type A struct { B }
      // type B struct { A } // Indirect recursive embedding

    ```

  The Fix: To create recursive data structures like linked lists or trees,
  you must use a POINTER to the type, not the type itself.

  example
    ```go
      type LinkedListNode struct {
        Value int
        Next  *LinkedListNode // OK: Pointers have a known, fixed size.
      }

    ```





-- The Purpose of Type Embedding (Composition)
--
  The primary purpose of type embedding in Go is to extend the functionality
  of a type by absorbing the fields and methods of another type. This provides
  powerful code reuse without needing to re-implement existing logic.


  -- Embedding vs. Inheritance (Has-A vs. Is-A)
  --
    Many classic Object-Oriented languages (like Java or C++) use "Inheritance"
    to reuse code. Go entirely rejects inheritance in favor of "Composition"
    via Type Embedding.


    -- Inheritance (The "Is-A" Relationship)
    --
      If type `Singer` inherits from `Person`, then a `Singer` IS A `Person`.
      You can pass a `Singer` object to any function that expects a `Person`.


    -- Go's Embedding (The "Has-A" Relationship)
    --
      If type `Singer` embeds `Person`, the `Person` becomes an internal part of
      `Singer`. The `Singer` HAS A `Person`.
      The `Singer` obtains the abilities (methods and fields) of the `Person`,
      but a `Singer` IS NOT a `Person` in the eyes of the type system.


    -- Code Demonstration: Extending Functionality
    --
      ```go
        // -- The Base Type
        type Person struct {
          Name string
          Age  int
        }


        func (p Person) PrintName() {
          fmt.Println("Name:", p.Name)
        }


        func (p *Person) SetAge(age int) {
          p.Age = age
        }


        // -- The Embedding Type
        type Singer struct {
          Person   // Anonymous field: Embeds Person to extend Singer's capabilities
          Genre    string
        }


        func extendingFunctionalityDemo() {
          // Notice how we initialize the embedded field using its implicit name "Person"
          var gaga = Singer{
            Person: Person{Name: "Gaga", Age: 30},
            Genre:  "Pop",
          }


        // -- "Promotion"
        // Because `Person` is embedded, its fields and methods are "promoted" to
        // the top level of `Singer`. We can access them directly!

        gaga.PrintName() // Output: Name: Gaga


        // We can mutate the promoted fields directly.
        gaga.Name = "Lady Gaga"


        // Go's standard pointer syntactic sugar works on promoted methods too!
        gaga.SetAge(31)


        gaga.PrintName()      // Output: Name: Lady Gaga
        fmt.Println(gaga.Age) // Output: 31


          // What is actually happening under the hood?
          // The direct access is just syntactic sugar. The compiler translates it:
          // gaga.Name       ---> gaga.Person.Name
          // gaga.PrintName() ---> gaga.Person.PrintName()
        }

      ```





-- The Strict Typing Rule (No Polymorphism via Embedding)
--
  As stated in the "Has-A vs Is-A" rule, embedding does NOT create a type
  hierarchy. A `Singer` is a completely distinct type from `Person`.


  example
    ```go
      func strictTypingDemo() {
        var gaga = Singer{Person: Person{Name: "Lady Gaga"}}


        // You CANNOT use an embedded type where the base type is expected.
        // This prevents the fragile base class problems common in OOP languages.


        // var _ Person = gaga
        // COMPILE ERROR: cannot use gaga (type Singer) as type Person in assignment


        // If you need to extract the base type, you must access it explicitly
        // via its implicit field name:
        var thePerson Person = gaga.Person
        _ = thePerson
      }

    ```





-- Field and Method Promotion in Type Embedding
--
   When one type embeds another, a process called "Promotion" occurs. This is
   the mechanism that allows us to access the internals of the embedded type
   as if they belonged to the outer type.


  -- The Concept of Promotion
  --
    Promotion provides a shorthand (syntactic sugar) for accessing fields and
    methods of an embedded field.


    - If a struct `S` embeds a type `T`.
    - And `T` has a field `F` and a method `M`.
    - Then `F` and `M` are "promoted" to the top level of `S`.


    Accessing `s.F` is automatically translated by the compiler to `s.T.F`.
    Accessing `s.M()` is automatically translated to `s.T.M()`.





  -- Fields vs. Methods: A Subtle Distinction
  --
    There is a significant technical difference in how Go treats promoted
    fields versus promoted methods.


    -- Field Promotion (Shorthand Only)
    --
      The embedding type does NOT "own" the promoted fields. They do not appear
      in the embedding type's field list. They remain strictly part of the
      internal embedded struct.


    -- Method Promotion (Actual Ownership)
    --
      Unlike fields, promoted methods ARE added to the "Method Set" of the
      embedding type. For all intents and purposes, the embedding type "owns"
      these methods.


    -- The Reflection Proof
    --
      ```go
        type Person struct {
          Name string // Field
          Age  int    // Field
        }


        func (p Person) PrintName() {
          fmt.Println("Name:", p.Name)
        }


        func (p *Person) SetAge(age int) {
          p.Age = age
        }


        type Singer struct {
          Person // Embedded Field
          works  []string
        }


        func main() {
          t := reflect.TypeOf(Singer{})


          // --  Field Inspection

          // Even though we can call `singer.Name`, reflection shows that `Singer`
          // only has TWO fields: `Person` (the anonymous field) and `works`.
          // `Name` is NOT a field of `Singer`.

          fmt.Printf("%v has %d fields:\n", t, t.NumField())
          for i := 0; i < t.NumField(); i++ {
            fmt.Printf(" - field#%d: %v\n", i, t.Field(i).Name)
          }


          // --  Method Inspection (Value Receiver)

          // `PrintName` is promoted to the method set of `Singer`.
          // Reflection confirms that `Singer` effectively "owns" this method.

          fmt.Printf("\n%v has %d methods:\n", t, t.NumMethod())
          for i := 0; i < t.NumMethod(); i++ {
            fmt.Printf(" - method#%d: %v\n", i, t.Method(i).Name)
          }


          // --  Method Inspection (Pointer Receiver)

          // `*Singer` inherits all methods of `Singer` (PrintName) PLUS
          // the promoted pointer methods of `*Person` (SetAge).

          pt := reflect.TypeOf(&Singer{})
          fmt.Printf("\n%v has %d methods:\n", pt, pt.NumMethod())
          for i := 0; i < pt.NumMethod(); i++ {
            fmt.Printf(" - method#%d: %v\n", i, pt.Method(i).Name)
          }
        }

      ```

      -- explaining the selector `gaga.Name`:
      --
        If `Singer` doesn't own the `Name` field, why is `gaga.Name` legal?


        Because the Go compiler performs an automatic search:
          1. Does `Singer` have a field named `Name`? -> No.
          2. Does `Singer` have an embedded field that has a field/method named `Name`?
          3. It finds `Name` inside the embedded `Person` field.
          4. It rewrites `gaga.Name` to `gaga.Person.Name` internally.


        This is strictly a convenience for the developer (Syntactic Sugar).





-- Selector Shorthands & Type Promotion
--
  A "selector" is an expression used to access a field or method of a value (e.g., x.y).
  When using Type Embedding, Go allows you to omit the names of embedded fields
  in these selectors. This mechanism is called "Promotion".


  -- The Shorthand Rule
  --
    If a middle name in a selector chain corresponds to an embedded (anonymous)
    field, that name can be omitted from the selector.


    Full Form:  x.EmbeddedField.Target
    Shorthand:  x.Target


    This applies to both fields and methods. This is precisely why embedded
    fields are often called "anonymous"—they act as invisible bridges in
    the selector path.


    -- Code Demonstration: Multi-Level Shorthands
    --
      ```go
        type A struct {
          FieldX int
        }


        func (a A) MethodA() {
          fmt.Println("Executing MethodA")
        }


        // B embeds a pointer to A
        type B struct {
          *A
        }


        // C embeds B (which embeds *A)
        type C struct {
          B
        }


        func selectorShorthands() {
          // Construction: C -> B -> *A
          var c = &C{
            B: B{
              A: &A{FieldX: 5},
            },
          }


          // --  Field Promotion

          // These 4 lines are logically identical and evaluate to the same value:
          _ = c.B.A.FieldX // Full Form
          _ = c.B.FieldX   // Omitted 'A'
          _ = c.A.FieldX   // Omitted 'B'
          _ = c.FieldX     // Omitted 'B' and 'A' (The "Shortest" Shorthand)


          // --  Method Promotion --

          // These 4 lines are logically identical:
          c.B.A.MethodA() // Full Form
          c.B.MethodA()   // Omitted 'A'
          c.A.MethodA()   // Omitted 'B'
          c.MethodA()     // Omitted 'B' and 'A' (The "Shortest" Shorthand)
        }

      ```


  -- Terminology: Promoted Fields & Methods
  --
    - Promoted Field: A field belonging to an embedded type that is accessible
      on the embedding type without using the embedded type's name.
    - Promoted Method: A method belonging to an embedded type that is added to
      the Method Set of the embedding type.


    NOTE: Even though you can use the shorthand, the "Full Form" is always legal
    and remains the explicit way to refer to the data.


  -- Selector Depth
  --
    The "Depth" of a selector is the number of middle names that correspond
    to embedded fields in the Full Form of that selector.


    Example using the code above (C -> B -> A):
      Selector: c.FieldX
      Full Form: c.B.A.FieldX
      Omitted Names: [B, A]
      Depth: 2

      Selector: c.B.FieldX
      Full Form: c.B.A.FieldX
      Omitted Names: [A]
      Depth: 1


  -- Interaction with Pointer Sugaring
  --

    Go's method call syntactic sugar (automatic address-taking) works perfectly
    with promoted methods.

    example
      ```go
        type Person struct {
          Age int
        }


        func (p *Person) SetAge(a int) {
          p.Age = a
        }


        type Singer struct {
          Person
        }


        func addressabilityAndPromotion() {
          var gaga Singer = Singer{Person: Person{Age: 30}}

          // 'SetAge' is a promoted method from '*Person'.
          // Even though 'gaga' is a value (Singer), not a pointer (*Singer),
          // Go automatically takes its address because 'gaga' is addressable.


          gaga.SetAge(31) // Sugar for: (&gaga.Person).SetAge(31)


          fmt.Println("Age:", gaga.Age) // Output: 31
        }

      ```





-- Selector Shadowing and Colliding
--
  When using type embedding, multiple fields or methods with the same name can 
  exist at different levels of the embedding hierarchy. Go resolves these 
  potential conflicts using two rules: Shadowing and Colliding.


  -- The Rule of Depth
  --
    The "depth" of a selector is the number of embedded fields traversed to reach
    the target. 
    - Depth 0: A field/method defined directly on the struct.
    - Depth 1: A field/method promoted from an immediately embedded type.
    - Depth 2+: A field/method promoted from a type embedded within an embedded type.

    The Principle: The compiler always looks for the "shallowest" possible path.


  -- Selector Shadowing (The Shallowest Wins)
  --
    If there is exactly ONE full-form selector at the shallowest depth, that 
    selector is chosen. Any other selectors with the same name at deeper levels 
    are "shadowed" (hidden).


  -- Method Overriding 
  --
    If a method shadows another method with the exact same signature, we say the 
    shallow method "overrides" the deeper one.


  example
    ```go
      type Base struct {
        Name string
      }


      func (Base) Describe() { fmt.Println("I am Base") }


      type Derived struct {
        Base // Embedded
        Name int  // This field shadows Base.Name (Depth 0 vs Depth 1)
      }


      // This method shadows Base.Describe (Depth 0 vs Depth 1)
      func (Derived) Describe() { fmt.Println("I am Derived") }


      func shadowingDemo() {

        d := Derived{
          Name: 42,
          Base: Base{Name: "Old Name"},
        }


        // Shadowed Field: Accesses Derived.Name (int), not Base.Name (string)
        fmt.Printf("Type of Name: %T\n", d.Name) // Output: int


        // Shadowed Method: Accesses Derived.Describe
        d.Describe() // Output: I am Derived


        // You can still access shadowed items using their full-form selectors:
        fmt.Println("Shadowed Base Name:", d.Base.Name) // Output: Old Name
      }

    ```


  -- Selector Colliding (Ambiguity Error)
  --
    If multiple full-form selectors exist at the SAME shallowest depth, they 
    "collide." The compiler cannot decide which one you intended to use.


    Result: The shorthand selector (x.y) becomes illegal and causes a 
    compile-time error: "ambiguous selector."

  
    example
      ```go
        type A struct {
          x string
        }
        func (A) y(int) bool { return false }


        type B struct {
          y bool
        }
        func (B) x(string) {}
        

        func collidingDemo() {
          var v1 struct {
            A
            B
          }


          // COLLISION: 
          // v1.x could be v1.A.x (Depth 1) or v1.B.x (Depth 1).
          // v1.y could be v1.A.y (Depth 1) or v1.B.y (Depth 1).
          

          // _ = v1.x // COMPILE ERROR: ambiguous selector v1.x
          // _ = v1.y // COMPILE ERROR: ambiguous selector v1.y


          // Fix: Use the full-form selector to resolve ambiguity.
          _ = v1.A.x
          _ = v1.B.x
        }

      ```





-- Deep Promotion & Package-Scoped Identifier Identity
--
  Type embedding rules for selectors (fields/methods) contain two advanced 
  nuances:
  1. Collisions at one level do not block promotion from deeper levels.
  2. Unexported identifiers are unique to their package, preventing collisions
     that would otherwise occur with exported identifiers.


  -- Deep Promotion Through Collisions
  --
    If two identifiers at the same depth "collide" (e.g., v.A.x and v.B.x), the 
    shorthand selector (v.x) becomes invalid. 
    

    However, this "broken bridge" does NOT prevent deeper unique fields or methods 
    from being promoted to the top level. The compiler continues to search 
    all paths for unique identifiers.

    
    example
      ```go
        type x string

        func (x) M() { fmt.Println("Method M called") }


        type y struct {
          z byte
        }
        

        type A struct {
          x // x is embedded here
        }

        func (A) y(int) bool { return false }


        type B struct {
          y // y is embedded here
        }

        func (B) x(string) {}


        func promotionDemo() {
          var v struct {
            A
            B
          }


          // -- Collision Level 
          // _ = v.x // ERROR: Ambiguous. Path 1: v.A.x (Depth 1). Path 2: v.B.x (Depth 1).
          // _ = v.y // ERROR: Ambiguous. Path 1: v.A.y (Depth 1). Path 2: v.B.y (Depth 1).

          // -- Deep Promotion Level 
          // Even though 'x' and 'y' collided at Depth 1, their children are unique 
          // relative to the root 'v'.
          
          _ = v.M // OK: Promoted from v.A.x.M (Depth 2). No other 'M' at Depth 2.
          _ = v.z // OK: Promoted from v.B.y.z (Depth 2). No other 'z' at Depth 2.
          
          fmt.Println("Unique deep selectors promoted successfully.")
        }

      ```


  -- Package-Scoped Identity (Unexported Identifiers)
  --
    In Go, the identity of an unexported identifier (starting with a lowercase 
    letter) is the combination of its (Name + PackagePath). 

    
    This means `pkgA.m` and `pkgB.m` are distinct identifiers. They will NEVER 
    collide or shadow each other when their owner types are embedded together.


    example
      ```go
        // -- Scenario: Two Packages with the same unexported method name --

        
        // -- FILE: foo/foo.go 
        // --
         package foo
         type A struct{ N int }
         func (a A) m() { fmt.Println("A's internal m:", a.N) }
         

         type I interface { m() }
         func Bar(i I) { i.m() } // Calls the m() visible to package foo


        // -- FILE: main.go 
        // --
        package main
        import "x.y/foo"

        type B struct{ N bool }
        func (b B) m() { fmt.Println("B's internal m:", b.N) }

        type C struct {
            foo.A
            B
        }


          -- Why this works without collision 
          --
            1. In package 'main', the selector `c.m()` only sees `B.m`. 
               `foo.A.m` is unexported and therefore invisible in package 'main'.
            2. When `c` is passed to `foo.Bar(I)`, the interface `I` looks for `m()`.
               Inside package 'foo', `foo.A.m` IS visible. 


          -- The "Exported" Difference 
          --
            If these were capitalized `M()`, the program would FAIL to compile. 
            Exported identifiers do not carry package-path identity for collision logic; 
            `c.M` would be ambiguous because both `A.M` and `B.M` are visible.

            func packageIdentityExplanation() {
              // Logic demonstration based on the provided packages:
              // var c C
              
              // c.m()      
              // Result: "B false" 
              // (Only B.m is visible here; no collision)

              // foo.Bar(c) 
              // Result: "A 0" 
              // (Inside package foo, only A.m is visible to the interface check)
            }





-- Implicit Methods via Type Embedding
--
  When a struct embeds a type, it doesn't just gain "shortcuts" to the embedded 
  fields; it actually acquires new methods in its own Method Set. 
  

  These are called "Implicit" or "Promoted" methods because the compiler 
  automatically generates them for you.


  -- How Methods are Promoted
  --
    Assume a struct type `S` embeds a type `T`. The rules for which methods 
    appear on `S` and its pointer `*S` depend on whether `T` was embedded 
    as a value or a pointer.


  -- Scenario A: Embedding by VALUE (type S struct { T })
  --
    1. Methods of T:   Promoted to BOTH S and *S.
    2. Methods of *T:  Promoted to *S ONLY.
  
    Rationale: If you have a value `S`, you can access `T`. But you cannot 
    necessarily take the address of `T` to call its pointer methods if `S` is 
    unaddressable. However, if you have `*S`, you can always get to `*T`.


  -- Scenario B: Embedding by POINTER (type S struct { *T }) 
  --
    1. Methods of T:   Promoted to BOTH S and *S.
    2. Methods of *T:  Promoted to BOTH S and *S.
    
    Rationale: Since the field is already a pointer (`*T`), any method requiring 
    a pointer is immediately available regardless of whether the outer struct 
    `S` is a value or a pointer.


  -- The Compiler's "Hidden Wrappers"
  --
    You can think of these implicit methods as the compiler writing "wrapper" 
    functions that simply delegate the call to the full selector path.


    ```go
      // --- Conceptual Pseudo-code (Not valid Go syntax) ---

      // Given: type Singer struct { Person }

      // The compiler effectively writes these for you:

      // 1. Promoted Value Method:
      func (s Singer) PrintName() {
          s.Person.PrintName() // Calls the embedded field's method
      }

      // 2. Promoted Pointer Method (on the Pointer type):
      func (s *Singer) PrintName() {
          (*s).Person.PrintName()
      }

      // 3. Promoted Pointer Method:
      func (s *Singer) SetAge(age int) {
          (&((*s).Person)).SetAge(age) // Takes address of embedded field to call pointer method
      }

    ```


  -- Code Demonstration: Proving the Method Set
  --

    ```go
      type Person struct {
        Name string
      }


      // Value Receiver
      func (p Person) PrintName() {
        fmt.Println("Name:", p.Name)
      }


      // Pointer Receiver
      func (p *Person) SetAge(age int) {
        fmt.Println("Setting age...")
      }


      type Singer struct {
        Person // Embedded by VALUE
      }


      func main() {
        tSinger := reflect.TypeOf(Singer{})
        tpSinger := reflect.TypeOf(&Singer{})

        // --  Method Set of Singer (Value) 

        // Includes only PrintName (Value method of Person).
        // SetAge is MISSING because it requires a pointer to Person, which 
        // the value Singer doesn't necessarily provide.
        fmt.Printf("Singer (Value) has %d methods:\n", tSinger.NumMethod())
        for i := 0; i < tSinger.NumMethod(); i++ {
          fmt.Println(" -", tSinger.Method(i).Name)
        }


        // --  Method Set of *Singer (Pointer) 
      
        // Includes BOTH PrintName and SetAge.
        fmt.Printf("\n*Singer (Pointer) has %d methods:\n", tpSinger.NumMethod())
        for i := 0; i < tpSinger.NumMethod(); i++ {
          fmt.Println(" -", tpSinger.Method(i).Name)
        }
      }



      // -- Why This Matters: Interface Satisfaction (Info Gain)

      // Implicit methods are not just about typing fewer characters. Their primary 
      // importance is satisfying Interfaces.
      //
      // If you have an interface:
      type AgeSetter interface {
        SetAge(int)
      }

      // A `Singer` value DOES NOT implement `AgeSetter`.
      // A `*Singer` pointer DOES implement `AgeSetter`.
      //
      // Without implicit method declaration, `*Singer` would not satisfy the 
      // interface unless you manually wrote a `SetAge` method for it. Go's 
      // embedding does this work for you automatically.

    ```





-- Interface Implementation and Method Set Chain
--
  Go's type system uses embedding to satisfy interfaces and build method sets.
  A crucial distinction must be made between "promoted" methods (from embedding)
  and "explicitly declared" methods when creating new defined types.


  -- Methods for Unnamed Types
  --
    You cannot explicitly declare methods for unnamed types (like a raw struct 
    literal `struct{ x int }`). 

    
    However, an unnamed struct CAN "own" methods through embedding. If an 
    unnamed struct embeds a type that has methods, those methods are promoted 
    to the unnamed struct.


    Example
      ```go
        var data struct { MyInt } 
        data.IsOdd() // Valid if MyInt has an IsOdd method.

      ```




-- Interface Satisfaction via Embedding
--
  If a struct embeds a type `T`, and `T` implements an interface `I`, the 
  embedding struct automatically implements `I` as well.
  
  This works because:
  1. The methods of `T` are promoted to the outer struct.
  2. The outer struct's Method Set now contains the methods required by `I`.
  
  The Exception: If the outer struct defines its own field or method with 
  the same name as one required by the interface, it might "shadow" or 
  "collide" with the promoted method, potentially breaking interface 
  implementation.





-- Defined Types vs. Method Promotion
--
  This is a common point of confusion. Let's look at the rules:
  1. A defined type (type New T) does NOT inherit methods declared for T.
  2. A struct type ALWAYS obtains methods from the types it EMBEDS.
  3. When you define a type from a struct (type Z X), the new type Z inherits 
     the structure of X, including its embedded fields and their promoted methods.
  4. However, Z does NOT inherit methods that were explicitly declared for X.


  example
    ```go
      // -- Base Type 
      type MyInt int


      func (mi MyInt) IsOdd() bool {
        return mi%2 == 1
      }


      // -- Type Definition 1: Age 
      // Age is a new type. It does NOT inherit IsOdd from MyInt.
      type Age MyInt


      // --- Type Definition 2: X (Embedding) ---
      type X struct {
        MyInt // Embedded field
      }


      // Double is explicitly declared for type X.
      func (x X) Double() MyInt {
        return x.MyInt + x.MyInt
      }


      // -- Type Definition 3: Y (Embedding Age) 
      type Y struct {
        Age // Embedded field
      }


      // -- Type Definition 4: Z (Defined from X) 
      // Z has the same underlying structure as X.
      type Z X


      func main() {
        // 1. Check MyInt: Has [IsOdd]
        fmt.Println("MyInt methods:", listMethods(MyInt(0)))


        // 2. Check Age: Has [None] 
        // Methods don't pass through standard type definitions.
        fmt.Println("Age methods:  ", listMethods(Age(0)))


        // 3. Check X: Has [Double, IsOdd]
        // IsOdd is promoted because MyInt is embedded.
        fmt.Println("X methods:    ", listMethods(X{}))


        // 4. Check Y: Has [None]
        // Age has no methods, so nothing is promoted to Y.
        fmt.Println("Y methods:    ", listMethods(Y{}))


        // 5. Check Z: Has [IsOdd] (and NOT Double)
        // - Z inherits X's structure, which includes the embedded MyInt. 
        // - Therefore, IsOdd is promoted to Z.
        // - However, 'Double' was declared for X, not as part of X's structure.
        // - Therefore, Z does NOT obtain 'Double'.
        fmt.Println("Z methods:    ", listMethods(Z{}))
      }


      // Helper function to inspect method sets via reflection
      func listMethods(i any) []string {
        t := reflect.TypeOf(i)
        methods := []string{}
        for i := 0; i < t.NumMethod(); i++ {
          methods = append(methods, t.Method(i).Name)
        }
        if len(methods) == 0 {
          return []string{"<none>"}
        }
        return methods
      }

    ```





-- Normalization and Evaluation of Promoted Method Values
--
  When you use a method value (assigning a method to a variable without calling it),
  Go performs two critical steps: Normalization (at compile time) and 
  Evaluation (at run time). 

  
  For "Promoted" methods (those coming from embedded fields), these rules 
  determine exactly which data is "captured" and how future changes to the 
  struct affect the method call.


  -- Compile-Time Normalization
  --
    "Normalization" is the process where the compiler converts shorthand selectors 
    into their "Full Form," making all implicit pointer dereferences (`*`) and 
    address-taking (`&`) operations explicit.

    Assume: `s` is a struct, and `s.M` is a promoted method from a deep-nested field.

    -- Rule 
    --
      1. Value Method: The compiler dereferences any pointers in the path to reach 
          the final value receiver.
      2. Pointer Method: The compiler dereferences pointers in the path and then 
          takes the address (`&`) of the final field to satisfy the pointer receiver.


  -- Runtime Evaluation (Binding)
  --
    The moment you assign a method to a variable (e.g., `f := s.M`), the 
    receiver is EVALUATED.

    - If the normalized path results in a VALUE: A copy of that value is saved.
    - If the normalized path results in a POINTER: A copy of that pointer (address) is saved.

    Crucial Insight: Once the evaluation happens, the variable `f` is "locked" 
    to that specific data. If you later point the outer struct to a different 
    embedded object, the method value `f` still points to the old one.



  -- example
    ```go
      type X int


      // M1 has a VALUE receiver
      func (x X) M1() {
        fmt.Println("M1 (Value Capture):", x)
      }


      // M2 has a POINTER receiver
      func (x *X) M2() {
        fmt.Println("M2 (Pointer Capture):", *x)
      }


      type T struct {
        X // X is embedded
      }


      type S struct {
        *T // Pointer to T is embedded
      }


      func main() {
        var t = &T{X: 1}
        var s = S{T: t}


        // -- Analysis of f := s.M1

        // Normalization: M1 needs a value. The path is s -> T (ptr) -> X (val).
        // Normalized Form: (*s.T).X.M1
        // Evaluation: Go looks at s.T, follows it to the T object, looks at X, 
        // and sees the value '1'. It saves a COPY of the value '1'.
        var f = s.M1 


        // -- Analysis of g := s.M2 

        // Normalization: M2 needs a pointer. The path is s -> T (ptr) -> X (val).
        // Normalized Form: (&(*s.T).X).M2
        // Evaluation: Go finds the field X inside the current T object and 
        // saves the MEMORY ADDRESS of that specific X field.
        var g = s.M2 


        // -- Scenario 1: Modify the value inside the existing T --

        s.X = 2 


        // f() prints 1: Because f captured a copy of the value '1' earlier.
        f() 
        

        // g() prints 2: Because g captured the ADDRESS of X. It follows that 
        // address and sees the value is now 2.
        g() 


        // -- Scenario 2: Change the pointer s.T to a NEW T object --

        s.T = &T{X: 3} 


        // f() still prints 1: It has its own independent copy of the original value.
        f() 

        // g() still prints 2: This is the most subtle part! 
        // g captured the address of X inside the OLD T object (where X is 2). 
        // Even though the struct 's' now points to a NEW T (where X is 3), 
        // the function value 'g' is still bound to the old T object's memory.
        g() 
      }

    ```





-- Recursive Promotion and Interface Cycles
--
  This example demonstrates a fascinating (and dangerous) edge case where 
  Type Embedding and Polymorphism combine to create an infinite recursive loop, 
  resulting in a Stack Overflow.


  -- The Setup: Embedding an Interface
  --
    Consider a struct 'T' that embeds an interface 'I'.
    
    1. Interface 'I' specifies a method 'm()'.
    2. Struct 'T' embeds 'I'.
    3. Because of promotion rules, 'T' now has a method 'm()'.
    4. Therefore, 'T' implicitly implements interface 'I'.

    The internal implementation of the promoted method T.m() is essentially:
      ```go
        func (t T) m() {
            t.I.m() // Call the method on the internal boxed interface
        }

      ```





  -- The Cycle: Pointing a Box at Itself
  --
    In the example below, we create an instance of 'T' and an interface 'i' 
    pointing to that instance. Then, we set the internal field of 'T' to be 
    that same interface 'i'.

    -- The Execution Chain
    --
      1. We call i.m()
      2. Go looks at the dynamic value inside 'i' (which is 't') and calls its 'm()' method.
      3. T.m() is a promoted method, so it calls its embedded field: t.I.m().
      4. But t.I contains 'i'!
      5. So we are back to step 1: calling i.m().

      This cycle repeats until the program runs out of stack memory.


    -- Code Demonstration
    --
      ```go
        type I interface {
          m()
        }


        type T struct {
          I // Embedded Interface
        }


        func main() {
          fmt.Println("Starting recursive interface call...")


          var t T
          var i I = &t // i is an interface pointing to t


          // Here is the cycle: 
          // We fill T's embedded interface field with a reference to T itself.
          t.I = i 


          // Trigger the infinite loop
          i.m() 
        }

      ```
