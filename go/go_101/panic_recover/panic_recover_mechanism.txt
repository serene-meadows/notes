-- The Exiting Phase & Panic/Recover Mechanics
--
  In Go, a function call doesn't always end abruptly. Most calls undergo an 
  "Exiting Phase" (or returning phase) before they are fully cleared from 
  the stack. Understanding this phase is critical for mastering defers, 
  error handling, and goroutine management.





-- The Exiting Phase Defined
--
  The Exiting Phase is the period of time after a function has stopped its 
  normal execution but before it has officially returned control to its caller.


  -- What happens during this phase? 
  --
    1. The Go runtime looks at the "defer-call stack" for that specific function.
    2. It executes all deferred functions in LIFO (Last-In, First-Out) order.
    3. Only after ALL deferred calls have fully finished does the function exit.





-- The Three Triggers for the Exiting Phase
--
  A function enters its exiting phase in exactly one of three ways:

  --  Trigger A: Normal Return
  --  
    The function reaches a `return` statement or the final closing brace `}`.
    This is the most common path.


  --  Trigger B: Panic
  --  
    A runtime error occurs (like dividing by zero) or the `panic()` function is 
    called explicitly. Execution stops immediately, and the function begins 
    "unwinding" by running its defers.


  --  Trigger C: runtime.Goexit()
  --
    The `runtime.Goexit()` function is called. This terminates the entire current 
    goroutine. However, before the goroutine dies, the runtime ensures every 
    function currently on that goroutine's stack enters its exiting phase and 
    runs its defers.


  -- Code Demonstration: Triggers in Action
  --
    ```go
      // -- Example 1: Normal Return 
      func f0() int {
        var x = 1
        // Captured immediately (1), but printed during the exiting phase
        defer fmt.Println("f0 exits normally, x:", x) 
        x++
        return x
      }


      // -- Example 2: Panic 
      func f2() {
        var x, y = 1, 0
        defer fmt.Println("f2 exits due to panic, x:", x)
        
        // Runtime error: integer divide by zero.
        // Normal execution stops HERE. The exiting phase begins.
        x = x / y 
        
        x++ // This line is UNREACHABLE
      }


      // -- Example 3: Goexit 
      func f3() {
        x := 1
        defer fmt.Println("f3 exits due to Goexit, x:", x)
        x++
        
        // Terminate the goroutine.
        runtime.Goexit()
        
        fmt.Println("This is unreachable")
      }


      func main() {
        fmt.Println("--- Start of main ---")


        // 1. Test Normal Return
        val := f0()
        fmt.Println("f0 returned:", val)


        // 2. Test Panic (wrapped in a recover so main doesn't die)
        func() {
          defer func() { recover() }()
          f2()
        }()


        // 3. Test Goexit
        // Note: runtime.Goexit should not be called in the main goroutine, 
        // as it will crash the program with a "no goroutines running" error.
        // We run it in its own goroutine instead.
        go f3()

        // Give the goroutine a moment to run
        time.Sleep(100 * time.Millisecond)
        fmt.Println("--- End of main ---")
      }


      // EXPECTED OUTPUT:
      // --- Start of main ---
      // f0 exits normally, x: 1
      // f0 returned: 2
      // f2 exits due to panic, x: 1
      // f3 exits due to Goexit, x: 1
      // --- End of main ---

    ```





-- Associating Panics and Goexit Signals
--
  In Go, the lifecycle of an error or a termination signal is managed through 
  "Association" with a specific function call. This association dictates 
  when a function enters its exiting phase and how the signal propagates 
  up the call stack.


  -- The Concept of Association
  --
    -- Panic Association 
    --
      When a panic occurs (either via `panic()` or a runtime error) within a 
      function call, that unrecovered panic becomes "associated" with that call. 
      This triggers the Exiting Phase (running `defer` statements) immediately.


    -- Goexit Association 
    --
      When `runtime.Goexit()` is called, a Goexit signal becomes associated with 
      the function call once the `Goexit` call itself completes. Like a panic, 
      this triggers the Exiting Phase.


    -- The Key Difference
    --
      - Panics: Can be removed (recovered) via the `recover()` function.
      - Goexit: CANNOT be cancelled or recovered. The goroutine is guaranteed to 
        terminate after all defers in the stack finish.





  -- The Panic Replacement Rule
  --
    At any given time, a specific function call can be associated with at most 
    ONE unrecovered panic.

    1. Recovery: If `recover()` is called, the association is cleared.
    2. Replacement: If a NEW panic occurs while the function is already 
       associating with an unrecovered panic (e.g., a panic inside a `defer`), 
       the NEW panic replaces the old one. The old panic is discarded forever.


    example
      ```go
        func panicReplacementDemo() {
          // This defer will finally catch the panic currently associated 
          // with the function.
          defer func() {
            fmt.Println("Recovered:", recover()) 
          }()


          // LIFO Order of Defers:
          // 4. This runs third. It panics with 3, replacing panic 2.
          defer panic(3) 


          // 3. This runs second. It panics with 2, replacing panic 1.
          defer panic(2) 


          // 2. This runs first. It panics with 1, replacing panic 0.
          defer panic(1) 


          // 1. Original panic initiates the exiting phase.
          panic(0)
        }

        // EXPECTED OUTPUT:
        // Recovered: 3

      ```





-- Panic Spreading (Propagation)
--
  While one function call has at most one panic, a single goroutine can have 
  multiple unrecovered panics coexisting—one for each function call currently 
  in its exiting phase.


  -- The Spreading Mechanism 
  --
    When a function call finishes its exiting phase (all its defers are done) 
    but still has an unrecovered panic associated with it, that panic "spreads" 
    to the nesting call (the caller).


  -- Consequences of Spreading 
  --
    1. To a Non-Panicking Caller: The caller now associates with the spread 
       panic and enters its own exiting phase immediately.
    2. To an Already Panicking Caller: The spread panic REPLACES the caller's 
       existing panic. The caller continues its exiting phase, moving to its 
       next deferred function.





-- Initial State of New Calls
--
  It is a fundamental rule that every time a function is invoked, it starts 
  with NO associated panics or Goexit signals.
  

  This is true even if the caller is currently in an exiting phase due to a 
  panic. The newly called function executes normally until/unless it 
  encounters its own panic or signal.





-- Complex Panic Propagation and Goexit Interaction
--
  This section explores advanced scenarios of panic spreading, the coexistence
  of multiple panics in a single goroutine, and the unique (and somewhat 
  counter-intuitive) way Goexit signals interact with active panics.


  -- Multi-Panic Coexistence and Replacement
  --
    While a single function call can only associate with ONE panic at a time,
    a goroutine can have multiple unrecovered panics coexisting across its 
    call stack. 

    
    When a nested function finishes its exiting phase while still panicking, 
    that panic "spreads" to the caller. If the caller is already panicking, 
    the new spread panic replaces the caller's current panic.


  example
    ```go
      func complexPanicChain() {
        // 1. Start a new goroutine to observe a full crash cycle.
        go func() {
          // --- Entry Function Call ---
          
          // 3. This defer is pushed.
          defer func() {
            // 6. Inside this defer, we register ANOTHER defer.
            defer panic(2) // This will eventually replace Panic 1, then Panic 0
            
            // 4. We call a nested anonymous function.
            func() {
              // 5. This triggers Panic 1. 
              // Now the goroutine has TWO panics:
              // - Panic 0 (associated with entry func)
              // - Panic 1 (associated with this anonymous func)
              panic(1)
            }()
            // When the function above exits, Panic 1 spreads to the 
            // nesting anonymous deferred call.
          }()

          // 2. Initial panic starts the exiting phase for the entry function.
          panic(0) 
        }()
      }

    ```

    -- understanding the crash output:
    --
      When the above goroutine crashes, the output looks like this:

      panic: 0
        panic: 1
        panic: 2

      critical insight: 
      The Go runtime lists panics in the order they occurred, but the LAST panic 
      listed (Panic 2) is the actual "active" one that replaced all others. 
      Panic 0 was replaced by Panic 2 when the deferred function fully exited.







-- Goexit vs. Panic: The Overwrite Rule
--
  A Goexit signal spread behaves similarly to a panic spread—it makes the 
  nesting call enter its exiting phase immediately. 


  -- The "Strange" Rule 
  --
    If a function call is currently associating with an unrecovered panic, 
    and a Goexit signal starts associating with that same call (via a defer), 
    the panic is effectively recovered (silenced) by the Goexit signal.


  example
    ```go
      func goexitAbsorbsPanic() {
        // This function will exit the goroutine but will NOT crash the program.
        defer runtime.Goexit()
        panic("This panic will be silenced by Goexit")
      }


      func goexitDemo() {
        fmt.Println("Starting Goexit demo...")

        
        go func() {
          fmt.Println("  Goroutine: starting...")
          goexitAbsorbsPanic()
          fmt.Println("  Goroutine: this is unreachable")
        }()


        // Wait for the goroutine to terminate
        for runtime.NumGoroutine() > 1 {
          runtime.Gosched()
        }

        
        fmt.Println("Main: Goroutine terminated peacefully (no panic crash).")
      }

    ```
      // expected output:
      // 
      // Starting Goexit demo...
      //   Goroutine: starting...
      // Main: Goroutine terminated peacefully (no panic crash).


      -- why? 
      --
        When `panic("bye")` occurs, the function enters the exiting phase. 
        It runs the defer, which calls `runtime.Goexit()`. 
        Because a Goexit signal is now associating with the function, the Go runtime 
        prioritizes the "terminate goroutine" instruction and discards the active panic.





-- Why Some `recover()` Calls Fail (No-Ops)
--
  In Go, simply calling `recover()` inside a `defer` statement is not always 
  enough to stop a panic. The Go runtime enforces strict "Context Rules" for 
  where `recover()` must be placed to be effective. 
  

  If these rules are not met, `recover()` returns `nil` and does nothing 
  (a "No-Op").


  -- The Official Requirements for Successful Recovery
  --
    According to the Go specification, `recover()` returns `nil` (fails) if:
    1. The argument passed to `panic()` was `nil` (Note: Go 1.21+ changed this 
       behavior so `panic(nil)` now throws a `*runtime.PanicNilError`).
    2. The goroutine is not currently in a panicking state.
    3. `recover()` was not called directly by a deferred function.


  -- The "Direct Context" Rule (The One-Level Depth Rule)
  --
    For `recover()` to work, it must be called directly within a deferred 
    function that was registered by the function call currently associating 
    with the panic.

    -- The Failure Scenarios (Nested Contexts) 
    --
      - Nested Defers: A defer inside a defer cannot recover the original panic.
      - Nested Functions: A normal function call inside a defer that calls 
        recover() will fail.
      - Immediate Calls: Calling recover() outside of a defer (in the normal flow) 
        is always a no-op.


    example
      ```go
        func galleryOfNoOps() {
          // -- FAIL 1: Nested Defer 
          defer func() {
            defer func() {
              recover() // NO-OP: This defer was not called by the panicking func.
            }()
          }()


          // -- FAIL 2: Nested Function Call 
          defer func() {
            func() {
              recover() // NO-OP: recover() is not called DIRECTLY by the deferred func.
            }()
          }()


          // -- FAIL 3: Defer within a normal block 
          func() {
            defer func() {
              recover() // NO-OP: This catches panics in THIS anonymous func, not main.
            }()
          }()


          // -- FAIL 4: Direct recover call 
          recover() // NO-OP: Not in a defer.


          panic("bye")
        }

      ```



-- The "Newest Panic" Target Rule
--
  A single goroutine can have multiple panics (e.g., a panic occurs inside 
  a defer while another panic is active). 

  
  Rule: Every `recover()` call always attempts to recover the newest
  (most recent) unrecovered panic in the current goroutine.


  example
    ```go
      func targetNewestPanic() {
        defer func() {
          // This defer runs while Panic 1 is active.

          
          defer func() {
            // This successfully recovers the "NEWEST" panic (Panic 2).
            if r := recover(); r != nil {
              fmt.Println("Recovered newest panic:", r)
            }
          }()


          // -- FAIL: Target Miss 
          // This recover attempts to catch the newest panic (Panic 2). 
          // However, it is not inside a defer called BY the function 
          // associating with Panic 2. It is a no-op.
          defer recover() 

          panic(2) // Triggers a second, newer panic
        }()

        panic(1) // Original panic
      }

    ```

      // EXPECTED OUTPUT (targetNewestPanic):
      // 
      // Recovered newest panic: 2
      // panic: 1
      // 
      // [Program Crashes with Panic 1]


    -- why did it crash? 
    --
      1. `panic(1)` starts. 
      2. The outer defer starts and triggers `panic(2)`.
      3. The innermost `recover()` successfully catches `panic(2)`.
      4. The `defer recover()` line was a no-op.
      5. `panic(2)` is now gone, but the goroutine still has the original `panic(1)`.
      6. Since `panic(1)` is never recovered, the program crashes.
