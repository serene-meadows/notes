Unlike languages like Java or C++ that rely heavily on `try/catch` exceptions,
Go strongly prefers explicit, returned error values for normal control flow.

However, for truly exceptional, unexpected situations (like severe bugs or 
corrupted state), Go provides the `panic` and `recover` mechanisms.





-- Panic: Initiating an Exception
--
  `panic(v any)` is a built-in function that stops the ordinary flow of control 
  and makes the current goroutine enter a "panicking" state.
  Once a panic occurs in a function call, the function call returns immediately and enters its exiting phase.

  (Note: `any` is an alias for `interface{}`).


  -- The Panicking Process
  --
    1. The function where `panic` is called stops executing its normal code immediately.
    2. Any `defer` functions within that function are executed normally.
    3. The function returns to its caller, which then also panics, running its own
       deferred functions.
    4. This "unwinding" continues up the call stack.
    5. If the panic reaches the top of the goroutine (e.g., the `main` function)
       without being recovered, the entire program crashes and prints a stack trace.





-- Recover: Regaining Control
--
  `recover() any` is a built-in function that regains control of a panicking 
  goroutine and restores normal execution.


  -- The Golden Rule of Recover
  --
    `recover()` is ONLY useful when called directly inside a `defer` function. 
    Calling it during normal execution returns `nil` and has no effect.


    When a panic occurs, the deferred functions run. If one of those deferred
    functions calls `recover()`, the panic is stopped. The value returned by
    `recover()` is the exact value that was originally passed to `panic()`.





-- example
--
  ```go
    func main() {
      // 1. This defer runs last (LIFO order). Because the panic is caught 
      // by the defer below this one, the program is back to a normal state here.
      defer func() {
        fmt.Println("3. main() exiting normally.")
      }()

      fmt.Println("1. Starting program...")

      // 2. This defer catches the panic.
      defer func() {
        // Calling recover() stops the panicking sequence.
        if r := recover(); r != nil {
          // 'r' holds the value passed to panic("bye!")
          fmt.Printf("2. Recovered from panic! The error was: %v\n", r)
        }
      }()

      // Trigger a panic. Normal execution stops here.
      panic("bye!")

      // This code is UNREACHABLE because the panic interrupts the flow.
      fmt.Println("This will never be printed.")
    }

  ```
    // EXPECTED OUTPUT:
    // 1. Starting program...
    // 2. Recovered from panic! The error was: bye!
    // 3. main() exiting normally.





-- Runtime Panics
--
  You don't always have to call `panic()` manually. The Go runtime will 
  automatically trigger panics for certain illegal operations.


  Common examples include:
    - Dividing an integer by zero.
    - Dereferencing a `nil` pointer.
    - Accessing an array or slice out of bounds.
    - Writing to a closed channel.


  These runtime panics can be caught and handled with `recover()` just like 
  manual panics.





-- Fatal Errors (Unrecoverable)
--
  It is crucial to distinguish between Panics and Fatal Errors.


  Fatal errors are severe, low-level system failures generated by the runtimme.
  They bypass the `panic`/`recover` mechanism completely and immediately crash 
  the program. You CANNOT catch them with `recover()`.


  Examples of Fatal Errors:
    - Out of Memory (OOM): The system cannot allocate more memory.
    - Stack Overflow: Infinite recursion exceeding the maximum stack size.
    - Concurrent Map Writes: Two goroutines writing to a map simultaneously
      without a mutex.
