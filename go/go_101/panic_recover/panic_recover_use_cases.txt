-- Panic & Recover - Practical Use Cases
--
  Go emphasizes explicit error handling (returning error values). However, the
  panic/recover mechanism is essential for handling truly exceptional cases or
  ensuring system robustness.


--  USE CASE 1: Preventing Crashes in Long-Running Programs
--
  This is arguably the most critical application of `recover`. In concurrent
  systems—especially servers—a panic in a single worker goroutine must not
  be allowed to terminate the entire process.


  -- The "Goroutine Isolation" Principle
  --
    Rule: If a goroutine panics and is not recovered within that same goroutine,
    the entire program crashes, killing every other goroutine immediately.

    Solution: Implement a "Recovery Middleware" or a deferred safety net at the
    beginning of every long-lived or entry-point goroutine.


  -- Code Demonstration: Robust TCP Server
  --
    ```go
      func main() {
        // We set up a simple TCP listener.
        listener, err := net.Listen("tcp", ":12345")
        if err != nil {
          log.Fatalln("Failed to start listener:", err)
        }


        log.Println("Server started on :12345. Waiting for connections...")


        for {
          conn, err := listener.Accept()
          if err != nil {
            log.Println("Accept error:", err)
            continue
          }

          // Handle each client connection in a separate goroutine.
          // If one client causes a panic, we don't want the server loop to die.
          go ClientHandler(conn)
        }
      }


      // ClientHandler processes individual connections.
      func ClientHandler(c net.Conn) {
        // --- THE SAFETY NET

        // This deferred function acts as a boundary.
        defer func() {
          if r := recover(); r != nil {
            // Instead of crashing the whole server, we log the incident.
            log.Printf("CRITICAL ERROR: Captured a panic in handler: %v\n", r)
            log.Println("Goroutine isolated. Program continuing...")
          }

          // Ensure the resource is always cleaned up.
          c.Close()
        }()


        // Simulate a process that might fail unexpectedly.
        log.Println("Handling new connection from:", c.RemoteAddr())


        // In a real app, this might be a null pointer dereference or
        // an out-of-bounds slice access. Here, we trigger it manually.
        doSomethingDangerous()
      }

      func doSomethingDangerous() {
        // Triggering a panic for demonstration.
        panic(errors.New("unexpected database corruption or logic bug"))
      }



      // EXPECTED BEHAVIOR & OUTPUT:
      // 1. Start the server.
      // 2. In another terminal, connect via: `telnet localhost 12345` (or similar).
      //
      // Terminal 1 (Server Log):
      // 2024/05/20 10:00:00 Server started on :12345. Waiting for connections...
      // 2024/05/20 10:00:05 Handling new connection from: 127.0.0.1:54321
      // 2024/05/20 10:00:05 CRITICAL ERROR: Captured a panic in handler: unexpected database corruption or logic bug
      // 2024/05/20 10:00:05 Goroutine isolated. Program continuing...
      //
      // Observation:
      // The server remains running. You can connect again, and the main `for` loop
      // in main() continues to accept new clients.
      //
      // If the `defer recover()` block was missing:
      // The entire program would exit with a stack trace the moment the first
      // connection was handled, effectively shutting down the service for all users.

    ```





-- Use Case 2: Supervisor Pattern (Auto-Restarting Goroutines)
--
  In Go, if a goroutine panics and is not recovered, the entire application
  crashes. While Use Case 1 focused on isolation, Use Case 2 focuses on
  "Self-Healing": detecting a crash and immediately spawning a replacement
  goroutine to ensure high availability of background tasks.


  -- The Supervisor Logic
  --
    The supervisor function (e.g., `NeverExit`) acts as a wrapper. It:
    1. Defers a recovery function.
    2. Executes the intended worker logic.
    3. If the worker logic panics, the deferred function catches it and
       re-launches itself in a new goroutine.


  -- Code Demonstration: The Self-Healing Worker
  --
    ```go
      // shouldNotExit simulates a worker that performs a task but occasionally
      // encounters a "Critical Failure" (panic).
      func shouldNotExit() {

        for {
          // Simulate meaningful workload
          time.Sleep(time.Second)


          // Simulate a random, unexpected panic based on time bits
          if time.Now().UnixNano()&0x3 == 0 {
            log.Println("...Critical logic error encountered! Panicking...")
            panic("unexpected internal state")
          }


          log.Println("Worker is processing data successfully...")
        }
      }


      // NeverExit is the "Supervisor". It ensures that if the function 'f' crashes,
      // a new instance of 'f' is started immediately.
      func NeverExit(name string, f func()) {
        defer func() {
          if r := recover(); r != nil {
            // 1. Log the failure
            log.Printf("SUPERVISOR: %s crashed due to: %v. Restarting now...\n", name, r)


            // 2. RESTART: Launch a brand new goroutine to replace the dead one.
            // This creates a "self-healing" loop.
            go NeverExit(name, f)
          }
        }()

        // Start the actual work
        f()
      }


      func main() {
        // Remove timestamp noise for cleaner log output in this demo
        log.SetFlags(0)


        log.Println("Starting system with supervised workers...")


        // Launch two instances of our job.
        go NeverExit("Worker-Alpha", shouldNotExit)
        go NeverExit("Worker-Beta", shouldNotExit)


        // select{} is a common Go idiom used to block the 'main' goroutine
        // forever without consuming CPU cycles. This keeps the background
        // workers alive.
        select {}

      }


      // REPRESENTATIVE OUTPUT:
      // Starting system with supervised workers...
      // Worker is processing data successfully...
      // Worker is processing data successfully...
      // ...Critical logic error encountered! Panicking...
      // SUPERVISOR: Worker-Alpha crashed due to: unexpected internal state. Restarting now...
      // Worker is processing data successfully...
      // Worker is processing data successfully...

    ```


  -- Avoid "Rapid-Fire" Restart Loops
  --
    If a worker crashes immediately upon starting (e.g., due to a persistent
    config error), the code above will enter an infinite loop of crashing and
    restarting as fast as the CPU allows.

    Best Practice: Implement a "Back-off" strategy (e.g., `time.Sleep`) before
    restarting to prevent overwhelming system resources.


  -- Clean Shutdowns
  --
    A `select{}` block prevents the program from ever exiting. In a real-world
    application, you would typically use a `context.Context` or a signal
    listener (like SIGTERM) to tell the supervisor to stop restarting
    and exit gracefully.


  -- State Loss
  --
    Remember: When a goroutine crashes and restarts, any local, in-memory state
    that was not persisted to an external database or a shared synchronized
    struct is lost.





-- Use Case 3: Simulating Long Jumps with Panic and Recover
--
  In some programming languages, "Long Jumps" (like setjmp/longjmp in C) allow
  execution to jump from a deeply nested function back to a much higher point 
  in the call stack instantly.

  
  While Go does not have a formal `longjump` keyword, the `panic`/`recover` 
  mechanism can be used to simulate this behavior.


  -- The "Long Jump" Use Case
  --
    -- The Concept 
    --
      Instead of returning an error through every single layer of a deeply nested 
      function tree (which can be verbose), you can `panic` with a specific value 
      at the bottom and `recover` that value at the top.


    -- Pros 
    --
      - Reduces verbosity in complex, deeply nested logic (like parsers or 
        deep recursive searches).
      - Can make the core logic look "cleaner" by removing repetitive error checks.


    -- Cons (High Risks) 
    --
      - readability: It obscures the standard control flow, making the code 
        difficult for other developers to follow.
      - Performance: `panic` involves stack unwinding, which is significantly 
        slower than a standard `return`.
      - Safety: If you forget to `recover`, the entire program crashes. 
      Recommendation: Use this pattern very sparingly, or better yet, avoid 
      it in favor of idiomatic error returns.


  -- Code Demonstration: Deeply Nested Return
  --
    ```go
      func main() {
        // We use an anonymous function that returns an int to demonstrate
        // how a panic can be turned into a return value.
        finalResult := func() (result int) {
          
          // -- THE RECOVERY POINT 
          
          defer func() {
            if r := recover(); r != nil {
              // We check if the panic value is of the type we expect (int).
              // This ensures we only "catch" our simulated long jump and
              // don't accidentally silence real runtime errors.
              if val, ok := r.(int); ok {
                result = val // Set the named return parameter
                fmt.Printf("Jump caught! Simulating return of: %d\n", val)
              } else {
                // If it's not an int, it's a real panic. Re-panic!
                panic(r)
              }
            }
          }()


          // -- DEEPLY NESTED EXECUTION 
          func() {
            fmt.Println("Entering Level 1")
            func() {
              fmt.Println("  Entering Level 2")
              func() {
                fmt.Println("    Entering Level 3")
                
                // Instead of returning through every level, 
                // we "Jump" out immediately.
                panic(123) 
                
                fmt.Println("    This will never be reached.")
              }()
              fmt.Println("  This will never be reached.")
            }()
            fmt.Println("This will never be reached.")
          }()

          // Default return if no panic occurs
          return 0 
        }()

        fmt.Printf("Final Result in Main: %d\n", finalResult)
      }



        // EXPECTED OUTPUT:
        // Entering Level 1
        //   Entering Level 2
        //     Entering Level 3
        // Jump caught! Simulating return of: 123
        // Final Result in Main: 123

    ```

  -- Comparison: Simulation vs. Idiomatic Go
  --
    -- Simulated Long Jump 
    --
      ```go
        func deepSearch() {
            if found { panic(result) }
            deepSearch()
        }

      ```


    -- Idiomatic Go (Recommended)
    --
      ```go
        func deepSearch() (Result, bool) {
            if found { return result, true }
            res, ok := deepSearch()
            if ok { return res, true }
            return nil, false
        }

      ```

    While the simulated version is shorter, the idiomatic version makes it 
    explicitly clear how data and control flow through the application.





-- Use Case 4: Using Panic/Recover to Reduce Error Check Verbosity
--
  In standard Go, error handling is explicit: you call a function and immediately 
  check "if err != nil". In complex sequences of operations, this can lead to 
  very repetitive code.

  
  Use Case 4 explores using panic/recover as a way to "flatten" this logic 
  by allowing steps to panic on failure, which is then caught and converted 
  back into a standard error return value.


  -- The Idiomatic (Verbose) Approach
  --
    This is the standard way to write Go code. It is clear and safe, but can 
    become "noisy" when many steps are involved.


    ```go
      func doWork() (err error) {
        shouldContinue, err := doStep1()
        if !shouldContinue { return err }

        shouldContinue, err = doStep2()
        if !shouldContinue { return err }

        shouldContinue, err = doStep3()
        if !shouldContinue { return err }

        return nil
      }

    ```


  -- The Panic/Recover (Compact) Approach
  --
    This pattern uses a central recovery block to manage flow. 
    - If a step succeeds but wants to stop: panic(nil) [See Go 1.21 note below].
    - If a step fails: panic(err).
    - If a step wants to continue: Do nothing (don't panic).

    
    example
      ```go
        // doStep simulates a task that might fail or finish early.
        func doStep(id int, triggerError bool, triggerEarlyExit bool) {
          if triggerError {
            panic(fmt.Errorf("error in step %d", id))
          }
          if triggerEarlyExit {
            // In Go < 1.21, this was panic(nil).
            // In Go >= 1.21, this is handled via a specific error type.
            panic(nil) 
          }
          fmt.Printf("Step %d completed...\n", id)
        }


        // PerformWork wraps the steps and converts panics back into a standard error.
        func PerformWork(failAt int, exitEarlyAt int) (err error) {
          defer func() {
            r := recover()
            if r == nil {
              return // No panic, return normally
            }


            // -- Go 1.21+ Logic 
            // panic(nil) now results in a *runtime.PanicNilError

            switch e := r.(type) {
            case *runtime.PanicNilError:
              fmt.Println("[Recovered] Early exit requested (nil panic)")
              err = nil 
            case error:
              fmt.Println("[Recovered] A functional error occurred")
              err = e
            default:
              // This was an unexpected panic (e.g., index out of bounds)
              // Re-panic to let the program crash as it's a genuine bug.
              panic(e) 
            }
          }()

          doStep(1, failAt == 1, exitEarlyAt == 1)
          doStep(2, failAt == 2, exitEarlyAt == 2)
          doStep(3, failAt == 3, exitEarlyAt == 3)

          return nil
        }

      ```


  -- Critical Change: panic(nil) in Go 1.21+
  --
    Before Go 1.21, `recover()` would return `nil` if `panic(nil)` was called. 
    This made it impossible to tell if a panic had actually occurred or if 
    the function was just executing normally.
    
    Now, calling `panic(nil)` automatically throws a `*runtime.PanicNilError`.

    Updated Recovery Block Logic:
      ```go
        defer func() {
          switch e := recover().(type) {
          case nil, *runtime.PanicNilError: 
            // Handle normal exit or legacy panic(nil)
          case error:
            err = e
          default:
            panic(e)
          }
        }()

      ```
