-- Expression Evaluation Orders
--
  In Go, the order in which expressions are evaluated is governed by a
  combination of lexical order (declaration order), dependency analysis,
  and specific rules for package-level vs. local-level code.





-- Fundamental Dependency Rule
--
  An expression is ALWAYS evaluated after all the expressions it depends on.
  Practically, this means sub-expressions are evaluated before the parent
  expression.


  Example: f(x, y[n])
    1. To evaluate y[n], the program must first evaluate 'y' and 'n'.
    2. To evaluate f(...), the program must first evaluate 'f', 'x', and 'y[n]'.





-- Package-Level Variable Initialization
--
  When a package is loaded, Go initializes package-level variables using an
  iterative dependency-check algorithm.


  -- The Algorithm
  --
    1. Find the first variable (by declaration order) that does not depend on
       any uninitialized variables.
    2. Initialize it.
    3. Repeat until all variables are initialized.
    4. Blank identifiers (_) are treated like named variables during this process.



  example
    ```go
      var (
        // Dependencies: _ depends on 'a' (via function f)
        _ = f()
        // Dependencies: 'a' depends on 'b'
        a = b / 2
        // Dependencies: 'b' depends on nothing
        b = 6
        // Dependencies: 'c' depends on 'a' (via function f)
        c = f()
      )


      func f() int {
        a++
        return a
      }


      func initializationOrderDemo() {
        // Let's Trace:
        // 1. 'b' is the first variable with no dependencies. b = 6.
        // 2. 'a' now has no uninitialized dependencies. a = 6 / 2 = 3.
        // 3. '_' is the first in order needing init. Calls f().
        //    Inside f: a++ (a becomes 4). Returns 4.
        // 4. 'c' is the last needing init. Calls f().
        //    Inside f: a++ (a becomes 5). Returns 5.

        fmt.Println(a, b, c)
      }

    ```





-- Grouping and Disassembly Rules
--
  -- Rule A: Single Multi-valued Expression (Grouped)
  --
    If multiple variables are initialized by one function call (multi-value),
    they are initialized as an atomic group. No other variable can be
    initialized between them.
    `var x, y = f()` // x and y are bound together in the init timeline.


  -- Rule B: Multiple Source Expressions (Disassembled)
  --
    If one line contains multiple assignments from multiple expressions, the
    compiler "disassembles" them into separate lines before calculating order.

    `var m, n = expr1, expr2`
    is treated as:
    `var m = expr1`
    `var n = expr2`





-- Hidden Dependencies and Unspecified Orders
--
  A "Hidden Dependency" occurs when a variable's initialization calls a method
  or an interface that references other package variables.


  -- warning
  --
    If hidden dependencies exist, the relative order of initialization between
    those variables and unrelated variables (with side effects) is UNSPECIFIED.


  example
    ```go
      func sideEffect() int {
        fmt.Println("Side effect executed!")
        return 0
      }


      type I interface    { ab() []int }
      type T struct{}
      func (T) ab() []int { return []int{a_val, b_val} }


      var x_val = I(T{}).ab() // x_val has a hidden dependency on a_val and b_val
      var _ = sideEffect()    // Execution moment is unspecified relative to x_val
      var a_val = b_val       // Guaranteed: a_val initialized after b_val
      var b_val = 42


      // -- Result of Unspecified Order
      // Different compilers (or different versions of the Go toolchain) might
      // execute 'sideEffect()' before OR after 'x_val' is initialized.

    ```





-- Risks & Best Practices
--
  1. Cross-File Danger: Go does not strictly define the compilation order of
     different files within the same package.

     Best Practice: Do not spread variables with complex dependencies
     across different files. Keep them in one file to ensure the
     "declaration order" rule acts as a predictable tie-breaker.


  2. Toolchain Bugs:
     - Go 1.22.0 - 1.22.4 had specific bugs in dependency analysis.
     - Toolchains prior to 1.23 had edge cases with complex init logic.
     Best Practice: Keep package-level initialization as simple as
     possible. Avoid using complex logic or functions with side-effects
     inside global `var` blocks.





-- Expression & Operand Evaluation Orders
--
  -- Logical Operations: Short-Circuit Evaluation
  --
    For boolean logical operators, the evaluation of the right-hand operand is
    strictly conditional based on the result of the left-hand operand.


    -- Operator: && (Conditional AND)
    --
      If 'a' is false, the result is already false. 'b' is NEVER evaluated.
      `a && b` -> Evaluation order: a (always), then b (only if a is true).


    -- Operator: || (Conditional OR)
    --
      If 'a' is true, the result is already true. 'b' is NEVER evaluated.
      `a || b` -> Evaluation order: a (always), then b (only if a is false).


    example
      ```go
        func trace(msg string, result bool) bool {
          fmt.Println("Evaluating:", msg)
          return result
        }


        func shortCircuitDemo() {
          fmt.Println("--- Logical AND ---")
          _ = trace("Left side", false) && trace("Right side", true)
          // Output: Evaluating: Left side (Right side is skipped)

          fmt.Println("\n--- Logical OR ---")
          _ = trace("Left side", true) || trace("Right side", false)
          // Output: Evaluating: Left side (Right side is skipped)
        }

      ```



  -- The "Usual Order": Lexical Left-to-Right
  --
    When evaluating operands in an expression, assignment, or return statement,
    Go guarantees that all "Active Operations" are evaluated in lexical
    (textual) left-to-right order.


    -- What are "Active Operations"?
    --
      1. Function calls: `f()`
      2. Method calls: `obj.m()`
      3. Channel communications: `<-ch` or `ch <- v`


    Note: Explicit type conversions (e.g., `string(65)`) are NOT function
    calls and do not follow this strict left-to-right rule relative to calls.

    func usualOrderDemo() {
      fa := func() int { fmt.Print("fa "); return 1 }
      fb := func() int { fmt.Print("fb "); return 2 }

      // Guaranteed: fa() executes before fb() because fa appears to the left of fb.
      vals := []int{fa(), fb()}
      fmt.Println("\nResult:", vals)
    }


  -- Unspecified Orders: The Ambiguity Traps
  --
    While calls are ordered left-to-right, the evaluation of basic variables
    relative to those calls is NOT specified. This can lead to unpredictable
    state if a function call modifies a variable used in the same statement.


    example
      ```go
        func ambiguityDemo() {
          a := 1
          f := func() int { a++; return a }


          // -- Trap 1: Variable vs. Function Call
          //
          // Evaluation order of 'a' and 'f()' is NOT specified.
          // Potential Result A: [1, 2] (a is read, then f adds 1)
          // Potential Result B: [2, 2] (f adds 1, then a is read)
          x := []int{a, f()}
          fmt.Println("Ambiguous Slice:", x)


          // -- Trap 2: Map Element Assignments
          //
          // The order of map literal entries is not specified.
          // m might be {2: 1} or {2: 2}.
          a = 1
          m := map[int]int{a: 1, a: 2}
          fmt.Println("Ambiguous Map Keys:", m)


          // -- Trap 3: Map Key vs. Map Value
          // The order between evaluating the key and the value is not specified.
          // n might be {2: 3} or {3: 3}.
          a = 2
          n := map[int]int{a: f()}
          fmt.Println("Ambiguous Map Value:", n)
        }

      ```


  -- Complex Assignment Case Study
  --
    ```go
      // Consider this line from the Go Specification:
      // y[z.f()], ok = g(h(a, b), i() + x[j()], <-c), k()
      //
      // Guaranteed Left-to-Right Order (for calls and channel ops):
      // 1. z.f()
      // 2. h()
      // 3. i()
      // 4. j()
      // 5. <-c
      // 6. g()
      // 7. k()
      //
      // Guaranteed Dependency Order:
      // - h() is evaluated AFTER its arguments h, a, and b.
      // - y[...] is evaluated AFTER its index z.f().
      // - z.f() is evaluated AFTER the expression z.
      //
      // UNSPECIFIED Order (Potential bugs!):
      // - Whether 'y' is evaluated before 'z'.
      // - Whether 'a' is evaluated before 'g'.
      // - Whether index 'x[j()]' is evaluated before the channel receive '<-c'.

    ```





-- Evaluation and Assignment Order in Assignment Statements
--
  Go's rules for assignment statements are designed to be predictable, even
  when the left-hand side (LHS) and right-hand side (RHS) share variables.
  Every assignment statement occurs in two distinct, sequential phases.


  -- The Two-Phase Execution Model
  --
    -- Phase 1: The Evaluation Phase
    --
      The runtime prepares for the assignment by evaluating:
      1. All operands of index expressions on the LHS (e.g., the 'i' in a[i]).
      2. All pointer indirections and selectors on the LHS (e.g., *p or obj.Field).
      3. ALL expressions on the RHS.

      Note: These follow the "Usual Order" (Lexical Left-to-Right for calls/channels).


    -- Phase 2: The Carry-out Phase
    --
      After everything is evaluated and "locked in," the actual assignments
      (the data movement) are carried out in strict Left-to-Right order.


  -- The "Freeze" Principle: Elementary Forms
  --
    A critical concept in Go is that assignments performed in Phase 2 DO NOT
    affect the evaluation results obtained in Phase 1.


    Before Phase 2 starts, each target on the LHS is resolved to its
    "Elementary Form." You can think of this as the runtime "pointing its finger"
    at a specific memory location and saying "The result goes here, regardless
    of what happens to the variables used to find this spot."


  -- Elementary Form Rules
  --
    1. Blank Identifier: Stays a blank identifier (the value is discarded).
    2. Index Expression (c[k]): Resolved to the memory address of the k-th
       element of container c. Even if 'c' or 'k' change later in the same
       statement, the destination is already fixed.
    3. Other Addressable Values: Resolved to a direct pointer to the variable's
       memory address.


  -- Code Demonstration: The Index Shadowing Trap
  --
    ```go
      func assignmentOrderDemo() {
        // -- Scenario 1: Basic Indexing
        i := 0
        x := []int{100, 200}


        // Question: Does this update x[0] or x[1]?
        i, x[i] = 1, 999


        // Trace:
        // Phase 1 (Evaluation):
        //   - LHS index operand 'i' is evaluated. Result: 0.
        //   - RHS expressions are evaluated. Result: 1, 999.
        //   - Target 1 is variable 'i'. Target 2 is x[0].
        // Phase 2 (Carry-out):
        //   - i = 1
        //   - x[0] = 999 (Even though i is now 1, the target was fixed at x[0])


        fmt.Printf("Scenario 1: i=%d, x=%v\n", i, x)


        // -- Scenario 2: Swapping via Pointer
        a, b := 10, 20
        p := &a


        // We change what 'p' points to while simultaneously assigning to *p
        p, *p = &b, 30


        // Trace:
        // Phase 1 (Evaluation):
        //   - LHS pointer indirection *p is evaluated. It points to 'a'.
        //   - RHS expressions are evaluated. Result: &b, 30.
        // Phase 2 (Carry-out):
        //   - p = &b
        //   - *p (the old p pointing to 'a') = 30

        fmt.Printf("Scenario 2: a=%d, b=%d, *p=%d\n", a, b, *p)
      }

        // EXPECTED OUTPUT:
        //
        // Scenario 1: i=1, x=[999 200]
        // Scenario 2: a=30, b=20, *p=20

    ```


  -- Practical Implementation Nuance
  --
    When an index expression `c[k]` is evaluated in Phase 1, Go effectively
    creates a hidden pointer to the container and the specific index.


    If the container is a Map:
    Go ensures that if `m[k]` is the target, the map entry for `k` is created
    (if it didn't exist) during the evaluation phase, returning the address
    where the value will be stored in Phase 2.


    example
      ```go
        func mapAssignmentDemo() {
          m := make(map[string]int)
          k := "A"

          // k is changed to "B", but m["A"] was already locked in as the target.
          k, m[k] = "B", 100

          fmt.Printf("Map: %v, k: %s\n", m, k)
        }


        // EXPECTED OUTPUT:
        //
        // Map: map[A:100], k: B

      ```





-- Deep Dive - Multi-Value Assignment Execution Order
--
  Go's multi-value assignment is powerful but relies on a strict two-phase 
  execution model. Understanding how Go "snapshots" values and "resolves" 
  memory addresses is vital for predicting behavior in complex assignments.


  -- The Anatomy of a Multi-Value Assignment
  --
    Every assignment statement follows these two steps:
    1. EVALUATION PHASE: All indices on the LHS and all values on the RHS are 
       evaluated in lexical order (left-to-right). Memory addresses for targets 
       are locked in.
    2. CARRY-OUT PHASE: The actual values are moved into the locked-in memory 
       addresses in left-to-right order.


  -- Practical Case Study 1: The Variable Swap
  --
    ```go
      func swapDemo() {
        a, b := 1, 2

        // How Go executes: a, b = b, a

        // 1. Evaluation Phase:
        //    - Get address of 'a' (P0) and 'b' (P1).
        //    - Evaluate values of 'b' (R0=2) and 'a' (R1=1).
        // 2. Carry-out Phase:
        //    - *P0 = R0 (a becomes 2)
        //    - *P1 = R1 (b becomes 1)
        
        a, b = b, a
        fmt.Printf("Swap: a=%d, b=%d\n", a, b) 
      }

      // EXPECTED OUTPUT:
      // Swap: a=2, b=1

    ```
	




  -- Practical Case Study 2: Index Dependency
  --
    ```go
      func indexDependencyDemo() {
        x := []int{0, 0}
        i := 0
        

        // Question: Does this update x[0] or x[1]?
        i, x[i] = 1, 2


        // 1. Evaluation Phase:
        //    - Evaluate i on LHS. Result: 0. 
        //    - Target for index expression is resolved to x[0].
        //    - Evaluate RHS: 1, 2.
        // 2. Carry-out Phase:
        //    - i = 1
        //    - x[0] = 2 (The index was locked to 0 before i became 1)
        

        fmt.Println("Index Dependency x:", x) 


        // EXPECTED OUTPUT:
        // Index Dependency x: [2 0]

      }

    ```


  -- Practical Case Study 3: Complex Mixed Assignment
  --
    ```go
      func complexMixedDemo() {
        m := map[string]int{"Go": 0}
        s := []int{1, 1, 1}
        olds := s // Keep reference to original slice
        n := 2
        p := &n
        

        // Massive multi-assignment
        s, m["Go"], *p, s[n] = []int{2, 2, 2}, s[1], m["Go"], 5


        // -- Step-by-Step Breakdown 

        // 1. Evaluation Phase:
        //    - LHS Targets: &s, m["Go"] (address), p, s[2] (address)
        //    - RHS Values:  []int{2,2,2}, s[1]=1, m["Go"]=0, 5
        // 2. Carry-out Phase:
        //    - s = []int{2,2,2} (Variable 's' now points to a NEW array)
        //    - m["Go"] = 1
        //    - *p = 0 (n becomes 0)
        //    - s[2] = 5 (CRITICAL: This targets the index 2 of the OLD 's' array!)

        fmt.Printf("Map: %v, New Slice: %v, n: %d\n", m, s, n)
        fmt.Printf("Original Slice array: %v\n", olds)
      }

      // EXPECTED OUTPUT:
      // Map: map[Go:1], New Slice: [2 2 2], n: 0
      // Original Slice array: [1 1 5]

    ```


  -- Advanced Pattern: Slice Rotation via Range
  --
    ```go
      func rotationDemo() {
        x := []int{2, 3, 5, 7, 11}
        t := x[0]
        var i int
        

        // This loop effectively shifts all elements left by one.
        // In each step, x[i] refers to the index from the PREVIOUS iteration.
        for i, x[i] = range x {}
        

        x[i] = t
        fmt.Println("Rotated x:", x) 
      }


      // EXPECTED OUTPUT:
      // Rotated x: [3 5 7 11 2]

    ```


  -- The Nil/Panic Trap
  --
    ```go
      func safetyDemo() {
        // -- Scenario A: No Panic 
        x := []int{123}
        
        // Evaluation phase: x[0] is evaluated and resolved to its address.
        // Carry-out phase: x becomes nil. Then the resolved address is set to 456.
        // Since the address was found BEFORE x became nil, it succeeds.
        x, x[0] = nil, 456
        fmt.Println("Scenario A: Success (x is nil)")
      }

    ```





-- Non-Deterministic Evaluation & Unspecified Orders
--
  In Go, the "Usual Order" rule guarantees that function calls, method calls, 
  and channel operations are evaluated from left-to-right. 

  
  HOWEVER, the evaluation of basic variables or expressions that do not contain 
  calls is NOT strictly sequenced against those calls. If a function call 
  has a side effect (modifies a variable), and that variable is used elsewhere 
  in the same statement, the result is non-deterministic (ambiguous).


  -- Ambiguity: Variable vs. Function Call in Assignment
  --
    When you have a multi-value assignment like `y, z := x + 1, f(&x)`, 
    the Go specification does not dictate whether the variable `x` is read 
    before or after the function `f` executes its side effect on `x`.

  
  example
    ```go
      func variableVsCallDemo() {
        f := func(p *int) int {
          *p = 99
          return *p
        }


        x := 0
        // Ambiguity:
        // Path A: `x + 1` is evaluated first (x=0). Result: y=1, z=99.
        // Path B: `f(&x)` is evaluated first (x becomes 99). Result: y=100, z=99.
        y, z := x+1, f(&x)

        fmt.Printf("Assignment Ambiguity: y=%d, z=%d\n", y, z)
      }

      // possible outputs (Compiler Dependent):
      // 
      // A: Assignment Ambiguity: y=1, z=99
      // B: Assignment Ambiguity: y=100, z=99

    ```





  -- Ambiguity: Mixed Arguments in Function Calls
  --
    The "Usual Order" says `f1(), f2()` are called left-to-right. 
    But if you have `f1(), y, f2()`, the evaluation of the variable `y` 
    is not sequenced. It could be read at the very beginning, in the middle, 
    or at the very end.


    example
      ```go
        func mixedArgumentsDemo() {
          x, y := 0, 7
          f := func() int {
            x++
            y++
            return x
          }


          // Ambiguity:
          // The two f() calls are guaranteed to happen in order: f()_1 then f()_2.
          // However, 'y' can be read at three different times:
          // 1. Before both calls:  y=7 -> Output: 1 7 2
          // 2. Between calls:      y=8 -> Output: 1 8 2
          // 3. After both calls:   y=9 -> Output: 1 9 2
          fmt.Printf("Argument Ambiguity: %d %d %d\n", f(), y, f())
        }

        // possible outputs (Compiler Dependent):
        // 
        // A: Argument Ambiguity: 1 7 2
        // B: Argument Ambiguity: 1 8 2
        // C: Argument Ambiguity: 1 9 2

      ```





  -- Critical Takeaway: The "Side-Effect" Danger
  --
    These examples illustrate why "Clever" single-line statements are dangerous 
    in Go if they rely on side effects.

    -- The Golden Rules for Predictable Code 
    --
      1. avoid side effects in functions if you can.
      2. If a function MUST modify a shared variable, do not reference that 
         variable again in the same statement.
      3. secuence explicitly: If you need a specific order, use multiple lines.


    example
      ```go
        func safeVersion() {
          x, y := 0, 7
          f := func() int {
            x++
            y++
            return x
          }
          

          // By splitting this into multiple lines, the behavior is 100% 
          // predictable across all compilers and versions.
          res1 := f()
          valY := y
          res2 := f()
          fmt.Printf("Safe Version: %d %d %d\n", res1, valY, res2)
        }

      ```





-- Expression Evaluation Orders in select-case Blocks
--
  The execution of a `select` block involves a very specific sequence of 
  evaluations. Go distinguishes between the expressions needed to identify 
  "ready" channels and the expressions used to store received data.


  -- The Pre-Selection Phase (Evaluation)
  --
    Before the runtime determines which case to execute, it evaluates all 
    necessary components in source order (top-to-bottom, left-to-right).

    -- What gets evaluated upfront (Exactly Once): 
    --
      1. Channel Operands: In both `case <-ch` and `case ch <- v`, the expression 
         identifying the channel (`ch`) is evaluated.
      2. Send Values: In `case ch <- val()`, the expression for the value being 
         sent (`val()`) is evaluated.

    -- What is NOT evaluated upfront: 
    --
      1. Receive Targets: In `case target() = <-ch`, the expression for the 
         destination (`target()`) is NOT evaluated during the initial phase.


  -- The Selection Phase (Lazy Target Evaluation)
  --
    Once the initial evaluations are done and the runtime selects a branch:
    
    - If the selected branch is a receive operation with an assignment target, 
      that target expression is evaluated ONLY at that moment.
    - If the branch was not selected, its receive target expression is NEVER 
      evaluated. This prevents unnecessary side effects.


  -- Code Demonstration: Tracking the Evaluation Timeline
  --
    ```go
      func main() {
        // Setup: a buffered channel with one value so one case is non-blocking.
        c := make(chan int, 1)
        c <- 0


        // fchan tracks which channel expression is being evaluated.
        fchan := func(info string, c chan int) chan int {
          fmt.Println("Channel Eval:", info)
          return c
        }


        // fptr tracks when an assignment target (pointer) is being evaluated.
        fptr := func(info string) *int {
          fmt.Println("Target Eval: ", info)
          return new(int)
        }


        fmt.Println("--- Start Select ---")


        select {
        // Case 1: Channel evaluated (bbb), Target skipped initially (aaa)
        case *fptr("aaa") = <-fchan("bbb", nil): 

        // Case 2: Channel evaluated (ddd), Target skipped initially (ccc)
        case *fptr("ccc") = <-fchan("ddd", c):   

        // Case 3: Channel evaluated (eee), Send-Value evaluated (fff)
        case fchan("eee", nil) <- *fptr("fff"):  

        // Case 4: Channel evaluated (ggg), Send-Value evaluated (hhh)
        case fchan("ggg", nil) <- *fptr("hhh"):  
        }
        
        fmt.Println("--- End Select ---")
      }

    ```

    -- explaining the evaluation trace:
    --
      Step 1: Lexical Evaluation (Upfront)
      1. bbb (Channel for Case 1)
      2. ddd (Channel for Case 2)
      3. eee (Channel for Case 3)
      4. fff (Value to send for Case 3 - Send values are evaluated early!)
      5. ggg (Channel for Case 4)
      6. hhh (Value to send for Case 4)

      Step 2: Decision
      The runtime polls the channels. 'ddd' (c) has data, so Case 2 is selected.

      Step 3: Post-Selection Evaluation
      7. ccc (The target for the chosen Case 2 is evaluated NOW).

      Note: 'aaa' was NEVER evaluated because its branch was not selected.

    -- EXPECTED OUTPUT:
    --

      --- Start Select ---
      Channel Eval: bbb
      Channel Eval: ddd
      Channel Eval: eee
      Target Eval:  fff
      Channel Eval: ggg
      Target Eval:  hhh
      Target Eval:  ccc
      --- End Select ---
