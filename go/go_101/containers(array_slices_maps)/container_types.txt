In Go, Arrays, Slices, and Maps are the primary "container" types. 
A container is a data structure used to store a collection of elements.





-- Core Common Traits
--
  1. Homogeneous: All elements inside a single container must be of the exact 
     same type (called the "element type").
  2. Length: Every container has a length property, accessible via the built-in
     `len()` function, indicating how many elements it currently holds.






-- Keys and Access
--
  Every element in a container is accessed or modified using an "associated key".


  --  Arrays and Slices (Indexed Containers)
  --
    - Key Type: Always the built-in `int`.
    - Range: Valid keys (indexes) are strictly non-negative integers ranging
      from `0` (inclusive) to `len(container)` (exclusive).

  
  --  Maps (Associative Containers)
  --
    - Key Type: Can be ANY "comparable" type (e.g., string, int, struct). 
      (Slices, maps, and functions cannot be map keys).
    - Range: Keys are arbitrary and not bound to a sequential sequence.





-- Memory Arrangement: Direct vs. Indirect
--
  -- Arrays (Value Semantics) 
  --
    Elements are stored contiguously (side-by-side) in memory.
    The data IS the direct part of the value. There are no hidden pointers.


  -- Slices (Reference Semantics)
  --
    Elements are also stored contiguously in memory.
    However, the slice itself is just a small header (Direct Part). It contains
    a pointer that references an underlying array (Indirect Part) where the
    contiguous data actually lives.


  -- Maps (Reference Semantics) 
  --
    Implemented by the Go runtime as a Hashtable.
    Like a slice, the map variable is just a pointer (Direct Part) to the
    hashtable (Indirect Part). The elements in the hashtable occupy a continuous
    chunk of memory, but they are NOT contiguous (there are "holes" or gaps
    between elements depending on the hash distribution).





--  The Copy Behavior Gotcha
--
  All assignments make a SHALLOW copy of the Direct Part of the value. 


  Because Arrays have no indirect part, copying an array copies ALL its data.
  Because Slices and Maps are just headers/pointers, copying them only copies
  the header. Both the original and the copy will point to the SAME underlying data!


  example
    ```go
      // --- Array Copying (Deep Data Copy) ---
      arr1 := [3]int{1, 2, 3}
      arr2 := arr1 // Copies the entire 3-integer block of memory.
      arr2[0] = 99
      
      fmt.Println(arr1) // [1 2 3]  (Original is UNTOUCHED)
      fmt.Println(arr2) // [99 2 3]

      // --- Slice Copying (Shared Underlying Data) ---
      slc1 :=[]int{1, 2, 3}
      slc2 := slc1 // Copies the 24-byte header ONLY.
      slc2[0] = 99
      
      fmt.Println(slc1) // [99 2 3] (Original IS MUTATED)
      fmt.Println(slc2) // [99 2 3]

      // --- Map Copying (Shared Underlying Data) ---
      map1 := map[string]int{"a": 1}
      map2 := map1 // Copies the 8-byte pointer ONLY.
      map2["a"] = 99
      
      fmt.Println(map1) // map[a:99] (Original IS MUTATED)
      fmt.Println(map2) // map[a:99]

    ```





-- Complexity and Performance
--
  -- Time Complexity
  --
    Accessing an element by its key in ANY of these three containers is O(1) 
    (constant time) on average.


  -- Array/Slice vs. Map Performance
  --
    While both are O(1), Map access requires calculating a hash and following 
    pointers. Therefore, Map lookups are generally several times SLOWER than 
    Array/Slice lookups (which are simple pointer math).


  -- Why use Maps
  --
    1. Flexibility: You can use non-integer keys (like strings or User IDs).
    2. Sparse Data: If you have data at index 1 and index 1,000,000, an Array
       would allocate 1 million slots of memory (mostly zeros). A Map would only 
       allocate memory for the 2 actual entries, saving massive amounts of RAM.





-- Variable Size (`unsafe.Sizeof`)
--
  How much memory does the container variable itself take up? 
  (This measures the Direct Part only).


  -- slice
  --
    All slice types have the EXACT SAME size (typically 24 bytes
    on a 64-bit system: 8 for pointer, 8 for length, 8 for capacity).


  -- maps
  --
    All map types have the EXACT SAME size (typically 8 bytes, as 
    it is just a single pointer to the runtime hashtable).


  -- arrays
  --
    The size depends heavily on the type and length. 
    `Size = Length * SizeOf(ElementType)`.


  -- example
  --
    ```go
      var slc []int64           // Size: 24 bytes
      var m map[string]int64    // Size: 8 bytes
      
      var arr1 [10]int64        // Size: 80 bytes (10 * 8)
      var arr2 [100]int64       // Size: 800 bytes (100 * 8)
      
      // -- Special Cases (Zero Size) --
      // A zero-length array takes up 0 bytes of memory.
      var emptyArr [0]int       // Size: 0 bytes
      
      // An array of zero-sized elements takes up 0 bytes, regardless of length!
      var structArr [1000]struct{} // Size: 0 bytes

    ```
