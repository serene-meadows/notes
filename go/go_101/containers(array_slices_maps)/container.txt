In Go, container types (arrays, slices, and maps) can be represented using
"Type Literals" (which define the structure/blueprint) and "Value Literals"
(which instantiate the actual data in memory).






-- Type Literals (Defining the Blueprint)
--
  Type literals describe the exact shape of an unnamed container type.


  -- Array Type Literals: `[N]T`
  --
    - `T`: The element type. Can be ANY arbitrary type.
    - `N`: The length. MUST be a non-negative integer CONSTANT.


    The length `N` is an inherent part of the array's type.
    Therefore, `[5]int` and `[8]int` are two completely distinct, incompatible types.


    example
      ```go
        type Person struct{ Name string }

        var _ [5]string    // Array of 5 strings
        var _ [16][]byte   // Array of 16 byte-slices
        var _ [100]Person  // Array of 100 Person structs

      ```


  -- Slice Type Literals: `[]T`
  --
    - `T`: The element type.
    Notice there is no length `N`. This is what makes a slice dynamic.


    example
      ```go
        var _ []bool           // Slice of booleans
        var _ []int64          // Slice of int64s
        var _ []map[int]bool   // Slice of maps
        var _ []*int           // Slice of integer pointers

      ```


  -- Map Type Literals: `map[K]T`
  --
    - `K`: The key type. MUST be a "comparable" type (cannot be a slice, map, or func).
    - `T`: The element (value) type. Can be ANY type.


    example
      ```go
        var _ map[string]int            // String keys, Int values
        var _ map[int16][6]string       // Int16 keys, Array of 6 strings as values
        var _ map[bool][]string         // Boolean keys, Slice of strings as values
        var _ map[struct{x int}]*int8   // Struct keys, Int8 pointer values

      ```





-- Value Literals (Instantiating the Data)
--
  Value literals use the Composite Literal syntax: `Type{values...}`

  ```go
    // Array literal (must provide exactly 4 elements if indices aren't specified)
    _ = [4]bool{false, true, true, false}

    // Slice literal
    _ =[]string{"break", "continue", "fallthrough"}

    // Map literal (Key-Value pairs are called "entries")
    _ = map[string]int{"C": 1972, "Python": 1991, "Go": 2009}

  ```





-- Array & Slice Variants (Keyed / Indexed Literals)
--
  You can explicitly specify the index for elements. This allows you to skip
  elements (which become zero-valued) and write them in any order.


  example
    ```go
      //["break", "continue", "fallthrough"]
      _ =[]string{"break", "continue", "fallthrough"}
      _ =[]string{0: "break", 1: "continue", 2: "fallthrough"}
      _ =[]string{2: "fallthrough", 1: "continue", 0: "break"} // Order doesn't matter
      _ =[]string{2: "fallthrough", 0: "break", "continue"}    // Mixed!

      // -- The "Missing Index" Rule --
      // If an index is omitted, it assumes `previous_index + 1`.
      // If the very first index is omitted, it defaults to `0`.

      // ALL 4 of these array literals result in:[false, true, true, false]
      _ = [4]bool{false, true, true, false}
      _ =[4]bool{0: false, 1: true, 2: true, 3: false}
      _ = [4]bool{1: true, true}       // Index 1 is true, Index 2 (1+1) is true. Rest are false.
      _ = [4]bool{2: true, 1: true}    // Index 1 is true, Index 2 is true. Rest are false.

    ```





-- `...` Length Deduction (Arrays Only)
--
  If you use `...` instead of a number for an array length, the compiler will
  automatically count the elements and deduce the length for you.


  example
    ```go
      // The compiler counts 4 elements, so this becomes a `[4]bool`.
      _ = [...]bool{false, true, true, false}

      // The highest index provided is 3, so the array must have 4 elements.
      // This also becomes a `[4]bool`.
      _ = [...]bool{3: false, 1: true, true}

    ```




-- The "Constant Key" Rule (Arrays/Slices vs. Maps)
--
  When initializing an Array or Slice using keyed indices, the index MUST be a
  non-negative CONSTANT that is representable as an `int`.

  When initializing a Map, the keys can be NON-CONSTANTS (runtime variables).


  example
    ```go
      // OKAY: Maps evaluate their keys at runtime, so variables are perfectly fine.
      _ = map[uint]int{dynamicKey: 123}

      // ERROR: Array and Slice literals are constructed differently by the compiler.
      // Their memory layout must be resolvable at compile time, so variables are forbidden.

      // _ =[]int{dynamicKey: 100}  // COMPILE ERROR: index must be a constant
      // _ = [5]int{dynamicKey: 100} // COMPILE ERROR: index must be a constant

    ```





-- Zero Values & Memory Allocation
--
  -- Arrays: Memory is pre-allocated
  --
    When an array is declared, its memory is allocated IMMEDIATELY, and all
    elements are initialized to their zero values.
    The composite literal `[100]int{}` represents the zero value.

      ```go
        func zeroArrays() {
          var a[100]int // Memory for 100 ints is allocated immediately.
          _ = a
        }

      ```





-- Slices & Maps: `nil` vs. Empty
--
  The zero value of a slice or a map is the untyped `nil` identifier.
  When a slice or map is `nil`, NO underlying memory has been allocated for data.

  This creates a critical difference between a `nil` container and an `empty` container.


  ```go
      // 1. The `nil` slice (No underlying array allocated)
      var nilSlice[]int
      var nilSliceExplicit =[]int(nil) // Equivalent to above

      // 2. The `empty` slice (Underlying array of length 0 IS allocated)
      var emptySlice =[]int{}

      fmt.Printf("nilSlice is nil? %t\n", nilSlice == nil)                 // true
      fmt.Printf("nilSliceExplicit is nil? %t\n", nilSliceExplicit == nil) // true
      fmt.Printf("emptySlice is nil? %t\n", emptySlice == nil)             // false

  ```





-- Addressability of Composite Literals
--
  By default, composite literals (like `[]int{1, 2}` or `map[string]int{}`) are
  unaddressable. They represent values in transit, not fixed memory locations.


  However, just like with structs, Go provides highly convenient syntactic sugar
  that allows you to use the address-of operator (`&`) directly on a composite
  literal


  When the compiler sees `&[]string{"a"}`, it silently translates it to:
    tmp :=[]string{"a"}  // 1. Create a hidden, addressable temporary variable
    ptr := &tmp           // 2. Take the address of that temporary variable



  example
    ```go
      // Using `&` directly on map, slice, and array literals.
      pm := &map[string]int{"C": 1972, "Go": 2009}
      ps := &[]string{"break", "continue"}
      pa := &[...]bool{false, true, true, false}

      // Using %T to print the type of the variables.
      // Notice they are all pointers to their respective container types.
      fmt.Printf("%T\n", pm) // Output: *map[string]int
      fmt.Printf("%T\n", ps) // Output: *[]string
      fmt.Printf("%T\n", pa) // Output: *[4]bool

    ```




-- Simplifying Nested Composite Literals
--
  When a composite literal contains other composite literals, and the compiler
  can unambiguously infer the inner types from the outer type definition, you
  can omit the inner type declarations


  This allows you to simplify the inner elements down to just `{...}`, making
  your code much cleaner and easier to read



  --  Example A: Slice of Pointers to Arrays
  --
    ```go
      // Verbose (Without simplification):
      // The element type is `*[4]byte`. We explicitly write `&[4]byte{...}` every time.
      var verboseHeads = []*[4]byte{
        &[4]byte{'P', 'N', 'G', ' '},
        &[4]byte{'G', 'I', 'F', ' '},
      }
      _ = verboseHeads

      // Simplified (Idiomatic Go):
      // The compiler knows each element MUST be a `*[4]byte`.
      // We can drop the `&[4]byte` entirely. The compiler handles the allocation
      // and takes the address automatically.
      var cleanHeads = []*[4]byte{
        {'P', 'N', 'G', ' '},
        {'G', 'I', 'F', ' '},
        {'J', 'P', 'E', 'G'},
      }
      _ = cleanHeads

    ```


  -- Example B: Array of Structs
  --
    ```go
      type Language struct {
        name string
        year int
      }

      func simplifyStructArray() {
        // Verbose:
        var _ = [...]Language{
          Language{"C", 1972},
          Language{"Python", 1991},
        }

        // Simplified:
        // The compiler infers that each `{}` represents a `Language` struct.
        var _ = [...]Language{
          {"C", 1972},
          {"Python", 1991},
          {"Go", 2009},
        }
      }

    ```


  --  Example C: Complex Maps (Struct Keys and Map Values)
  --
    ```go
      type LangCategory struct {
        dynamic bool
        strong  bool
      }

      func simplifyComplexMap() {
        // Verbose:
        // A map where the key is a `LangCategory` struct, and the value is a `map[string]int`.
        var _ = map[LangCategory]map[string]int{
          LangCategory{true, true}: map[string]int{
            "Python": 1991,
            "Erlang": 1986,
          },
          LangCategory{false, false}: map[string]int{
            "C": 1972,
          },
        }

        // Simplified:
        // The compiler infers the key type from the left side of the colon `:`,
        // and the value type from the right side of the colon `:`.
        var _ = map[LangCategory]map[string]int{
          {true, true}: {
            "Python": 1991,
            "Erlang": 1986,
          },
          {true, false}: {
            "JavaScript": 1995,
          },
          {false, true}: {
            "Go":   2009,
            "Rust": 2010,
          },
          {false, false}: {
            "C": 1972,
          },
        }
      }

    ```








-- Comparing Container Values
--
  Go has strict rules about which types can be compared using the equality
  operators (`==` and `!=`). A type that can be compared is called "comparable".
  This is critical because ONLY comparable types can be used as map keys.


	-- Slices and Maps (Incomparable)
  --
    Slices and Maps are INCOMPARABLE. The language designers intentionally
    forbade comparing them to avoid hidden, expensive O(N) operations and
    complications with cyclic data structures.

    - They CANNOT be compared to another slice/map.
    - They CANNOT even be compared to themselves.
    - They CANNOT be used as keys in a map.

    Exception: The ONLY valid comparison for a slice or map is against the
    untyped `nil` identifier to check if it is uninitialized.

    example
      ```go
        var s[]int
        var m map[string]int

        fmt.Println(s == nil) // true
        fmt.Println(m == nil) // true

        // Important Distinction: `nil` vs. Empty Literal
        // `[]int{}` allocates an empty backing array. Therefore, it is NOT nil.
        fmt.Println(nil ==[]int{})          // false
        fmt.Println(nil == map[string]int{}) // false

        // -- ILLEGAL COMPARISONS --
        /*
        _ = s == s                    // COMPILE ERROR: slice can only be compared to nil
        _ = m == m                    // COMPILE ERROR: map can only be compared to nil
        _ = s ==[]int(nil)           // COMPILE ERROR: comparing to a TYPED nil is forbidden
        */

      ```


  -- Arrays (Conditionally Comparable)
  --
    Arrays are value types. An array is COMPARABLE if and only if its
    element type is comparable.


    When you compare two arrays, Go automatically compares each pair of
    corresponding elements one by one.


    example
      ```go
        var a1 [16]byte
        var a2 [16]byte
        fmt.Println(a1 == a2) // true (All elements are 0, and bytes are comparable)


        // -- The "Infection" Rule
        // --
        // If an array contains an incomparable type (like a slice), the array
        // itself becomes incomparable!
        /*
        var x [16][]int
        _ = x == x // COMPILE ERROR: array of slices is not comparable


        var y [16]map[int]bool
        _ = y == y // COMPILE ERROR: array of maps is not comparable
        */

      ```





-- Lengths and Capacities (`len` and `cap`)
--
  Go provides two built-in functions to inspect the size of containers:
  - `len(v)`: Returns the number of elements currently stored in the container.
  - `cap(v)`: Returns the capacity (the maximum potential size) of the container.

  Both return an `int`. If the argument is a constant, the result is a constant.


  -- Arrays (Fixed Size)
  --
    Because an array's length is part of its type (e.g., `[5]int`), its
    length and capacity are ALWAYS equal and can NEVER change at runtime.


    example
      ```go
        var a [5]int
        fmt.Printf("Array - len: %d, cap: %d\n", len(a), cap(a)) // len: 5, cap: 5

      ```


  -- Maps (Dynamic Length, No Capacity)
  --
    Maps grow dynamically as you add items. Conceptually, their capacity is
    unlimited (bound only by system memory).
    Therefore, the `cap()` function is NOT APPLICABLE to maps.


    example
      ```go
        var m map[int]bool
        fmt.Printf("Map (nil)   - len: %d\n", len(m)) // len: 0

        m2 := map[int]bool{1: true, 0: false}
        fmt.Printf("Map (items) - len: %d\n", len(m2)) // len: 2

        // _ = cap(m) // COMPILE ERROR: invalid argument m (type map[int]bool) for cap

      ```


  -- Slices (Dynamic Length AND Capacity)
  --
    Slices are dynamic windows into an underlying array.
    - `len`: The number of elements currently visible in the slice's window.
    - `cap`: The total number of elements in the underlying array, counting
             from the start of the slice's window to the end of the array.

    For slices, Capacity is ALWAYS >= Length. Both can change at runtime.


    example
      ```go
        var s[]int // A nil slice
        fmt.Printf("Slice (nil)   - len: %d, cap: %d\n", len(s), cap(s)) // len: 0, cap: 0


        s2 :=[]int{2, 3, 5} // An initialized slice
        fmt.Printf("Slice (items) - len: %d, cap: %d\n", len(s2), cap(s2)) // len: 3, cap: 3


        // We create an array of 5 elements.
        underlyingArray := [5]int{10, 20, 30, 40, 50}


        // We create a slice that "looks" at indices 1 and 2 (values 20, 30).
        windowSlice := underlyingArray[1:3]


        // The length is 2 (it contains two items: 20, 30).
        // The capacity is 4 (from index 1 to the end of the array: 20, 30, 40, 50).
        // This means the slice can safely "grow" up to 2 more elements without
        // needing to allocate new memory.
        fmt.Printf("Slice (window)- len: %d, cap: %d\n", len(windowSlice), cap(windowSlice))

      ```





-- Retrieving & Modifying Container Elements
--
  In Go, the syntax `v[k]` is used to retrieve or modify an element within a 
  container `v` using its associated key or index `k`.


  The rules governing what happens when you use `v[k]` depend heavily on whether 
  `v` is an Array, a Slice, or a Map.


  -- Arrays and Slices (Indexed Access)
  --
    For arrays and slices, `k` represents an integer index.

    
    -- Constant Indices (Compile-Time Checks)
    --
      If `k` is a hardcoded constant (e.g., `v[2]`), the compiler checks it strictly:
      - It must be a non-negative integer.
      - For Arrays: It MUST be strictly less than the array's fixed length. If it 
        is out of bounds, the program WILL NOT COMPILE.


    --  Variable Indices (Runtime Checks)
    --
      If `k` is a variable (e.g., `v[n]`), the compiler cannot predict its value, 
      so the check is deferred to runtime:
      - It must be a basic integer type.
      - It must be `>= 0` and `< len(v)`.
      - If it is out of bounds, the program will PANIC at runtime.


    -- The `nil` Slice Rule
    --
      Accessing or modifying an element in a `nil` slice ALWAYS causes a runtime panic, 
      because a `nil` slice has a length of 0 and no underlying memory.


  -- Maps (Key-Value Access)
  --
    For maps, `k` must be a value assignable to the map's defined key type.


    -- Reading from Maps (Extremely Forgiving)
    --
      - Missing Keys: If you try to read a key `k` that does not exist in the 
        map, Go does NOT panic. Instead, it gracefully returns the "zero value" of 
        the map's element type.
      - Nil Maps: You can safely READ from a `nil` map. It acts exactly like 
        an empty map and simply returns zero values.


    -- Distinguishing Missing vs. Zero
    --
      Because missing keys return zero values, how do you know if a key was actually 
      missing, or if its value was genuinely just `0` (or `""`, or `false`)?
      Go allows `v[k]` to optionally return a second untyped boolean value:
      `value, isPresent := v[k]`


    -- Writing to Maps (Strict Memory Rules)
    --
      Nil Maps: While reading from a `nil` map is safe, WRITING to a `nil` 
      map causes a runtime PANIC. (A nil map has no initialized hashtable to store the new entry).


    -- Interface Keys 
    --
      If the map's key type is an interface (e.g., `any`), and you pass a dynamic 
      type that is incomparable (like a slice) as the key `k`, a runtime PANIC occurs.


  -- example
  --
    ```go
      // Setup Containers
      //
        a := [3]int{-1, 0, 1}                    // Array (len 3)
        s :=[]bool{true, false}                 // Slice (len 2)
        m := map[string]int{"abc": 123, "xyz": 789} // Map


      // Valid Retrievals & Modifications
      //
        fmt.Println("Initial:", a[2], s[1], m["abc"]) // Output: 1 false 123

        a[2] = 999       // Modify Array
        s[1] = true      // Modify Slice
        m["abc"] = 567   // Modify Map


      // Reading a missing key returns the zero value (0 for int).
      //
        val1, present1 := m["hello"]
        fmt.Printf("Key 'hello': Value=%d, Present=%t\n", val1, present1) // Output: 0, false


      // Reading from a `nil` Map (Safe)
      //
      m = nil // Set the map to nil
      fmt.Println("Read from nil map:", m["abc"]) // Output: 0 (Safe! Returns zero value)


      // Compile time Errors (Constants)
      //
      _ = a[3]  // COMPILE ERROR: invalid array index 3 (out of bounds for 3-element array)
      _ = s[-1] // COMPILE ERROR: invalid slice index -1 (must be non-negative)


      // Runtime Panics (Variables & State)
      //
      n := 5 
        
      // PANIC: index out of range[5] with length 3
      // _ = a[n] 
      
      // PANIC: index out of range [5] with length 2
      // _ = s[n] 
      
      // PANIC: assignment to entry in nil map
      // m["hello"] = 555 

    ```





-- Internal Structure Definition of Slice Types
--
  the internal structure of slice types defined by the standard Go compiler/runtime is like

    ```go
      type _slice struct {
        elements unsafe.Pointer // referencing underlying elements
        len      int            // length
        cap      int            // capacity
      }

    ```





-- Slice Memory Layout & Append Mechanics
--
  -- The Slice "Header" (Direct Part)
  --
    As discussed previously, a slice is just a small "header" (the direct part) 
    that describes a contiguous section of a backing array (the indirect part).

    At runtime, a slice header consists of exactly three fields:
    1. A Pointer: Points to the first element of the underlying array that this
       slice can "see".
    2. Length (`len`): The number of active, accessible elements in the slice.
    3. Capacity (`cap`): The total number of elements in the underlying array,
       counting from the pointer to the very end of the allocated memory block.


  -- The "Window" and Redundant Slots
  --
    You can think of a slice as a "window" looking into a larger array.

    - The elements from index `0` to `len - 1` belong to the slice.
    - The elements from index `len` to `cap - 1` are "redundant" or hidden slots.
      The current slice cannot directly read or write them, but the memory is
      reserved for future growth.
    - WARNING: Those hidden slots might actually be active elements belonging to
      a DIFFERENT slice that shares the same underlying array!


  -- The `append()` Function
  --
    The built-in `append()` function adds new elements to the end of a slice.
    Because slices are just headers, `append` always returns a NEW slice header
    (which you typically assign back to the original variable).

      `s = append(s, 10, 20)`


    The most important question `append` asks is:
    "Does the base slice have enough capacity to hold these new elements?"

    -- The Two Scenarios of Append (Shared vs. New Memory)
    --
      -- Scenario A: Sufficient Capacity (In-Place Append)
      --
        If (number of new elements) <= (cap - len):
        - NO new memory is allocated.
        - `append` simply places the new elements into the "hidden slots" of the
          existing underlying array.
        - The returned slice and the original slice SHARE the same underlying memory.

    -- Scenario B: Insufficient Capacity (Reallocation)
    --
      If (number of new elements) > (cap - len):
      - The underlying array is full. `append` CANNOT fit the new elements.
      - Go allocates a brand NEW, larger underlying array somewhere else in memory.
      - It copies all the old elements over, adds the new ones, and returns a slice
        pointing to the NEW array.
      - The returned slice and the original slice DO NOT share memory anymore.


  -- example
  --
    ```go
      // Create a slice with Length 2, Capacity 4.
      // We use make() to explicitly set len and cap.
      baseSlice := make([]int, 2, 4)
      baseSlice[0] = 10
      baseSlice[1] = 20
      

      fmt.Printf("Base   - len: %d, cap: %d, data: %v\n", len(baseSlice), cap(baseSlice), baseSlice)


      // -- SCENARIO A: Sufficient Capacity 
      // We append 1 element. We have 2 empty slots in capacity, so it fits.
      // `sharedSlice` and `baseSlice` will point to the SAME underlying array.
      sharedSlice := append(baseSlice, 30)


      // If we mutate the shared portion, it affects both!
      sharedSlice[0] = 999 


      fmt.Println("\n-- After appending 1 element (fits in capacity)")
      fmt.Printf("Base   - len: %d, cap: %d, data: %v  <-- Notice index 0 changed!\n", len(baseSlice), cap(baseSlice), baseSlice)
      fmt.Printf("Shared - len: %d, cap: %d, data: %v\n", len(sharedSlice), cap(sharedSlice), sharedSlice)


      // -- SCENARIO B: Insufficient Capacity 
      // We append 3 elements to `baseSlice`. 
      // `baseSlice` only has 2 empty slots left in its capacity. IT DOES NOT FIT.
      // Go allocates a completely new array for `newSlice`.
      newSlice := append(baseSlice, 40, 50, 60)


      // If we mutate `newSlice`, it does NOT affect `baseSlice` anymore, 
      // because they point to different memory locations.
      newSlice[1] = 777 


      fmt.Println("\n-- After appending 3 elements (exceeds capacity)")
      fmt.Printf("Base   - len: %d, cap: %d, data: %v  <-- Unaffected by newSlice mutation!\n", len(baseSlice), cap(baseSlice), baseSlice)
      fmt.Printf("New    - len: %d, cap: %d, data: %v\n", len(newSlice), cap(newSlice), newSlice)

    ```





-- Container Assignments
--
  -- map
  --
  If a map is assigned to another map, then the two maps will share all (underlying) elements.
  Appending elements into (or deleting elements from) one map will reflect on the other map.


  -- slice
  --
  Like map assignments, if a slice is assigned to another slice, they will share all (underlying)
  elements. Their respective lengths and capacities equal to each other. However, if the
  length/capacity of one slice changes later, the change will not reflect on the other slice.


  -- array
  --
    When an array is assigned to another array, all the elements are copied from the source one to
    the destination one. The two arrays don't share any elements.


  -- example
  --
    ```go
      m0 := map[int]int{0:7, 1:8, 2:9}
      m1 := m0
      m1[0] = 2
      fmt.Println(m0, m1) // map[0:2 1:8 2:9] map[0:2 1:8 2:9]


      s0 := []int{7, 8, 9}
      s1 := s0
      s1[0] = 2
      fmt.Println(s0, s1) // [2 8 9] [2 8 9]


      a0 := [...]int{7, 8, 9}
      a1 := a0
      a1[0] = 2
      fmt.Println(a0, a1) // [7 8 9] [2 8 9]

    ```





-- Appending & Deleting Container Elements
--
  -- Maps (Dynamic Key-Value Pairs)
  --
    Maps are inherently dynamic. You can easily add, update, and remove entries.


    -- Appending / Modifying (Upsert)
    --
      The syntax for adding a new entry is identical to modifying an existing one:
        `m[k] = e`
      If the key `k` does not exist, Go appends the new key-value pair.
      If the key `k` already exists, Go overwrites (modifies) the existing value.


    -- Deleting
    --
      Go provides the built-in `delete(map, key)` function to remove entries.
      IMPORTANT SAFETY RULE: `delete` is extremely forgiving. If you call it with a
      key that doesn't exist, or even if you call it on a `nil` map, it acts as a
      safe "no-op" (it does nothing) and will NOT panic.

      example
        ```go
          m := map[string]int{"Go": 2007}

          // Append new entries
          m["C"] = 1972
          m["Java"] = 1995
          fmt.Println(m) // map[C:1972 Go:2007 Java:1995]

          // Modify an existing entry
          m["Go"] = 2009

          // Delete an entry
          delete(m, "Java")

          // Delete a non-existent key (Safe, no panic)
          delete(m, "Rust")

          fmt.Println(m) // map[C:1972 Go:2009]

        ```


  -- Arrays (Fixed Size)
  --
    By definition, arrays have a fixed size determined at compile time.
    Therefore, elements CANNOT be appended to or deleted from an array.
    You can only modify the values of existing elements (if the array is addressable).



  -- Slices (Dynamic Sequences)
  --
    Slices are dynamic windows over underlying arrays. We use the built-in,
    variadic `append` function to add elements:
    `newSlice := append(baseSlice, elem1, elem2...)`


    -- The "No Delete" Rule
    --
      Go does not have a built-in `delete` function for slices. To delete an element,
      you must combine `append` with slice re-slicing (e.g., `append(s[:i], s[i+1:]...)`).

    
    example
      ```go
        // Idiomatic slice appending: Reassign the result back to the same variable.
        var s []string = append([]string(nil), "array", "slice") 
        // Note: The first argument to append CANNOT be the untyped `nil` identifier.
        // It must be a typed nil slice (e.g., `[]string(nil)` or just a declared variable).

        fmt.Printf("Len: %d, Cap: %d, Data: %v\n", len(s), cap(s), s) // Len: 2, Cap: 2
        
        s = append(s, "map")
        fmt.Printf("Len: %d, Cap: %d, Data: %v\n", len(s), cap(s), s) // Len: 3, Cap: 4

      ```


    -- How `append` Manages Memory
    --
      Because `append` returns a new slice header, you must understand how it
      manages the underlying memory to avoid accidental data sharing or corruption.


      When you call `append(baseSlice, elements...)`:
      -- Scenario A: Sufficient Capacity
      --
        If the `baseSlice` has enough redundant capacity (cap - len >= new_elements):
        `append` overwrites those redundant slots. NO new memory is allocated.
        The new slice and the base slice SHARE the same underlying array!


      -- Scenario B: Insufficient Capacity
      --
        If the `baseSlice` lacks capacity:
        `append` allocates a brand NEW, larger underlying array, copies the old data over,
        and adds the new elements.
        The new slice and the base slice DO NOT share memory!
        (To optimize, the Go compiler usually doubles the capacity to prevent frequent allocations).


      example
        ```go
          s0 :=[]int{2, 3, 5} // len: 3, cap: 3
          
          // s0 lacks capacity for a new element.
          // SCENARIO B: Allocates NEW memory. `s1` and `s0` are totally independent.
          s1 := append(s0, 7)  // s1 len: 4, cap: 6 (Go doubled the capacity)
          
          // s1 has capacity of 6, current length 4. We append 2 elements.
          // SCENARIO A: Has sufficient capacity!
          // `s2` and `s1` now SHARE the same underlying array.
          s2 := append(s1, 11, 13) // s2 len: 6, cap: 6
          
          // `s3` is just a copy of `s0` (append with no extra elements).
          s3 := append(s0)
          
          // Appending `s0` to itself. Exceeds capacity. Allocates NEW memory.
          s4 := append(s0, s0...) // s4 len: 6, cap: 6

          // -- Let's prove the memory sharing! 
          
          // We modify index 0 of `s0` and `s1`.
          s0[0] = 99
          s1[0] = 789 

          // Output: 789 99 2
          fmt.Println(s2[0], s3[0], s4[0]) 


        // s0[0] modification affected s3[0] because they share an array.
        // s1[0] modification affected s2[0] because they share an array.
        // s4 remains untouched because it was allocated a brand new array.


        ```
	
      -- visualizing the memory
      --

           [ Slice Headers ]                [ Underlying Arrays ]
            +--------------+                 +------------------+
        s0  | elements ptr |---------------> |        99        | <-----
            | len (3)      |                 |        3         |      |
            | cap (3)      |                 |        5         |      |
            +--------------+                 +------------------+      |
                                                                       |
            +--------------+                 +------------------+      |         s3
        s1  | elements ptr |---------+-----> |       789        |      |  +--------------+
            | len (4)      |         |       |        3         |      ---| elements ptr |
            | cap (6)      |         |       |        5         |         | len (3)      |
            +--------------+         |       |        7         |         | cap (3)      |
                                     |       |        11        |         +--------------+
            +--------------+         |       |        13        |
        s2  | elements ptr |---------|       +------------------+
            | len (6)      |
            | cap (6)      |                 +------------------+
            +--------------+         ------> |        2         |
                                     |       |        3         |
            +--------------+         |       |        5         |
        s4  | elements ptr |---------|       |        2         |
            | len (6)      |                 |        3         |
            | cap (6)      |                 |        5         |
            +--------------+                 +------------------+





-- Creating Slices and Maps with `make`
--
  While composite literals (`[]int{...}`) are common, the built-in `make` function
  is the idiomatic way to create and pre-allocate memory for slices, maps, and channels.
  `make` CANNOT be used to create arrays.


  -- Using `make` with Maps
  --
    ```go
      make(map[K]T) 

      // or 

      make(map[K]T, n)


      // - The first form (`make(map[K]T)`) creates a new, non-nil, empty map with a
      //   small, default amount of pre-allocated space.
      // - The second form (`make(map[K]T, n)`) creates a new, non-nil, empty map but
      //   provides a "size hint" to the runtime, pre-allocating enough memory to hold
      //   at least `n` entries. This is a performance optimization to avoid repeated
      //   reallocations if you know the map will grow large.

    ```


  -- Using `make` with Slices
  --
    ```go
      make([]T, length) 

      // or 

      make([]T, length, capacity)


      // - `length`: The initial number of elements in the slice.
      // - `capacity`: The total size of the underlying array that is allocated.
      // - RULE: `0 <= length <= capacity`.
      //
      // All elements in the newly created slice are initialized to the zero value of `T`.

    ```


  example 
    ```go
      // -- Maps 
      m1 := make(map[string]int)
      fmt.Println("Empty map:", m1) // map[]


      m2 := make(map[string]int, 3) // Pre-allocate for 3 entries
      fmt.Printf("Pre-alloc map: %v, len: %d\n", m2, len(m2)) // map[], len: 0
      m2["Go"] = 2009
      fmt.Printf("After add: %v, len: %d\n", m2, len(m2)) // map[Go:2009], len: 1
      

      // -- Slices 
      // Create a slice of length 3 and capacity 5. It is filled with zero values (0 for int).
      s1 := make([]int, 3, 5)
      fmt.Printf("Slice 1: %v, len: %d, cap: %d\n", s1, len(s1), cap(s1)) // [0 0 0], 3, 5


      // If capacity is omitted, it defaults to being equal to the length.
      s2 := make([]int, 2)
      fmt.Printf("Slice 2: %v, len: %d, cap: %d\n", s2, len(s2), cap(s2)) // [0 0], 2, 2

    ```





  -- Why `new` is Rarely Used for Slices and Maps
  --
    The `new(T)` function allocates memory for a zero value of type `T` and returns
    a pointer (`*T`) to it.


    The zero value of a slice or map is `nil`.


    Therefore, `new([]int)` returns a `*[]int` that points to a `nil` slice. This
    is rarely useful and is more verbosely equivalent to `var s []int`.

    example
      ```go
        // Using `new` just gives you a pointer to a nil map.
        pm := new(map[string]int)
        fmt.Println(*pm == nil) // true


        // This is equivalent to just declaring a nil map variable.
        var m map[string]int
        fmt.Println(m == nil) // true


        // Using `new` with an array IS valid, as its zero value is a fully
        // initialized array. However, using a composite literal is far more common.
        pa := new([5]bool)
        fmt.Println(*pa == [5]bool{}) // true

      ```





-- Addressability of Container Elements
--
  The ability to take the address of an element (`&v[k]`) depends on the container.


  -- Arrays: Addressability Cascades
  --
    If an array is addressable (it's a variable), its elements are addressable.
    If an array is unaddressable (it's a literal), its elements are unaddressable.


    example
      ```go
        a := [2]int{1, 2}
        _ = &a[0]         // OK


        // _ = &[2]int{1, 2}[0] // COMPILE ERROR: cannot take address of literal's element

      ```



  -- Slices: Elements are ALWAYS Addressable
  --
    Slice elements reside in the underlying array, which is an allocated block of
    memory on the heap. Therefore, slice elements are always addressable, even if
    the slice header itself is from an unaddressable literal.


    example
      ```go
        s := make([]bool, 2)
        _ = &s[0]                       // OK
        _ = &[]string{"Go", "C"}[0]    // OK!

      ```


  -- Maps: Elements are NEVER Addressable
  --
    You can NEVER take the address of a map element.
    WHY?: The Go runtime may move map elements in memory as the map grows
    (a process called "rehashing"). If Go allowed you to take a pointer to `m["key"]`,
    that pointer could become a "dangling pointer" to invalid memory the next time
    an element is added to the map. To prevent this entire class of bugs, it is forbidden.


    example
      ```go
        m := map[int]bool{1: true}
        _ = &m[1] // COMPILE ERROR: cannot take the address of m[1]

      ```


    -- The Consequence of Unaddressable Map Elements
    --
      Because map elements are unaddressable, you cannot modify parts of them directly
      (e.g., fields of a struct or elements of an array).


      example
        ```go
          mt := map[string]T{}
          mt["John"] = T{age: 29}
          

          ma := map[int][5]int{}
          ma[1] = [5]int{1: 789}


          // Accessing is fine.
          fmt.Println(mt["John"].age) // 29
          fmt.Println(ma[1][1])       // 789

          
          // The following modifications are ILLEGAL because `mt["John"]` is unaddressable.
          // You are trying to modify a part of a value that has no fixed memory location.
          // mt["John"].age = 30 // COMPILE ERROR
          // ma[1][1] = 123      // COMPILE ERROR


          // THE IDIOMATIC FIX: Copy, Modify, Re-assign
          // 1. Copy the element to a temporary, addressable local variable.
          tempT := mt["John"]
          // 2. Modify the copy.
          tempT.age = 30
          // 3. Assign the entire modified copy back to the map.
          mt["John"] = tempT
          fmt.Println(mt["John"].age) // 30

        ```





-- Deriving Slices (Reslicing)
--
  You can create a new slice from an existing slice or an addressable array.
  This is called "reslicing". The new slice SHARES the same underlying memory.


  -- The Subslice Syntax
  --
    - Two-Index Form*
      `base[low : high]`
      Creates a slice with `len = high - low` and `cap = cap(base) - low`.

    - Three-Index Form*
      `base[low : high : max]`
      Creates a slice with `len = high - low` and `cap = max - low`.
      This form is used to explicitly control the capacity of the new slice.


  -- Constraints
  --
    `0 <= low <= high <= max <= cap(base)` // len(base) doesn't matter


  -- Omission Rules
  --
    - low 
      if the low index is equal to zero
      it can be omitted,
      either for two-index or three-index forms.

    - high
      if the high is equal to len(baseContainer)
      it can be omitted, but only for two-index forms.

    - max
      the max can never be omitted in three-index forms.


    For example, the following expressions are equivalent.
      ```go
        baseContainer[0 : len(baseContainer)]
        baseContainer[: len(baseContainer)]
        baseContainer[0 :]
        baseContainer[:]
        baseContainer[0 : len(baseContainer) : cap(baseContainer)]
        baseContainer[: len(baseContainer) : cap(baseContainer)]

      ```





-- Reslicing Example and Memory Walkthrough
--
  ```go
    // Underlying Array: [0 1 2 3 4 5 6]
    a := [...]int{0, 1, 2, 3, 4, 5, 6}

    // s0 and s1 point to the full array.
    s0 := a[:]     // len: 7, cap: 7, data: [0 1 2 3 4 5 6]
    s1 := s0[:]    // len: 7, cap: 7, data: [0 1 2 3 4 5 6]

    // s2 is a window from index 1 up to (but not including) 3.
    s2 := s1[1:3]  // len: 2, cap: 6, data: [1 2]

    // s3 is a window from index 3 to the end.
    s3 := s1[3:]   // len: 4, cap: 4, data: [3 4 5 6]

    // s4 is a window from index 3 up to 5.
    s4 := s0[3:5]  // len: 2, cap: 4, data: [3 4]

    // s5 is derived from s4, but its capacity is restricted to 2.
    s5 := s4[:2:2] // len: 2, cap: 2, data: [3 4]

    // s6 appends to s4. It has capacity, so it writes into the shared array.
    s6 := append(s4, 77) // data: [3 4 77]

    // s7 appends to s5. s5's capacity was restricted to 2, so this must allocate NEW memory.
    s7 := append(s5, 88) // data: [3 4 88]

    // s8 appends to s7, which has enough capacity. It shares memory with s7.
    s8 := append(s7, 66) // data: [3 4 88 66]

    // This modification to s3 writes to index 4 of the original array `a`.
    s3[1] = 99

    // WALKTHROUGH of final state of underlying array `a`: [0 1 2 3 99 77 6]
    // s2 points to indices 1-2 of `a`. cap is 7-1=6 -> cap(6), data [1 2]
    // s3 points to indices 3-6 of `a`. cap is 7-3=4 -> cap(4), data [3 99 77 6]
    // s4 points to indices 3-4 of `a`. cap is 7-3=4 -> cap(4), data [3 99]
    // s5 points to indices 3-4 of `a`. cap was reset to 5-3=2 -> cap(2), data [3 99]
    // s6 points to indices 3-5 of `a`. cap is 7-3=4 -> cap(4), data [3 99 77]
    // s7 points to a NEW array.
    // s8 points to the SAME new array as s7.

  ``` 
      
      

    

-- The Memory Leak Risk
--
  Be careful when returning a small subslice from a function that created a very
  large slice. The returned subslice holds a reference to the entire large
  underlying array, preventing the garbage collector from reclaiming it.


  eample
    ```go
      func potentialLeak() []byte {
        largeSlice := make([]byte, 1000000) // Allocate 1MB

        // ... do some work ...

        return largeSlice[900000:900010] // Return a tiny 10-byte window
        // The entire 1MB array is still "live" because the returned slice points into it.
      }

    ```






-- Conversions Between Slices and Arrays
--
  Go allows for explicit conversions between slices and arrays/array pointers,
  but these conversions come with strict rules and potential runtime panics.


  -- Convert Slice to Array POINTER (`*[N]T`)
  --
    A slice `s` can be converted to an array pointer `*[N]T`. This conversion
    does NOT copy the data; the resulting pointer points directly into the
    slice's underlying array.


    Rule:
      The length of the target array (`N`) must be less than or equal to
      the length of the source slice (`len(s)`).
      `N <= len(s)`
      If this rule is violated, a runtime PANIC occurs.

    
    example
      ```go
        s := make([]int, 3, 5) // len: 3, cap: 5


        // OK: Target array length (3) is equal to slice length (3).
        _ = (*[3]int)(s)


        // OK: Target array length (2) is less than slice length (3).
        _ = (*[2]int)(s)


        // PANIC: Target array length (4) is greater than slice length (3).
        // _ = (*[4]int)(s)


        var x []int // nil slice
        var y = make([]int, 0) // empty slice


        // OK: A nil slice can be converted to a pointer to a zero-length array. The pointer is nil.
        var x0 = (*[0]int)(x) // x0 == nil


        // OK: An empty slice can also be converted. The pointer is non-nil.
        var y0 = (*[0]int)(y) // y0 != nil
        _, _ = x0, y0

      ```


  -- Convert Slice to Array VALUE (`[N]T`)
  --
    A slice `s` can be converted to an array value `[N]T`. This conversion
    ALWAYS copies the data. The resulting array and the source slice do NOT
    share memory.

    Rule: 
      The length of the target array (`N`) must be less than or equal to
      the length of the source slice (`len(s)`).
      `N <= len(s)`
      If this rule is violated, a runtime PANIC occurs.


    example
      ```go
        s := []int{0, 1, 2, 3}

        // OK: We are converting a subslice of `s` (which is `[]int{1, 2, 3}`)
        // to a [3]int. Lengths match. This is a DATA COPY.
        a := [3]int(s[1:])

        // Modifying the original slice does not affect the copied array.
        s[2] = 99
        fmt.Println("Original Slice:", s) // [0 1 99 3]
        fmt.Println("Copied Array:", a)   // [1 2 3] (Unaffected)

        // PANIC: The subslice `s[:2]` has length 2, which is less than the
        // target array length of 3.
        // _ = [3]int(s[:2])

      ```





-- The Built-in `copy` Function
--
  `copy(dst, src)` is the idiomatic way to copy elements from a source slice `src`
  to a destination slice `dst`.
  - It returns the number of elements copied, which is `min(len(dst), len(src))`.
  - The two slices can overlap in memory.
  - The element types of `dst` and `src` must be identical.

  A special case allows copying bytes from a `string` to a `[]byte`.


  example
    ```go
      dest := []int{1, 2, 3}
      src := []int{50, 60, 70, 80, 90}


      // Copy from src to dest. `dest` only has room for 3 elements.
      n := copy(dest, src)
      fmt.Printf("Copied %d elements. dest is now: %v\n", n, dest) // 3, [50 60 70]


      // Copy from an array to a slice.
      a := [4]int{}
      n = copy(a[:], src) // `a[:]` creates a slice that points to the array `a`.
      fmt.Printf("Copied %d elements. array 'a' is now: %v\n", n, a) // 4, [50 60 70 80]

    ```
  




-- Iteration with `for-range`
-- 
  `for-range` is the idiomatic way to iterate over containers.


  -- variants
  --
    ```go
      aContainer := []string{"a", "b", "c"}

      // 1. Get both index and value.
      for index, value := range aContainer { _, _ = index, value }

      // 2. Get only the index.
      for index := range aContainer { _ = index }

      // 3. Get only the value (ignore index with `_`).
      for _, value := range aContainer { _ = value }

      // 4. Ignore both (useful for just counting or for channels).
      for range aContainer { }

    ```


  -- Iteration Semantics (The "Copy" Gotcha)
  --
    When you start a `for-range` loop, two things are copied:
    1. The Container Header: The loop operates on a shallow copy of the container.
    2. The Iteration Variables: In each step, the key/value are copied into `i` and `p`.


    example
      ```go
        type Person struct { name string; age int }


        func rangeCopyGotcha() {
          // -- Case: Ranging over an ARRAY 
          // The loop works on a complete copy of the array.

          personsArray := [2]Person{{"Alice", 28}, {"Bob", 25}}
          for i, p := range personsArray {
            _ = i
            // `p` is a copy of the element FROM the copied array.
            // Modifying `p` has no effect on the original `personsArray`.
            p.age = 99
            // Modifying `personsArray` directly is not visible to this loop.
            personsArray[1].name = "Jack"
          }

          fmt.Println("Final Array:", personsArray) // [{Alice 28} {Jack 25}]



          // -- Case: Ranging over a SLICE 
          // The loop works on a copy of the slice HEADER. Both headers point to the SAME underlying data.

          personsSlice := []Person{{"Alice", 28}, {"Bob", 25}}
          for i, p := range personsSlice {
            _ = i
            // `p` is still a copy of the element. Modifying it has no effect.
            p.age = 99
            // HOWEVER, modifying the slice directly IS visible because the underlying data is shared.
            if i == 0 { personsSlice[1].name = "Jack" }
          }
          fmt.Println("Final Slice:", personsSlice) // [{Alice 28} {Jack 25}]

      ```

      -- Performance
      --
        example
          ```go
            // Because the iteration variable is a copy, avoid ranging over large struct arrays/slices
            // with the value variable if you don't need it. It creates unnecessary memory copies.

            type Buffer [1024]byte


            // INEFFICIENT: `buf` is a 1KB copy made on every iteration.
            func fa(buffers []Buffer) { for _, buf := range buffers { _ = buf } }

            // EFFICIENT: No copy is made. We access the original via index.
            func fb(buffers []Buffer) { for i := range buffers { _ = buffers[i] } }

          ```

  
  -- The Loop Variable Scope Change (Go 1.22+)
  --
    Go <= 1.21: The `i` and `n` variables were declared ONCE and reused for every iteration.
    Go >= 1.22: A NEW `i` and `n` variable is created for each iteration.


    example
      ```go
        // -- Example with Closures 

        for i, n := range []int{0, 1, 2} {
          defer func() {
            fmt.Println(i, n)
          }()
        }
        // Go 1.21 Output: 2 2 | 2 2 | 2 2 (All closures see the final state of the shared variables)
        // Go 1.22 Output: 2 2 | 1 1 | 0 0 (Each closure captures its own distinct variables)


        // -- Example with Pointers 
        var m = make(map[*int]bool)
        for i, n := range []int{1, 2, 3} {
          m[&i] = true
          m[&n] = true
        }
        fmt.Println(len(m))
        // Go 1.21 Output: 2 (Only 2 unique memory addresses were ever created)
        // Go 1.22 Output: 6 (6 unique memory addresses were created)

      ```





-- Using Array Pointers as Arrays
--
  In most contexts, Go lets you use a pointer to an array (`*[N]T`) as if it
  were the array itself. This is a powerful feature for performance optimization.


  -- Performance Benefit
  --
    `for-range` loops operate on a copy of the ranged value
      - `for _, v := range largeArray`: Copies the ENTIRE large array. Very expensive.
      - `for _, v := range &largeArray`: Copies only the 8-byte pointer. Extremely cheap.
      - `for _, v := range largeArray[:]`: Copies only the 24-byte slice header. Also cheap.


    example
      ```go
        var a [100]int

        // EFFICIENT: Ranging over a pointer copies only the pointer.
        for i, n := range &a {
          _, _ = i, n
        }

        // EFFICIENT: Ranging over a slice derived from the array copies only the slice header.
        for i, n := range a[:] {
          _, _ = i, n

      ```





  -- Interacting with Array Pointers
  --
    Indexing: You can index through an array pointer as if it were the array.
    Slicing: You can derive slices from an array pointer.
    `len` and `cap`: These built-in functions work on array pointers.


    Rule:   
      Any operation (indexing, slicing) on a `nil` array pointer
      will cause a runtime PANIC. `len` and `cap` are the only safe exceptions.

    
    ```go
        a := [5]int{2, 3, 5, 7, 11}
        p := &a


        // -- Indexing
        p[0], p[1] = 17, 19 // OK: Implicitly dereferences and modifies the array.
        fmt.Println(a)      // [17 19 5 7 11]


        // -- Slicing 
        s := p[1:3]      // OK: Derives a slice from the array pointer.
        fmt.Println(s)   // [19 5]


        // -- `len` and `cap` on a nil pointer (SAFE) 
        var nilPtr *[5]int
        fmt.Println(len(nilPtr), cap(nilPtr)) // 5 5


        // -- Operations that PANIC on a `nil` pointer 
        // p = nil
        // _ = p[0]   // PANIC
        // _ = p[:]   // PANIC


        // -- The `range` Panic Exception 
        // Ranging over a nil array pointer is safe ONLY if you do not access the element value.
        for i := range nilPtr { // OK
          fmt.Println(i)
        }

        // for i, n := range nilPtr { ... } // PANIC on first iteration!

    ```





-- The `clear` Built-in Function (Go 1.21+)
--
  The `clear` function provides a standardized way to zero out the contents of
  a map or slice.

  - `clear(s []T)`: Resets all elements from index 0 to `len(s)-1` to their zero value.
  - `clear(m map[K]T)`: Deletes all entries from the map, making it empty.


  ```go
    s := []int{1, 2, 3}
    clear(s)
    fmt.Println(s) // [0 0 0]


    a := [4]int{5, 6, 7, 8}
    clear(a[1:3])
    fmt.Println(a) // [5 0 0 8]


    m := map[float64]float64{}
    x := 0.0
    m[x] = x
    x /= x // x is NaN now
    m[x] = x

    fmt.Println(len(m)) // 2

    for k := range m {
      delete(m, k)
    }

    fmt.Println(len(m)) // 1

    clear(m)

    fmt.Println(len(m)) // 0

  ```





-- compile-Time Evaluation of `len` and `cap`
--
  If the argument to `len()` or `cap()` is an array or an array pointer, the
  compiler evaluates the function at compile time. The result is an `int`-typed
  constant, which means it can be used to declare other constants.


  ```go
    var a [5]int
    var p *[7]string


    const N = len(a) // OK
    const M = cap(p) // OK
    // const S = len(make([]int, 10)) // ERROR: `make` is a runtime call, not a constant.

  ```





-- Concurrency Safety
--
  Rule:
    All built-in container operations are NOT synchronized internally.
      - It is SAFE for multiple goroutines to read from the same container concurrently.
      - It is UNSAFE for any goroutine to write to a container while any other
        goroutine is reading from or writing to it. This will cause a data race.

    You MUST use synchronization primitives (like channels or `sync.Mutex`) to
    coordinate access to shared containers.





-- Go's Design Philosophy (What's Missing)
--
  Go intentionally keeps its set of built-in functions small and composable.

  - No `clone`, `delete`, or `insert` for Slices: These operations must be
    composed using `append`, `copy`, and slicing syntax. This makes the operations
    explicit and avoids hidden memory allocations.

  - No Built-in `Set` Type: A set can be easily and efficiently simulated
    using a map with a zero-sized `struct{}` as the value type.
    `mySet := make(map[string]struct{})`
    Using `struct{}` is memory efficient, as it occupies zero bytes of storage.

  - No Individual Slice Property Modification: You cannot modify `len` or
    `cap` individually. A slice value is immutable and must be overwritten by
    re-slicing or `append`. (While possible with the `reflect` package, it is
    unsafe, slow, and strongly discourage).

    example
      ```go
        func main() {
          s := make([]int, 2, 6)
          fmt.Println(len(s), cap(s)) // 2 6

          reflect.ValueOf(&s).Elem().SetLen(3)
          fmt.Println(len(s), cap(s)) // 3 6

          reflect.ValueOf(&s).Elem().SetCap(5)
          fmt.Println(len(s), cap(s)) // 3 5
        }

      ```
      The secon argument passed to the reflect.SetLen function must not be larger than the
      current capacity of the argument slice s. The second argument passed to the
      reflect.SetCap function must not be smaller than the current length of the argument slice s
      and larger than the current capacity of the argument slice s. Otherwise, a panic will occur.
