-- Memory Leaking Scenarios - Substrings
--
  While Go features automatic Garbage Collection (GC), certain coding patterns
  can lead to "Kind-of" memory leaks. These occur when a small, long-lived
  object references a large, otherwise unused memory block, preventing the GC
  from reclaiming that large block.


  -- The Substring Trap: Shared Memory
  --
    In Go, strings are immutable. For efficiency, the standard Go compiler
    implements substrings by sharing the same underlying byte array as the
    base string.


    -- The Mechanism
    --
      A string is a header: { Data *byte, Len int }.
      When you take a substring `s[low:high]`, Go creates a new header pointing
      to the same memory address (plus an offset), with a smaller length.


    -- The Leak
    --
      If you take a tiny substring of a massive 1GB string and store that tiny
      substring in a global variable, the ENTIRE 1GB byte array remains "reachable"
      and cannot be garbage collected.


    example
      ```go
        var s0 string // Long-lived package-level variable

        // f demonstrates the leak.
        func f(s1 string) {
          // s0 now points to the same 1MB block as s1.
          // Even after f() returns and s1 is out of scope, the 1MB block stays in RAM.
          s0 = s1[:50]
        }


        //
        // Visualizing the Leak:
        // [ s1 Header ] ----> [ 1,000,000 bytes of data in one Memory Block ]
        //                        ^
        // [ s0 Header ] ---------|  (Only uses 50 bytes, but holds the whole block)
        //

      ```


  -- Solutions: Breaking the Reference
  --
    ```go
      // To avoid this leak, we must "Clone" the substring into a brand-new,
      // smaller memory block.


      // -- Method A: The Byte Slice Hop (Legacy / Manual) --

      func fixWithConversion(s1 string) {
        // 1. Convert to []byte (Copy 1)
        // 2. Convert back to string (Copy 2)
        // Pros: Works in all Go versions.
        // Cons: Inefficient (performs two copies of the 50 bytes).
        s0 = string([]byte(s1[:50]))
      }


      // -- Method B: The Concatenation Trick (Clever but Fragile) --

      func fixWithOptimization(s1 string) {
        // Adding a space and then slicing it off triggers a compiler optimization
        // that creates a new underlying array.
        // Cons: Relies on specific compiler behavior; may break or not exist in
        // other compilers.
        s0 = (" " + s1[:50])[1:]
      }


      // -- Method C: Strings Builder (Pre-1.18 Efficient) --

      func fixWithBuilder(s1 string) {
        var b strings.Builder
        b.Grow(50)
        b.WriteString(s1[:50])
        s0 = b.String() // b.String() performs a clean allocation
      }


      // -- Method D: strings.Repeat (The Go 1.12+ Hack) --

      func fixWithRepeat(s1 string) {
        // Repeating a string 1 time effectively clones it.
        // In Go 1.12+, this uses strings.Builder internally to be efficient.
        s0 = strings.Repeat(s1[:50], 1)
      }


      // -- Method E: strings.Clone (The Modern Standard - Go 1.18+)

      // This is the absolute best way to handle this in modern Go. It is clear,
      // efficient, and explicitly designed to solve this exact memory leak.

      func fixWithClone(s1 string) {
        // Allocates exactly 50 bytes, copies the data, and returns a new string.
        // The original 1MB block can now be garbage collected.
        s0 = strings.Clone(s1[:50])
      }

    ```





-- Memory Leaking Scenarios - Slices & Pointers
--
  Go's Garbage Collector (GC) tracks memory based on reachability. A memory
  block can only be reclaimed if no "live" pointers refer to it.


  Because slices are headers pointing to an underlying backing array, it is
  easy to accidentally keep large amounts of memory "alive" even when only
  a tiny fraction is actually being used.


  -- Subslice Memory Leaks (Backing Array Retention)
  --
    Like substrings, a subslice `s[low:high]` does not create a new data block.
    It creates a new header that points to a specific segment of the ORIGINAL
    backing array.


    -- The Leak
    --
      If you have a slice with 1 million elements and create a subslice of just
      10 elements, the ENTIRE 1-million-element array remains in memory as
      long as that 10-element subslice is alive.


  example
    ```go
      var s0 []int // Long-lived global slice


      // LEAKY: The entire array of s1 is kept alive by s0.
      func gLeaky(s1 []int) {
        if len(s1) > 30 {
          s0 = s1[len(s1)-30:] // s0 holds a reference to s1's huge backing array
        }
      }


      // FIXED: Copy the data to a brand new, smaller backing array.
      func gFixed(s1 []int) {
        if len(s1) > 30 {
          // 1. Allocate a new slice of the exact size needed
          s0 = make([]int, 30)
          // 2. Copy only the relevant data
          copy(s0, s1[len(s1)-30:])
          // Now s1's large backing array can be garbage collected.
        }
      }

    ```


  -- Pointers in "Lost" Elements (Reference Retention)
  --
    This is a more subtle leak. Even if you "remove" elements from a slice by
    re-slicing, the "lost" elements in the underlying backing array still
    hold their values.


    -- The Leak 
    --
      If the slice elements are pointers (or interfaces, channels, maps), those
      pointers in the "invisible" part of the backing array still count as
      active references to heap objects.

      
      example
        ```go
          // LEAKY: The objects pointed to by s[0] and s[3] are still reachable.
          func hLeaky() []*int {
            s := []*int{new(int), new(int), new(int), new(int)}
            // s[1:3:3] creates a slice of length 2 and capacity 2.
            // Index 0 and 3 are no longer "in" the slice, but they still exist
            // in the backing array, holding pointers to the heap.
            return s[1:3:3]
          }


          // FIXED: Manually clear the references (nil them out).
          func hFixed() []*int {
            s := []*int{new(int), new(int), new(int), new(int)}

            // 1. Explicitly break the references to allow GC to collect the objects.
            s[0], s[len(s)-1] = nil, nil

            // 2. Return the subslice safely.
            return s[1:3:3]
          }

        ```


  -- Practical Pattern: Slice Element Deletion
  --
    When deleting an element from a slice of pointers, you must always "nil out"
    the slot that is being shifted/removed to avoid memory leaks.

    
    example
      ```go
        func deleteElement(s []*int, i int) []*int {
          // Standard deletion logic: shift elements left
          copy(s[i:], s[i+1:])

          // CRITICAL: The last element of the original slice is now a duplicate
          // of the new last element, but it still holds a reference.
          s[len(s)-1] = nil

          return s[:len(s)-1]
        }

      ```





-- Real Memory Leaks (Goroutines and Tickers)
--
  Unlike "Kind-of" leaks (where memory is held by shared backing arrays), 
  "Real" memory leaks in Go involve resources that the runtime cannot reclaim 
  until the program terminates. These are primarily caused by abandoned 
  goroutines and active tickers.


  -- Real Memory Leaks: Hanging Goroutines
  --
    A "Hanging Goroutine" is a goroutine that remains in a blocking state forever.
    Because the Go runtime cannot determine if a blocked goroutine will ever
    be unblocked, it must keep the goroutine's entire stack (minimum 2KB) and 
    all variables it references alive in memory.


  -- Why the Runtime doesn't "kill" them 
  --
    1. Indeterminacy: The runtime doesn't know if a channel will eventually 
       receive data or if a Mutex will eventually be unlocked.
    2. Deliberate Blocking: Idioms like `select{}` in the main goroutine are 
       valid ways to keep a process alive while background workers do the work.


  -- Common Causes
  --
    - Receiving from a channel that is never closed and never sent to.
    - Sending to an unbuffered channel that no one is reading from.
    - Deadlocks (circular dependencies between goroutines).


  example
    ```go
      func hangingGoroutineDemo() {
        // Let's track the number of goroutines
        fmt.Printf("Starting goroutines: %d\n", runtime.NumGoroutine())

        // This function spawns a goroutine that will hang forever.
        leak := func() {
          ch := make(chan int)
          go func() {
            // This goroutine will block here forever because 
            // the 'ch' variable is lost to the outer scope 
            // and no one will ever send a value to it.
            val := <-ch 
            fmt.Println(val)
          }()
        }

        leak()

        // Give the scheduler a moment to catch up
        time.Sleep(time.Millisecond * 10)
        
        fmt.Printf("Goroutines after leak: %d\n", runtime.NumGoroutine())
      }

    ```


  -- Real Memory Leaks: Unstopped Tickers
  --
    There is a critical difference between `time.Timer` and `time.Ticker`.


    -- time.Timer 
    --
      A `Timer` is a one-shot event. Once it fires (or is stopped), it can be 
      garbage collected relatively easily. (Note: Go 1.23 improved this further).


    -- time.Ticker 
    --
      A `Ticker` is a recurring event. It tells the Go runtime to send a signal 
      every N duration. Even if the variable holding the `Ticker` goes out of 
      scope, the underlying runtime resource continues to fire until 
      `Stop()` is called. 


    -- The Rule 
    --
      You must ALWAYS call `Stop()` on a `time.Ticker` when it is no longer 
      needed, typically via `defer ticker.Stop()`.


    example
      ```go
        func tickerLeakDemo() {
          // DANGER: If we create a ticker in a loop or a short-lived function
          // and don't stop it, we leak runtime resources.
          
          doWork := func() {
            ticker := time.NewTicker(time.Millisecond * 500)
            
            // In a real app, we might perform logic here...
            fmt.Println("Ticker created, doing one-off work...")
            
            // If we return now without calling ticker.Stop(), 
            // the ticker continues to live in the runtime's internal 
            // scheduler, consuming CPU and memory.
            // defer ticker.Stop() // <--- THE FIX
          }

          doWork()
          fmt.Println("Function exited, but the ticker is still 'alive' in the runtime.")
        }

      ```





-- Memory Leaking via Improper Finalizers
--
  Go provides `runtime.SetFinalizer(obj, func)`, which schedules a function to 
  run when an object is about to be garbage collected. While intended for 
  cleaning up non-Go resources (like file descriptors from a system level), 
  it is easily misused, leading to permanent memory leaks.


  -- The Cyclic Reference Trap
  --
    The Go Garbage Collector (GC) uses a "mark-and-sweep" algorithm. 
    A "Finalizer" adds a layer of complexity: the GC must first determine an 
    object is unreachable, then run the finalizer, and only then reclaim 
    the memory in a subsequent GC cycle.


    -- The Conflict 
    --
      If an object with a finalizer is part of a "Cyclic Reference Group" (e.g., 
      Object A points to B, and B points to A), the GC may never be able to 
      prove that the group is safe to collect. 

      Why? Because to run the finalizer for A, the GC must ensure the memory A 
      points to (B) remains valid. But B points back to A. This inter-dependency 
      can "pin" the entire group in memory forever.


  -- Code Demonstration: The Finalizer Leak
  --
    ```go
      type Node struct {
        data [1 << 20]int // Large payload (8MB on 64-bit) to make the leak obvious
        next *Node
      }


      func createCycleWithFinalizer() {
        var x, y Node


        // 1. Define a finalizer.
        finalizer := func(n *Node) {
          fmt.Println("Finalizer executed for node at", n)
        }


        // 2. Set the finalizer on 'x'. 
        // This forces 'x' to escape to the heap.
        runtime.SetFinalizer(&x, finalizer)


        // 3. Create a CYCLIC REFERENCE.
        // x points to y, and y points to x.
        // This makes 'y' also escape to the heap.
        x.next = &y
        y.next = &x


        // 4. Function exits. In standard circumstances, x and y should be 
        // eligible for GC. However, because of the finalizer + the cycle, 
        // they are NOT guaranteed to be collected.
        fmt.Println("Function createCycleWithFinalizer has exited.")
      }


      func main() {
        createCycleWithFinalizer()

        // Force multiple GC cycles to attempt collection
        for i := 0; i < 3; i++ {
          fmt.Printf("GC Cycle %d...\n", i+1)
          runtime.GC()
          time.Sleep(time.Second)
        }
        
        fmt.Println("Program terminating. If 'Finalizer executed' didn't print, memory leaked.")
      }

    ```

      -- EXPECTED OUTPUT:
      --
        Function createCycleWithFinalizer has exited.
        GC Cycle 1...
        GC Cycle 2...
        GC Cycle 3...
        Program terminating. If 'Finalizer executed' didn't print, memory leaked.

      Note: In most runs of this specific logic, the finalizer will NEVER be called, 
      and the 16MB of data remains leaked until the program ends.


  -- Finalizers are NOT Destructors
  --
    One of the most common mistakes is treating finalizers as C++-style 
    destructors. They are not.

    -- Why Finalizers are Unreliable 
    --
      1. No Execution Guarantee: Go does not guarantee that a finalizer will 
         ever run (e.g., if the program exits before the next GC cycle).
      2. Timing Uncertainty: You cannot predict when a finalizer will run. 
         This makes them useless for time-sensitive cleanup like releasing 
         database locks.
      3. Performance Cost: They add significant overhead to the GC process.


  -- The Idiomatic Alternative: Explicit Cleanup
  --
    Instead of relying on the GC to clean up resources via finalizers, use the 
    "Closer" pattern. 


    example
      ```go
        type Resource struct { ... }

        func (r *Resource) Close() error {
            // Perform manual cleanup here (close files, sockets, etc.)
            return nil
        }

        func main() {
            res := OpenResource()
            defer res.Close() // Explicitly manage the lifecycle
        }

      ```
