-- Value Sizes and Copy Costs
--
  In Go, every time you assign a variable, pass an argument to a function, 
  or send a value into a channel, a "copy" occurs. Understanding the "size" 
  of a type is critical because it directly correlates to the CPU and memory 
  cost of these operations.





-- The "Direct Part" Rule
--
  The size of a value refers strictly to its "Direct Part"—the memory 
  occupied by the variable itself. 
  

  Many Go types (Slices, Maps, Channels, Strings, Interfaces) have an 
  "Indirect Part" (data stored elsewhere on the heap). This indirect part 
  does NOT contribute to the value's size and is NOT duplicated during a 
  standard copy.


  -- Native Words and Architecture
  --
    Many sizes in Go are measured in "Words".
    - 32-bit Architecture: 1 Word = 4 bytes.
    - 64-bit Architecture: 1 Word = 8 bytes.


  -- Value Size Table (Standard Go Compiler v1.25.n)
  --
    +-----------------------+-------------------------+----------------------------+
    | Type Kind             | Value Size              | Internal Structure         |
    +-----------------------+-------------------------+----------------------------+
    | bool                  | 1 byte                  | Logical value              |
    | int8, uint8 (byte)    | 1 byte                  | 8-bit integer              |
    | int16, uint16         | 2 bytes                 | 16-bit integer             |
    | int32 (rune), float32 | 4 bytes                 | 32-bit value               |
    | int64, float64        | 8 bytes                 | 64-bit value               |
    | complex128            | 16 bytes                | Two float64s               |
    |                       |                         |                            |
    | int, uint, uintptr    | 1 Word                  | Platform-dependent         |
    | Pointer (*T)          | 1 Word                  | Memory address             |
    | Map, Chan, Func       | 1 Word                  | Pointer to runtime object  |
    |                       |                         |                            |
    | String                | 2 Words                 | {Pointer, Length}          |
    | Interface             | 2 Words                 | {Type/itab, Data Pointer}  |
    | Slice                 | 3 Words                 | {Pointer, Len, Cap}        |
    |                       |                         |                            |
    | Array                 | (ElemSize * Length)     | Contiguous block           |
    | Struct                | (FieldSum + Padding)    | Aligned fields             |
    +-----------------------+-------------------------+----------------------------+


  -- Copying Costs of Composite Types
  --
    -- Structs: The Padding Factor 
    --
      The size of a struct is the sum of its fields PLUS padding bytes used for 
      memory alignment. Fields are aligned to their own size (e.g., an int64 
      must start at an address divisible by 8).


    -- Arrays: The Value-Type Burden 
    --
      Unlike slices, arrays are full values. If you have `[1000]int`, copying it 
      moves 8000 bytes (on 64-bit). This is why passing large arrays by pointer 
      or using slices is preferred.


  -- Code Demonstration: Verifying Sizes
  --
    func main() {
      // -- Headers (Headers have fixed sizes) 

      var s string = "This is a very long string, but its header size is fixed."
      var slc []int = make([]int, 1000) // 1000 elements, but header size is fixed.
      var m map[int]int = make(map[int]int, 1000)


      fmt.Println("--- Header Sizes (64-bit) ---")
      fmt.Printf("String size:    %d bytes\n", unsafe.Sizeof(s))   // Output: 16 (2 words)
      fmt.Printf("Slice size:     %d bytes\n", unsafe.Sizeof(slc)) // Output: 24 (3 words)
      fmt.Printf("Map size:       %d bytes\n", unsafe.Sizeof(m))   // Output: 8  (1 word)


      // -- Arrays (Size scales with length) 

      var a1 [10]int64
      var a2 [100]int64
      fmt.Println("\n--- Array Sizes ---")
      fmt.Printf("[10]int64 size:  %d bytes\n", unsafe.Sizeof(a1)) // Output: 80
      fmt.Printf("[100]int64 size: %d bytes\n", unsafe.Sizeof(a2)) // Output: 800


      // -- Structs and Padding 

      type Optimized struct {
        a int64 // 8 bytes
        b int32 // 4 bytes
        c bool  // 1 byte
      } // Total: 13 -> Padded to 16


      type Unoptimized struct {
        c bool  // 1 byte
        // 7 bytes padding
        a int64 // 8 bytes
        b int32 // 4 bytes
        // 4 bytes padding to make total multiple of 8
      } // Total: 24


      fmt.Println("\n--- Struct Padding ---")
      fmt.Printf("Optimized size:   %d bytes\n", unsafe.Sizeof(Optimized{}))   // Output: 16
      fmt.Printf("Unoptimized size: %d bytes\n", unsafe.Sizeof(Unoptimized{})) // Output: 24
    }

    -- expected output (on 64-bit architecture):
    --
      -- Header Sizes (64-bit) 
      --
        String size:    16 bytes
        Slice size:     24 bytes
        Map size:       8 bytes


      -- Array Sizes 
      --
        [10]int64 size:  80 bytes
        [100]int64 size: 800 bytes


      -- Struct Padding 
      --
        Optimized size:   16 bytes
        Unoptimized size: 24 bytes





-- Strategic Value Copying & Performance
--
  In Go, value copying is ubiquitous. To write high-performance code, one must 
  understand not just the size of data, but the architectural and runtime 
  costs associated with moving that data across function boundaries, channels, 
  and loops.


  -- Defining "Small" vs. "Large" Values
  --
    While the cost of copying is generally proportional to size, modern CPUs 
    and compilers have "sweet spots" where copying is extremely efficient.


    -- Rule of Thumb: The "Small Value"
    --
      A value is considered "small" if it meets these criteria:
      1. Size: Not larger than 4 native words (32 bytes on 64-bit systems).
      2. Complexity: Structs with fewer than 5 fields.


    -- Small-Size Types (Always Cheap to Copy) 
    --
      Most types in Go are small because they are either basic types or small 
      descriptors (headers):
      - Basic types (int, bool, float, etc.)
      - Pointers
      - Strings and Interfaces (2 words)
      - Slices (3 words)
      - Maps, Channels, and Functions (1 word/pointer)


    -- Large-Size Types (Potentially Expensive) 
    --
      These are primarily large structs and arrays. 
      Example: `[1024]int64` (8KB) or `struct { a, b, c, d, e [100]int }`.


  -- Strategic Use of Pointers
  --
    To avoid the CPU overhead of copying large data blocks, we use pointers.


    -- Parameters and Receivers
    --
      If a struct is large, pass it as `*T`. This reduces the copy cost to a 
      single native word (8 bytes), regardless of the struct's size.


    -- Channel Elements 
    --
      Sending a large struct through a channel triggers two copies (Sender -> Buffer 
      -> Receiver). Using `chan *LargeStruct` is significantly faster.


    -- The GC Trade-off (Pressure) 
    --
      Crucial Insight: Pointers are not a "free lunch." 
      While pointers save CPU time during copying, a high density of pointers 
      increases "GC Pressure." The Garbage Collector must scan every pointer to 
      determine if memory is still in use. 
      
      Best Practice: 
      - Use values for small data to keep them on the stack and invisible to the GC.
      - Use pointers for large data to save copy costs.
      - Seldom use pointers to "header types" (e.g., `*[]int` or `*map[K]V`) because 
        those types are already small and efficient to copy as values.





-- The Iterator Performance Trap
--
  A common performance leak in Go occurs when using the `for-range` loop with 
  two iteration variables on a collection of large elements.


  -- The Problem 
  --
    `for i, v := range largeElements`
    In every single iteration, Go COPIES the element into the variable `v`. 
    If the element is a large struct, this loop will be significantly slower 
    than an index-only loop.


  example
    ```go
      // S is a large-ish struct (12 * 8 bytes = 96 bytes)
      type S [12]int64

      var sX = make([]S, 1000)
      var sumX int64


      // -- Benchmark Comparison 

      // Approach 1: Classic C-style Loop
      // Very efficient: No element copying.
      func Benchmark_Loop(b *testing.B) {
        for i := 0; i < b.N; i++ {
          sumX = 0
          for j := 0; j < len(sX); j++ {
            sumX += sX[j][0]
          }
        }
      }


      // Approach 2: Range with One Variable (Index only)
      // Very efficient: No element copying.
      func Benchmark_Range_OneIterVar(b *testing.B) {
        for i := 0; i < b.N; i++ {
          sumX = 0
          for j := range sX {
            sumX += sX[j][0]
          }
        }
      }


      // Approach 3: Range with Two Variables (Index and Value)
      // INEFFICIENT: The 96-byte struct is copied to 'v' every iteration.
      func Benchmark_Range_TwoIterVar(b *testing.B) {
        for i := 0; i < b.N; i++ {
          sumX = 0
          for _, v := range sX {
            sumX += v[0]
          }
        }
      }

    ```

  -- representative benchmark results (Go 1.25.n on 64-bit):
  --
    Benchmark_Loop-4             424342    2708 ns/op
    Benchmark_Range_OneIterVar-4 407905    2808 ns/op
    Benchmark_Range_TwoIterVar-4 214860    3915 ns/op

  -- observation:
  --
    The Two-Iteration-Variable form is ~40% slower than the index-based forms
    due to the repeated copying of the 96-byte struct.
