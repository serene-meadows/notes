-- The Core Concept: PackagePath + Name
--
  In Go, the true identity of a non-exported (lowercase) identifier is not just
  its name. The compiler internally tracks it as a combination of its package
  path and its name: `PackagePath.name`.

  -- Why does this matter?
  --
    This means that if two different packages both define a private field named
    `data`, they are treated as two entirely distinct entities by the compiler
    (`pkgA.data` vs. `pkgB.data`). They will never collide or override each other.


  This is a critical safety feature when utilizing struct embedding across
  different packages. It allows package authors to use common internal names
  (like `id`, `state`, or `mu`) without worrying about breaking code that embeds
  their structs.





-- Multi-Package Demonstration
--
  The following pseudo-code represents three separate files in three different
  packages to illustrate how package-scoped identity prevents collisions.


  example
    ```go
      --  FILE 1: internal/alpha/provider.go
      --
        package alpha

        type Provider struct {
          // Internal Identity: "alpha.data"
          data int 
        }

        // Exported method to allow external mutation of the private field.
        func (p *Provider) SetAlphaData(v int) { 
          p.data = v 
        }


      -- FILE 2: internal/beta/consumer.go
      --
        package beta

        type Consumer struct {
          // Internal Identity: "beta.data"
          // Because of the package prefix, this has NO relationship to alpha's `data`.
          data string 
        }

        // Exported method to allow external mutation of the private field.
        func (c *Consumer) SetBetaData(v string) { 
          c.data = v 
        }


      -- FILE 3: main.go (The Embedder)
      -- 
        package main


        import (
          "fmt"
          // "alpha"
          // "beta"
        )


        // Assume `alpha` and `beta` packages are imported here.

        // System embeds both types. It inherits methods from both, and internally
        // holds both unexported `data` fields.
        type System struct {
          alpha.Provider // Contains `alpha.data`
          beta.Consumer  // Contains `beta.data`
        }


        func main() {
          s := System{}

          -- CASE 1: Interacting with Unexported Fields
          --
            s.data = 10 // COMPILE ERROR: s.data undefined (cannot refer to unexported field)


          // You cannot access `data` directly from `main` because it is unexported.
          // However, the embedded types still hold their distinct state safely.
          // We interact with them via their exported methods:
          s.SetAlphaData(100)
          s.SetBetaData("Active")


          -- CASE 2: The "Ambiguous Selector" Contrast (If fields were Exported)
          --
            	// Imagine if the authors of `alpha` and `beta` had capitalized their fields 
              // (making them `Data` instead of `data`).
              
              // Exported fields do NOT include the package path in their identity. 
              // Their identity is simply `"Data"`.
              
              // Because `System` embeds both structs, both `Data` fields would be "promoted"
              // to the top level of `System`. 
              
              // s.Data = "test" // COMPILE ERROR: ambiguous selector s.Data
              
              // The compiler would throw an error because it wouldn't know if you meant
              // `s.Provider.Data` or `s.Consumer.Data`.
              
              // By keeping internal state unexported (`data`), package authors protect 
              // consumers from these ambiguous selector collisions.

    ```
