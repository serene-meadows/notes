-- Methods and Receivers
--
   While Go does not have "classes" or "objects" in the traditional Object-Oriented
   Programming (OOP) sense, it heavily supports OOP paradigms.
   The "Method" is the primary feature that enables this behavior.





-- What is a Method? (The Anatomy)
--
  A method is simply a function that has a special "receiver" argument declared
  between the `func` keyword and the method name.

  When a method is declared for a type, we say that type "owns" the method.

  func   (r ReceiverType)   MethodName(args...)   (results...) { ... }
    ▲            ▲                ▲                      ▲
  Keyword     The Receiver       Name           Standard Function Signature





-- Strict Rules for the Receiver Base Type (`T`)
--
  You can declare a method on a type `T` (Value Receiver) or `*T` (Pointer Receiver).
  However, the "Receiver Base Type" `T` MUST satisfy 4 strict conditions:

  1. Defined Type: `T` must be a custom type defined using the `type` keyword.
  2. Same Package: `T` must be defined in the EXACT SAME package as the method.
     - This is why you cannot add methods to built-in types like `int`
       or `string`, nor can you add methods to types from other packages (like `time.Time`).
  3. Not a Pointer: `T` itself cannot be a pointer type (e.g., `type Ptr *int`).
  4. Not an Interface: `T` cannot be an interface type. Interfaces define
     behavior (method signatures), they do not implement it.


  example
    ```go
      // -- Condition 1 & 2: Same Package & Defined Type
      type Age int // `Age` is a defined type in this package.

      // -- Condition 3: Not a Pointer
      type AgePtr *int
      // func (ap AgePtr) DoSomething() {} // COMPILE ERROR: invalid receiver type

    ```





-- Value Receivers vs. Pointer Receivers
  - Value Receiver (`T`): The method receives a COPY of the value.
    Modifications inside the method DO NOT affect the original caller.
  - Pointer Receiver (`*T`): The method receives a COPY OF THE POINTER.
    Modifications inside the method DO affect the original caller's underlying data.


  example
    ```go
      type Book struct {
        pages int
      }


      // Value Receiver: Used for read-only operations, or when the struct is very small.
      func (b Book) Pages() int {
        return b.pages
      }


      // Pointer Receiver: Required when the method needs to MUTATE the receiver's state,
      // or to avoid expensive memory copies if the struct is very large.
      func (b *Book) SetPages(pages int) {
        b.pages = pages
      }

    ```





-- Non-Struct Receivers (Go's Ultimate Flexibility)
--
  Unlike Java or C++, methods in Go are not restricted to structs (classes).
  You can attach methods to ALMOST ANY defined type!

  example
    ```go
      type Age int


      // -- Methods on Basic Types
      func (age Age) LargerThan(a Age) bool {
        return age > a
      }


      func (age *Age) Increase() {
        *age++ // Dereference and increment the original value
      }


      // -- Methods on Function Types

      // This is an advanced and highly idiomatic Go pattern (e.g., `http.HandlerFunc`).
      type FilterFunc func(in int) bool


      func (ff FilterFunc) Filter(in int) bool {
        // The receiver `ff` is literally a function, so we execute it!
        return ff(in)
      }


      // -- Methods on Container Types (Maps, Slices)

      type StringSet map[string]struct{} // struct{} takes 0 bytes of memory


      func (ss StringSet) Has(key string) bool {
        _, present := ss[key]
        return present
      }


      // Maps and Slices are already pointers internally, so a Value Receiver (`StringSet`)
      // is usually sufficient to mutate the contents of the container!

      func (ss StringSet) Add(key string) {
        ss[key] = struct{}{}
      }


      func (ss StringSet) Remove(key string) {
        delete(ss, key)
      }



      func main() {
        // -- Testing the Struct
        b := Book{pages: 100}
        b.SetPages(200) // Modifies via pointer receiver
        fmt.Println("Book pages:", b.Pages()) // Output: 200


        // -- Testing the Basic Type
        var myAge Age = 25
        myAge.Increase() // Modifies via pointer receiver
        fmt.Println("Age larger than 20?", myAge.LargerThan(20)) // Output: true

        // -- Testing the Map Type
        set := make(StringSet)
        set.Add("Golang")
        fmt.Println("Set has Golang?", set.Has("Golang")) // Output: true

        // -- Testing the Function Type
        // Define an anonymous function matching the signature
        isEven := FilterFunc(func(n int) bool {
          return n%2 == 0
        })
        fmt.Println("Is 4 even?", isEven.Filter(4)) // Output: true
      }

    ```





-- Idiomatic Naming Rules
--
  1. Never use `this` or `self`: This is a strict Go convention. Because Go
     does not have classes, using terms from traditional OOP languages is highly
     discouraged.
  2. Use short abbreviations: The receiver name should be a 1 or 2 letter
     abbreviation of the type's name (e.g., `b` for `Book`, `ss` for `StringSet`).
  3. Exported vs. Unexported: Just like functions, if a method name starts
     with a Capital letter (e.g., `SetPages`), it can be called from other packages.





-- Edge Cases: Aliases and Blank Identifiers
--
  -- Type Aliases
  --
    If you create a type alias (`type MyAge = Age`), you can declare methods using
    `MyAge` as the receiver, because the compiler treats `MyAge` as perfectly
    identical to `Age`.


  -- The Blank Identifier
  --
    You can name a method with the blank identifier `_`.

      func (b Book) _() {}

    You can even have multiple methods named `_` on the same type.
    However, because it has no readable name, a blank method can NEVER be called.
    (This is sometimes used in highly advanced code generation or
    type-checking to ensure a type implements an interface secretly without
    polluting the public API).





-- Internal Method Mechanics & Method Expressions
--
  While we write methods attached to types, the Go compiler fundamentally treats
  them as regular functions under the hood. Understanding how the compiler
  translates and generates these methods is key to mastering Go's type system.


  -- The Implicit Function (Method Expressions)
  --
    For every method you declare, the Go compiler automatically generates a
    corresponding implicit, standalone function.

    How it works: The receiver parameter is simply extracted from its special
    position and placed as the very first argument in the standard parameter list.


  -- Explicit Method Declaration
  --
    ```go
      func (b Book) Pages() int {
        return b.pages
      }

    ```

  -- Compiler's Implicit Function Generation
  --
    The compiler internally generates a function named `Book.Pages`.
    Notice that the receiver `b Book` is now the first parameter.

    ```go
      func Book.Pages(b Book) int {
        return b.pages
      }

    ```



  In official Go terminology, `Book.Pages` is called a
  "Method Expression". While you cannot declare a function with a dot `.` in
  its name (it's an illegal identifier), you CAN absolutely call these
  compiler-generated functions in your code!


  -- The Mental Model of Compiler Rewrites
  --
    Once the compiler generates these implicit standalone functions, you can
    mentally model your original method declarations as simple wrappers that
    immediately delegate to the implicit functions.

    ```go
      // How we write it:
      func (b *Book) SetPages(pages int) { b.pages = pages }

      // How the compiler essentially executes it:
      func (b *Book) SetPages(pages int) {
        (*Book).SetPages(b, pages) // Calls the implicit standalone function
      }
    ```





-- The Cascade Effect: Value Receivers Spawn Pointer Receivers
--
  This is a CRITICAL rule in Go:
  When you declare a method with a Value Receiver (`T`), the compiler
  automatically and implicitly generates a matching method for the
  Pointer Receiver (`*T`).

  ```go
    // If you write:
    func (b Book) Pages() int { return b.pages }

    // The compiler silently adds this for you:
    func (b *Book) Pages() int {
      return (*b).pages // It automatically dereferences the pointer
    }

  ```


  -- Why does it do this?
  Because if you have a pointer, you can ALWAYS safely dereference it to get
  the underlying value. Therefore, it's perfectly safe for the compiler to
  create a pointer method that just dereferences itself and calls the value method.

  Note: The reverse is NOT true. Declaring a pointer receiver does NOT generate
  a value receiver. You cannot always safely take the address of a value
  (e.g., unaddressable literal values).


  -- The "3-for-1" Deal of Value Receivers
  --
    Because of the rules above, declaring a single method with a Value Receiver
    actually prompts the compiler to create THREE things under the hood:

    1. The implicit standalone function for the value:  `func Book.Pages(b Book)`
    2. The implicit method for the pointer:             `func (b *Book) Pages()`
    3. The implicit standalone function for the pointer:`func (*Book).Pages(b *Book)`


  -- Code Demonstration
  --
    ```go
      type Book struct {
        pages int
      }

      // Explicitly declared Value Receiver
      func (b Book) Pages() int {
        return b.pages
      }

      // Explicitly declared Pointer Receiver
      func (b *Book) SetPages(pages int) {
        b.pages = pages
      }


      func main() {
        var book Book

        // -- Calling Standard Methods
        // Normal method calls utilizing Go's syntactic sugar.
        book.SetPages(100)
        fmt.Println("Method Call:", book.Pages()) // Output: 100


        // -- Calling Implicit Functions (Method Expressions)
        // We bypass the syntactic sugar and call the compiler-generated
        // standalone functions directly!
        // Call (*Book).SetPages. First argument MUST be the receiver pointer.
        (*Book).SetPages(&book, 123)


        // Call Book.Pages. First argument MUST be the receiver value.
        result := Book.Pages(book)
        fmt.Println("Method Expression Call:", result) // Output: 123


        // -- Proving the "Cascade Effect"
        // We never explicitly defined a method called `Pages()` for `*Book`.
        // We only defined it for `Book`.
        // However, because of the Cascade Effect, `(*Book).Pages` exists!

        ptrToBook := &book

        // This calls the implicitly generated standalone function for the pointer!
        cascadedResult := (*Book).Pages(ptrToBook)
        fmt.Println("Cascaded Implicit Call:", cascadedResult) // Output: 123
      }

    ```





-- Method Specifications & Method Sets
--
  To understand how Go implements polymorphism (Interfaces), you must first
  understand how the compiler categorizes methods using "Method Sets".


  -- Method Specifications
  --
    A "Method Specification" is the abstract signature of a method.
    You can think of it as the function prototype MINUS the `func` keyword and
    MINUS the receiver parameter.

    For example, given this method declaration:
      `func (b Book) Pages() int { ... }`

    The Method Specification is simply:
      `Pages() int`


  -- Method Sets
  --
    Every type in Go has an associated "Method Set".
    A Method Set is an unordered collection of all the method specifications
    declared for that type (both explicitly and implicitly).


    -- The "Blank Identifier" Exception
    --
      If a method is declared with the blank identifier (e.g., `func (T) _()`),
      it is explicitly EXCLUDED from the type's Method Set.


  -- Method Set of `T` vs. `*T`
  --
    Because of the "Cascade Effect" (where declaring a Value Receiver implicitly
    generates a Pointer Receiver, but NOT vice versa), there is a strict
    mathematical subset relationship between `T` and `*T`.

    RULE: For any non-pointer, non-interface type `T`:
    The Method Set of `T` is ALWAYS a subset of the Method Set of `*T`.


    example
      ```go
        type Book struct {
          pages int
        }

        // Explicitly declared for `Book` (Value Receiver)
        func (b Book) Pages() int { return b.pages }

        // Explicitly declared for `*Book` (Pointer Receiver)
        func (b *Book) SetPages(p int) { b.pages = p }

      ```

      // +---------------------------------------+
      // |   Method Set of `Book` (Value `T`)    |
      // +---------------------------------------+
      // | 1. Pages() int                        |
      // +---------------------------------------+

      // +---------------------------------------+
      // |  Method Set of `*Book` (Pointer `*T`) |
      // +---------------------------------------+
      // | 1. Pages() int       <-- (Implicitly Generated!)
      // | 2. SetPages(p int)   <-- (Explicitly Declared)
      // +---------------------------------------+
      //
      // Conclusion: MethodSet(Book) is a subset of MethodSet(*Book).
      // This becomes incredibly important when assigning types to Interfaces





-- Package-Scoped Identity (Unexported Methods)
--
  Just like struct fields, unexported (lowercase) method names carry their
  package path as part of their identity.

  `packageA.Type.read()` is considered a completely different method
  specification than `packageB.Type.read()`, even if the text matches exactly.

  Exported methods (`Read()`) do not carry package paths and are universally
  identical across packages.





-- Types with Blank (Empty) Method Sets
--
  Not all types can have methods. The compiler strictly enforces that the
  Method Sets for the following types are ALWAYS blank (empty):

  1. Built-in Basic Types (`int`, `string`, `bool`, etc.)
  2. Defined Pointer Types (`type MyPtr *int`)
  3. Pointer Types whose base types are interfaces or other pointers (`**int`)
  4. Unnamed composite types (`[]int`, `map[string]int`, `func()`)





-- Code Demonstration: Proving the `T` vs `*T` Rule
--
  ```go
    // To prove method sets exist and behave as described, we use an
    // Interface. An interface is simply a named Method Set. If a type's method set
    // contains all the interface's specifications, it "implements" the interface.



    // PageManager is an interface requiring TWO methods.
    type PageManager interface {
      Pages() int
      SetPages(int)
    }


    type Book struct {
      pages int
    }


    // Explicitly declared for `Book` (Value Receiver)
    func (b Book) Pages() int { return b.pages }


    // Explicitly declared for `*Book` (Pointer Receiver)
    func (b *Book) SetPages(p int) { b.pages = p }


    func manipulatePages(pm PageManager) {
      pm.SetPages(999)
      fmt.Println("Pages set to:", pm.Pages())
    }


    func main() {
      var bookVal Book  = Book{pages: 10}
      var bookPtr *Book = &Book{pages: 10}


      // --  Testing the Pointer (`*T`)
      // `*Book` has both Pages() and SetPages() in its Method Set.
      // Therefore, it perfectly satisfies the PageManager interface.
      manipulatePages(bookPtr) // OK: Success!


      // --  Testing the Value (`T`)
      // `Book` ONLY has Pages() in its Method Set. It lacks SetPages().
      // Therefore, it DOES NOT satisfy the PageManager interface.

      // manipulatePages(bookVal)

      // COMPILE ERROR: cannot use bookVal (type Book) as type PageManager in argument
      // to manipulatePages: Book does not implement PageManager (SetPages method
      // has pointer receiver)

      _ = bookVal
    }

  ```





-- Method Values and Method Calls
--
  In Go, methods act like special member functions attached to types.
  However, the way Go handles method binding, dispatch, and nil receivers
  is quite unique compared to traditional Object-Oriented languages.


  -- Method Values and Selectors
  --
    When a type "owns" a method, every value of that type automatically owns an
    immutable member of a function type.

    We access this member using the selector syntax: `value.MethodName`

    When you do this WITHOUT invoking it (no parentheses `()`), you get a
    "Method Value". This is a function value that is inherently "bound" to that
    specific receiver instance.


    example
      ```go
        type Book struct {
          pages int
        }


        func (b Book) Pages() int {
          return b.pages
        }


        func (b *Book) SetPages(pages int) {
          b.pages = pages
        }


        func methodValuesDemo() {
          var book Book


          // These are Method Values. The receiver (`book` or `&book`) is already
          // bound to the function. Notice the types!


          fmt.Printf("%T \n", book.Pages)       // Output: func() int
          fmt.Printf("%T \n", (&book).SetPages) // Output: func(int)


          // Due to the "Cascade Effect", a pointer also has access to the value methods.
          fmt.Printf("%T \n", (&book).Pages)    // Output: func() int


          // We can assign a method value to a variable and call it later.
          // It remembers that it belongs to `book`.
          boundMethod := book.Pages
          fmt.Println("Bound call:", boundMethod()) // Output: 0
        }

      ```





-- Method Calls & Syntactic Sugar
--
  A method call is simply invoking that bound function value: `value.MethodName()`

  -- No `->` Operator
  --
    Unlike C or C++, Go uses the dot `.` operator for EVERYTHING. You use `.`
    regardless of whether the receiver is a direct value or a pointer.
    `(&book)->SetPages(123)` is ILLEGAL syntax in Go.


  -- The Syntactic Sugar (Pointer Methods on Values)
  --
    You can call a method with a Pointer Receiver on a standard Value
    how does `book.SetPages(123)` work if `SetPages` requires `*Book`?
      The Go compiler provides syntactic sugar. If the value is
      ADDRESSABLE, the compiler automatically injects the address-of operator (`&`).


  example
    ```go
      func methodCallsDemo() {
        var book Book // `book` is a variable, so it is addressable.


        // Explicitly calling via pointer
        (&book).SetPages(123)


        // Syntactic Sugar: The compiler translates this to `(&book).SetPages(456)`
        book.SetPages(456)


        fmt.Println(book.Pages()) // Output: 456


        // -- When the sugar FAILS
        // If the value is UNADDRESSABLE (e.g., a raw struct literal), the compiler
        // cannot take its address, so the syntactic sugar fails.


        // Book{}.SetPages(999) // COMPILE ERROR: cannot call pointer method on Book literal
      }

    ```





-- Methods on Zero Values & `nil` Receivers
--
  This is a massive paradigm shift from Java/C# (where calling a method on a
  null object instantly throws a NullPointerException).

  In Go, method dispatch DOES NOT PANIC on a `nil` receiver!
  The method is called normally, passing `nil` in as the receiver argument.
  A panic will ONLY occur if the code inside the method body illegally
  tries to dereference that nil pointer or write to a nil map.


  example
    ```go
      type StringSet map[string]struct{}

      func (ss StringSet) Has(key string) bool {
        // Reading from a nil map is perfectly safe in Go. It just returns false/zero.
        _, present := ss[key]
        return present
      }


      type Age int


      func (age *Age) IsNil() bool {
        // We can safely check if the receiver itself is nil!
        return age == nil
      }


      func (age *Age) Increase() {
        // If `age` is a nil pointer, dereferencing it here WILL cause a panic.
        *age++
      }


      func nilReceiverDemo() {
        // 1. Method Values from nil receivers (No Panic)
        _ = (StringSet(nil)).Has
        _ = ((*Age)(nil)).IsNil
        _ = ((*Age)(nil)).Increase


        // 2. Invoking methods on nil receivers (No Panic)
        fmt.Println("Nil Map Has:", (StringSet(nil)).Has("key")) // Output: false
        fmt.Println("Nil Ptr IsNil:", ((*Age)(nil)).IsNil())     // Output: true


        // 3. Invoking a method that dereferences nil (PANIC!)
        // The panic happens INSIDE `Increase()` at the line `*age++`,
        // NOT at the method call itself.


        // ((*Age)(nil)).Increase() // RUNTIME PANIC: invalid memory address or nil pointer dereference
      }

    ```





-- Receiver Arguments Are Passed by Copy
--
  Go's universal "pass-by-value" rule applies equally to method receivers.
  When you call a method, the receiver argument is copied, just like any
  other function argument. This has critical implications for whether a method
  can mutate its receiver.


  -- The Core Rule: Pass-by-Value (Shallow Copy)
  --
    - Value Receiver (`func (v T) ...`): The method receives a complete copy
      of the direct part of the value.
    - Pointer Receiver (`func (v *T) ...`): The method receives a copy of the
      pointer (the memory address).

    This distinction determines whether modifications inside the method are
    reflected in the caller's scope.


    -- Value Receivers: Modifications are Lost
    --
      When you use a value receiver, the method operates on a copy. Any changes it
      makes are made to that copy and are thrown away when the method returns.

      example
        ```go
          type Book struct {
            pages int
          }


          // `SetPages` receives a COPY of the Book struct.
          func (b Book) SetPages(pages int) {
            // This line only modifies the `pages` field of the local copy `b`.
            b.pages = pages
          }


          func valueReceiverExample() {
            var book Book // book.pages is 0


            // A copy of `book` is passed to SetPages.
            book.SetPages(123)


            // The original `book` was never touched.
            fmt.Println(book.pages) // Output: 0
          }

        ```


    -- Pointer Receivers: Modifications are Reflected
    --
      To allow a method to modify its receiver, you MUST use a pointer receiver.
      This gives the method the memory address of the original data.

      example

        ```go
          func (b *Book) SetPagesWithPointer(pages int) {
            // This follows the pointer and modifies the `pages` field of the original struct.
            b.pages = pages
          }

          func pointerReceiverExample() {
            var book Book // book.pages is 0

            // Go's syntactic sugar automatically passes `&book`.
            book.SetPagesWithPointer(123)

            // The original `book` is now modified.
            fmt.Println(book.pages) // Output: 123
          }

        ```




-- The "Slice and Map" Nuance: Direct vs. Indirect Parts
--
  Slices and maps are "reference types".
  The variable itself is just a header/pointer (the direct part).


  With a value receiver on a slice, you can modify the contents of the slice,
  but you cannot modify the slice *header* (its length, capacity, or pointer).

  example
    ```go
      type Books []Book


      // `Modify` receives a COPY of the slice header.
      // Both the original and the copy point to the SAME underlying array of books.
      func (books Books) Modify() {
        // This modification IS VISIBLE to the caller because it changes the
        // shared underlying array data.
        books[0].pages = 500


        // This modification is NOT VISIBLE. The `append` function might reallocate
        // memory and return a NEW header. This new header is assigned only to the
        // local `books` copy. The original slice header in `main` is never updated.
        books = append(books, Book{789})
      }


      func sliceReceiverExample() {
        var books = Books{{123}, {456}}
        books.Modify()


        // We see the change to page 500, but we do not see the appended book.
        fmt.Println("Slice after Modify():", books) // Output: [{500} {456}]
      }

    ```


  --  The "Append-First" Trap
  --
    ```go
      // The order of operations inside the method matters immensely!

      func (books Books) ModifyTrap() {
        // 1. `append` is called. The capacity of the `books` copy might be exceeded.
        // 2. A NEW backing array is allocated just for the `books` copy.
        // 3. The local `books` variable is updated to point to this NEW array.
        books = append(books, Book{789})


        // 4. This modification now happens on the NEW, unshared array.
        //    The original array seen by `main` is no longer being touched.
        books[0].pages = 500
      }


      func sliceTrapExample() {
        var books = Books{{123}, {456}}
        books.ModifyTrap()


        // NEITHER modification is visible to the caller!
        fmt.Println("Slice after ModifyTrap():", books) // Output: [{123} {456}]
      }

    ```

    --  The Fix: Use a Pointer Receiver
    --
      ```go
        // If you need to modify the slice header itself (e.g., via `append`), you must
        // use a pointer receiver for the slice.


        func (books *Books) ModifyWithPointer() {
          // Dereference the pointer to get the original slice header...
          // ...and assign the new header returned by `append` back to it.
          *books = append(*books, Book{789})


          // Dereference the pointer to access the slice and modify its contents.
          (*books)[0].pages = 500
        }


        func slicePointerFixExample() {
          var books = Books{{123}, {456}}


          // Go automatically passes `&books`.
          books.ModifyWithPointer()


          // Both modifications are now visible.
          fmt.Println("Slice after ModifyWithPointer():", books) // Output: [{500} {456} {789}]
        }

      ```




-- Method Value Normalization & Evaluation
--
  To understand the behavior of method values, it is crucial to know how the
  Go compiler "normalizes" method expressions and exactly when the receiver
  argument is evaluated and bound.


  -- Compile-Time Normalization
  --
    "Normalization" is the process where the compiler rewrites a method value
    expression to make all implicit address-of (`&`) and dereference (`*`)
    operations explicit. This happens entirely at compile time.


    Assume `v` is a value of type `T` and `p` is a pointer of type `*T`.


    -- Rule 1: Pointer Method on a Value 
    --
      If you access a pointer-receiver method on an addressable value `v`, the compiler
      normalizes it by taking the address of `v`.
    
      Your Code:   `v.PointerMethod`
      Normalized:  `(&v).PointerMethod`


    -- Rule 2: Value Method on a Pointer 
    --
      If you access a value-receiver method on a pointer `p`, the compiler
      normalizes it by dereferencing `p`.
    
      Your Code:   `p.ValueMethod`
      Normalized:  `(*p).ValueMethod`


    -- No Change Needed 
    --
      `v.ValueMethod` and `p.PointerMethod` are already in their normalized forms.





-- Runtime Evaluation and Binding
--
  This is the most critical concept: When a method value expression (e.g., `b.Pages`)
  is evaluated, the receiver argument is evaluated at that exact moment, and a
  copy of that evaluation result is "bound" to the resulting function value.


  The method value then carries this bound receiver with it for all future calls.
  Any subsequent changes to the original variable will have NO effect on the
  method value's bound receiver if it was a value receiver.


  example
    ```go
      type Book struct {
        pages int
      }


      // Value Receiver Method
      func (b Book) Pages() int {
        return b.pages
      }


      // Pointer Receiver Method
      func (b *Book) SetPages(p int) {
        b.pages = p
      }


      func main() {
        var b = Book{pages: 123}
        var p = &b


        // -- Case 1: Value method on a value (`b.Pages`) 
        // Normalization: `b.Pages` (already normalized)
        // Evaluation: A COPY of `b` (value: Book{pages: 123}) is made and bound to `f1`.
        var f1 = b.Pages


        // -- Case 2: Value method on a pointer (`p.Pages`) 
        // Normalization: `(*p).Pages`
        // Evaluation: `p` is dereferenced to get `b`, and a COPY of that `b`
        // (value: Book{pages: 123}) is made and bound to `f2`.
        var f2 = p.Pages


        // -- Case 3: Pointer method on a pointer (`p.SetPages`) 
        // Normalization: `p.SetPages` (already normalized)
        // Evaluation: A COPY of the pointer `p` (value: address of `b`) is made
        // and bound to `g1`.
        var g1 = p.SetPages


        // -- Case 4: Pointer method on a value (`b.SetPages`) 
        // Normalization: `(&b).SetPages`
        // Evaluation: The address of `b` is taken, and a COPY of that pointer
        // (value: address of `b`) is made and bound to `g2`.
        var g2 = b.SetPages


        // Now, we mutate the original `b` AFTER all method values have been created.
        b.pages = 789
        fmt.Println("Original `b` is now:", b) // Original `b` is now: {789}


        // Let's call the method values and see what they captured.

        // f1 and f2 are bound to COPIES of the original `b` from BEFORE the change.
        fmt.Println("f1() result:", f1()) // Output: 123
        fmt.Println("f2() result:", f2()) // Output: 123


        // g1 and g2 are bound to COPIES of the POINTER to `b`. When they are called,
        // they follow that pointer to the current state of `b`.
        // We need to see the effect of g1 and g2, so let's reset `b` and call them.
        b.pages = 100

        g1(555) // Modifies `b` through the bound pointer
        fmt.Println("b after g1 call:", b) // b after g1 call: {555}

        g2(999) // Modifies `b` through the bound pointer
        fmt.Println("b after g2 call:", b) // b after g2 call: {999}
      }

    ```





-- Method Inheritance & Receiver Choice
--
  -- "No Inheritance" Rule for Defined Types
  --
    When you create a new defined type from a source type, the new type does NOT
    inherit the methods explicitly declared for the source type.

    Each defined type starts with a blank Method Set.


    example
      ```go
        // `MyInt` is a new type with an underlying type of `int`.
        type MyInt int


        // We declare a method exclusively for `MyInt`.
        func (mi MyInt) IsOdd() bool {
          return mi%2 == 1
        }


        // `Age` is a new type with an underlying type of `MyInt`.
        // `Age` does NOT inherit the `IsOdd` method from `MyInt`.
        type Age MyInt


        func main() {
          var x MyInt = 3
          _ = x.IsOdd() // OK: `IsOdd` is in the Method Set of `MyInt`.


          var y Age = 36
          // The following line causes a compiler error because `Age` has its own
          // distinct (and currently empty) Method Set.
          // _ = y.IsOdd() // COMPILE ERROR: y.IsOdd undefined (type Age has no method IsOdd)
          _ = y
        }

      ```





-- When to Use a Pointer vs. a Value Receiver
--
  --  Rule 1: If the method needs to MUTATE the receiver, use a Pointer.
  --
    This is the most important rule. A value receiver gets a copy, so any
    modifications it makes will be lost. A pointer receiver gets the memory
    address, allowing it to modify the original value.


  --  Rule 2: If the receiver is a slice or map, and you only modify its
  --  CONTENTS (not the header), a Value receiver is often sufficient.
    Slices and maps are reference types. The value itself is just a small header.
    A value receiver copies this header, but both headers still point to the
    same underlying data. However, if you need to `append` and potentially change
    the slice header, you need a pointer receiver.


  --  Rule 3: If the struct is VERY LARGE, use a Pointer to avoid expensive copies.
  --
    Copying a large struct on every method call can impact performance. Passing an
    8-byte pointer is much cheaper.


  --  Rule 4: If the receiver contains a `sync.Mutex` or similar, use a Pointer.
  --
    The types in the `sync` package (like `Mutex`) should never be copied after
    their first use. A value receiver would copy the mutex, leading to subtle bugs
    where each method call locks a different, useless copy of the mutex.


  --  Rule 5: For consistency, if SOME methods need a pointer, make them ALL pointers.
  --
    While not a strict requirement, it is common Go practice to keep the method
    set for a type consistent. If any method on `T` has a pointer receiver, the
    rest of them probably should too. This makes the type's usage less ambiguous.


  --  Rule 6: When in doubt, use a Pointer.
  --
    A pointer receiver is always a safe choice. It avoids unexpected copies,
    allows for mutation, and works for all types. The potential performance
    overhead of an extra pointer indirection is usually negligible compared to
    the cost of an accidental large struct copy.
    
    Use a value receiver only when you have a clear reason to do so (e.g., the
    type is small, immutable, and has no pointers).
