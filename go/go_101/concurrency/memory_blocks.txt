-- Memory Blocks and Reference Logic
--
  Go features automatic memory management, handling both allocation and garbage
  collection. While developers don't need to manage pointers manually as they
  would in C, understanding how the Go runtime organizes data into "Memory
  Blocks" is essential for writing high-performance, memory-efficient code.





-- Defining the Memory Block
--
  A "Memory Block" is a continuous segment of memory allocated by the runtime
  to host one or more "Value Parts."


  -- Core Properties
  --
    1. Contiguous: Each block is a single, uninterrupted stretch of RAM.
    2. No Crossing: A single value part (like a struct field or an array element)
       is always contained entirely within ONE memory block. It never spans
       across two blocks.
    3. Multi-Hosting: A single memory block can host multiple value parts
       simultaneously.


  -- When Blocks Host Multiple Value Parts
  --
    There are three primary scenarios where Go bundles multiple values into
    a single memory block:


    -- Structs and Fields
    --
      When you allocate a struct, Go allocates a single block large enough for
      the struct and ALL its fields. The direct parts of those fields live
      together in that block.


    -- Arrays and Elements
    --
      Similar to structs, an array is a single allocation. Every element in the
      array is hosted in the same continuous memory block.


    -- Slices and Shared Backing Arrays
    --
      Multiple slices can point to different sections of the same backing array.
      That backing array is a single memory block hosting elements for multiple
      different slice values.


  -- Code Demonstration: Shared Memory Blocks
  --
    ```go
      func blockSharingDemo() {
        // --- Example A: Struct Fields ---
        type User struct {
          ID   int
          Age  int
        }

        u := User{ID: 1, Age: 30}


        // 'u', 'u.ID', and 'u.Age' all inhabit the same memory block.
        // We can verify this by checking that their addresses are contiguous.
        fmt.Printf("User Addr: %p\n", &u)
        fmt.Printf("ID   Addr: %p\n", &u.ID)
        fmt.Printf("Age  Addr: %p\n", &u.Age)


        // --- Example B: Overlapping Slices ---
        // One backing array (one memory block)
        backingArray := [5]int{10, 20, 30, 40, 50}


        // Two slices "viewing" parts of the same block
        s1 := backingArray[0:3] // [10, 20, 30]
        s2 := backingArray[2:5] // [30, 40, 50]


        // They overlap at the value '30'.
        fmt.Println("Slice 1:", s1)
        fmt.Println("Slice 2:", s2)


        // Modifying s1 affects s2 because they share the same Memory Block!
        s1[2] = 999
        fmt.Println("After Modifying S1[2]:")
        fmt.Println("Slice 1:", s1)
        fmt.Println("Slice 2:", s2)
      }

    ```

    -- EXPECTED OUTPUT (Addresses will vary):
    --
      User Addr: 0xc0000140d0
      ID   Addr: 0xc0000140d0
      Age  Addr: 0xc0000140d8
      Slice 1: [10 20 30]
      Slice 2: [30 40 50]
      After Modifying S1[2]:
      Slice 1: [10 20 999]
      Slice 2: [999 40 50]





-- The Reference Model
--
  The Garbage Collector (GC) doesn't just look at values; it looks at
  Memory Blocks.


  -- The "Hosting" Reference
  --
    A memory block is considered "referenced" (and thus "alive") as long as
    AT LEAST ONE value part it hosts is still reachable in your code.


  -- Direct vs. Indirect References
  --
    1. Direct: A variable `x` lives in Block A. `x` references Block A.
    2. Indirect: If variable `y` contains a pointer to `x`, then `y`
       references `x`, which in turn means `y` references Block A.

  Practical Implication:
    If you have a massive array of 1 million elements and you keep a single
    pointer to just ONE element, the ENTIRE memory block (all 1 million
    elements) cannot be garbage collected. This is a common cause of
    "hidden" memory leaks in Go.





-- When are Memory Blocks Allocated?
--
  Go's runtime manages memory allocation automatically. While much of this
  happens behind the scenes, knowing which operations trigger the creation
  of a new memory block allows you to write more efficient, garbage-collector
  friendly code.


  -- Explicit Allocation: `new` and `make`
  --
    -- The `new(T)` Function
    --
      - Always allocates exactly ONE memory block.
      - This block is large enough to hold a zero value of type T.
      - Returns a pointer (*T) to that block.


    -- The `make(T, ...)` Function
    --
      - Often allocates MULTIPLE memory blocks.
      - One block for the "Direct Part" (the header: slice, map, or channel struct).
      - One or more blocks for the "Underlying Part" (the actual data buffer).


  example
    ```go
      func explicitAllocationDemo() {
        // Allocates 1 block for an int
        p := new(int)

        // Allocates at least 2 blocks:
        // 1 for the slice header, 1 for the backing array of 10 ints.
        s := make([]int, 10)

        _, _ = p, s
      }

    ```





-- Literals and Declarations
--
  - Variable Declarations: `var x T` triggers an allocation for the value.
    Whether it is on the stack or heap depends on "Escape Analysis."


  - Composite Literals: Creating maps, slices, or structs using `T{...}`
    syntax. Similar to `make`, these often allocate both headers and data buffers.


  - Anonymous Functions (Closures): Creating a closure may allocate memory
    to store the "captured" variables from the surrounding environment.





-- Interface Boxing
--
  When you assign a non-interface value to an interface, Go "boxes" the value.

  Rule: If the value being assigned is NOT a pointer, Go must allocate
  a new memory block to hold a copy of that value inside the interface.


  example
    ```go
      func interfaceBoxingDemo() {
        var i any
        x := 100

        // BOXING: 'x' is not a pointer.
        // A new memory block is allocated to store a copy of '100'.
        i = x

        fmt.Printf("Interface type: %T, value: %v\n", i, i)
      }

    ```





-- String Operations and Conversions
--
  Strings are immutable byte sequences. Most modifications require creating
  an entirely new string (and thus a new memory block).


  -- Non-Constant Concatenation
  --
    `s1 + s2` where strings are variables requires a new memory block to
    hold the combined result.


  -- Integer to String
  --
    `string(65)` allocates a new block to hold the UTF-8 encoded "A".


  -- Type Conversions
  --
    Converting `string <-> []byte` or `string <-> []rune` usually requires
    a deep copy into a new memory block to preserve the string's immutability.
    (Note: The compiler may optimize this away if the result is only used
    in a loop or as a map key).


  example
    ```go
      func stringAllocationDemo() {
        a, b := "Hello ", "World"

        // Allocates a new block for the result
        joined := a + b

        // Allocates a new block for the slice
        byteSlice := []byte(joined)

        fmt.Println(joined, byteSlice)
      }

    ```





-- Dynamic Container Growth
--
  -- Slice `append`
  --
    If you append to a slice and its backing array is full (len == cap),
    Go allocates a brand NEW, larger memory block, copies the old data,
    and discards the old block.


  -- Map Inserts
  --
    When adding entries to a map, the underlying hash table may reach a
    "Load Factor" threshold. When this happens, Go allocates a new, larger
    memory block for the buckets and "rehashes" the existing items into it.


  example
    ```go
      func growthDemo() {
        s := make([]int, 2, 2) // Full slice
        fmt.Printf("Before append: cap=%d, ptr=%p\n", cap(s), s)

        // Backing array is full. This triggers a NEW allocation.
        s = append(s, 3)
        fmt.Printf("After append:  cap=%d, ptr=%p\n", cap(s), s)
      }

    ```
      -- EXPECTED OUTPUT:
      --
        Before append: cap=2, ptr=0xc000016150
        After append:  cap=4, ptr=0xc00001a120  <-- Note the address changed!





-- Memory Allocation - Stack vs. Heap
--
  In Go, memory blocks are allocated in two primary locations: the Stack or
  the Heap. Understanding the difference between them is vital for optimizing
  performance and understanding how Go manages concurrency and memory safety.


  -- The Goroutine Stack
  --
    Every goroutine maintains its own dedicated "Stack." This is a segment of
    memory used as a local pool for that specific goroutine.


    -- Stack Characteristics
    --
      - Goroutine-Local: Memory on a stack is reserved for its goroutine.
        It is NOT safe to share stack-allocated memory across goroutine boundaries.
      - High Performance: Allocation and deallocation on the stack are extremely
        fast (incrementing/decrementing a stack pointer).
      - Self-Cleaning: Memory is automatically reclaimed when the function returns.
        The Garbage Collector (GC) does NOT need to scan the stack for cleanup.
      - Cache Friendly: Stack memory stays "hot" in CPU caches.


    -- Stack Sizing (Go 1.19+)
    --
      - Initial Size: Adaptive (starts small, typically 2KiB; 8KiB on Windows).
      - Dynamic Scaling: Stacks grow and shrink based on the goroutine's needs.
      - Minimum Size: Strictly 2KiB (8KiB on Windows).


    -- Stack Limits and Program Safety
    --
      While stacks grow, they have a global maximum limit. If a goroutine attempts
      to grow its stack beyond this limit, the program will crash.


      -- Default Limits (Go 1.25.n)
      --
        - 64-bit systems: 1 GB (Effective max is usually 512 MiB due to Power-of-2 logic).
        - 32-bit systems: 250 MB (Effective max is usually 128 MiB).


      -- Configuration
      --
        You can adjust this using `debug.SetMaxStack` from the `runtime/debug` package.
        Note: The actual limit used is the largest power of 2 that is not larger
        than the requested value.


  -- The Program Heap
  --
    The Heap is a global singleton memory pool shared by all goroutines.

    -- Heap Characteristics
    --
      - Shared: Values on the heap can be accessed by multiple goroutines
        simultaneously. Access often requires synchronization (Mutex, Channels).
      - GC Managed: The Garbage Collector must scan and manage the heap to
        free memory that is no longer in use.
      - Conservative: If the compiler isn't 100% sure a value stays local, it
        defaults to the heap for safety.


  -- Escape Analysis: Deciding the Destination
  --
    "Escape Analysis" is the compiler's process for determining whether a
    variable can be safely allocated on the stack or must "escape" to the heap.


    -- Checking for Escapes
    --
      Use the following command to see the compiler's decisions:
        go build -gcflags -m main.go


    -- Rules of Propagation
    --
      Escape is "infectious." If a part of a structure escapes, the whole does:
      1. Struct Field: If a field escapes, the whole struct escapes.
      2. Array/Slice: If one element escapes, all elements (and the container) escape.
      3. Pointer Reference: If variable V is referenced by a value that escapes,
         V also escapes.


    -- Go vs. C++ (New Keyword)
    --
      Unlike C++, where `new` always implies heap allocation, in Go, a value
      created with `new(T)` can live on the stack if it does not escape.


  -- Implementation: Indirect Stack-to-Heap References
  --
    If a local variable of type T escapes to the heap:
    1. Go allocates T on the heap.
    2. Go creates an implicit pointer (*T) on the stack.
    3. The compiler replaces all standard variable usage with pointer
       dereferences in the machine code.
    4. Once that stack pointer is marked "dead," the heap value becomes
       eligible for Garbage Collection.


  -- Code Demonstration: Shifting Memory Addresses
  --
    Because Go stacks are dynamic, they are sometimes reallocated to a larger
    segment. When this happens, the memory addresses of stack-allocated
    variables CHANGE.


    example
      ```go
        //go:noinline
        func forceStackGrowth(i int) byte {
          // Allocate a large local array to force the stack to resize/grow.
          var a [1 << 20]byte
          return a[i]
        }

        func stackMovementDemo() {
          var x int

          // Print address before growth
          fmt.Printf("Addr before growth: %p\n", &x)

          // Call function that forces stack to move to a larger memory segment
          forceStackGrowth(100)

          // Print address after growth
          // In modern Go, you will see that &x has changed!
          fmt.Printf("Addr after growth:  %p\n", &x)
        }

      ```

    -- REPRESENTATIVE OUTPUT (Go 1.25.n):
    --
      Addr before growth: 0xc0000a6f38
      Addr after growth:  0xc0004fcf38





-- Memory Collection and Garbage Collection (GC)
--
  Go manages memory automatically, but knowing when memory becomes "eligible"
  for collection is vital for understanding performance and avoiding leaks.
  The rules differ depending on where the memory block is located.


  -- The Three Tiers of Collection
  --
    -- Package-Level Variables (Permanent)
    --
      Memory blocks allocated for the direct parts of package-level variables are
      never collected. They stay in the "Global Memory Zone" for the entire
      duration of the program's execution.


    -- Stack Memory (Automatic)
    --
      Blocks allocated on a goroutine's stack are NOT managed by the Garbage
      Collector. Instead, the entire stack is reclaimed by the OS/Runtime as a
      single unit when the goroutine exits. This makes stack cleanup extremely fast.


    -- Heap Memory (GC Managed)
    --
      Blocks on the heap are only collected when they become "unused."
      A block is unused if it is no longer reachable (referenced) by:
      1. Any active goroutine stack.
      2. The Global Memory Zone.


  -- Code Demonstration: Heap Reachability
  --
    ```go
      var p *int // Global variable (never collected)

      func heapLifecycleDemo() {
        done := make(chan bool)
        // 'done' is used across goroutines, so it is allocated on the heap.


        go func() {
          x, y, z := 123, 456, 789
          _ = z  // 'z' stays on stack; cleaned up when goroutine exits.


          // x and y escape to the heap because the global 'p' references them.
          p = &x


          // -- State A
          // 'x' is alive (referenced by p).


          p = &y
          // -- State B
          // 'y' is now alive.
          // 'x' is no longer referenced by anything. Its block is now "unused"
          // and can be collected by the next GC cycle.


          p = nil
          // -- State C
          // 'y' is now also unused and eligible for collection.

          done <- true
        }()


        <-done
        // Once the goroutine above exits, the 'done' channel is no longer
        // reachable. It becomes eligible for collection.
      }

    ```


  -- Liveness Analysis (Smart Compiler Optimizations)
  --
    Modern Go compilers use "Liveness Analysis." A variable may be eligible
    for collection even if its scope (the curly braces it lives in) hasn't
    ended yet, provided the compiler can prove it is never used again.


    example
      ```go
        func optimizationDemo() {
          // Allocate a massive slice (backing array goes to heap)
          bs := make([]byte, 1<<31) // 2GB


          // -- THE OPTIMIZATION --

          // A slice is a header: { Pointer, Len, Cap }.
          // The function below ONLY needs the 'Len' field from the header.
          // It does NOT need the actual bytes in the massive backing array.


          fmt.Println("Slice length:", len(bs))


          // Because the backing array (the indirect part) is never used again
          // after the allocation line, a smart compiler allows the GC to
          // reclaim those 2GB immediately, even before Println is called!
        }

      ```


  -- Manual Control: `runtime.KeepAlive`
  --
    Sometimes, especially when dealing with finalizers or `unsafe` pointers,
    the compiler's liveness analysis is "too smart" and collects an object
    while you still technically need it to exist in memory.

    
    `runtime.KeepAlive(v)` explicitly tells the GC: "Do not collect 'v'
    until this specific line of code is reached."


    example
      ```go
        func keepAliveDemo() {
          bs := make([]int, 1000000)

          fmt.Println("Working with slice...")

          // We perform an operation that doesn't strictly need the data
          fmt.Println("Capacity:", cap(bs))

          // Ensure the backing array isn't collected before this point.
          runtime.KeepAlive(bs)
        }

      ```





-- Internal Mechanics of Garbage Collection (GC)
--
  Go uses a concurrent, tri-color, mark-and-sweep garbage collector. Its primary
  goal is to minimize "Stop-the-World" (STW) latency, allowing the application
  (the "mutator") to continue running alongside the collector.


  -- The Tri-Color Mark-Sweep Algorithm
  --
    The GC identifies used vs. unused memory by traversing the "object graph"
    and assigning one of three colors to every memory block on the heap.


  -- The Three Colors
  --
    - WHITE: The "Candidate" set. Objects that might be unreachable. At the 
      start of a cycle, every object on the heap is White.
    - GREY:  The "Worklist" set. Objects known to be reachable, but whose 
      pointers to other objects have not yet been scanned.
    - BLACK: The "Safe" set. Reachable objects that have been fully scanned. 
      The GC is finished with these for the current cycle.


    -- The Marking Process (Step-by-Step)
    --
      1. Initialization: All heap objects are White.
      2. Root Scanning: The GC identifies "Roots" (global variables and variables 
         on every goroutine's stack). These roots are colored GREY.
      3. Traversal: The GC picks a GREY object, scans it for pointers:
         - Any WHITE object found via this scan is turned GREY.
         - Once the scan is complete, the original object is turned BLACK.
      4. Convergence: Step 3 repeats until the GREY set is empty.
      5. Termination: Any objects still remaining WHITE are unreachable and 
         eligible for collection.


  -- The Sweep Phase and Memory Reuse
  --
    Once marking is finished, the "Sweep" phase begins. 
      - Collection: The GC reclaims the memory blocks identified as WHITE.
      - Non-Compacting: Unlike some other languages (e.g., Java), Go's GC 
        is non-compacting. It does NOT move memory blocks to "defragment" them. 
        The addresses of heap objects stay the same until they are destroyed.
      - Lazy Release: Reclaimed memory is not always returned to the OS 
        immediately. It is usually kept in the Go runtime's "free list" to be 
        efficiently reused for future allocations.


  -- Triggering the GC (The Pacer)
  --
    The "Pacer" is the internal logic that decides when a new GC cycle should 
    start. It balances CPU usage against memory growth.


    -- GOGC (The Percentage Target) 
    --
      This environment variable sets the relative growth of the heap. 
      Default is 100. 
      - If GOGC=100, the GC will trigger when the heap size reaches 200% of 
        the size it was after the last collection.
      - Higher GOGC = More memory used, less CPU spent on GC.
      - Lower GOGC  = Less memory used, more CPU spent on GC.


    -- GOMEMLIMIT (The Hard Cap) 
    --
      Introduced in Go 1.19, this allows you to set a maximum memory limit for 
      the entire process (e.g., 2GiB). The Pacer will trigger GC cycles more 
      aggressively to avoid exceeding this limit.



  -- Code Demonstration: Interacting with the GC
  --
    ```go
      func gcInteractionDemo() {
        // -- Manual Triggering 
        // Generally, you should let the Pacer handle this. 
        // However, runtime.GC() forces an immediate cycle.
        fmt.Println("Manually triggering GC...")
        runtime.GC()


        // -- Monitoring Statistics 
        var stats runtime.MemStats
        runtime.ReadMemStats(&stats)
        fmt.Printf("Heap Allocated: %d KiB\n", stats.HeapAlloc/1024)
        fmt.Printf("Number of GC Cycles: %d\n", stats.NumGC)


        // -- Adjusting the Pacer at Runtime 
        // SetGOGC returns the previous value. 
        // Setting it to -1 disables the GC entirely (DANGEROUS).
        oldGOGC := debug.SetGCPercent(200)
        fmt.Printf("Changed GOGC from %d to 200\n", oldGOGC)
      }

    ```
      -- representative output:
      --
        Manually triggering GC...
        Heap Allocated: 1240 KiB
        Number of GC Cycles: 1
        Changed GOGC from 100 to 200
