-- Gracefully Closing Channels
--
  Closing channels in Go is a frequent source of confusion and runtime panics.
  Unlike many other resources, the "close" operation on a channel has strict
  rules that can crash your program if violated.


  -- The Three Deadly Channel Sins
  --
    To avoid panics, you must strictly obey these three rules:
    1. Never close a channel from the receiver side.
    2. Never close a channel if it has multiple concurrent senders.
    3. Never close a channel that is already closed.


  -- Why these rules exist
  --
    Closing a closed channel results in a PANIC.
    Sending to a closed channel results in a PANIC.
    Closing is a signal to receivers that no more data is coming. If multiple
    senders exist, closing the channel would cause other senders to panic when
    they try to send their remaining data.


  -- The "IsClosed" Fallacy
  --
    A common request from new Gophers is a built-in `closed(ch)` function.
    However, even if such a function existed, it would be dangerous due to
    TOCTOU (Time-of-Check to Time-of-Use) race conditions.


    example
      ```go
        type T int

        // IsClosed is a common non-blocking implementation using select.
        // WARNING: This is NOT a universal solution.
        func IsClosed(ch <-chan T) bool {
          select {
          case <-ch:
            // If we receive something, it's either a zero-value (closed)
            // or a real value (which we just accidentally consumed!).
            return true
          default:
          }

          return false
        }

      ```

    -- WHY AN "IsClosed" FUNCTION IS UNRELIABLE:
    --
      Imagine this logic:
        ```go
          if !IsClosed(ch) {
              // <--- Logic Gap: Another goroutine could close 'ch' RIGHT HERE.
              ch <- value // PANIC: send on closed channel
          }

        ```

        The status of a channel is volatile in a concurrent environment. Information
        returned by a check is potentially obsolete by the time the next line executes.


  -- The Golden Rule of Channel Closing
  --
    "The Sender owns the channel."

    In almost all cases, the goroutine that produces data (the sender) should be
    the one that closes the channel. This naturally satisfies the rule of never
    sending to a closed channel.


  -- Scenario: Single Sender, Multi-Receiver
  --
    This is the simplest "Graceful" scenario. Since there is only one sender,
    that sender knows exactly when it is done and can safely close the channel.


    ```go
      func singleSenderDemo() {
        ch := make(chan int)


        // The Producer
        go func() {
          for i := 0; i < 5; i++ {
            ch <- i
          }
          // SENDER closes because it knows it is the only one sending.
          close(ch)
        }()


        // The Receivers (Multiple)
        for i := 0; i < 3; i++ {
          go func(id int) {
            for val := range ch {
              fmt.Printf("Receiver %d got %d\n", id, val)
            }
          }(i)
        }
      }

    ```



  -- How to Handle "Dangerous" Closures
  --
    If you find yourself in a situation where you feel you need to check
    "is this channel closed?", it usually indicates a flaw in your concurrency
    design.

    Instead of checking state, you should:
    1. Use a `sync.WaitGroup` to coordinate multiple senders (only close
       when the WaitGroup reaches 0).
    2. Use a "Signal Channel" (like `stopCh`) to tell senders to stop sending
       without closing the data channel itself.
    3. Use `sync.Once` if you absolutely must ensure a channel is only
       closed once by whichever worker finishes first.





-- The Channel Closing Principle
--
  Closing channels in Go requires a disciplined approach to avoid runtime
  panics. Because sending to a closed channel or closing an already closed
  channel causes a panic, the "ownership" of the close operation is vital.


-- The Channel Closing Principle
--
  -- The Principle
  --
    1. Never close a channel from the receiver side.
    2. Never close a channel if it has multiple concurrent senders.


  -- The Logic
  --
    You should only close a channel in a sender goroutine, and only if that
    sender is the EXCLUSIVE sender of the channel.


  -- The Universal Rule
  --
    The underlying requirement is simple: Don't close or send values to a
    closed channel.


    While you could technically coordinate receivers or multiple senders to
    close a channel safely using complex synchronization, it is significantly
    easier and more idiomatic to simply follow the "Sender Ownership" principle.


  -- "Rude" Solutions: Using Recover
  --
    If a developer chooses to ignore the closing principle, they might attempt
    to use Go's `recover` mechanism to "safely" handle the resulting panics.
    These are often called "rude" solutions because they treat the symptoms
    of a bad design rather than fixing the architecture.


    ```go
      // SafeClose attempts to close a channel and returns true if it succeeded,
      // or false if the channel was already closed.
      func SafeClose[T any](ch chan T) (justClosed bool) {
        defer func() {
          if recover() != nil {
            // If we panicked, the channel was already closed.
            // We intercept the return value here.
            justClosed = false
          }
        }()

        // If ch is already closed, this line panics.
        close(ch)
        return true
      }


      // SafeSend attempts to send to a channel and returns true if the channel
      // was already closed.
      func SafeSend[T any](ch chan T, value T) (closed bool) {
        defer func() {
          if recover() != nil {
            // If we panicked, we know the channel is closed.
            closed = true
          }
        }()

        // If ch is closed, this line panics.
        ch <- value
        return false
      }

    ```





-- Polite Channel Closing Patterns
--
  "Polite" closing refers to using standard synchronization primitives to
  ensure a channel is closed exactly once, thereby avoiding the panics
  associated with closing a closed channel.


  However, even "polite" closing doesn't solve every concurrency issue,
  particularly the "send-on-closed-channel" problem.


  -- Pattern A: Using `sync.Once`
  --
    This is the most common and idiomatic way to ensure an operation (like
    closing a channel) is executed exactly once, regardless of how many
    goroutines attempt it.


  example
    ```go
      type T int

      type OnceChannel struct {
        C    chan T
        once sync.Once
      }

      func NewOnceChannel() *OnceChannel {
        return &OnceChannel{C: make(chan T)}
      }

      // SafeClose ensures the channel is closed only once.
      func (oc *OnceChannel) SafeClose() {
        // sync.Once.Do is thread-safe and guarantees the function runs once.
        oc.once.Do(func() {
          close(oc.C)
        })
      }

    ```


  -- Pattern B: Using `sync.Mutex`
  --
    Using a Mutex allows you to maintain an explicit 'closed' state. This is
    slightly more verbose than `sync.Once` but allows for additional checks
    like `IsClosed()`.


    example
      ```go
        type MutexChannel struct {
          C      chan T
          closed bool
          mutex  sync.Mutex
        }


        func NewMutexChannel() *MutexChannel {
          return &MutexChannel{C: make(chan T)}
        }


        func (mc *MutexChannel) SafeClose() {
          mc.mutex.Lock()
          defer mc.mutex.Unlock()

          if !mc.closed {
            close(mc.C)
            mc.closed = true
          }
        }


        func (mc *MutexChannel) IsClosed() bool {
          mc.mutex.Lock()
          defer mc.mutex.Unlock()
          return mc.closed
        }

      ```


  -- The Remaining Danger: Data Races and Sends
  --
    While these patterns prevent the "closing a closed channel" panic, they
    do NOT inherently prevent the "sending to a closed channel" panic.


    -- The Data Race Reality
    --
      As of current Go specifications, if a `SafeClose` (using either Mutex or Once)
      runs concurrently with a `send` operation (`c <- val`) on the same channel,
      a data race technically occurs.


    -- Why?
    --
      The Go memory model and specification do not explicitly guarantee that a
      concurrent send and close are perfectly synchronized without external
      coordination.


    -- Practical Implication
    --
      Even if your close is "Safe" (called once), if a sender is still active
      while the close happens, that sender will PANIC.


      example
        ```go
          go func() {
            // Even if SafeClose is used, this may PANIC if the close
            // happens precisely while this line is executing.
            mc.C <- 42
          }()
          mc.SafeClose()

        ```





-- Graceful Channel Closing
--
  Closing channels "gracefully" means avoiding runtime panics and ensuring that
  all goroutines terminate cleanly. While panic/recover solutions (the "rude"
  way) or sync.Once (the "polite" way) exist, the most idiomatic Go approach
  uses pure channel signaling and ownership.


  -- Scenario: M Receivers, One Sender
  --
    -- The Architecture
    --
      1. The Sender: Owns the lifecycle. It decides when the data stream ends.
      2. The Receivers: Use `for range` loops to consume data until the channel
         is closed.


    -- The "Graceful" Logic
    --
      In this scenario, closing is straightforward. Because there is only ONE
      sender, that sender can safely call `close(ch)` without any risk of another
      goroutine attempting to send to a closed channel.


    -- Why it works
    --
      When the sender closes the channel:
      - All receivers currently blocked on `<-ch` are immediately unblocked.
      - They receive the zero-value of the channel type and `ok == false`.
      - `for range` loops exit automatically.


  -- Code Demonstration: One Sender, M Receivers
  --
    ```go
      func main() {
        // Configuration
        const MaxValue = 100
        const NumReceivers = 5

        // Synchronization for the demo
        var wg sync.WaitGroup
        wg.Add(NumReceivers)

        // The Data Channel
        dataCh := make(chan int)


        // --  The Single Sender (Owner)

        go func() {
          // Seed the random number generator
          source := rand.New(rand.NewSource(time.Now().UnixNano()))


          for {
            // Logic: Stop sending if we roll a 0
            if value := source.Intn(MaxValue); value == 0 {
              fmt.Println("[Sender] Stopping: Rolled a zero.")

              // SENDER OWNERSHIP:
              // Since this is the only sender, it is 100% safe to close.
              close(dataCh)
              return
            } else {
              dataCh <- value
            }
          }
        }()


        // --  The Multiple Receivers 

        for i := 1; i <= NumReceivers; i++ {
          go func(id int) {
            defer wg.Done()

            // The 'range' loop is the idiomatic way to consume until close.
            // It automatically exits once the channel is closed and drained.
            for value := range dataCh {
              fmt.Printf("[Receiver #%d] Got: %d\n", id, value)
            }

            fmt.Printf("[Receiver #%d] Detected channel close. Exiting.\n", id)
          }(i)
        }

        // Wait for all receivers to acknowledge the close and exit.
        wg.Wait()
        fmt.Println("Main: All goroutines finished gracefully.")
      }

    ```





-- Graceful Shutdown (One Receiver, N Senders)
--
  In a scenario with multiple senders and one receiver, the receiver cannot 
  safely close the data channel. If it did, any sender attempting to send 
  its next value would immediately trigger a runtime PANIC.

  
  To handle this gracefully while adhering to the "Channel Closing Principle," 
  we introduce an auxiliary "Signal Channel."


  -- The "Signal Channel" Strategy
  --
    -- The Architecture 
    --
      1. Data Channel (`dataCh`): Used for the actual workload.
      2. Stop Channel (`stopCh`): A `chan struct{}` used strictly for signaling.


    -- The Roles 
    --
      - Senders: They are the "Receivers" of the `stopCh`. They use a `select` 
        block to check if `stopCh` is closed before (or during) every send.
      - Receiver: It is the "Sender" (Owner) of the `stopCh`. When it has seen 
        enough data, it closes `stopCh`.


    -- Why it works 
    --
      Closing a channel is a broadcast. When the receiver closes `stopCh`, ALL 
      senders are notified simultaneously via their `select` blocks. They stop 
      sending to `dataCh` and exit peacefully.


  -- Code Demonstration: One Receiver, N Senders
  --
    ```go
      func main() {
        const MaxValue = 1000
        const NumSenders = 100

        var wg sync.WaitGroup
        wg.Add(1) // We only wait for the receiver to finish

        dataCh := make(chan int)
        stopCh := make(chan struct{}) // The signal channel


        // -- The Multiple Senders 

        for i := 0; i < NumSenders; i++ {
          go func(id int) {
            for {
              // Optimization: Check if we are already stopped before 
              // even attempting to generate data.
              select {
              case <-stopCh:
                return
              default:
              }

              // The main work block
              select {
              case <-stopCh:
                // If the receiver closed the signal, we exit.
                return
              case dataCh <- rand.Intn(MaxValue):
                // Successfully sent data to the receiver.
              }
            }
          }(i)
        }


        // -- The Single Receiver (Owner of the Stop Signal) 

        go func() {
          defer wg.Done()

          for value := range dataCh {
            // Logic: If we receive the "Magic Number", stop the whole system.
            if value == 777 {
              fmt.Println("[Receiver] Received 777! Signaling all senders to stop...")
              
              // BROADCAST: This unblocks the 'case <-stopCh' in every sender.
              close(stopCh) 
              return
            }
            
            // Process data...
            _ = value 
          }
        }()

        wg.Wait()
        fmt.Println("Main: System shut down gracefully.")
      }

    ```

    -- explaining the mechanics:
    --
      1. Why close(stopCh) instead of close(dataCh)?
         If the receiver closed 'dataCh', 100 senders would likely still be trying 
         to send. The next sender to hit 'dataCh <- val' would crash the program.
         By closing 'stopCh', we give the senders a "polite" way to exit via select.

      2. Does 'dataCh' ever get closed?
         No. And that is perfectly fine! 
         In Go, you do NOT have to close every channel. If no goroutines are 
         referencing a channel anymore, the Garbage Collector will reclaim its 
         memory, whether it was closed or not.

      3. The "Try-Receive" Optimization:
         The first 'select' with 'default' inside the sender loop is a performance 
         hint. It allows the sender to exit immediately if the stop signal was 
         already sent, avoiding the overhead of generating a random number or 
         waiting on the 'dataCh'.





-- The Moderator Pattern (M Receivers, N Senders)
--
  This is the most complex graceful shutdown scenario. Any participant (sender 
  or receiver) can trigger a total system shutdown. 
  
  -- The Challenge 
  --
    1. We cannot close the data channel because there are multiple senders 
       (violates the Closing Principle).
    2. We cannot let a random receiver close a shared signal channel because 
       multiple receivers might try to close it at once (causing a panic).
  

  -- The Solution: The Moderator 
  --
    We introduce a third party: the "Moderator" goroutine.
    1. Participants send a "stop request" to the Moderator.
    2. The Moderator receives the FIRST request and closes the signal channel.
    3. The signal channel (closed) broadcasts the shutdown to everyone else.


  -- The Component Architecture
  --
    1. dataCh (chan int): The main pipeline for data. Never explicitly closed.
    2. stopCh (chan struct{}): The broadcast signal. Only the Moderator closes this.
    3. toStop (chan string): The "Request for Shutdown" channel. 
       - Senders: Any worker (Sender or Receiver).
       - Receiver: The Moderator.
       - Buffer: Must be at least 1 to ensure the first request doesn't block.


  -- Code Demonstration: The "Game Over" Pattern
  --
    ```go
      func main() {
        const MaxRange = 100000
        const NumReceivers = 10
        const NumSenders = 100

        var wg sync.WaitGroup
        wg.Add(NumReceivers)

        dataCh := make(chan int)
        stopCh := make(chan struct{})           // Broadcast Kill-Switch
        toStop := make(chan string, 1)          // Shutdown Request Conduit


        // -- The Moderator 

        // It waits for the first person to "press the button," then cuts the power.
        var stoppedBy string
        go func() {
          stoppedBy = <-toStop // Wait for first request
          close(stopCh)        // Broadcast: Game Over
        }()


        // -- The N Senders 

        for i := 0; i < NumSenders; i++ {
          go func(id string) {
            for {
              value := rand.Intn(MaxRange)

              // LOGIC: If a sender rolls a 0, it wants to stop the game.
              if value == 0 {
                // "TRY-SEND" TRICK: 
                // We try to tell the moderator. If the toStop buffer is full,
                // someone else already requested a stop. We just exit.
                select {
                case toStop <- "sender#" + id:
                default:
                }
                return
              }

              // Standard Check: Exit as early as possible if signal received.
              select {
              case <-stopCh:
                return
              default:
              }

              // The Work: Send data unless stop signal arrives.
              select {
              case <-stopCh:
                return
              case dataCh <- value:
              }
            }
          }(strconv.Itoa(i))
        }


        // -- The M Receivers 

        for i := 0; i < NumReceivers; i++ {
          go func(id string) {
            defer wg.Done()
            for {
              // Exit early if system stopped.
              select {
              case <-stopCh:
                return
              default:
              }

              select {
              case <-stopCh:
                return
              case value := <-dataCh:
                // LOGIC: If a receiver gets the MaxValue-1, it stops the game.
                if value == MaxRange-1 {
                  select {
                  case toStop <- "receiver#" + id:
                  default:
                  }
                  return
                }
                // Process value...
                _ = value 
              }
            }
          }(strconv.Itoa(i))
        }

        // Wait for all receivers to finish processing and exit.
        wg.Wait()
        fmt.Println("Main: System shutdown complete.")
        fmt.Println("Shutdown was initiated by:", stoppedBy)
      }

    ```

    -- critical concepts & logic:
    --
      1. The "Try-Send" toStop Logic:
         The toStop channel has a buffer of 1. When the first worker sends its 
         identity, the buffer becomes full. If other workers simultaneously try to 
         request a stop, their 'default' case executes immediately. This avoids 
         blocking multiple workers on a channel that only one goroutine (the 
         moderator) is reading from.

      2. Double Select Pattern:
         Inside the loops, we use a standalone non-blocking 'select' to check stopCh 
         followed by a blocking 'select'. The first check is an optimization to avoid 
         performing expensive data generation (rand.Intn) if the game is already over.

      3. Violation Check:
         - Does a receiver close a channel? No.
         - Does a sender close a channel? No.
         - Does anyone send to a closed channel? No, the 'select' protects them.
         - Does anyone close a closed channel? No, the Moderator only reads from 
           toStop once, so it only calls close(stopCh) once.

      4. Buffer Strategy Variation:
         If you don't want to use the "Try-Send" trick, you can set the capacity of 
         toStop to (NumSenders + NumReceivers). This guarantees there is always 
         enough room for every worker to send a request, but it consumes more memory.





-- Third-Party Triggered Graceful Shutdown
--
  In this scenario, we have one sender and multiple receivers, but the decision
  to stop the data flow is made by one or more external "third-party" 
  goroutines (e.g., a timeout watcher, an interrupt handler, or a monitor).
  

  -- The Challenge 
  --
    We must adhere to the "Sender Closes" principle. Third parties cannot close
    the data channel directly. We need a way for them to "request" a shutdown
    that the sender can acknowledge and execute.


  -- The Dual-Signal Architecture
  --
    To handle this safely, we use two signaling channels:
    1. closing (chan struct{}): Used by third parties to signal a stop request.
    2. closed  (chan struct{}): Used by the sender to broadcast that it has
       actually finished and closed the data channel.


    -- The "Stop" Function Logic 
    --
      This pattern uses a clever 'select' block (attributed to Roger Peppe) that 
      allows multiple goroutines to call a stop function simultaneously without 
      blocking or panicking.


  -- Code Demonstration: Third-Party Trigger
  --
    ```go
      func main() {
        const MaxRange = 100000
        const NumReceivers = 50
        const NumThirdParties = 5

        var wg sync.WaitGroup
        wg.Add(NumReceivers)

        dataCh := make(chan int)
        closing := make(chan struct{}) // The "Request" line
        closed := make(chan struct{})  // The "Confirmation" line


        // -- The Idempotent Stop Function 

        // This function can be called by any number of goroutines safely.
        stop := func() {
          select {
          case closing <- struct{}{}:
            // We successfully sent the FIRST stop request.
            // Now we wait for the sender to confirm it has closed everything.
            <-closed
          case <-closed:
            // The system is already closed. Do nothing and return.
          }
        }


        // -- Third-Party "Watchdogs" 

        for i := 0; i < NumThirdParties; i++ {
          go func(id int) {
            // Simulate an external event triggering a shutdown (e.g. a timeout)
            waitTime := time.Duration(rand.Intn(3)+1) * time.Second
            time.Sleep(waitTime)
            
            fmt.Printf("[Third-Party #%d] Triggering shutdown after %v...\n", id, waitTime)
            stop()
          }(i)
        }


        // -- The Single Sender (Owner of dataCh) 

        go func() {
          // Ensure confirmation and data stream are closed on exit
          defer func() {
            close(closed) // Confirm shutdown to anyone calling stop()
            close(dataCh) // Close the actual data stream
          }()

          for {
            // Pre-check: Exit immediately if a stop request is waiting
            select {
            case <-closing:
              return
            default:
            }

            // Main Work: Send data OR exit if stop request arrives
            select {
            case <-closing:
              return
            case dataCh <- rand.Intn(MaxRange):
            }
          }
        }()


        // -- The Receivers 

        for i := 0; i < NumReceivers; i++ {
          go func() {
            defer wg.Done()
            for value := range dataCh {
              // Process value (suppressed for demo noise)
              _ = value
            }
          }()
        }

        // Wait for receivers to drain dataCh and exit
        wg.Wait()
        fmt.Println("Main: System exited gracefully.")
      }

    ```

    -- logic analysis:
    --
      1. Why the 'select' in the stop() function?
         - If multiple third parties call stop() at once, only one will succeed in 
           sending to the unbuffered 'closing' channel (since the sender only 
           performs one receive from 'closing' before returning).
         - The others will then block on 'case closing <-'. 
         - However, once the sender hits its defer and calls 'close(closed)', 
           the 'case <-closed' branch in the stop() function becomes ready for 
           EVERYONE. All callers of stop() are released.

      2. Violation Check:
         - Does a third party close a channel? No.
         - Does a receiver close a channel? No.
         - Does the sender close a closed channel? No, defer only runs once.
         - Does anyone send to a closed channel? No, dataCh is only closed after 
           the sender exits its loop.

      3. Efficiency:
         The "double select" in the sender (first non-blocking, then blocking) is a 
         standard optimization. It prevents the sender from starting a new iteration 
         of expensive work if a shutdown request is already pending.
