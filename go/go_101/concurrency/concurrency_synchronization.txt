Go makes it incredibly easy to run code concurrently using "goroutines".
However, concurrency introduces the risk of unpredictable behavior when 
multiple goroutines interact with the same shared resources (like memory).





-- The Problem: Data Races
--
  A "Data Race" is a critical concurrency bug. It occurs when two or more 
  concurrent computations (goroutines) access the exact same memory segment 
  simultaneously, and AT LEAST ONE of those accesses is a WRITE operation.


  -- Circumstance 1: Read/Write Conflict
  --
    Goroutine A is writing to variable X while Goroutine B is reading from X.
    Result: Goroutine B might read partially written, corrupted, or stale data. 
    The integrity of the READ is not preserved.


  -- Circumstance 2: Write/Write Conflict
  --
    Goroutine A and Goroutine B are both writing to variable X at the exact same time.
    Result: The final value of X is unpredictable. Data gets overwritten or corrupted.
    The integrity of the STORED DATA is not preserved





-- data race
--
  Concurrent computations may share resources, generally memory resource
  The following are some circumstances that may occur during concurrent computing:

    In the same period that one computation is writing data to a memory segment,
    another computation is reading data from the same memory segment.
    Then the integrity of the data read by the other computation might be not preserved.

    In the same period that one computation is writing data to a memory segment
    another computation is also writing data to the same memory segment
    Then the integrity of the data stored at the memory segment might be not preserved.

  These circumstances are called data races

  To prevent data races, we need concurrency synchronization

  
  example
    ```go
      // Example of a Data Race (DO NOT DO THIS)
      func dataRaceExample() {
        var counter int // Shared memory resource

        // Launch 1000 goroutines that all try to write to 'counter' at the same time
        for i := 0; i < 1000; i++ {
          go func() {
            counter++ // Write/Write and Read/Write conflicts happen here!
          }()
        }

        time.Sleep(1 * time.Second)
        // You might expect 1000, but because of data races, it will likely be 
        // a random, lower number (e.g., 942).
        fmt.Println("Unsafe Counter:", counter) 
        
        // TIP: Run Go code with the `-race` flag to detect these bugs automatically!
        // `go run -race main.go`
      }

    ```





-- Concurrency Synchronization Techniques
--
  -- Technique A: Mutexes (Mutual Exclusion)
  --
    Mutexes (`sync.Mutex`) are the direct 
    solution to the Read/Write and Write/Write conflicts.

    They "lock" a piece of code so only one goroutine can execute it at a time.


    example
      ```go
        func mutexExample() {
          var counter int
          var mu sync.Mutex // The synchronization primitive

          for i := 0; i < 1000; i++ {
            go func() {
              mu.Lock()   // Only ONE goroutine can pass this point at a time.
              counter++   // Safe from data races!
              mu.Unlock() // Let the next goroutine in.
            }()
          }
          
          time.Sleep(1 * time.Second)
          fmt.Println("Safe Counter (Mutex):", counter) // Will ALWAYS be 1000
        }

      ```


  -- Technique B: WaitGroups (`sync.WaitGroup`)
  --
    A WaitGroup is used to synchronize the COMPLETION of multiple goroutines.
    It allows the main program to wait until a specific number of background
    tasks have finished before proceeding.


    - `Add(int)`: Sets the number of goroutines to wait for.
    - `Done()`: Decrements the counter by 1 (called when a goroutine finishes).
    - `Wait()`: Blocks the current goroutine until the counter hits 0.


    example
      ```go
        func waitGroupExample() {
          var wg sync.WaitGroup

          // We are going to launch 3 worker goroutines.
          wg.Add(3)

          for i := 1; i <= 3; i++ {
            go func(workerID int) {
              // Ensure Done() is called even if the worker panics
              defer wg.Done() 
              
              fmt.Printf("Worker %d starting...\n", workerID)
              time.Sleep(100 * time.Millisecond) // Simulate work
              fmt.Printf("Worker %d done!\n", workerID)
            }(i)
          }

          // This blocks the main thread from exiting until wg.Done() has been 
          // called exactly 3 times. No `time.Sleep` hacks needed!
          wg.Wait() 
          fmt.Println("All workers finished. Main program exiting.")
        }

      ```


  -- Technique C: Channels
  --
    Channels are Go's most famous and idiomatic synchronization technique.
    Instead of locking shared memory (like a Mutex), channels allow goroutines
    to pass data safely between each other.


    Go Proverb: "Do not communicate by sharing memory; instead, share memory by communicating."


    example
      ```go
        func channelExample() {
          // Create a channel that transports strings
          messages := make(chan string)


          // Launch a goroutine
          go func() {
            time.Sleep(200 * time.Millisecond)
            // Send data INTO the channel safely
            messages <- "Data processed successfully!"
          }()


          // The main goroutine will BLOCK here until data is received from the channel.
          // This implicitly synchronizes the two goroutines without any locks.
          result := <-messages 
          
          
          fmt.Println("Received from channel:", result)
        }

      ```
