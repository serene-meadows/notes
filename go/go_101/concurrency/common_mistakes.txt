-- Common Concurrent Programming Mistakes
--
  Go's concurrency primitives (goroutines and channels) make concurrent
  programming approachable, but they do not eliminate the risk of fundamental
  synchronization errors. Carelessness or a lack of understanding of the
  Go Memory Model can lead to subtle, non-deterministic bugs.


  -- Mistake: Missing Synchronization (The "Visibility" Trap)
  --
    A common mistake is assuming that code will execute exactly in the order
    it appears in the source file across different goroutines.


    -- The Two Main Issues
    --
      1. Data Races: Occur when one goroutine reads a variable while another
         writes to it simultaneously without coordination.
      2. Instruction Reordering: For performance, compilers and CPUs may swap the
         execution order of independent statements. While this reordering is
         invisible to the goroutine doing the work, it can be observed by others.


    example
      ```go
        // DANGER: This function contains intentional concurrency bugs!
        func buggySynchronization() {
          var a []int // Initialized to nil
          var b bool  // Initialized to false

          go func() {
            // STATEMENT 1
            a = make([]int, 3)
            // STATEMENT 2
            b = true

            // A compiler or CPU may decide to execute Statement 2
            // BEFORE Statement 1 because they are independent within this closure.
          }()


          // Busy-waiting on 'b'
          for !b {
            // runtime.Gosched() yields the CPU but provides NO memory synchronization.
            runtime.Gosched()
          }

          // TRAP: Just because 'b' is observed as true here, it does NOT guarantee
          // that 'a' has been initialized. If reordering happened in the goroutine,
          // 'a' might still be nil.

          // a[0], a[1], a[2] = 0, 1, 2 // POTENTIAL RUNTIME PANIC: nil pointer dereference
          fmt.Println("Buggy check finished (might have crashed if unlucky)")
        }

      ```

      -- Corrected Code: Using Channels for "Happens-Before"
      --
        To fix this, we must use a synchronization primitive. Channels provide
        a "Happens-Before" guarantee: a send to a channel happens-before the
        corresponding receive completes.

        ```go
          func correctSynchronization() {
            var a []int
            // A signal channel (struct{} consumes no memory)
            done := make(chan struct{})

            go func() {
              // 1. Initialize the data
              a = make([]int, 3)

              // 2. SIGNAL: This acts as a memory fence. Everything done
              // above this line is guaranteed to be visible to the receiver.
              done <- struct{}{}
            }()

            // 3. WAIT: This blocks until the signal is received.
            <-done

            // 4. GUARANTEED: 'a' is initialized and visible here.
            a[0], a[1], a[2] = 0, 1, 2
            fmt.Println("Safe access results:", a)
          }

        ```





-- The "Sleep" and "Evaluation" Traps
--
  Go provides robust tools for synchronization (Channels, Mutexes, Atomics).
  A common mistake among beginners is attempting to use "time.Sleep" to
  coordinate memory access or failing to account for when a variable in a
  closure is actually evaluated.


  -- Mistake: Using `time.Sleep` as a Synchronization Primitive
  --
    -- The Trap
    --
      You launch a goroutine to write a value and "Sleep" in the main thread for
      a second, assuming the write will be finished by the time the sleep ends.


    -- Why it is WRONG
    --
      1. No "Happens-Before" Guarantee: The Go Memory Model does NOT define
         time.Sleep as a synchronization point. There is no technical guarantee
         that the write will be visible to the reader.
      2. Resource Dependency: If the OS is under heavy load, the scheduler might
         not even start the second goroutine within that one second.
      3. Flaky Code: This leads to "heisenbugs"—bugs that disappear when you
         try to debug them but crash in production.


    example
      ```go
        func sleepSyncMistake() {
          var x = 123

          go func() {
            x = 789 // Write to x
          }()

          // DANGER: We are guessing that 1 second is enough.
          time.Sleep(time.Second)

          // Technically a data race! There is no formal link between the write and read.
          fmt.Println("Result of sleep sync:", x)
        }

      ```





-- Mistake: Data Races in Expression Evaluation
--
  When a goroutine uses a variable from its outer scope (closure), the
  moment that variable is "read" matters immensely.


  example
    ```go
      func evaluationOrderMistake() {
        var n = 123
        c := make(chan int)

        go func() {
          // When is 'n' read?
          // If the compiler treats this as: tmp := n; c <- tmp
          // ...then it sends 123.
          // If the compiler waits until the channel is ready to send:
          // ...and main() has already updated n to 789, it sends 789.
          c <- n + 0
        }()

        // Give the goroutine a head start (but no guarantee!)
        time.Sleep(time.Millisecond * 10)

        n = 789 // Concurrent write to n

        fmt.Println("Received from channel:", <-c)
      }

    ```
    -- the compiler dependency:
    --
      - With `c <- n + 0`: Standard Go 1.25.n often outputs 123. The math '+0'
        forces an earlier evaluation in current implementations.
      - With `c <- n`: Standard Go 1.25.n often outputs 789.
      - THEORY: Both are data races. The Go spec does not define the order here.
        The program is non-deterministic and broken.





  --  The Correct Way: Value Snapshotting (Capturing)
  --
    To fix data races involving closures, you must "snapshot" the value so the
    goroutine has its own private, immutable copy.


    example
      ```go
        func correctSynchronization() {
          var n = 123
          c := make(chan int)


          // -- FIX A: Local variable snapshot --
          temp := n
          go func() {
            c <- temp // 'temp' is a copy of 123, it never changes.
          }()


          // -- FIX B: Pass as function parameter (Recommended) --
          // This is the cleanest way. Parameters are evaluated at the moment
          // the 'go' statement is executed.
          go func(val int) {
            // val is a local copy of n at the time the goroutine was born.
            // c <- val
          }(n)


          n = 789
          fmt.Println("Corrected received value:", <-c)
        }

      ```





-- Hanging Goroutines (Goroutine Leaks)
--
  A "Hanging Goroutine" is a goroutine that enters a blocking state and stays
  there forever. In Go, this is effectively a memory leak. Because the Go
  runtime cannot know if a blocked goroutine will eventually be unblocked,
  it never garbage collects them or the resources (stack memory, local
  variables) they hold.


  -- Common Causes of Hanging Goroutines
  --
    1. Ghost Receives: Receiving from a channel that will never be sent to or closed.
    2. Ghost Sends: Sending to a channel that will never be read from (or a nil channel).
    3. Self-Deadlock: A goroutine attempting to acquire a lock it already holds.
    4. Circular Deadlock: Goroutine A waits for B, while B waits for A.
    5. Case Starvation: A 'select' block where no channel operations ever become ready.


  -- Case Study: The "First Response Wins" Leak
  --
    This is a frequent mistake when implementing competitive requests. If you
    use an unbuffered channel to collect results but only read one result,
    the losing goroutines will "hang" on the send operation.


    example
      ```go
        // DANGER: This function leaks goroutines!
        func leakyRequest() int {
          // Unbuffered channel
          c := make(chan int)

          for i := 0; i < 5; i++ {
            go func(n int) {
              // Only one receiver exists in leakyRequest().
              // 1 goroutine will succeed, the other 4 will stay here FOREVER.
              c <- n
            }(i)
          }

          // We return after the first result. The other 4 workers are abandoned.
          return <-c
        }

      ```

      -- THE FIX:
      --
        To prevent the leak, the channel must be buffered with a capacity of at least N-1.
        This allows the "losers" to finish their send into the buffer and exit.

        // c := make(chan int, 5) // Capacity 5 ensures no one hangs.


  -- Case Study: The "Try-Send" Race Condition
  --
    Using 'select' with 'default' (try-send) on an unbuffered channel is
    extremely timing-sensitive.


    example
      ``go
        func buggyTrySendRequest() int {
          c := make(chan int) // Unbuffered


          for i := 0; i < 5; i++ {
            go func(n int) {
              select {
              case c <- n:
                // Success
              default:
                // If the receiver is not ready EXACTLY when this line runs,
                // the value is dropped and the goroutine exits.
              }
            }(i)
          }


          // RISK: If all goroutines reached their 'default' case before the
          // line below was ready to receive, this line will block FOREVER
          // because no one is left to send.
          return <-c
        }

      ``

      -- THE FIX:
      --
        Change 'c' to a buffered channel. This ensures that even if the workers start
        before the receiver is ready, they can successfully "deposit" their values
        into the buffer.


  -- Diagnostic Demonstration
  --
    ```go
      func main() {
        fmt.Printf("Initial goroutines: %d\n", runtime.NumGoroutine())

        // Call the leaky function
        _ = leakyRequest()

        // Give the runtime a moment to process exits
        time.Sleep(time.Millisecond * 100)

        // NumGoroutine will show that 4 goroutines are still alive and leaking.
        fmt.Printf("Goroutines after leak: %d\n", runtime.NumGoroutine())

        fmt.Println("\n--- Analysis ---")
        fmt.Println("The 4 extra goroutines are stuck in 'chan send' state.")
        fmt.Println("They will occupy memory until the program terminates.")
      }

    ```

    -- expected output:
    --
      Initial goroutines: 1
      Goroutines after leak: 5

    -- Analysis
    --
      The 4 extra goroutines are stuck in 'chan send' state.
      They will occupy memory until the program terminates.





-- Copying sync Package Values (A Critical Mistake)
--
  A fundamental rule in Go concurrency is: NEVER copy values of types defined
  in the "sync" standard package (e.g., Mutex, RWMutex, WaitGroup, Cond, Once).


  These types maintain internal state that is strictly tied to their specific
  memory address. Copying them creates a new instance with a "cloned" state
  that is no longer synchronized with the original, leading to deadlocks,
  data races, or corrupted state.


  -- Why Copying is Broken
  --
    1. Memory Identity: Many primitives in the 'sync' package (like Mutex) are
       implemented using low-level runtime machinery that tracks the address
       of the lock. If you copy the lock, the new copy is technically a
       different lock, but it may think it is in the same state as the original.


    2. Meaningless Protection: Even if a copied lock works, it only protects
       the data inside the COPY. It provides zero protection for the original
       shared data.


  -- Buggy Code: The "Value Receiver" Trap
  --
    ```go
      type Counter struct {
        // Embedding or including Mutex as a field
        sync.Mutex
        n int64
      }


      // -- CORRECT: Pointer Receiver
      //
      // This method receives a pointer (*Counter). No copy of the Mutex happens.
      // It correctly locks the original Mutex.
      func (c *Counter) Increase(d int64) {
        c.Lock()
        c.n += d
        c.Unlock()
      }


      // -- BUGGY: Value Receiver
      //
      // When this method is called, the ENTIRE Counter struct is copied.
      // This includes a copy of the Mutex and a copy of the value 'n'.
      func (c Counter) Value() int64 {
        // DANGER: This locks a local, temporary copy of the Mutex.
        // It protects a local, temporary copy of 'n'.
        // It does NOT synchronize with other goroutines calling Increase().
        c.Lock()
        defer c.Unlock()
        return c.n
      }

    ```


  -- Detecting the Mistake: `go vet`
  --
    You don't have to find these bugs manually. The Go toolchain includes
    a static analysis tool that catches "copylocks" errors.


    COMMAND:
      go vet main.go


    OUTPUT for the code above:
      ./main.go:31:9: Value passes lock by value: main.Counter contains sync.Mutex
      ./main.go:33:2:  call of c.Lock copies lock value: main.Counter contains sync.Mutex


  -- The Exception: sync.Locker
  --
    The only thing in the 'sync' package that is safe to copy is the
    'sync.Locker' INTERFACE. Since an interface is just a pointer-like
    header (itab + data pointer), copying it does not copy the underlying
    Mutex value itself.





-- sync.WaitGroup Placement Mistakes
--
  A `sync.WaitGroup` is essentially a concurrent-safe counter.
  - Add(n): Increments the counter.
  - Done(): Decrements the counter (shorthand for Add(-1)).
  - Wait(): Blocks the current goroutine IF the counter is > 0.


  The most common mistake is calling `Add()` in a place that is not
  guaranteed to execute before `Wait()` is called.


  -- The Mistake: Calling Add() Inside the Goroutine
  --
    In this scenario, the main goroutine launches many background goroutines.
    It then immediately hits `wg.Wait()`.


  -- The Race Condition
  --
    Because goroutines take time to initialize and schedule, the main goroutine
    often reaches the `Wait()` call BEFORE any of the background goroutines
    have actually started executing.

    If the background goroutine hasn't started, it hasn't called `wg.Add(1)`.
    The counter is still 0. `Wait()` sees 0, returns immediately, and the
    program exits before any work is actually done.


  example
    ```go
      func buggyWaitGroup() {
        var wg sync.WaitGroup
        var x int32 = 0

        for i := 0; i < 100; i++ {
          go func() {
            // DANGER: The main goroutine might hit Wait() before this line runs!
            wg.Add(1)
            atomic.AddInt32(&x, 1)
            wg.Done()
          }()
        }

        fmt.Println("Main: Waiting...")
        wg.Wait()

        // This will often print a number much smaller than 100 (e.g., 0, 3, or 12).
        fmt.Println("Buggy Result:", atomic.LoadInt32(&x))
      }

    ```


  -- The Solution: Calling Add() in the Parent Goroutine
  --
    To guarantee correctness, you must "reserve" the counter space in the
    parent goroutine BEFORE the child goroutine is launched.


    By calling `Add(1)` inside the loop but outside the `go func()`, we
    ensure the counter is incremented before the main goroutine can
    possibly reach the `Wait()` statement.


    example
      ```go
        func correctWaitGroup() {
          var wg sync.WaitGroup
          var x int32 = 0


          for i := 0; i < 100; i++ {
            // SUCCESS: Increment happens in the main goroutine.
            // It is strictly sequenced before the next Wait() call.
            wg.Add(1)

            go func() {
              defer wg.Done()
              atomic.AddInt32(&x, 1)
            }()
          }


          fmt.Println("Main: Waiting...")
          wg.Wait()


          // This is guaranteed to print 100 every single time.
          fmt.Println("Correct Result:", atomic.LoadInt32(&x))
        }

      ```





-- Common Architecture & Atomic Mistakes
--
  Go provides powerful concurrency primitives, but subtle mistakes in how
  they are sequenced or how memory is laid out can lead to performance
  bottlenecks, runtime panics, or platform-specific crashes.


  -- Mistake: Improper Future/Promise Evaluation
  --
    When using the "Future" pattern (a function that returns a channel),
    the way you invoke the receive operation (`<-`) determines whether your
    code runs in parallel or strictly in sequence.


  -- The Trap: Sequential Execution
  --
    doSomething(<-fa(), <-fb())
    Because Go evaluates function arguments from left-to-right:
    1. fa() is called and returns a channel.
    2. The program BLOCKS waiting for a value from that channel.
    3. ONLY THEN is fb() called.
    Result: Total time = Time(A) + Time(B). No concurrency achieved.


    example
      ```go
        func future(id string, d time.Duration) <-chan string {
          c := make(chan string)
          go func() {
            time.Sleep(d)
            c <- id
          }()
          return c
        }


        func improperFutures() {
          start := time.Now()

          // DANGER: Arguments are evaluated one after another.
          // It waits for 'A' to finish before even starting the goroutine for 'B'.
          fmt.Printf("Results: %s, %s\n", <-future("A", time.Second), <-future("B", time.Second))

          fmt.Printf("Improper took: %v\n", time.Since(start).Round(time.Second))
        }


        func properFutures() {
          start := time.Now()

          // 1. Call both functions immediately to launch both goroutines.
          ca, cb := future("A", time.Second), future("B", time.Second)

          // 2. Now wait for both. They are running in parallel.
          fmt.Printf("Results: %s, %s\n", <-ca, <-cb)

          fmt.Printf("Proper took:   %v\n", time.Since(start).Round(time.Second))
        }

      ```


-- Mistake: Closing Channels with Active Senders
--
  A common source of production panics (even in massive projects like Kubernetes)
  is closing a channel while other goroutines are still potentially sending.


  -- The Golden Rule
  --
    Only close a channel when you are CERTAIN no more sends will occur.
    This usually means the "Last Active Sender" should be the one to close.


  example
    ```go
      func earlyClosePanic() {
        ch := make(chan int)


        go func() {
          time.Sleep(time.Millisecond * 50)
          // 3. This will trigger a PANIC because the channel is closed.
          // panic: send on closed channel
          ch <- 1
        }()


        // 1. Work happens...
        time.Sleep(time.Millisecond * 10)


        // 2. Early close!
        close(ch)
        fmt.Println("Channel closed early.")
      }

    ```


  -- Mistake: 64-bit Atomic Alignment on 32-bit Systems
  --
    For performance reasons, 64-bit atomic operations (on `int64` or `uint64`)
    require the variable's memory address to be "8-byte aligned."


    -- The Platform Gap 
    --
      - 64-bit Architectures: Variables are usually 8-byte aligned automatically.
      - 32-bit Architectures: Variables might only be 4-byte aligned.


  -- Legacy Danger (atomic.AddInt64) 
  --
    If you use `atomic.AddInt64(&myVar, 1)` on a 32-bit system and `myVar`
    is not aligned, the program will CRASH with a panic.


  -- The Modern Fix (Go 1.19+) 
  --
    Go 1.19 introduced atomic wrapper types like `atomic.Int64`.
    These types are GUARANTEED to be correctly aligned by the compiler,
    even on 32-bit operating systems.


    exmaple
      ```go
        func alignmentSafetyDemo() {
          // LEGACY: Risky on 32-bit ARM/x86
          // var count int64
          // atomic.AddInt64(&count, 1)


          // MODERN (Recommended): Safe on all platforms
          var safeCount atomic.Int64
          safeCount.Add(1)
          fmt.Println("Atomic counter value:", safeCount.Load())
        }

      ```





-- Managing Timers and Avoiding Resource Leaks
--
  The `time` package provides powerful tools for delays and timeouts, but 
  improper use—specifically with `time.After` is a leading cause of memory 
  bloat in high-throughput Go applications. 


  Recent changes in Go 1.23 have significantly simplified timer management, 
  but understanding the "legacy" pitfalls is vital for maintaining older 
  codebases and understanding Go's evolution.


  -- The Problem: `time.After` in High-Frequency Loops
  --
    `time.After(d)` is a convenience function that returns a channel. Under the 
    hood, it creates a `time.Timer`. 


    -- The Trap (Pre-Go 1.23) 
    --
      Before Go 1.23, a `Timer` created by `time.After` stayed alive in memory until 
      it reached its expiration time, even if the program was no longer listening 
      to its channel.


      example
        ```go
          // longRunningLeaky demonstrates how a high-volume channel can cause 
          // thousands of timers to accumulate in memory before Go 1.23.
          func longRunningLeaky(messages <-chan string) {
            for {
              select {
              case <-time.After(time.Minute):
                // If no message arrives for 1 minute, exit.
                return
              case msg := <-messages:
                // If a message arrives every millisecond, we call time.After()
                // 1000 times per second. In Pre-1.23 Go, each call leaves 
                // a timer object "alive" for a full minute, even though we 
                // loop and stop listening to the previous one.
                fmt.Println(msg)
              }
            }
          }

        ```


    -- The Legacy Fix: Manual Timer Reuse (Pre-Go 1.23)
    --
      To avoid the leak before Go 1.23, developers had to manually manage a 
      single timer instance. This required complex "draining" logic.


      ```go
        func longRunningLegacyFix(messages <-chan string) {
          // 1. Create a single timer instance.
          timer := time.NewTimer(time.Minute)
          defer timer.Stop()

          for {
            select {
            case <-timer.C:
              // Timer expired, exit.
              return
            case msg := <-messages:
              fmt.Println(msg)

              // 2. CRITICAL STEP: Reset the timer for the next iteration.
              // Before Go 1.23, you had to stop the timer and manually "drain" 
              // the channel if it had already fired during the processing step.
              if !timer.Stop() {
                // If Stop() returns false, the timer already expired/fired.
                // We must drain the channel to keep it ready for Reset().
                select {
                case <-timer.C:
                default:
                }
              }
              timer.Reset(time.Minute)
            }
          }
        }

      ```


  -- The Modern Solution: Go 1.23 Improvements
  --
    Go 1.23 introduced two major improvements:
    1. Garbage Collection: Unreachable timers can now be garbage collected 
       even if they haven't expired or been stopped. This makes `time.After` 
       safe for most use cases!
    2. Automatic Draining: `timer.Reset()` now automatically drains the 
       internal channel, making the manual `if !timer.Stop()` logic obsolete.


    example
      ```go
        func longRunningModern(messages <-chan string) {
          // Even though Go 1.23 makes time.After safer, using a single timer 
          // is still more CPU efficient than creating new ones.
          timer := time.NewTimer(time.Minute)

          for {
            select {
            case <-timer.C:
              return
            case msg := <-messages:
              fmt.Println(msg)
            }

            // In Go 1.23+, Reset handles everything internally.
            timer.Reset(time.Minute)
          }
        }

      ```
