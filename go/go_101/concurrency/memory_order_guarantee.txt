-- Memory Ordering and Guarantees
--
  In concurrent programming, the order in which instructions are written in
  the source code is not necessarily the order in which they are executed
  by the CPU or the compiler. This concept is known as "Memory Ordering."





-- Why Reordering Happens
--
  Modern compilers (at compile time) and CPUs (at runtime) frequently reorder
  instructions to optimize performance.


  -- The Golden Rule of Reordering
  --
    Reordering is allowed ONLY if the result is indistinguishable to the
    goroutine performing the operation.


  From the perspective of Goroutine A, its own execution always appears to
  follow the exact order of the code. This is called "As-If-Serial" semantics.





-- The Concurrency Problem: Observability
--
  While reordering is invisible to a single goroutine, it becomes a major
  problem when multiple goroutines share data.


  If Goroutine A reorders its instructions, Goroutine B might observe those
  changes in the "wrong" order. Without proper synchronization, the behavior
  of your program becomes dependent on the specific compiler version, the
  CPU architecture (x86 vs ARM), and even runtime timing.





-- Case Study: The "Setup Trap" (Unsafe Code)
--
  The following example is a classic "unprofessional" Go program that fails
  because it ignores memory reordering and data races.


  example
    ```go
      var a string
      var done bool


      func setup() {
        // Intended logic: Initialize 'a' THEN signal via 'done'
        a = "hello, world"
        done = true

        // Internal view: This goroutine will always see 'a' as initialized
        // because it can't observe its own reordering.
        if done {
          log.Println("Setup sees len(a):", len(a))
        }
      }


      func main() {
        go setup()

        // Busy-waiting loop.
        // runtime.Gosched() simply yields the processor; it does NOT provide
        // any memory synchronization guarantees.
        for !done {
          runtime.Gosched()
        }

        // DANGER: The main goroutine might observe 'done == true' BEFORE
        // 'a' has been assigned its value because the compiler or CPU
        // flipped the two lines inside the setup() function!
        log.Println("Main sees a:", a)
      }

    ```
    -- why this program is broken:
    --
      1. MEMORY REORDERING:
         The compiler or CPU is free to transform the setup() logic to this:
             done = true
             a = "hello, world"
         If 'main' reads 'done' in the tiny window between these two lines,
         it will exit the loop and print an empty string for 'a'.

      2. DATA RACES:
         Both 'a' and 'done' are being accessed concurrently (one write, one read)
         without any synchronization (like Mutexes or Atomic operations).
         This constitutes a formal Data Race.

      3. ARCHITECTURE DEPENDENCE:
         This program might work 99% of the time on your local machine but fail
         instantly when deployed to a server with a different CPU (e.g., ARM has
         weaker memory ordering than x86).


  -- The Solution: Synchronization and Tooling
  --
    To fix these issues, Go provides the "Go Memory Model" document, which
    defines "Happens-Before" relationships. These relationships guarantee
    that memory writes in one goroutine are visible to another.


    -- Standard Synchronization Tools
    --
      1. Channels (sending/receiving)
      2. sync.Mutex / sync.RWMutex
      3. sync/atomic operations
      4. sync.WaitGroup


    -- The Race Detector
    --
      Go includes a powerful tool to detect these mistakes at runtime.
      ALWAYS run your tests and benchmarks with the race flag:

        go run -race main.go
        go build -race
        go test -race ./...

      This tool will report exactly which variables are causing the data race
      and provide a stack trace for the conflicting goroutines.





-- The Memory Model & Happens-Before
--
  In concurrent programming, "Memory Model" refers to the formal rules that
  specify the conditions under which a read of a variable in one goroutine
  is guaranteed to observe a write to the same variable in another goroutine.


  -- The Design Philosophy: Abstraction over Fences
  --
    -- CPU Fences
    --
      Different CPU architectures (x86, ARM, etc.) use "Fence" or "Barrier"
      instructions to prevent hardware-level reordering. Some languages (like C++)
      allow developers to insert these manually.


    -- The Go Way
    --
      Go abstracts these low-level complexities. You cannot manually insert CPU
      fences. Instead, Go provides high-level synchronization primitives
      (Channels, Mutexes, WaitGroups, and Atomics).


    These primitives use fences internally. To ensure correct execution order,
    you must use these tools rather than assuming the order of code lines.


  -- The "Happens-Before" Relationship
  --
    To describe memory safety, Go uses the term "Happens-Before."

    If event A is guaranteed to happen before event B:
    1. Any memory write that occurred before A in the source code is guaranteed
       to be visible to any code executing after B.
    2. The compiler and CPU are forbidden from reordering instructions across
       that specific boundary in a way that violates this visibility.


  -- Rule: Goroutine Creation
  --
    The Rule: The 'go' statement that starts a new goroutine happens-before
    the start of that goroutine's execution.


    -- Safe Example: Guaranteed Visibility
    --
      ```go
        var x, y int


        func f1() {
          // 1. These writes happen-before the 'go' statement below.
          x, y = 123, 789

          // 2. The start of this goroutine is guaranteed to see x=123 and y=789.
          go func() {
            // 3. Guaranteed to print 123.
            fmt.Println("x:", x)

            // 4. This nested creation also follows the rule.
            go func() {
              // 5. Guaranteed to print 789.
              fmt.Println("y:", y)
            }()
          }()


          time.Sleep(time.Millisecond * 100)
        }

      ```

        -- EXPECTED OUTPUT (f1):
        --
          x: 123
          y: 789


    -- Unsafe Example: Data Races & Non-Determinism
    --
      ```go
        func f2() {
          // DANGER: The goroutines are started BEFORE the variables are assigned.
          go func() {
            // This might print 0 (initial value) or 123 (if the write below
            // happened to finish first).
            fmt.Println("Ambiguous x:", x)
          }()

          go func() {
            fmt.Println("Ambiguous y:", y)
          }()

          // These writes are concurrent with the reads above.
          x, y = 123, 789

          time.Sleep(time.Millisecond * 100)
        }

      ```

      -- WHY f2() IS BROKEN:
      --
        1. No Happens-Before: There is no synchronization between the assignment
           (x, y = ...) and the goroutine execution.
        2. Data Race: This is a textbook data race. One goroutine is writing to
           memory while others are reading it without coordination.
        3. Use the Race Detector: Running 'go run -race' on this code will
           immediately flag the error.





-- Channel Memory Order Guarantees
--
  Channels are the primary synchronization primitive in Go. The "Go 1 Memory
  Model" defines very specific "Happens-Before" relationships for channel
  operations to ensure that memory writes in one goroutine are visible to
  another.


  -- The Three Cardinal Rules of Channel Order
  --
    -- Rule 1: Send Happens-Before Receive
    --
      The nth successful SEND to a channel happens-before the completion of the
      nth successful RECEIVE from that channel.

      This applies to BOTH buffered and unbuffered channels. It ensures that
      the data being sent is ready before the receiver tries to read it.


    -- Rule 2: Receive Happens-Before Send Completion (Capacity m)
    --
      The nth successful RECEIVE from a channel with capacity 'm' happens-before
      the completion of the (n+m)th successful SEND to that channel.


      -- Special Case: Unbuffered Channels (m=0):
      --
        The nth successful RECEIVE happens-before the completion of the nth
        successful SEND. This creates a "rendezvous" point where both goroutines
        are strictly synchronized.


    -- Rule 3: Close Happens-Before Receive of Zero Value
    --
      The closing of a channel happens-before a receive completes if that
      receive returns a zero value because the channel is closed.


  -- Code Demonstration: Unbuffered Channel Rendezvous
  --
    ```go
      func unbufferedGuarantees() {
        var a, b int
        var c = make(chan bool)


        // -- Goroutine 1
        go func() {
          a = 1      // Write A
          c <- true  // SEND (Rendezvous point)

          // Because of Rule 2 (m=0):
          // The receive in Goroutine 2 happens-before this send finishes.
          // Since 'b = 1' happens before that receive, 'b = 1' is guaranteed
          // to be visible here.
          if b != 1 {
            panic("This will never happen: b is guaranteed to be 1")
          }
          fmt.Println("G1: Successfully saw b=1")
        }()


        // -- Goroutine 2
        go func() {
          b = 1      // Write B
          <-c        // RECEIVE (Rendezvous point)

          // Because of Rule 1:
          // The send in Goroutine 1 happens-before this receive completes.
          // Since 'a = 1' happens before that send, 'a = 1' is guaranteed
          // to be visible here.
          if a != 1 {
            panic("This will never happen: a is guaranteed to be 1")
          }
          fmt.Println("G2: Successfully saw a=1")
        }()
      }

    ```

    -- EXPECTED OUTPUT (unbufferedGuarantees):
    --
      G1: Successfully saw b=1
      G2: Successfully saw a=1
      (Order of lines may vary, but no panics will occur)





  -- The "Irrelevant Goroutine" Trap (Unsafe Code)
  --
    A critical warning: "Happens-Before" relationships only apply to the
    goroutines directly involved in the synchronization event. A third,
    unrelated goroutine does NOT inherit these guarantees.


    ```go
      func irrelevantGoroutineDanger() {
        var a, b, x, y int
        c := make(chan bool)


        go func() {
          a = 1
          c <- true // Sync with G2
          x = 1
        }()


        go func() {
          b = 1
          <-c       // Sync with G1
          y = 1
        }()


        // --- GOROUTINE 3: The Danger Zone ---
        // This goroutine has NO relationship with the channel 'c'.
        // It is observing variables 'x', 'y', 'a', and 'b' across a data race.
        go func() {
          // Even if x == 1, there is NO guarantee that 'a' is seen as 1 here.
          // The compiler or CPU could have reordered 'a=1' and 'x=1' in G1,
          // or the cache might not have synced 'a' to G3 yet.
          if x == 1 {
            if a != 1 {
              // This panic IS possible!
              fmt.Println("CRITICAL ERROR: G3 saw x=1 but a=0")
            }
          }


          if y == 1 {
            if b != 1 {
              // This panic IS possible!
              fmt.Println("CRITICAL ERROR: G3 saw y=1 but b=0")
            }
          }
        }()
      }

    ```


  -- Buffered Channel Guarantees (The n+m Rule)
  --
    For a channel with capacity 'm':
      -- Rule A: Send -> Receive
      --
        The nth successful SEND happens-before the completion of the nth
        successful RECEIVE. (Same as unbuffered).


      -- Rule B: Receive -> Send Completion (The Buffer Limit)
      --
        The nth successful RECEIVE happens-before the completion of the (n+m)th
        successful SEND.

        This rule defines the "backpressure" mechanism. It ensures that the sender
        cannot finish its (n+m)th send until the receiver has freed up enough
        space by completing the nth receive.


    example
      ```go
        func bufferedGuaranteesDemo() {
          var k, l, m, n, x, y int
          c := make(chan bool, 2) // Capacity m = 2


          go func() {
            k = 1      // Write K
            c <- true  // Send #1
            l = 1      // Write L
            c <- true  // Send #2
            m = 1      // Write M
            c <- true  // Send #3 (This blocks until Receive #1 completes)
            n = 1      // Write N
          }()


          go func() {
            x = 1      // Write X
            <-c        // Receive #1
            y = 1      // Write Y
          }()



          // -- ANALYSIS OF f5() LOGIC:
          // --
          // 1. k=1 happens-before y=1:
          //    k=1 -> Send #1 -> Receive #1 completes -> y=1. (Guaranteed)
          //
          // 2. x=1 happens-before n=1:
          //    x=1 -> Receive #1 -> Send #3 completes (n+m rule: 1+2=3) -> n=1. (Guaranteed)
          //
          // 3. UNSPECIFIED:
          //    x=1 vs l=1: No guarantee. Send #2 can finish before G2 even starts.
          //    l=1 vs y=1: No guarantee. Send #2 is decoupled from Receive #1 by the buffer.
        }

      ```


  -- Channel Closing Guarantees
  --
    -- Rule C: Close -> Receive (Zero Value)
    --
      The closing of a channel happens-before a receive completes if that
      receive returns a zero value because the channel was closed.


      ```go
        func closingGuaranteesDemo() {
          var k, x, y int
          c := make(chan bool, 1)


          go func() {
            c <- true  // Send #1
            k = 1      // Write K
            close(c)   // CLOSE happens-after k=1
          }()


          go func() {
            <-c        // Receive #1 (Gets 'true')
            x = 1      // Write X
            <-c        // Receive #2 (Gets 'false' because channel is closed)
            y = 1      // Write Y
          }()


          // -- ANALYSIS OF f6() LOGIC:
          // --
          // 1. k=1 happens-before y=1:
          //    k=1 -> close(c) -> Receive #2 completes (due to close) -> y=1. (Guaranteed)
          //
          // 2. UNSPECIFIED:
          //    k=1 vs x=1: No guarantee.
          //    G2 can receive the first value and set x=1 while G1 is still
          //    working toward the k=1 assignment.
        }

      ```





-- Mutex and RWMutex Memory Order Guarantees
--
  In Go, Mutexes are not just tools for preventing concurrent access; they are
  "Memory Fences." They establish "Happens-Before" relationships, ensuring
  that memory writes made by one goroutine are visible to the next goroutine
  that acquires the same lock.


  -- The Core Guarantee: The Baton Pass
  --
    For any `sync.Mutex` or `sync.RWMutex` variable 'm':


    The Rule: The nth successful call to `m.Unlock()` happens-before the
    (n+1)th call to `m.Lock()` returns.


    This creates a "baton pass" effect. Everything the previous holder of the
    lock did before calling `Unlock()` is strictly guaranteed to be visible to
    the next holder once their `Lock()` call returns.


  example
    ```go
      // fab demonstrates the basic Mutex baton pass.
      func fab() {
        var a, b int
        var mu sync.Mutex

        // 1. Initial lock in the main goroutine.
        mu.Lock()


        go func() {
          // 4. This Lock() blocks until mu.Unlock() is called in the other goroutine.
          mu.Lock()
          // 5. Because of the baton pass, a=1 is guaranteed to be visible here.
          b = 1
          mu.Unlock()
        }()


        go func() {
          // 2. Perform a write.
          a = 1
          // 3. UNLOCK: This is the baton pass.
          // It makes 'a=1' visible to the next Locker.
          mu.Unlock()
        }()


        // Result: 'a = 1' always ends before 'b = 1' is executed.
      }

    ```


  -- RWMutex: Reader/Writer Coordination
  --
    `sync.RWMutex` adds specific rules for the interaction between readers
    (`RLock`) and writers (`Lock`).


    -- Rule A: Writer-to-Reader Visibility 
    --
      If a Writer finishes (Unlock), all subsequent Readers (RLock) are guaranteed
      to see the Writer's changes.


      example
        ```go
          func fxy() {
            var x, y int
            var rw sync.RWMutex


            rw.Lock() // Writer lock
            go func() {
              // 3. This RLock() waits for the Writer's Unlock().
              rw.RLock()
              // 4. Guaranteed to see x = 1.
              y = 1
              rw.RUnlock()
            }()


            go func() {
              // 1. Writer updates data.
              x = 1
              // 2. Writer releases lock.
              rw.Unlock()
            }()


            // Result: 'x = 1' always ends before 'y = 1' is executed.
          }

        ```


    -- Rule B: Reader-to-Writer Visibility 
    --
      A Writer (Lock) will not return until all active Readers (RLock) have
      released their locks (RUnlock). This ensures the Writer sees the state
      left by the Readers.


      example
        ```go
          func fmn() {
            var m, n int
            var rw sync.RWMutex

            rw.RLock() // Read lock
            go func() {
              // 3. This Lock() blocks until the Reader calls RUnlock().
              rw.Lock()
              // 4. Guaranteed to see m = 1.
              n = 1
              rw.Unlock()
            }()

            go func() {
              // 1. Reader processes data.
              m = 1
              // 2. Reader releases lock.
              rw.RUnlock()
            }()

            // Result: 'm = 1' always ends before 'n = 1' is executed.
          }

        ```


  -- The "Spec vs. Reality" Warning
  --
    It is tempting to assume that simply calling Lock/Unlock acts as a global
    memory sync. However, you must follow the official Memory Model, not
    implementation side-effects.

    
    example
      ```go
        func fpq() {
          var p, q int
          var l sync.Mutex

          p = 1
          // --- Potential Reordering Point ---
          l.Lock()
          l.Unlock()
          // ----------------------------------
          q = 1

          // OFFICIAL SPEC: The execution of 'p = 1' is NOT guaranteed to end before
          // the execution of 'q = 1' from the perspective of an IRRELEVANT goroutine.
          //
          // Although most compilers will not reorder across a Lock/Unlock pair,
          // the official documentation does not guarantee it. To ensure 'p' is visible
          // before 'q' in another goroutine, that other goroutine MUST also use
          // the SAME mutex to create a formal Happens-Before link.
        }

      ```





-- Memory Order Guarantees (Sync Primitives & Atomics)
--
  In addition to Channels and Mutexes, Go provides memory order guarantees 
  for WaitGroups, Once, Condition Variables, and Atomic operations. These 
  guarantees ensure that memory writes in one goroutine are correctly 
  synchronized and visible to others.


  -- sync.WaitGroup: Counter-to-Wait Visibility
  --
    -- The Rule 
    --
      Assume the counter of a `sync.WaitGroup` is not zero. The completion of the 
      call to `wg.Done()` (or `wg.Add(n)` with a negative delta) that drops the 
      counter to zero is guaranteed to happen-before the return of any 
      `wg.Wait()` call.


    example
      ```go
        func waitGroupOrderDemo() {
          var data int
          var wg sync.WaitGroup

          wg.Add(1)
          go func() {
            // 1. Data write happens here
            data = 42
            // 2. wg.Done() marks the counter as zero
            wg.Done()
          }()

          // 3. wg.Wait() returns ONLY after the counter reaches zero
          wg.Wait()
          
          // 4. Guaranteed: data=42 is visible here because wg.Done() 
          //    synchronized with wg.Wait().
          fmt.Println("WaitGroup Visibility:", data)
        }

      ```


  -- sync.Once: Initialization Visibility
  --
    -- The Rule 
    --
      A call to `once.Do(f)` will execute `f` exactly once. The completion of that 
      single call to `f` is guaranteed to happen-before any return from 
      `once.Do(f)`, regardless of which goroutine made the call.


      example
        ```go
          func onceOrderDemo() {
            var config string
            var once sync.Once

            setup := func() { config = "Initialized" }

            var wg sync.WaitGroup
            wg.Add(2)

            for i := 0; i < 2; i++ {
              go func() {
                defer wg.Done()
                // Only one goroutine runs setup(), but BOTH are guaranteed 
                // to see config="Initialized" when Do() returns.
                once.Do(setup)
                fmt.Println("Config State:", config)
              }()
            }
            wg.Wait()
          }

        ```


  -- Atomic Operations: Sequential Consistency (Go 1.19+)
  --
    -- The Rule (Go 1.19 Memory Model Update) 
    --
      All atomic operations behave as though they are executed in a single 
      "sequentially consistent" order across the entire program. 

    
    If the effect of atomic operation A (e.g., a Store) is observed by 
    atomic operation B (e.g., a Load), then A is synchronized-before B.


    example
      ```go
        func atomicOrderDemo() {
          var a, b int32 = 0, 0

          go func() {
            // 1. Normal write to 'a'
            a = 2
            // 2. Atomic Store to 'b' acts as a memory fence
            atomic.StoreInt32(&b, 1)
          }()

          for {
            // 3. Atomic Load of 'b'
            // If we observe b == 1, we are guaranteed to see a == 2.
            if atomic.LoadInt32(&b) == 1 {
              // 4. Result is guaranteed to be 2.
              fmt.Println("Atomic Sync Visibility (a):", a)
              break
            }
            runtime.Gosched()
          }
        }

      ```


  -- sync.Cond: Signal/Wait Coordination
  --
    While harder to define in a single sentence, `sync.Cond` relies on the 
    Mutex it wraps. 
    1. The unlocking of the mutex inside `cond.Wait()` allows other goroutines 
       to acquire the lock and modify the state.
    2. The `cond.Signal()` or `cond.Broadcast()` call unblocks the waiter.
    3. The `cond.Wait()` call only returns after re-locking the mutex, ensuring 
       the waiter sees the changes made by the signaler (provided the signaler 
       held the lock while modifying).





-- Finalizers
--
  -- The Rule 
  --
    A call to `runtime.SetFinalizer(x, f)` happens-before the finalization 
    call `f(x)` is executed by the runtime. This ensures that the setup for 
    the finalizer is always visible to the finalizer itself when the 
    object `x` is eventually garbage collected.
