-- Advanced Deferred Function Calls
--
  -- The "Discarded Result" Rule
  --
    In Go, a 'defer' statement is technically a statement that executes a
    function call but ignores all of its return values.


  -- Custom Functions
  --
    For functions you define yourself, Go allows you to call them and discard
    the results. Therefore, they can always be deferred.

  -- Built-in Functions
  --
    Most built-in functions (like 'append', 'make', 'len', 'cap') have a strict
    requirement: their return values MUST NOT be ignored.


    Because 'defer' forces results to be ignored, you cannot defer most
    built-in functions directly.





-- Exceptions: Built-ins That CAN Be Deferred
--
  There are only two built-in functions that allow their return values to be
  discarded, and thus can be deferred directly:
    1. copy()
    2. recover()


  -- The Workaround: Anonymous Function Wrappers
  --
    If you encounter a rare scenario where you need to defer a built-in like
    'append', you must wrap it in an anonymous function (closure). Inside the
    closure, you can explicitly discard the result using the blank identifier (_).


    -- Code Demonstration: Deferring Append
    --
      ```go
        func main() {
          // Initialize a slice with length 4 and capacity 4
          s := []string{"a", "b", "c", "d"}


          // -- Analysis of the first Defer

          // 1. Arguments are evaluated IMMEDIATELY.
          // 2. 's' (the slice header) is copied.
          // 3. This header points to the underlying array ["a", "b", "c", "d"].
          defer fmt.Println("Result in main defer:", s)


          // -- Analysis of the second Defer

          // We cannot do: defer append(s[:1], "x", "y") // COMPILE ERROR

          defer func() {
            // We use a sub-slice s[:1] which shares the same underlying array.
            // append() will overwrite indices 1 and 2 of that shared array.
            _ = append(s[:1], "x", "y")
            fmt.Println("Append inside defer completed.")
          }()


          fmt.Println("End of main body reach. Executing defers...")
        }

        // EXPECTED OUTPUT:
        // End of main body reach. Executing defers...
        // Append inside defer completed.
        // Result in main defer: [a x y d]

      ```





-- Defer Evaluation Timing and the Nil Function Trap
--
  -- Evaluation of the Function Value
  --
    When you write `defer f()`, the evaluation of 'f' happens at the moment the
    'defer' statement is executed (when it's pushed onto the stack).


    If 'f' is a variable holding a function, the "value" captured is the function
    that 'f' points to at that specific line. Changing the variable 'f' later
    in the function will not change which function was deferred.


    example
      ```go
        func evaluationTimingDemo() {
          var f = func() {
            fmt.Println("Result of f(): false")
          }

          // At this point, 'f' is evaluated. The specific anonymous function above
          // is pushed onto the defer stack.
          defer f()

          // Changing the variable 'f' here has NO effect on the deferred call.
          f = func() {
            fmt.Println("Result of f(): true")
          }
        }

        // EXPECTED OUTPUT:
        // Result of f(): false

      ```



  -- The Nil Function Panic: Registration vs. Execution
  --
    This is a subtle but dangerous behavior in Go.


    -- Registration Phase (The defer line)
    --
      When you call `defer f()` and 'f' is nil, the registration SUCCEEDS.
      Go simply pushes the nil function pointer onto the defer stack. The
      program continues executing normally.


    -- Execution Phase (The exit phase)
    --
      When the surrounding function starts to exit and tries to INVOKE the
      deferred nil function, it hits a runtime PANIC.


    Crucial Note: Because the capture happened at the 'defer' line, trying
    to assign a real function to 'f' later in the code will NOT save you
    from the panic.


    example
      ```go
        func nilDeferPanicDemo() {
          // 1. This defer is registered and will run last.
          defer fmt.Println("Step 3: Reachable 1 (Last defer executed)")


          // 2. We declare a nil function.
          var f func()


          // 3. REGISTRATION: The nil value of 'f' is pushed onto the stack.
          // No panic happens on this line!
          defer f()


          // 4. These lines are perfectly reachable.
          fmt.Println("Step 1: Reachable 2 (Surrounding function code)")


          // 5. Attempting to "fix" the nil pointer.
          // This is USELESS because the defer stack already holds the nil value.
          f = func() { fmt.Println("I will never run") }


          fmt.Println("Step 2: Reaching the end of the function body...")


          // <-- PANIC OCCURS HERE as the function attempts to execute the nil 'f'
        }

          // EXPECTED OUTPUT:
          // Step 1: Reachable 2 (Surrounding function code)
          // Step 2: Reaching the end of the function body...
          // Step 3: Reachable 1 (Last defer executed)
          // panic: runtime error: invalid memory address or nil pointer dereference

      ```





-- Advanced Defer - Receiver Evaluation & Best Practices
--
  Beyond basic function calls, 'defer' has specific behaviors when dealing with 
  method chains and is an essential tool for writing robust, clean, and 
  panic-safe code.


  -- Evaluation Moment of Receiver Arguments
  --
    We know that arguments to a deferred function are evaluated immediately. 
    A critical corollary is: The receiver of a method call is ALSO an argument 
    (the first implicit argument) and is therefore evaluated IMMEDIATELY.

    example
      For methods:
        defer obj.Method()

      becomes:
        defer Method(obj)

    so obj is evaluated and captured


  -- The Method Chain Trap 
  --
    If you defer a chained method call like `defer object.MethodA().MethodB()`, 
    'MethodA()' is executed IMMEDIATELY because its result is needed as the 
    receiver for 'MethodB()'. Only 'MethodB()' is actually deferred.


  example
    ```go
      type T int


      func (t T) M(n int) T {
        fmt.Print(n)
        return t
      }


      func receiverEvaluationDemo() {
        var t T
        fmt.Print("Execution Order: ")


        // 1. To push the call ".M(2)" onto the defer stack, Go must know 
        //    what the receiver is. 
        // 2. It evaluates "t.M(1)" IMMEDIATELY to get that receiver.
        // 3. Output so far: 1
        defer t.M(1).M(2)


        // 4. Standard synchronous calls: M(3) runs, then M(4) runs.
        // 5. Output so far: 1342
        t.M(3).M(4)


        // 6. Function exits. The deferred call ".M(2)" finally runs.
      }

      // EXPECTED OUTPUT:
      // Execution Order: 1342

    ```





-- Resource Management: Cleaner Code
--
  One of the primary use cases for 'defer' is ensuring that resources 
  (files, network connections, database handles) are closed regardless of 
  how a function returns (success or error).


  -- Approach A: Without Defer (Fragile) 
  --
    ```go
      func withoutDefers(filepath string, data []byte) error {
        f, err := os.Create(filepath)
        if err != nil {
          return err
        }

        _, err = f.Write(data)
        if err != nil {
          f.Close() // Must remember to close here
          return err
        }

        err = f.Sync()
        if err != nil {
          f.Close() // And here...
          return err
        }

        f.Close() // And finally here.
        return nil
      }

    ```

  -- Approach B: With Defer (Robust & Clean) 
  --
    ```go
      func withDefers(filepath string, data []byte) error {
        f, err := os.Create(filepath)
        if err != nil {
          return err
        }
        // This single line guarantees the file is closed on ANY return path.
        defer f.Close()


        _, err = f.Write(data)
        if err != nil {
          return err
        }

        return f.Sync()
      }

    ```





-- Concurrency Safety: Mutexes and Panics
--
  'defer' is not just about cleanliness; it is a safety mechanism. 
  Standard execution requires that every 'Lock()' has a matching 'Unlock()'. 
  If code panics between the lock and unlock, the mutex stays locked forever, 
  causing a deadlock.


  example
    ```go
      var mu sync.Mutex

      // -- Safe Pattern: defer mu.Unlock() 

      func safeCounter() {
        mu.Lock()
        // Even if doSomethingDangerous() panics, the defer stack will unwind 
        // and execute mu.Unlock(), saving the program from a permanent deadlock.
        defer mu.Unlock()
        
        doSomethingDangerous()
      }

      
      // -- Risky Pattern: Manual Unlock 

      func riskyCounter() {
        mu.Lock()
        doSomethingDangerous() 
        // If the line above panics, this line is NEVER reached. 
        // The mutex remains locked.
        mu.Unlock() 
      }

      func doSomethingDangerous() {
        // Logic that might panic
      }

    ```





-- Performance & Resource Considerations for `defer`
--
  While 'defer' is an essential tool for clean and safe code, it is not free.
  Understanding its overhead and how it interacts with function scope is 
  critical for writing high-performance and resource-efficient Go applications.


  -- Performance Analysis: Then vs. Now
  --
    Historically, 'defer' had a reputation for being slow because it required 
    heap allocation for the defer-call stack.


    -- Legacy Go (Pre-1.13) 
    --
      Every 'defer' incurred a measurable overhead. Developers often avoided it 
      in "hot paths" (code that runs millions of times per second).

    -- Modern Go (1.13+) 
    --
      Significant optimizations (primarily by Dan Scales) have changed the game.
      1. Go 1.13 introduced "open-coded" defers, which allows the compiler to 
         inline many defer calls, making them nearly as fast as manual calls.
      2. Go 1.14+ further improved this by reducing the need for stack-to-heap 
         transitions.
      

    Conclusion: In modern Go, the performance cost of 'defer' is negligible 
    for the vast majority of applications. Cleanliness and safety should 
    almost always take priority over manual cleanup.





-- The "Resource Leak" Trap: Defer inside Loops
--
  This is the most common production bug related to 'defer'. 
  Remember: 'defer' executes when the SURROUNDING FUNCTION returns, 
  NOT when a loop iteration ends.

  
  example
    ```go
      // -- Dangerous Pattern: Accumulating Resources
      
      // If this function processes 10,000 files, it will keep 10,000 file descriptors 
      // open until the ENTIRE function finishes. This usually leads to 
      // "too many open files" errors.

      func writeManyFiles(files []File) error {
        for _, file := range files {
          f, err := os.Open(file.path)
          if err != nil {
            return err
          }
          // DANGER: This does NOT run at the end of the loop iteration.
          // It stacks up until the whole writeManyFiles function returns!
          defer f.Close() 

          _, err = f.WriteString(file.content)
          if err != nil {
            return err
          }
        }
        return nil
      }

    ```


  -- The Solution: Function Scoping (The Wrapper Pattern)
  --
    To ensure resources are released immediately at the end of every loop 
    iteration, we must wrap the loop body in an anonymous function. 


    example
      ```go
        type File struct {
          path    string
          content string
        }


        // writeManyFilesCorrected demonstrates the safe way to use defer in a loop.
        func writeManyFilesCorrected(files []File) error {

          for _, file := range files {
            // We wrap the iteration in an IIFE (Immediately Invoked Function Expression).
            // This creates a NEW function scope for every iteration.
            err := func() error {
              f, err := os.Create(file.path)
              if err != nil {
                return err
              }
              
              // SUCCESS: Because this defer is inside the anonymous function,
              // it executes as soon as the anonymous function returns (at the 
              // end of this specific loop iteration).
              defer f.Close()

              fmt.Printf("Writing to %s...\n", file.path)
              _, err = f.WriteString(file.content)
              return err
            }()


            if err != nil {
              return err
            }
          }
          return nil
        }

      ```
