-- typed and untyped values
--
  Understanding the distinction between "untyped" and "typed" values, especially
  regarding constants, is fundamental to writing clean and idiomatic Go code.





-- The Core Idea: Untyped Constants
--
  An untyped value, is a high-precision value that is not yet committed to a
  specific Go type. In Go, these are almost exclusively constants


  All literal constants (`123`, `3.14`, `"hello"`, `true`) are untyped values.
  The predeclared identifiers `true` and `false` are named boolean constants.


  While untyped, every constant has a default type that the compiler will
  use if the context doesn't provide a more specific type.
    - `"a string"`      -> default type `string`
    - `true` / `false`  -> default type `bool`
    - `123`             -> default type `int`
    - `'A'`             -> default type `rune` (an alias for `int32`)
    - `3.14`            -> default type `float64`
    - `3 + 4i`          -> default type `complex128`

  
  -- special case - nil
  --
    The predeclared identifier `nil` is the only untyped value
    that has no default type.





-- From Untyped to Typed: The Conversion Process
--
  An untyped constant must become a typed value before it can be stored in
  a variable or used at runtime. This "typing" happens in one of two ways:

  1. Implicitly, through type deduction from the context.
  2. Explicitly, through a type conversion `T(v)`.


  -- Implicit Conversion (Type Deduction)
  --
    When an untyped constant is used in a context that demands a specific type (like
    an assignment to a typed variable), the compiler will implicitly "type" the
    constant if its value is representable by the target type.

    If there is no specific type required by the context, the constant is given
    its default type.


    example
      ```go
        var f float32 = 3.14 // OK: `3.14` is an untyped float, representable by `float32`.
        var i int = 123      // OK: `123` is an untyped integer, representable by `int`.
        // var u uint8 = -1   // ERROR: `-1` is untyped, but NOT representable by `uint8`.


        var x = 3.14 // No type is specified, so `x` gets the default type of `3.14`, which is `float64`.
        var y = 10   // No type is specified, so `y` gets the default type of `10`, which is `int`.

      ```


  -- Explicit Conversion (`T(v)`)
  --
    You can explicitly convert an untyped constant `v` to a type `T`.
        

    both T(v) and (T)(v) are valid


    -- Rule A: Representability for Numeric Types
    --
      The conversion `T(v)` is legal if the constant value `v` is representable
      by the basic type `T`. Rounding is allowed for floating-point and complex types.


      example 
        ```go
          var f1 = float32(0.499999999) // OK -> becomes typed float32 value `0.5` (rounding allowed)
          var u1 = uint(1.0)           // OK -> becomes typed uint value `1`
          // var i1 = int(1.23)           // ERROR: `1.23` is not representable as an `int` (no rounding)
          // var f2 = float64(1e1000)     // ERROR: value `1e1000` overflows `float64`

        ```


    -- Rule B: Special Case for Integer-to-String Conversion
    --
      If the constant's default type is an integer type (`int` or `rune`), it can be
      converted to a string. The result is a string containing the UTF-8 representation
      of that integer code point.


      example 
        ```go
          var s1 = string(65)         // "A"
          var s2 = string('A')        // "A"
          var s3 = string(-1)         // "\uFFFD" (Unicode replacement character)
          // var s4 = string(65.0)     // ERROR: Default type is `float64`, not an integer type.

        ```





-- The Flexibility of Untyped Constants: "Overflowing" the Default Type
--
  this is a crucial concept, an untyped constant can hold a value that is
  too large for its default type, as long as it is eventually used in a context
  where the final result fits into a valid typed value.


  example
    ```go
      // These are VALID untyped constants, even though they overflow their default types.
      const n = 1 << 64          // Overflows `int` (on a 64-bit system)
      const x = 2e+308           // Overflows `float64`

      func overflowExample() {
        // The constant `n` is untyped. In this expression, the operation `n >> 2`
        // is performed with high precision at compile time. The result is `1 << 62`,
        // which *does* fit into a variable of type `uint64`.
        var a uint64 = n >> 2
        fmt.Println(a)

        // In contrast, a **typed** constant CANNOT overflow its type.
        // const typed_n int = 1 << 64 // COMPILE ERROR: constant overflows int
      }

    ```





-- Conversions for Typed Values (Non-Constants)
--
  Conversions for typed, non-constant values follow different rules, especially
  regarding overflow and precision loss.

  - Overflow: Allowed for integer types. The value is truncated (it "wraps around").
  - Precision Loss: Allowed. Converting a float to an integer discards the
    fractional part (truncates toward zero).


  example
    ```go
      var non_const_float float64 = -1.99
      // var z1 int32 = non_const_float // ERROR: Cannot assign float64 to int32 directly.


      // OK: Explicit conversion is required. The fractional part is discarded.
      var z2 int32 = int32(non_const_float) // z2 becomes -1


      var non_const_int int16 = 256
      // OK: `non_const_int` (256) overflows the max value of `uint8` (255).
      // The value is truncated. 256 in binary is `...0001 0000 0000`. The lower
      // 8 bits are all zero.
      var h uint8 = uint8(non_const_int) // h becomes 0

    ```





-- The Compile-Time Nature of Constants
--
  All constants, named or unnamed, are resolved at compile time. The compiler
  replaces every occurrence of a constant identifier with its literal value and
  evaluates constant expressions.


  example
    // This code at design time:
      ```go
        const X = 3
        const Y = X * 2
        var a = X

        func main() {
            b := Y
            println(a, b, X, Y) // 3 6 3 6
        }

      ```

    // Is treated by the compiler like this:
      ```go
        var a = 3 // `a` is a variable initialized at runtime

        func main() {
            b := 6 // `b` is a variable initialized at runtime
            println(a, b, 3, 6)
        }

      ```





example - tricky error 
  ```go
    // error: -1 overflows uint
    const MaxUint_b uint = ^0 
    // the reason ^0 gives -1, even though the consxt is uint(...)
    // its because the default type of a number is int, and since ^0 is an operation
    // it will use the default value of int (signed) which leads to -1, which not convertible
    // to uint
  ```
