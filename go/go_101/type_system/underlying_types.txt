-- underlying type
--
  Every type in Go has an underlying type. You can think of it as the
  fundamental blueprint or structure that the type is built upon. It determines
  the memory layout and the intrinsic properties of the type (e.g., a type with
  an underlying `int` supports arithmetic).





-- How to Find the Underlying Type
--
  Finding a type's underlying type involves a simple "tracing" process.

  The Golden Rule of Tracing:
    You trace through type definitions (`type NewType OldType`) until you reach
    a predeclared (built-in) type OR an unnamed type literal. That final type is
    the underlying type.

  This breaks down into three simple points:
    1. Predeclared types (`int`, `string`, `bool`, etc.) are their own
       underlying types. They are the end of the line.
    2. Unnamed type literals (`[]int`, `map[string]bool`, `struct{...}`, etc.)
       are their own underlying types. They are also the end of the line.
    3. A defined type shares the same underlying type as its source type in
       the definition.


  -- example - Tracing Examples in Detail
  --
    ```go
      type (
        MyInt int
        Age   MyInt
      )


      type (
        IntSlice   []int   // underlying type is []int
        MyIntSlice []MyInt // underlying type is []MyInt
        AgeSlice   []Age   // underlying type is []Age
      )


      type Ages AgeSlice



    // Example Trace 1: `Age` 
    //   - We start with `Age`. Its source type is `MyInt`.
    //   - We look at `MyInt`. Its source type is `int`.
    //   - We look at `int`. This is a predeclared type. The trace stops.
    //   - Result: The underlying type of `Age` (and `MyInt`) is `int`.
    //   - `Age -> MyInt -> int`


    // Example Trace 2: `MyIntSlice`
    //   - We start with `MyIntSlice`. Its source type is `[]MyInt`.
    //   - We look at `[]MyInt`. This is an unnamed type literal (a slice type). The trace stops.
    //   - Result: The underlying type of `MyIntSlice` is `[]MyInt`.
    //   - It is NOT `[]int`, because the trace stops as soon as it hits the unnamed `[]MyInt` literal.
    //   - `MyIntSlice -> []MyInt` 


    // Example Trace 3: `Ages` 
    //   - We start with `Ages`. Its source type is `AgeSlice`.
    //   - We look at `AgeSlice`. Its source type is `[]Age`.
    //   - We look at `[]Age`. This is an unnamed type literal (a slice type). The trace stops.
    //   - Result: The underlying type of `Ages` (and `AgeSlice`) is `[]Age`.
    //   - `Ages -> AgeSlice -> []Age` 

    ``` 





-- Why Underlying Types Matter: CONVERSION
--
  The concept of underlying types is most critical for understanding Go's
  rules on type conversion.


  The Rule of Conversion:
    A value of type `A` can be explicitly converted to type `B` if they share the
    same underlying type


  example
    ```go
      type MyInt int
      type Age MyInt

      func main() {
        var i MyInt = 20
        var j Age = 30

        // `i` and `j` are different types, so direct assignment is illegal.
        // i = j // COMPILE ERROR: cannot use j (type Age) as type MyInt in assignment

        // However, because both `MyInt` and `Age` share the same underlying
        // type (`int`), we can explicitly convert between them.
        i = MyInt(j)
        j = Age(i)

        fmt.Println(i, j) // Output: 30 30
      }

    ```





-- Common Type Categories
--
  Go often groups types into families based on their underlying type.
    - Boolean types: Any type whose underlying type is `bool`.
    - Integer types: Any type whose underlying type is one of the built-in integer types.
    - Floating-point types: Any type whose underlying type is `float32` or `float64`.
    - Complex types: Any type whose underlying type is `complex64` or `complex128`.
    - String types: Any type whose underlying type is `string`.
    - Numeric types: A collective term for integer, floating-point, and complex types
