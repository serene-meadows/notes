-- channel type
--
  Channels are a core feature of Go's concurrency model. They provide a typed
  conduit through which you can send and receive values between goroutines,
  acting like synchronized FIFO (First-In, First-Out) queues.

  
  The type of value that can be sent over a channel is determined by its
  element type, for example, `chan int` or `chan string`.


  A key feature of channel types is their direction, which specifies whether a
  channel can be used for sending, receiving, or both.





-- bidirectional channel (`chan T`)
--
  This is the most common channel type. A bidirectional channel can be used for 
  both sending and receiving values.

  `chan T` where `T` is the element type.

  example
    ```go
      var ch chan int // A bidirectional channel of integers

    ```





-- directional channel
--
  Directional channels restrict operations to either sending or receiving. They
  are extremely useful for improving the type safety of a program. By using
  directional channels in function signatures, you can clearly define the
  intended role of a goroutine, preventing accidental misuse (e.g., a receiver
  goroutine accidentally sending a value).


  -- send only channel (`chan<- T`)
  --
    A send-only channel can only be used to send values. You cannot receive from it.
    Mnemonic: The arrow `<-` points INTO the `chan` keyword.

    `chan<- T`


    example
      ```go
        var sendOnlyChan chan<- string // A channel you can only send strings into.


        val := <-sendOnlyChan   // COMPILE ERROR: invalid operation (receive from send-only type)

      ```


  -- receive only channel (`<-chan T`)
  --
    A receive-only channel can only be used to receive values. You cannot send to it.
    Mnemonic: The arrow `<-` points OUT of the `chan` keyword.

    `<-chan T`


    example 
      ```go
        val := <-recvOnlyChan 


        recvOnlyChan <- true  // COMPILE ERROR: invalid operation (send to receive-only type)

      ```


  -- Conversions and Practical Usage
  --
    The power of directional channels comes from this rule:
    A bidirectional channel (`chan T`) can be implicitly converted to either a
    send-only (`chan<- T`) or receive-only (`<-chan T`) channel, but not the
    other way around.


    This allows you to create a general-purpose channel and pass it to functions
    that operate on it in a more restricted, safer way.


    // This example shows a "ping-pong" exchange between two goroutines, where
    // directional channels enforce their roles.
    example
      ```go
        // `pinger` only needs to SEND a message. Its role is enforced by the
        // `chan<- string` type. It cannot receive from this channel.
        func pinger(pings chan<- string, msg string) {
          fmt.Println("Pinger sending:", msg)
          pings <- msg
        }

        // `ponger` needs to RECEIVE the ping and then SEND the pong.
        // Its roles are enforced by a receive-only and a send-only channel.
        func ponger(pings <-chan string, pongs chan<- string) {
          // Receive a message from the `pings` channel.
          msg := <-pings
          fmt.Println("Ponger received:", msg)

          // Send a reply on the `pongs` channel.
          fmt.Println("Ponger sending: pong")
          pongs <- "pong"
        }

        func main() {
          // Create two BIDIRECTIONAL channels.
          pings := make(chan string, 1)
          pongs := make(chan string, 1)

          // Pass the bidirectional `pings` channel to the pinger.
          // It is implicitly converted to a `chan<- string` here.
          go pinger(pings, "ping")

          // Pass both bidirectional channels to the ponger.
          // `pings` is converted to `<-chan string`.
          // `pongs` is converted to `chan<- string`.
          go ponger(pings, pongs)

          // Receive the final message from the `pongs` channel in the main function.
          finalMsg := <-pongs
          fmt.Println("Main received:", finalMsg)

          // Give goroutines time to print before main exits
          time.Sleep(10 * time.Millisecond)
        }

      ```
