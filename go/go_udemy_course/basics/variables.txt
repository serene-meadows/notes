Variables are named values.
Variables are stored in memory at run time.
The value represented by a variable can be modified at run time.

All variables are typed values.
When declaring a variable,
there must be sufficient information provided for compilers to deduce the type of the variable.

The variables declared within function bodies are called local variables.
The variables declared out of any function body are called package-level variables.
  We also often call package-level variables as global variables.



package level 
All package-level variables declared in a package are initialized before any init function declared in the same package is invoked.
Go runtime will try to initialize package-level variables in a package by their declaration order,
but 
a package-level variable will be initialized after all of its depended variables 
examle
  ```go
    // The initializations the four package-level variables happen in the order y, z, x, and w. 

    func f() int {
      return z + y
    }

    func g() int {
      return y/2
    }

    var (
      w       = x
      x, y, z = f(), 123, g()
    )
  ```


There are two basic variable declaration forms,
  the standard one
  and the short one. (can only be used inside functions)

standard declaration
  Each standard declaration starts with the var keyword
  which is followed by the declared variable name

  // this method is pretty verbose
  ```go
    var announceYear int = 2009
    // in this method, the types of the multiple variables declared

    // in the same declaration line must be identical
    var lang, website string = "Go", "https://golang.org"
    var compiled, dynamic bool = true, false
  ```

  // let compiler deduce type of variable
  // i.e. implicit conversion
  ```go
     // using this method
     // the types of the multiple variables declared in the same declaration line can be different.


    // The types of the lang and dynamic variables
    // will be deduced as built-in types "string"
    // and "bool" by compilers, respectively.
    var lang, dynamic = "Go", false

    // The types of the compiled and announceYear
    // variables will be deduced as built-in
    // types "bool" and "int", respectively.
    var compiled, announceYear = true, 2009

    // The types of the website variable will be
    // deduced as the built-in type "string".
    var website = "https://golang.org"
  ```

    // Both are initialized as blank strings.

short declaration
  short declaration can only be used to declare local variables (within functions)
  it cannot be used to declare package variables

  > must ommit the var keyword and the type
  > it uses the := syntax
  > both old and new variables can be used to the left side
    but if any one of them is a new variable you have to use :=

  ```go
    // Both lang and year are newly declared.
    lang, year := "Go language", 2007

    // Only createdBy is a new declared variable.
    // The year variable has already been
    // declared before, so here its value is just
    // modified, or we can say it is redeclared.
    year, createdBy := 2009, "Google Research"

    // This is a pure assignment.
    // a pure assignment means assigning values to already-declared variables without declaring new ones.
    lang, year = "Go", 2012
  ```


assignment
  In variable declarations, the sign = means assignment.

  The expression items at the left of = symbol in a pure assignment
  must be addressable values, map index expressions, or the blank identifier.
        // addressable values
        an addressable value is a value that has a stable memory location.

        because it has a memory location,
        the program can store a new value there.

        "addressable" means:
          the address of the value can be obtained using the address operator (&).

          example
            x := 10
            x = 20
            p := &x

            // &x means, to get the memory address where x is stored
            // so x is addressable

          another example
            // slice
            s := []int{1, 2, 3}
            s[0] = 10
            p := &s[0]
            // &s[0] refers to a memory address inside the slice's backing array
            // s[0] is addressable

          another example
            // struct field
            type User struct {
              Age int
            }
            u := User{Age:20}
            u.Age = 25
            p := &u.Age
            // &u.Age referes to memory address of the field
            // u.Age is addressable

          another example
            // pointer dereference
            x := 10
            p := &x
            *p = 20
            // *p refers to the memory address stored at p
            // *p is also addressable

        examples of "non addressable"
          some expressions produce values but do not refer to a stable memory location.
          those values are not addressable.

          example
            10 = x
            // invalid because literal values do not have assignable memory locations

          another example
            // function return
            func get() int { return 1 }
            get() = 10
            // invalid because the return value of a function is a temporary value, not a named
            // memory location

          another example
            map elements are not addressable
            // however we can use them in assignemnts because as mentioned it's allowed
            m := map[string]int{}
            m["a"] = 1   // allowed
            // but
            p := &m[a]   // not allowed, since it's not addressable
                         // reason
                         // map implementations may move elements in memory when the
                         // memory grows (during rehashing or resizing)
                         // therefor the memory address of a map element is not stable
                         // so map[key] is not addressable

        map index vs slice index (why one of them is not addressable)
          slice elements are addressable
          map elements are not addressable

          even though both of them can grow dynamically

          slice (internal representation)
            pointer to an array
            length
            capacity

            s[i] refers to an element in the backking array, array elements live in
            stable memory locations

            when a slice grows using append
            the runtime may allocate a NEW array
            ```go
              s := []int{1,2,3}
              p := &s[0]
              s = append(s, 4,5,6)
            ```
              // possible case
              // old array -> 1 2 3
              // new array -> 1 2 3 4 5 6
              // but
              // the pointer p still points to the old array
              // the pointer s points to the new array
              // so the memory locations are stable

          maps (internal representation)
            maps are implemented using hash tables
            they contain
              buckets
              hash tables
              metadata

            elements are stored inside buckets

            when the map grows elements may have to be moved between buckets
            thus, because their memory locations are not stable
            they are not addressable


  ```go
    const N = 123
    var x int
    var y, z float32

    N = 9 // error: constant N is not modifiable
    y = N // ok: N is deduced as a float32 value
    x = y // error: type mismatch
    x = N // ok: N is deduced as an int value
    y = x // error: type mismatch
    z = y // ok
    _ = y // ok

    x, y = y, x // error: type mismatch
    x, y = int(y), float32(x) // ok
    z, y = y, z               // ok
    _, y = y, z               // ok
    z, _ = y, z               // ok
    _, _ = y, z               // ok
    x, y = 69, 1.23           // ok
  ```

// Go does NOT do implicit type conversion.
// you need to do it explicitly
// If types differ → you must convert explicitly.
//   this doesn't apply to un typed consts because well they are "untyped" until they are in a scenario which requires a type
//   and it implicitly converts to the expected type, it should still be similar tho, cannot assign numeric unsigned typed const to a string
//   however for typed consts, you need to explicitly ocnvert to another type if need so
  ```go
    // In Go, values of different basic types cannot be assigned to each other.
      var a int = 10
      var b float64 = 20.5
      a = b // error

    // You must explicitly convert the source value to the destination type.
      var a int
      var b float64 = 20.5

      a = int(b) // OK

    // The same rule applies to operations between different types.
    var x int = 10
    var y float64 = 5.5

    z := x + y // error 

    // Convert one value so both operands have the same type.

    var x int = 10
    var y float64 = 5.5

    z := float64(x) + y // OK

    // tldr
    
    // Key rule in Go:
    // Basic types must match for:
      // 1. Assignments
      // 2. Arithmetic operations
      // 3. Comparisons
  ```








// Non-constant numeric conversions in Go behave differently from constants.
  Overflow is allowed when converting NON-constant values. It truncates the value
  // const would have given err
    var x int16 = 130
    var y int8

    y = int8(x)      
    // result overflows because int8 range is -128..127
    // value wraps around (implementation-defined but allowed)

  Converting a floating-point value to an integer may round by discarding the fraction.
  // rouding is towards zero
    var f float64 = 8.9
    var ff float64 = -8.9
    var i int

    i = int(f)      
    // i becomes 8
    i = int(ff) 
    // i become -8

  This rule applies only to NON-constant values.
    // Constants are checked strictly at compile time.
    const f = 300.0
    var x int8 = int8(f)  // error

  examples
    ```go
      const a = -1.23
      // The type of b is deduced as float64.
      var b = a
      // error: constant 1.23 truncated to integer.
      var x = int32(a)
      // error: cannot assign float64 to int32.
      var y int32 = b
      // okay: z == -1, and the type of z is int32.
      //       The fraction part of b is discarded.
      var z = int32(b)

      const k int16 = 255
      // The type of n is deduced as int16.
      var n = k
      // error: constant 256 overflows uint8.
      var f = uint8(k + 1)
      // error: cannot assign int16 to uint8.
      var g uint8 = n + 1
      // okay: h == 0, and the type of h is uint8.
      //       n+1 overflows uint8 and is truncated.
      var h = uint8(n + 1)
    ```



code block using { }  
  In Go, we can use a pair of { and } to form a code block. 
  A code block can nest other code blocks.

  A variable or a named constant declared in an inner code block 
  will shadow the variables and constants declared with the same name in outer code blocks.

  so inner var/ const shadow the outer var/ const

  The scope (visibility range in code) of a package-level variable (or a named constant) 
  is the whole package of the variable (or the named constant) is declared in.

  The scope of a local variable (or a named constant) 
  begins at the end of its declaration and ends at the end of its innermost containing code block.

  ```go
    const y = 789
    var x int = 123

    func main() {
      // The x variable shadows the above declared
      // package-level variable x.
      var x = true

      // A nested code block.
      {
        // Here, the left x and y are both
        // new declared variable. The right
        // ones are declared in outer blocks.
        x, y := x, y

        // In this code block, the just new
        // declared x and y shadow the outer
        // declared same-name identifiers.
        x, z := !x, y/10 // only z is new declared
        y /= 100
        println(x, y, z) // false 7 78
      }
      println(x) // true
      println(z) // error: z is undefined.
    }
  ```
