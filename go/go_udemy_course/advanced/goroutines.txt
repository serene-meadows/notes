go routine
  go routines are lightweight threads managed by the go runtime 

  Goroutines are also often called green threads. (since they are maintained and sceduled by the runtime instead of the OS)
  The cost of memory consumption and context switching, of a goroutine is much lesser than an OS thread.
  So, it is not a problem for a Go program to maintain tens of thousands goroutines at the same time (as long as memory is sufficient)

  Go doesn't support the creation of system threads in user code. So, using goroutines is the only way to do (program scope) concurrent programming in Go.



they enable concurrent execution of functions allowing you to perform multiple tasks concurrently within a single go program, it is non-blocking
  without us neededing the manage the threads

  Go encourages concurrent design, But whether it becomes parallel depends on:
    go runtime, number of cores

  concurrency vs parallelism 
    Concurrency = dealing with many tasks at once
    Parallelism = doing many tasks at the exact same time

    Concurrency means a program is structured to handle multiple tasks that overlap in time, but they might not run simultaneously.
      The system switches between tasks.
      example 
        Task A
        Task B
        Execution timeline:
          time →
            A A A
                  B B
                       A A
                            B
        Only one task runs at a time, but the program switches between them.
        This is called interleaving.

    Parallelism means tasks are literally running at the same time.
      This requires multiple CPU cores.

      Example:
        Task A → Core 1
        Task B → Core 2

        Execution timeline:
          time →
            A A A A
            B B B B

        Both tasks execute simultaneously.









go routine scheduling 
  Not all goroutines in running state are being executed at a given time.
  At any given time, the maximum number of goroutines being executed will not exceed the number of logical CPUs available for the current program

  We can call the runtime.NumCPU function to get the number of logical CPUs available for the current program
  each logical CPU can execute only one goroutine at any given time 

  Go runtime must frequently switch execution contexts between goroutines to let each running goroutine have a chance to execute.
  This is similar to how operating systems switch execution contexts between OS threads.


  go routine states
    create -> (running [executing <-> queued] <-> blocking) -> exit

    note, a go routine is considered to be running even after calling (time.Sleep)

  A blocking goroutine can only be unblocked by an operation made in another goroutine.
  if all go routine are in blocked state -> deadlock



  A goroutine in the queuing sub-state is waiting to be executed.
  A goroutine in the executing sub-state may enter the queuing sub-state again when it has been executed for a while (a very small piece of time).







The standard Go runtime adopts the M-P-G model to do the goroutine schedule job
  where 
    M represents the OS threads
    P represents the logical/virtual processors (not logical CPUs)
    G represents goroutines

  Most schedule work is made by logical processors (Ps)
    which act as brokers by attaching goroutines (Gs) to OS threads (Ms)

  Each OS thread can only be attached to at most one goroutine at any given time
  and each goroutine can only be attached to at most one OS thread at any given time


  A goroutine can only get executed when it is attached to an OS thread
  A goroutine which has been executed for a while will try to detach itself from the corresponding OS thread, 
  so that other running goroutines can have a chance to get attached and executed.


  At runtime. we can call the runtime.GOMAXPROCS function
    to get and set the number of logical processors (Ps).


  The default initial value of runtime.GOMAXPROCS
  can also be set through the GOMAXPROCS environment variable.

  At any time, the number of goroutines in the executing sub-state is no more than the smaller one of runtime.NumCPU and runtime.GOMAXPROCS


    
        


you cannot capture a return value from go routines
  if the function has a return value, it is discarded






Each Go program starts with only one goroutine (main)
  When a Go program starts, the runtime creates the main goroutine which runs the main() function.
    main goroutine → runs main()

  When we write:
    go sayHello()
  Go starts a new goroutine that will execute sayHello() concurrently.
  Now two goroutines exist:
    main goroutine
    sayHello goroutine
  However, the Go scheduler decides when each goroutine runs.

  After starting the goroutine, the main() function keeps running.

  In Go, when main() returns:
    the entire program exits
      When the program exits:
        all goroutines are terminated 
        all program memory is released by the OS

  so if main thread finishes before the goroutine is finished, the goroutine is also terminated
  
  

goroutine scheduling in go
  managed by the go runtime scheduler 

  uses M:N scheduling model
    goroutines are not OS threads
    a goroutine is a lightweight user-space thread managed by the Go runtime.

    M:N means Goroutine : os threads
      many goroutines : fewer OS threads

  
declaration   
  go keyword 
    add the go keyword preceding the function and execute that function immediately in the main function

````go
  package main 

  func main() {
    // sayHello() // waits for 1 second and prints
    go sayHello() // no output on the screen
        // the reason nothing was printed is because
          // After starting the goroutine, the main() function keeps running.
          // But in our program, main() has nothing else to do, so it reaches the end immediately.

             // In Go, when main() returns:
             //   the entire program exits
             //     When the program exits:
             //       all goroutines are terminated
             //       all program memory is released by the OS

         // So the sayHello goroutine never gets a chance to finish.
         // That is why no output appears.

        // a very impractical sol'n for now would be to time sleep for 2 seconds in main after go routine call
  }

  func sayHello() {
    time.Sleep(1 * time.Second)
    fmt.Println("hello from Goroutine")
  }
````


error handling in go routines 
  go routines execute functions concurrently 
  errors need to be communicated back to them main thread

  use 
    return values, shared error variables, channels 

````go
  func doWork() error {
    // work 

    return fmt.Errorf("an error occured T_T")
  }

  func main() {
    var err error 
    err = doWork() // works - normal function


    err = go doWork() // doesn't work
    // workaround 
      // not really safe, creates race condition
      // here race condition is between the goroutine and the main thread
      // suppose in the next lines, we check err, it may or may not be nil even if goroutine returns error
      // because of race condition
    go func() {
      err = doWork()
    }()
  }
````
