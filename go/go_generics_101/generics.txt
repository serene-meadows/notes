-- Custom Generics (Type Parameters)
--
  Generics, officially known as "Type Parameters," were introduced in Go 1.18.
  They represent one of the most significant changes to the language since 
  its inception, providing a formal mechanism for writing code that is 
  independent of the specific types being used.





-- The Core Purpose: Reusability and Type Safety
--
  The primary goals of custom generics are:
  1. Avoid Code Repetition: Write a logic once (e.g., a "Find" function) 
     instead of writing separate versions for int, string, and float.
  2. Type Safety: Unlike using 'any' (interface{}), generics allow the 
     compiler to verify types at compile time, eliminating the need for 
     risky runtime type assertions.
  3. Cleaner APIs: Simplifies signatures by removing redundant type-specific 
     function names (e.g., SumInts, SumFloats).
  4. Performance: In some cases, generics can be faster than interfaces 
     because they allow the compiler to perform "Monomorphization" (generating 
     optimized concrete code for each used type).






-- The Pre-1.18 Landscape: How We Coped
--
  Before Go 1.18, developers relied on three primary workarounds to achieve 
  generic-like behavior:


  -- Built-in Generics 
  --
    Go has always had built-in generics for its most fundamental types:
    - Containers: maps, channels, and slices are inherently generic.
    - Functions: `make`, `new`, `len`, `cap`, `append`, and `close` work on 
      multiple types.


  -- Reflection (interface{}) 
  --
    Using `interface{}` (now `any`) allowed functions to accept any type.
    DRAWBACK: Lost type safety and required expensive runtime assertions/reflection.


  -- Code Generation 
  --
    Using tools like `//go:generate` to automatically "copy-paste" code for 
    different types before compilation.
    DRAWBACK: Increased build complexity and "bloated" source files.





-- Code Demonstration: The "Pains" vs. The Solution
--
  ```go
    // -----------------------------------------------------------------------------
    // -- The Old Way: Code Repetition --
    // -----------------------------------------------------------------------------
    func SumInts(m map[string]int64) int64 {
      var s int64
      for _, v := range m { s += v }
      return s
    }

    func SumFloats(m map[string]float64) float64 {
      var s float64
      for _, v := range m { s += v }
      return s
    }

    // -----------------------------------------------------------------------------
    // -- The Generic Way: Single Implementation --
    // -----------------------------------------------------------------------------
    // This single function replaces both versions above. 
    // [K comparable, V int64 | float64] are the Type Parameters.
    func SumValues[K comparable, V int64 | float64](m map[K]V) V {
      var s V
      for _, v := range m {
        s += v
      }
      return s
    }

    func main() {
      ints := map[string]int64{"first": 34, "second": 12}
      floats := map[string]float64{"first": 35.98, "second": 26.99}

      // The compiler "instantiates" the function with the correct type automatically.
      fmt.Printf("Generic Sum (Ints):   %v\n", SumValues(ints))
      fmt.Printf("Generic Sum (Floats): %v\n", SumValues(floats))
    }

  ```

    -- EXPECTED OUTPUT:
    --
      Generic Sum (Ints):   46
      Generic Sum (Floats): 62.97
                                                                                
  


                                                                                
-- The Implementation: Type Parameters Proposal
--
  Go's team settled on the "Type Parameters" proposal, which uses square 
  brackets `[]` to define generic types.

  -- Key Design Philosophies --
  1. Orthogonality: Generics should work with Go's existing features 
     (methods, interfaces, etc.) without creating a "new language."
  2. Backward Compatibility: Code written in Go 1.0 still compiles 
     perfectly alongside generic code.
  3. Performance: The design avoids the "boxing" overhead typically 
     associated with interface-based polymorphism.





-- Evolution and Current Status (Go 1.25)
--
  - Generics are NOT a "ceiling": The current implementation is a robust 
    foundation, but the Go team explicitly stated that more features may 
    be added as real-world usage patterns emerge.
  - Ongoing Improvements: Since 1.18, the compiler's ability to infer types 
    and optimize the generated machine code has improved in every version.
  - Limitations: Some advanced generic patterns (like "Generic Methods" on 
    non-generic structs) are still not supported to keep the implementation 
    complexities manageable.





-- The Generic Type Blueprint
--
  A generic type is a "defined type" that includes a Type Parameter List. 
  This list acts as a placeholder for actual types that will be provided 
  when the type is used (instantiated).


  example
    ```go
      // Lockable is a Generic Type.
      // [T any] is the Type Parameter List.
      // T is the Type Parameter (the placeholder name).
      // any is the Type Constraint (the rule for what types can replace T).
      type Lockable[T any] struct {
        sync.Mutex
        Data T
      }

      func genericsBasicsDemo() {
        // -------------------------------------------------------------------------
        // -- Instantiation --
        // -------------------------------------------------------------------------
        // A generic type cannot be used directly as a value type. It must be 
        // "instantiated" by providing a Type Argument List.

        // Case 1: Instantiating with uint32
        // Lockable[uint32] is now a NAMED type and an ORDINARY type.
        var n Lockable[uint32]
        n.Lock()
        n.Data = 10
        n.Data++
        n.Unlock()
        fmt.Printf("uint32 Data: %v\n", n.Data)

        // Case 2: Instantiating with float64
        var f Lockable[float64]
        f.Lock()
        f.Data = 3.14
        f.Data += 1.23
        f.Unlock()
        fmt.Printf("float64 Data: %v\n", f.Data)

        // Case 3: Instantiating with a Slice
        var bs Lockable[[]byte]
        bs.Lock()
        bs.Data = append(bs.Data, "Go"...)
        bs.Unlock()
        fmt.Printf("[]byte Data:  %s\n", bs.Data)
      }

    ```

    -- EXPECTED OUTPUT:
    --
      uint32 Data: 11
      float64 Data: 4.37
      []byte Data:  Go
                                                                                
  



                                                                                
-- Key Components: Parameters, Arguments, and Constraints
--
  -- Type Parameters ([T any]) 
  --
    - Defined in square brackets `[]`.
    - The name (e.g., `T`) behaves like a type within the scope of the declaration.
    - Multiple parameters are comma-separated: `[T any, K comparable]`.


  -- Type Constraints (`any`) 
  --
    - A constraint is effectively a "type of a type."
    - All constraints are actually INTERFACE types.
    - `any`: A new predeclared identifier in Go 1.18. It is an alias for the 
      blank interface `interface{}`. Since every type implements the blank 
      interface, `any` allows the type parameter to be replaced by any type.


  -- Type Arguments ([uint32]) 
  --
    - The concrete types passed to the generic type during instantiation.
    - A type argument MUST implement the constraint of its corresponding parameter.






-- Type Categories (Post-Go 1.18)
--
  Since the introduction of generics, Go types are categorized into two groups:

  1. Type Parameter Types: 
     Placeholders (like 'T') used inside generic declarations.

  2. Ordinary Types: 
     Standard types (int, string, bool) AND "Instantiated Types" (Lockable[int]).
     Every fully instantiated generic type is considered a named ordinary type.





-- Methods and Code Reusability
--
  Go custom generics drastically reduce boilerplate code. This document 
  compares the "Before and After" of a common concurrency pattern and explores 
  how to attach methods to generic types.


  -- The Cost of Non-Generic Code (Boilerplate)
  --
    Without generics, if you need a "thread-safe" version of several different 
    data types, you are forced to declare a unique struct for every single one.


    example
      ```go
        --- NON-GENERIC REPETITION ---

        type LockableUint32 struct {
          sync.Mutex
          Data uint32
        }

        type LockableFloat64 struct {
          sync.Mutex
          Data float64
        }

        type LockableBool struct {
          sync.Mutex
          Data bool
        }

        // ... and so on. Every new type requires a new struct declaration.

      ```


  



-- The Generic Solution: Encapsulation with Methods
--
  A better design pattern involves hiding the data and the mutex behind 
  unexported fields and providing a method to interact with the data safely.


  ```go
    // Lockable is a generic struct. It wraps a mutex around data of type T.
    type Lockable[T any] struct {
      mu   sync.Mutex
      data T
    }

  ```

  -- Declaring Methods on Generic Types 
  --
    When declaring a method for a generic type, the receiver must include 
    the type parameter list.

    
    func (l *Lockable[T]) Do(f func(*T))
                     ▲
              Type Parameter List
    

  ```

  -- KEY RULES:
  --
    1. Omit Constraints: Unlike the struct definition [T any], the method 
       receiver MUST omit the constraint. It is [T], not [T any].
    2. Matching Parameters: The [T] refers to the type parameter used in the 
       base type.
    3. Thread Safety: By wrapping the lock/unlock logic inside 'Do', we ensure 
       users of this type cannot forget to unlock the mutex.


  ```go
    func (l *Lockable[T]) Do(f func(*T)) {
      l.mu.Lock()
      defer l.mu.Unlock()
      // We pass a pointer to the internal data to the callback function.
      f(&l.data)
    }

  ```


-- Flexibility in Naming 
--
  While it is best practice to keep names consistent, the type parameter 
  name in the method does NOT have to match the name in the struct definition.

  ```go
    func (l *Lockable[SomeOtherName]) GetValue() SomeOtherName {
      l.mu.Lock()
      defer l.mu.Unlock()
      return l.data
    }

  ```


-- The Blank Identifier for Unused Parameters 
--
  If a method doesn't need to reference the type parameter (e.g., it only 
  resets the mutex), you can use the blank identifier `_`.


  ```go
    func (l *Lockable[_]) ResetLock() {
      l.mu = sync.Mutex{}
    }

  ```



-- Code Demonstration & Output
--
  ```go
    func main() {
      // --- Instance 1: uint32 ---
      var n Lockable[uint32]
      n.Do(func(v *uint32) {
        *v = 10
        *v++
      })
      fmt.Printf("uint32 Result:  %v\n", n.GetValue())

      // --- Instance 2: float64 ---
      var fval Lockable[float64]
      fval.Do(func(v *float64) {
        *v = 3.14
        *v += 1.0
      })
      fmt.Printf("float64 Result: %v\n", fval.GetValue())

      // --- Instance 3: bool ---
      var b Lockable[bool]
      b.Do(func(v *bool) {
        *v = true
        *v = !*v
      })
      fmt.Printf("bool Result:    %v\n", b.GetValue())

      // --- Instance 4: Slice ---
      var bs Lockable[[]byte]
      bs.Do(func(v *[]byte) {
        *v = append(*v, "Golang"...)
      })
      fmt.Printf("Slice Result:   %s\n", bs.GetValue())
    }

  ```

  -- EXPECTED OUTPUT:
  --
    uint32 Result:  11
    float64 Result: 4.140000000000001
    bool Result:    false
    Slice Result:   Golang





-- Constraints and Type Parameters
--
  In Go's custom generics, we use "Type Parameters" to stand in for concrete 
  types. To ensure these parameters support the operations we need (like math 
  or specific methods), we apply "Constraints."






-- Core Definitions: Parameters vs. Constraints
--
  -- Type Parameter 
  --
    A distinct named type declared in a type parameter list (within `[]`). 
    It acts as a placeholder that can be used throughout the generic 
    specification, function, or method.


  -- Type Constraint 
  --
    A "template" for types. Just as a Type defines the shape of a Value, 
    a Constraint defines the requirements for a Type.


  Rule: In Go, all Type Constraints are actually Interface Types.





-- Enhanced Interface Syntax (Go 1.18+)
--
  To support generics, Go 1.18 expanded what an interface can contain. 
  Interfaces can now specify "Type Sets" using two new notations:


  -- The Tilde Form (~T) 
  --
    Syntax: `~T`
    Definition: Matches any type whose **underlying type** is T.
    Restriction: T must be a non-interface type and must be its own underlying type.

    ```go
      type MyString string

    ```

    Constraint: ~string 
    This set includes: 'string' and 'MyString'


  -- The Union Form (T1 | T2) 
  --
    Syntax: `T1 | T2 | ...`
    Definition: Matches any type that belongs to the type set of T1 OR T2.

    Example: int | int64 | float64





-- Modern Interface Composition
--
  Since Go 1.18, an interface can embed:
  1. Method Specifications (the "old" way).
  2. Type Names (interface or non-interface).
  3. Tilde Forms (~T).
  4. Term Unions (A | B).


  ```go
    // L is a standard "Basic" interface.
    type L interface {
      Run() error
      Stop()
    }

    // M embeds L and adds a method.
    type M interface {
      L
      Step() error
    }

    // N is a "Constraint-Only" interface because it uses ~ and |.
    type N interface {
      M                          // Embeds 3 methods
      interface{ Resume() }      // Embeds 1 method via anonymous interface
      ~map[int]bool              // Matches any map with these types
      ~[]byte | string           // Matches slices of bytes OR strings
    }

    // O demonstrates deep recursion and the 'any' keyword.
    type O interface {
      Pause()                    // Direct method
      N                          // Recursively expands all elements of N
      string                     // Matches EXACTLY the type string
      int64 | ~chan int | any    // Union including 'any' (which matches everything)
    }

  ```





-- The Principle of Expansion
--
  Embedding an interface is logically equivalent to "pasting" all its elements 
  into the new interface.


  example
    ```go
      // -- The Logical Expansion of 'N' above --
      type N_Expanded interface {
          // From L via M
        Run() error
        Stop()
          // From M
        Step() error
          // From the anonymous literal
        Resume()
          // From N's own type elements
        ~map[int]bool
        ~[]byte | string
      }

    ```

  



                                                                                
-- Constraint Terms vs. Methods
--
  - Method Elements: Define "What the type can DO."
  - Type Elements: Define "What the type can BE."

  If an interface contains ANY type elements (tilde or union), it becomes a 
  "Constraint-Only" interface. You cannot use it as a variable type; it can 
  only appear in square brackets `[]` as a generic constraint.


  ```go
    func main() {
      // any is an empty interface: interface{}
      // Its type set includes ALL non-interface types.
      var _ any = 42
      var _ any = "Golang"
      
      fmt.Println("Interface constraint logic processed.")
    }

  ```





// #############################################################################
// #
// #            Go Generics: Type Sets and Implementations
// #
// #############################################################################

// Since Go 1.18, the definition of an interface has evolved from a "Method Set"
// to a "Type Set." An interface now represents a collection of concrete 
// (non-interface) types that satisfy its requirements.


// =============================================================================
// :: 1. Rules for Calculating Type Sets
// =============================================================================

// An interface's Type Set is determined by its elements. Multiple elements 
// on different lines (separated by implicit semicolons) result in an 
// INTERSECTION of their respective type sets.

// -- 1.1. Non-interface type literal or name --
// The set contains exactly that one type.
// Example: interface { int } -> {int}

// -- 1.2. Tilde Form (~T) --
// The set contains all types whose underlying type is T.
// Example: interface { ~int } -> {int, MyInt, UserAge, ...}

// -- 1.3. Method Specification --
// The set contains all non-interface types that possess that specific method.
// Example: interface { M() } -> {AnyTypeWithMethodM}

// -- 1.4. Union of Terms (T1 | T2) --
// The set is the UNION of the type sets of each term.
// Example: interface { int | string } -> {int, string}

// -- 1.5. The Empty Interface (any) --
// The set contains all possible non-interface types.


// =============================================================================
// :: 2. Code Demonstration: Type Set Math
// =============================================================================

package main

type Bytes []byte    // Underlying: []byte
type Letters Bytes   // Underlying: []byte
type MyString string // Underlying: string
type Blank struct{}

func (MyString) M() {}
func (Bytes) M()    {}
func (Blank) M()    {}

// -----------------------------------------------------------------------------
// -- Interface Examples and their Type Sets --
// -----------------------------------------------------------------------------

// Type Set: { []byte } (Exactly)
type P interface { []byte }

// Type Set: { []byte, Bytes, Letters } (All with underlying []byte)
type Q interface { ~[]byte }

// Type Set: { []byte, string } (Explicit union)
type R interface { []byte | string }

// Type Set: EMPTY. 
// Logic: Intersection of R {[]byte, string} and types with method M().
// Neither []byte nor string have method M().
type S interface { R; M() }

// Type Set: { []byte, Bytes, Letters, string, MyString }
type T interface { ~[]byte | ~string }

// Type Set: { MyString, Bytes, Blank }
// Logic: All types that defined method M().
type U interface { M() }

// Type Set: { Bytes, MyString }
// Logic: Intersection of T and U.
type W interface { T; U }

// Type Set: All non-interface types.
// Logic: any is the universal set; union with anything remains the universal set.
type Z interface { ~[]byte | ~string | any }


// =============================================================================
// :: 3. Type Implementations (Set Theory)
// =============================================================================

// Implementation is now defined by subset relationships:
// "Type X implements Interface Y if TypeSet(X) is a SUBSET of TypeSet(Y)."

// -- Rules of Thumb --
// 1. Universal Implementation: Since 'any' is a superset of everything, 
//    all types implement 'any'.
// 2. The Empty Set: If an interface has an empty type set (like 'S' above), 
//    it technically implements every other interface ($\emptyset \subseteq AnySet$).
// 3. Method Sets: If Interface X implements Interface Y, then the Method Set 
//    of X must be a SUPERSET of the Method Set of Y.


// =============================================================================
// :: 4. Basic vs. Non-Basic (Constraint-Only) Interfaces
// =============================================================================

// -- Basic Interfaces --
// Definition: Interfaces defined entirely by methods (no ~ or |).
// Usage: Can be used as Value Types (variables) and Type Constraints.
var _ any           // OK: any is basic
var _ interface{M()} // OK: strictly methods

// -- Non-Basic Interfaces (Constraint-Only) --
// Definition: Interfaces containing type elements (~, |, or concrete types).
// Usage: Can ONLY be used as type constraints in generics. 
// They CANNOT be used as variable types.

/*
// var _ interface { ~[]byte } 
// COMPILE ERROR: interface contains type constraints (non-basic)
*/


// =============================================================================
// :: 5. Generic Type Aliases (Go 1.19+)
// =============================================================================

// Since Go 1.19, you can create aliases for non-basic interfaces, which 
// is helpful for complex constraints.

// A generic non-basic interface
type C[T any] interface {
	~int
	M() T
}

// Valid Aliases since 1.19:
type C1 = C[bool]                     // Specialized alias
type C2 = comparable                  // Alias to built-in constraint
type C3 = interface { ~[]byte | ~string } // Alias to literal constraint





// #############################################################################
// #
// #      Go Generics: The `comparable` Constraint & Type Satisfaction
// #
// #############################################################################

// In Go, the concept of "comparability" evolved significantly with the 
// introduction of generics. While standard Go allows comparing many types, 
// generics introduce a stricter requirement to ensure type safety and 
// prevent runtime panics.


// =============================================================================
// :: 1. The `comparable` Interface
// =============================================================================

// `comparable` is a predeclared identifier (Go 1.18+) that denotes an interface 
// type. Its type set consists of all "strictly comparable" types.

// -- Key Properties --
// 1. Constraint-Only: It is a non-basic interface. You cannot use it as a 
//    variable type. (e.g., `var x comparable` is ILLEGAL).
// 2. Filter Mechanism: It can be embedded in other interfaces to restrict 
//    their type sets to only comparable types.
// 3. Hierarchy: `comparable` implements `any` (its type set is a subset 
//    of the universal set), but `any` does not implement `comparable`.

type OnlyStrings interface {
	comparable
	// The union below includes incomparable types (slices, funcs, maps).
	// Because 'comparable' is intersected with this union, the final 
	// type set is reduced to ONLY 'string'.
	[]byte | func() | map[int]bool | string | [2]any
}


// =============================================================================
// :: 2. Comparable vs. Strictly Comparable
// =============================================================================

// Go distinguishes between types that can technically use `==` and those 
// that are guaranteed to never panic when doing so.

// -----------------------------------------------------------------------------
// -- 2.1. Why "Basic" Comparison is Dangerous --
// -----------------------------------------------------------------------------
// Interface values are "comparable," but they can cause a RUNTIME PANIC if 
// their dynamic types are incomparable (like a map or slice).

package main

import "fmt"

func comparisonPanicDemo() {
	var m any = map[int]string{}
	var a = [2]any{1, func() {}}
	var s = struct{ x any }{x: m}

	// The following will compile but PANIC at runtime:
	// _ = m == m // PANIC: runtime error: comparing uncomparable type map[int]string
	// _ = a == a // PANIC: runtime error: comparing uncomparable type func()
	// _ = s == s // PANIC: runtime error: comparing uncomparable type map[int]string
}

// -----------------------------------------------------------------------------
// -- 2.2. Strictly Comparable (Go 1.20+) --
// -----------------------------------------------------------------------------
// A type is "strictly comparable" if a comparison is guaranteed to be 
// panic-free. 

// -- NOT Strictly Comparable: --
// - Interface types (basic interfaces like 'any' or 'error').
// - Structs containing interface fields.
// - Arrays containing interface elements.


// =============================================================================
// :: 3. Implementation vs. Satisfaction (Go 1.20+)
// =============================================================================

// Before Go 1.20, "implements" and "satisfies" were synonymous. Since 1.20, 
// Go allows certain types to satisfy a constraint even if they don't 
// strictly implement it.

// -- Type Implementation --
// Type X implements Interface Y if: TypeSet(X) is a subset of TypeSet(Y).
// This is a strict compile-time check for assignments.

// -- Type Satisfaction --
// Type X satisfies Constraint Y if it can be used as a type argument for 
// a generic parameter constrained by Y.

// -----------------------------------------------------------------------------
// -- The Exception: Relaxed Satisfaction --
// -----------------------------------------------------------------------------
// If an ordinary type X is comparable (but NOT strictly comparable), it 
// satisfies a constraint Y if:
// 1. The constraint Y is basically: interface { comparable; E }
// 2. Type X implements the basic interface E.

// Example: 'any' satisfies 'comparable'
// 'any' does NOT implement 'comparable' (it contains incomparable types).
// But since Go 1.20, 'any' SATISFIES 'comparable', allowing us to pass 
// interfaces into generic functions that require comparability.

type A [2]any

func (a *A) M() {}

type B struct {
	A
	x any
}

// C requires comparability AND method M()
type C interface {
	comparable
	M()
}

func satisfyDemo() {
	// *A and *B satisfy C because:
	// 1. They are pointer types (inherently comparable).
	// 2. They implement method M().
	// Even though they contain 'any' (non-strict), they satisfy C's constraint.
}





// #############################################################################
// #
// #        Go Generics: Advanced Constraints & Type Parameter Lists
// #
// #############################################################################

// Go's custom generics implementation includes strict rules for how type 
// sets are composed in unions and how the compiler parses type parameter 
// lists to avoid syntactic ambiguity.


// =============================================================================
// :: 1. Requirements for Union Terms
// =============================================================================

// When creating a union constraint (e.g., T1 | T2), certain rules apply to 
// prevent ambiguity and simplify the compiler's work.

// -----------------------------------------------------------------------------
// -- 1.1. No Overlapping Type Sets (Non-Interface Terms) --
// -----------------------------------------------------------------------------
// The type sets of non-interface terms in a union must not intersect. 
// For example, you cannot include both a specific type and its approximation 
// in the same union.

/*
// ERROR: overlapping terms int and ~int
type Overlap interface {
	int | ~int 
}

// WORKAROUND: Wrapping terms in interfaces currently bypasses this check, 
// though this may be restricted in future Go versions.
type Workaround interface {
	interface{int} | interface{~int} // OK
}
*/

// -----------------------------------------------------------------------------
// -- 1.2. No `comparable` or Methods in Multi-term Unions --
// -----------------------------------------------------------------------------
// As of Go 1.25, a union with more than one term CANNOT contain the 
// predeclared 'comparable' identifier or any interface that defines methods.

/*
// ILLEGAL:
type BadUnion1 interface { []byte | comparable }
type BadUnion2 interface { string | error } // error has Error() method
*/


// =============================================================================
// :: 2. Type Parameter Lists
// =============================================================================

// Type parameter lists appear in generic type specs, generic function 
// declarations, and methods of generic types.

// -----------------------------------------------------------------------------
// -- 2.1. Syntax and Grouping --
// -----------------------------------------------------------------------------
// - Parameters are comma-separated inside square brackets `[]`.
// - Consecutive parameters with the same constraint can be grouped.

// These are equivalent:
// [A any, B any, X comparable, _ comparable]
// [A, B any, X, _ comparable]

// -----------------------------------------------------------------------------
// -- 2.2. Line Breaks and Trailing Commas --
// -----------------------------------------------------------------------------
// - If the closing `]` is on the same line as the last element, the comma is optional.
// - If the closing `]` is on a NEW line, a trailing comma is MANDATORY.

// Legal:
type G1[T any,] struct{}
type G2[
	T any,
] struct{}

// Illegal:
/*
type G3[
	T any
] struct{} // ERROR: missing ',' before newline
*/


// =============================================================================
// :: 3. Simplified Constraint Forms & Ambiguity
// =============================================================================

// If a constraint contains only one type element, you can omit the 
// `interface{}` wrapper.
// [X interface{string|[]byte}]  =>  [X string|[]byte]

// -----------------------------------------------------------------------------
// -- 3.1. The Pointer Ambiguity (*T) --
// -----------------------------------------------------------------------------
// Consider the following:
// type G[T *int] struct{}

// To a compiler, `T *int` looks like an expression (T multiplied by int) 
// used for an array size, rather than a generic constraint. To save 
// compilation time, Go defaults to treating this as a non-generic array 
// unless you prove otherwise.

// -- The Fix --
// 1. Use an explicit interface wrapper.
// 2. Add a trailing comma to force the "parameter list" interpretation.

type SafeG1[T interface{*int}] struct{} // Solution 1
type SafeG2[T *int,] struct{}           // Solution 2

// -----------------------------------------------------------------------------
// -- 3.2. Parentheses Ambiguity (cap, len) --
// -----------------------------------------------------------------------------
// Similarly, using built-ins or parentheses can confuse the parser into 
// thinking you are calling a function to determine an array size.

type K[cap (*int),] struct{} // Trailing comma resolves ambiguity
type L[len (string),] struct{}


// =============================================================================
// :: 4. Edge Cases: Shadowing Predeclared Identifiers
// =============================================================================

// Technically, you can name a type parameter after a built-in function, 
// though it is considered extremely poor practice.

// 'make' is the name of the type parameter here. 
// Its constraint is a channel.
type PtrToChan[make chan int] *make

// 'new' is the name of the type parameter.
// Its constraint is an array.
type Matrix33[new [3]float64] [3]new

// WARNING: Do not use these patterns in production. They are confusing 
// to both humans and tooling.





// #############################################################################
// #
// #            Go Generics: Type Parameters and Scopes
// #
// #############################################################################

// Since Go 1.18, the category of "Named Types" has expanded to include 
// Type Parameters. Understanding the lifecycle, scope, and structural 
// limitations of these placeholders is essential for writing valid 
// generic code.


// =============================================================================
// :: 1. Type Parameters as Distinct Named Types
// =============================================================================

// A "Type Parameter" (declared in a list like `[T any]`) is a first-class 
// member of Go's named type categories.

// -- Categories of Named Types --
// 1. Predeclared: int, bool, string, etc.
// 2. Defined Non-Generic: type MyInt int
// 3. Instantiated Types: Lockable[int]
// 4. Type Parameter Types: T (within its generic scope)

// **The Identity Rule**: Two different type parameters are NEVER identical, 
// even if they share the exact same name or constraints across different 
// declarations.


// =============================================================================
// :: 2. The "Interface" Nature of Type Parameters
// =============================================================================

// Technically, the "type of a type parameter" is its constraint (an interface). 
// This implies its underlying type is an interface. However, Type Parameters 
// do NOT behave like interface values:
// - They cannot "box" values.
// - You cannot perform a Type Assertion on them (e.g., `t.(int)`).
// - Pro-Tip: It is mostly meaningless to discuss the underlying type of a 
//   type parameter; they are strictly placeholders for common traits.


// =============================================================================
// :: 3. Structural Restrictions on Type Parameters
// =============================================================================

// Because type parameters are placeholders and not concrete definitions, 
// they cannot be used in several specific structural contexts.

// -----------------------------------------------------------------------------
// -- 3.1. No Tilde on Parameters (~T is forbidden) --
// -----------------------------------------------------------------------------
// You cannot use the approximation operator `~` on a type parameter. 
// It can only be used on concrete types (like `~int`).

/*
// INVALID:
[A int, B ~A] // ERROR: invalid use of ~ (A is a type parameter)
*/

// -----------------------------------------------------------------------------
// -- 3.2. No Embedding in Interfaces or Structs --
// -----------------------------------------------------------------------------
// You cannot embed a type parameter directly as a field in a struct 
// or as a term in an interface.

/*
// INVALID:
type G[T any] struct {
    T // ERROR: type parameter cannot be embedded
}

type I[T any] interface {
    T // ERROR: type parameter cannot be embedded
}
*/


// =============================================================================
// :: 4. The Scope of Type Parameters
// =============================================================================

// Type parameters have a specific lexical scope:
// - For Types: From the start of `[` to the end of the type specification.
// - For Functions: From the start of `[` to the end of the function body.

// **The "Look-Ahead" Property**: Within the `[]` list, the order doesn't 
// strictly matter. One parameter can reference another declared later in 
// the same list.

package main

import "fmt"

// S uses E in its constraint, even though E is defined later in the list.
// This is perfectly valid Go.
type G[S ~[]E, E any] struct {
	Data S
}

/*
================================================================================
UNDER THE HOOD:
================================================================================
- 'E' is a type parameter.
- 'S' is a type parameter.
- '[]E' is an ORDINARY slice type (even though it contains a parameter).
- The underlying type of 'S' is 'interface{~[]E}'.
================================================================================
*/


// =============================================================================
// :: 5. Composite Types Built from Parameters
// =============================================================================

// While 'T' is a type parameter, any composite type literal containing 'T' 
// (like `*T`, `[]T`, or `chan T`) is treated as an **Ordinary Type**.

// This allows parameters to depend on pointers or containers of other parameters.

// LEGAL:
type Node[T any, P *T] struct {
	Value   T
	Pointer P
}

// LEGAL:
type Pipeline[T any, C chan T, F func(T) bool] struct {
	In   C
	Test F
}


// =============================================================================
// :: 6. Defining Generic Types: Valid Source Types
// =============================================================================

// When declaring `type Name[T] SourceType`, the `SourceType` must be an 
// "Ordinary Type."

// -----------------------------------------------------------------------------
// -- 6.1. Valid Source Types --
// -----------------------------------------------------------------------------
type (
	// Source is a predeclared type
	FakeInt[T any] int 
	
	// Source is a composite literal (unnamed type)
	MyData[A any, B ~bool] struct {
		valA A
		valB B
	}
	
	// Source is an instantiated type (re-wrapping)
	AliasData[T any] MyData[T, bool] 
)

// -----------------------------------------------------------------------------
// -- 6.2. Invalid Source Types --
// -----------------------------------------------------------------------------
// You cannot use a type parameter as the direct source of a type definition.
/*
type Wrapper[T any] T // ERROR: cannot use type parameter T as source type
*/


// =============================================================================
// :: 7. Identifier Redeclaration and Go 1.24+ 
// =============================================================================

// You must avoid using the same name for the Generic Type and its parameter.
/*
type C any
func foo1[C C]() {} // ERROR: C used as both name and constraint
*/

// **Modern Go Note**: Since Go 1.24, method receivers are more flexible with 
// naming, allowing the receiver variable to share the type parameter's name.

type Box[T any] struct{ x T }

// This is valid in Go 1.24+
func (T Box[T]) Get() T {
	return T.x
}


// =============================================================================
// :: Demonstration Code
// =============================================================================

func main() {
	// Instantiate G: S=[]int, E=int
	val := G[[]int, int]{Data: []int{1, 2, 3}}
	
	fmt.Printf("Type: %T, Value: %v\n", val, val.Data)
	
	node := Node[int, *int]{Value: 10}
	node.Pointer = &node.Value
	fmt.Printf("Node Value: %d, Pointer Value: %d\n", node.Value, *node.Pointer)
}

/*
================================================================================
EXPECTED OUTPUT:
================================================================================
Type: main.G[[]int,int], Value: [1 2 3]
Node Value: 10, Pointer Value: 10
================================================================================
*/
