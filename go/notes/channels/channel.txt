-- The Philosophy of Channels
-- 
  Go provides a unique concurrency synchronization technique: the `channel`.
  While traditional languages use Mutexes and Locks (found in Go's `sync` and 
  `sync/atomic` packages) to protect shared memory, Go introduces a different 
  paradigm, famously summarized by Rob Pike:
    "Don't communicate by sharing memory; share memory by communicating."
  
  -- Sharing Memory by Communicating
  -- 
    Instead of locking a variable to safely
    read/write it across goroutines, you send the data itself through a channel.

  
  -- First-Class Citizens
  -- 
    Unlike the `sync` package, channels are built-in
    types in Go. You don't need to import anything to use them.

  -- FIFO Queue
  --
    Internally, a channel acts as a First-In, First-Out queue.


  -- Logical Ownership Transfer
  -- 
    When Goroutine A sends a value (e.g., a pointer) to a channel, it logically
    "releases" ownership of that data. When Goroutine B receives it, it "acquires"
    ownership. While Go's syntax doesn't strictly enforce this (you could still 
    technically create a data race if you keep modifying a sent pointer), 
    following this ownership pattern makes data-race-free code easy to write.

    example
      ```go
        type DataPacket struct {
            ID      int
            Payload string
        }

        func channelPhilosophyExample() {
            // A standard bidirectional channel of type *DataPacket
            ch := make(chan *DataPacket)

            // Producer Goroutine (Sends data)
            go func() {
                packet := &DataPacket{ID: 1, Payload: "Important Data"}
                // By sending 'packet', this goroutine logically transfers ownership.
                // It should NOT modify 'packet' after this line.
                ch <- packet 
            }()

            // Consumer Goroutine (Receives data)
            // The main goroutine blocks until it receives the packet, acquiring ownership.
            receivedPacket := <-ch
            fmt.Printf("Received: %v\n", receivedPacket.Payload)
        }

      ```





-- Channel Types & Directionality
--
  Like arrays, slices, and maps, channels are strongly typed. A channel can 
  ONLY transfer values of its defined "element type" (e.g., `chan int`).


  Channels can be Bidirectional or Single-Directional:
    1. `chan T`   : Bidirectional. You can send to and receive from it.
    2. `chan<- T` : Send-Only. The compiler prevents receiving from it.
    3. `<-chan T` : Receive-Only. The compiler prevents sending to it.


  -- Implicit Conversion Rules
  --
    - `chan T` CAN be implicitly converted to `chan<- T` or `<-chan T`.
      (This is highly useful for function parameters to restrict permissions).
    - `chan<- T` CANNOT be converted to `<-chan T` (or vice versa).
    - Single-directional channels CANNOT be converted back to `chan T`.


  example
    ```go
      // producer strictly requires a Send-Only channel.
      // It is impossible for this function to accidentally read from the channel.
      func producer(ch chan<- int, wg *sync.WaitGroup) {
        defer wg.Done()
        for i := 1; i <= 3; i++ {
          ch <- i // Sending is allowed
        }
        close(ch) // Closing is allowed on send-only channels
      }

      // consumer strictly requires a Receive-Only channel.
      // It is impossible for this function to accidentally write to the channel.
      func consumer(ch <-chan int, wg *sync.WaitGroup) {
        defer wg.Done()
        for val := range ch { // Receiving is allowed
          _ = val
        }
      }

      func directionalityExample() {
        // Create a standard Bidirectional channel
        biChan := make(chan int)
        var wg sync.WaitGroup

        wg.Add(2)
        // Implicitly converted to chan<- int when passed to producer
        go producer(biChan, &wg)
        
        // Implicitly converted to <-chan int when passed to consumer
        go consumer(biChan, &wg)

        wg.Wait()
      }

    ```





-- Initialization, Capacity, and States
--
  Zero Value: The zero value of any channel type is `nil`. 
  Sending to or receiving from a `nil` channel blocks forever!


  To create a usable channel, you MUST use the built-in `make` function:
  `make(chan Type, capacity)`


  -- capacity
  --
    - Unbuffered (Capacity 0): `make(chan int)` or `make(chan int, 0)`.
    Sends block until another goroutine is ready to receive
    Receives block until another goroutine is ready to send.
    Perfect for strict synchronization.
  
    - Buffered (Capacity > 0): `make(chan int, 5)`.
    Sends only block if the buffer is FULL. 
    Receives only block if the buffer is EMPTY. 

  
  example
    ```go
      // 1. Nil Channel
      var nilChan chan string 
      _ = nilChan
      // nilChan <- "hello" // FATAL ERROR: blocks forever (deadlock)

      // 2. Unbuffered Channel (Default capacity is 0)
      unbuffered := make(chan int)
      
      // go func() { unbuffered <- 1 }() // Requires a concurrent receiver
      // <-unbuffered

      // 3. Buffered Channel (Capacity of 3)
      buffered := make(chan string, 3)

      // Because it has a capacity of 3, we can send 3 items without blocking,
      // even if there is no concurrent receiver yet.
      buffered <- "A"
      buffered <- "B"
      buffered <- "C"
      // buffered <- "D" // This would block because the buffer is now full.

      // We can inspect the length (current items) and capacity (max items)
      fmt.Printf("Buffered Chan - Length: %d, Capacity: %d\n", len(buffered), cap(buffered))

      _, _ = unbuffered, buffered
      
    ```
