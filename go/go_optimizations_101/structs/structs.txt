-- Struct Performance & Optimization
--
  This document focuses on the physical reality of structs in memory and how
  coding patterns affect CPU execution speed. Specifically, we explore why
  local variables are superior to pointer-based field access in loops.


  -- Optimizing Struct Field Access in Loops
  --
    A "Pointer Dereference" is a memory operation. While modern CPUs have fast
    caches (L1/L2/L3), accessing a struct field via a pointer (`t.x`) is still
    significantly slower than accessing a CPU register.


    -- The "Hoisting" Strategy
    --
      When performing repeated calculations on a struct field inside a loop:
      1. Copy the field's value to a local variable (which the compiler can
         place in a CPU register).
      2. Perform the loop calculations using the local variable.
      3. Write the final result back to the struct field exactly once.


    example
      ```go
        const N = 1000

        type T struct {
          x int
        }

        // --- f: Naive Implementation (Slow) ---
        // This function forces the CPU to perform a memory read and a memory write
        // for 't.x' in every single iteration of the loop.
        //go:noinline
        func f(t *T) {
          t.x = 0
          for i := 0; i < N; i++ {
            // Memory Access: fetch address of t, find offset x, read, add, write.
            t.x += i
          }
        }

        // --- g: Optimized Implementation (Fast) ---
        // This function "hoists" the value into the local stack variable 'x'.
        // The Go compiler optimizes 'x' to live in a CPU register for the loop duration.
        //go:noinline
        func g(t *T) {
          var x = 0 // Local register-friendly variable
          for i := 0; i < N; i++ {
            // Register-only math: extremely fast.
            x += i
          }
          // Single memory write at the very end.
          t.x = x
        }

      ```
      -- BENCHMARK RESULTS (N=1000):
      --
        Benchmark_f-4    2402.0 ns/op    (Memory-heavy loop)
        Benchmark_g-4     461.3 ns/op    (Register-only loop)


      -- ANALYSIS:
      --
        The optimized version (g) is ~5.2x faster. The performance gap is purely due
        to avoiding pointer dereferencing inside the loop.


    -- Why doesn't the compiler fix this automatically?
    --
      You might assume a "clever" compiler would transform 'f' into 'g' for you.
      However, it cannot do this because of "Pointer Aliasing."

      The compiler cannot prove that some other part of the program (perhaps in
      a different goroutine or a different pointer) isn't looking at 't.x'
      WHILE the loop is running. To guarantee correctness, it must perform
      the memory writes exactly as written in 'f'.





-- Small-Size Struct Optimizations (Recap)
--
  As established in "Value Copy Costs," the Go compiler (gc) applies special
  optimizations to "small" structs:

  - Definition: Structs with <= 4 fields of native-word size.
  - Optimization: The compiler can pass these structs entirely in CPU
    registers during function calls, rather than using the stack/memory.
  - Practical Tip: Keep internal Data Transfer Objects (DTOs) small to
    maximize register utilization.





-- Struct Size and Field Ordering (Recap)
--
  Field order in a struct determines the amount of "Padding" the compiler
  must add to satisfy alignment requirements.


  -- Best Practice: Largest to Smallest
  --
    To minimize wasted space, order fields by their alignment requirement:
    1. 8-byte types (pointers, int64, float64, slices, maps, channels)
    2. 4-byte types (int32, float32)
    3. 2-byte types (int16)
    4. 1-byte types (int8, bool)


  example
    ```go
      // --- UNOPTIMIZED (24 bytes) ---
      type Bad struct {
          a bool   // 1 byte + 7 padding
          b *int   // 8 bytes
          c bool   // 1 byte + 7 padding
      }


      // --- OPTIMIZED (16 bytes) ---
      type Good struct {
          b *int   // 8 bytes
          a bool   // 1 byte
          c bool   // 1 byte + 6 padding at the end
      }

    ```
