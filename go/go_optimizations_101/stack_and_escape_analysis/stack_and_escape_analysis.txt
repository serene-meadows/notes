-- Stack and Escape Analysis
--
  Go's memory management relies on two zones: the Stack and the Heap.
  The "Escape Analysis" module in the compiler is responsible for deciding
  where every local variable should live.


  -- Goroutine Stacks
  --
    Every goroutine is created with its own dedicated "Stack" memory.

  -- Contiguous Stacks
  --
    Modern Go (since 1.3) uses "Contiguous Stacks." This means a goroutine's
    stack is one single continuous memory segment. If the stack runs out of
    space, Go allocates a new, larger segment and copies the current stack to it.


  -- Advantages of Stack Allocation
  --
    1. Speed: Allocation is a simple "pointer bump" (incrementing a register).
    2. No GC: When a function returns or a goroutine exits, its stack is
       reclaimed instantly without the Garbage Collector needing to scan it.
    3. Cache Locality: Stack data is small and frequently accessed, making it
       highly likely to stay in the CPU's L1/L2 cache.





-- Escape Analysis: The "Stay Local" Rule
--
  "Escape Analysis" is a compile-time process. The compiler looks at a local
  variable and asks: "Can I prove this value is only used within this specific
  goroutine and this specific call stack?"


  -- The Principle of Escape
  --
    If the compiler determines that a value part might be used by another
    goroutine, or if its lifetime exceeds the function call, it "Escapes"
    to the Heap.


  -- Why Not Everything is on the Stack?
  --
    1. Complexity: Proving "safety" for every variable is computationally expensive.
       To keep compile times fast, the analyzer is conservative—if it's unsure,
       it defaults to the Heap.
    2. Large Values: Even if a value doesn't "escape," if it is too large (exceeding
       internal thresholds), the compiler puts it on the Heap to avoid
       stack overflow.





-- Scoping and References
--
  -- Unit of Analysis
  --
    Escape analysis is performed function-by-function for local variables.
    NOTE: ALL package-level (global) variables are allocated on the Heap.


  -- The Reference Hierarchy
  --
    - A value on the HEAP can point to another value on the HEAP.
    - A value on the STACK can point to a value on the HEAP.
    - A value on the STACK can point to a value on the SAME STACK.
    - CRITICAL: A value on the HEAP can NEVER point to a value on a STACK.

    IMPLICATION: If a package-level variable (Heap) references a local variable,
    that local variable is immediately forced to escape to the Heap.





-- Code Demonstration: Escape Analysis in Action
--
  ```go
    type Data struct {
      ID int
    }


    // staysOnStack demonstrates a value that the compiler can prove
    // never leaves the function.
    func staysOnStack() int {
      d := Data{ID: 1} // Allocated on stack
      return d.ID      // Only the integer value is returned
    }


    // escapesToHeap demonstrates a value whose address is returned.
    // Since the function ends but the pointer remains valid for the caller,
    // the data MUST live on the heap.
    func escapesToHeap() *Data {
      d := Data{ID: 2}
      return &d // Pointer escapes the stack frame
    }


    // largeValueEscapes demonstrates a value too large for the stack.
    func largeValueEscapes() {
      // A massive array will escape to the heap even if not shared.
      var big [1 << 20]byte
      _ = big[0]
    }

  ```

    -- How to Verify
    --
      You can see the compiler's decisions by running:
        go build -gcflags="-m" main.go

      Output will look like:
        ./main.go:55:9: &d escapes to heap
        ./main.go:54:2: moved to heap: d
        ./main.go:61:6: moved to heap: big





-- Escape Analysis Examples
--
  Escape Analysis is the compile-time process that decides whether a variable
  can be allocated on the stack or must "escape" to the heap. We can inspect
  these decisions using the compiler flag: `-gcflags="-m"`.


  -- Case Study: Selective Escape in Closures
  --
    When a variable is used inside a closure (like a goroutine's anonymous function),
    the compiler must decide how to "capture" that variable.


    example
      ```go
        func example1() {
          var (
            a = 1                    // Logic: Modified in closure -> Escapes
            b = false                // Logic: Read-only in closure -> Stays on Stack
            c = make(chan struct{})  // Logic: Channel header -> Stays on Stack
          )


          go func() {
            // Variable 'a' is modified here.
            if b {
              a++
            }
            close(c)
          }()


          <-c
          fmt.Println(a, b)
        }

      ```


    -- COMPILER OUTPUT (-gcflags="-m"):
    --
      ./escape.go:13:3: moved to heap: a
      ./escape.go:17:5: func literal escapes to heap

    -- ANALYSIS:
    --
      1. Why does 'a' escape?
         The closure modifies 'a' (a++). Since the goroutine runs concurrently and
         might outlive the stack frame of 'example1', 'a' must be on the heap to
         ensure memory safety.

      2. Why does 'b' stay on the stack?
         The compiler sees that 'b' is never modified. Instead of creating a
         heap-allocated reference, the compiler simply passes a COPY of 'b's
         value to the goroutine.

      3. What about the channel 'c'?
         The "Direct Part" (the header) of the channel stays on the stack. The
         "Indirect Part" (the internal buffer/queues) is always on the heap.


  -- Case Study: Shared Mutability (The Cascade)
  --
    If we modify 'b' later in the main function, the compiler's "copy"
    optimization is no longer safe.


    example
      ```go
        func example2() {
          var (
            a = 1     // moved to heap: a
            b = false // moved to heap: b
            c = make(chan struct{})
          )


          go func() {
            if b {
              a++
            }
            close(c)
          }()


          <-c
          b = !b // Mutation added here!
          fmt.Println(a, b)
        }

      ```


      -- COMPILER OUTPUT (-gcflags="-m"):
      --
        ./escape.go:50:3: moved to heap: a
        ./escape.go:51:3: moved to heap: b

      -- ANALYSIS:
      --
        Now both 'a' and 'b' escape. Why? Because 'b' is now mutated after the
        goroutine starts. To ensure the goroutine and the main thread see the same
        memory for 'b', the compiler can no longer use a simple stack copy.





-- Optimization and Dead Code Elimination
--
  Escape analysis is closely tied to other compiler optimizations like
  Constant Folding and Dead Code Elimination.


  example
    ```go
      func optimizationEffect() {
        const b = false // 'b' is now a constant
        var a = 1       // Logic: a++ becomes unreachable code

        go func() {
          if b {
            a++ // The compiler knows this never runs!
          }
        }()
      }

    ```

    -- ANALYSIS:
    --
      In this case, 'a' will NOT escape. Because 'b' is a constant 'false', the
      compiler realizes the 'a++' branch is dead code. It removes the reference
      to 'a' from the goroutine entirely, allowing 'a' to remain on the stack.





-- Stack Frames of Function Calls
--
  A "Stack Frame" is the specific segment of a goroutine's stack allocated to a
  single function call. It contains the function's local variables, arguments,
  return values, and bookkeeping information (like the return address).


  -- Compile-Time Determinism
  --
    Unlike heap allocation, which is dynamic and happens at runtime, stack frame
    sizes are calculated entirely at COMPILE TIME.

    -- The Calculation Rule
    --
      The Go compiler examines every possible execution path within a function.
      It calculates the maximum possible memory the function could ever need.


      CRITICAL: Even if a branch of code is never executed at runtime (e.g., inside
      an 'if false' block), the compiler still reserves stack space for the
      variables declared in that branch.


  -- Performance: Why Stack Allocation is Faster
  --
    1. Fixed Offsets: Because the compiler knows the size of the frame, it
       calculates the memory address of every local variable as a fixed "offset"
       from the start of the frame (e.g., "variable x is at FrameStart + 8 bytes").

    2. Instant Allocation: At runtime, "allocating" the entire frame is just
       subtracting the frame size from the CPU's Stack Pointer (SP) register.
       This is significantly faster than the complex "find-a-free-block" logic
       required by the Heap.


  -- Code Demonstration: Analyzing Frame Sizes
  --
    ```go
      // bar contains a [5000]byte array.
      // Expected Frame: ~5000 bytes.
      func bar(s []uint32, n byte) (r byte) {
        s = s[:1250]
        var a [5000]byte // Large local allocation
        for i, v := range s {
          a[i*4+0] = byte(v >> 0)
          a[i*4+1] = byte(v >> 8)
          a[i*4+2] = byte(v >> 16)
          a[i*4+3] = byte(v >> 24)
        }
        var v = a[0]
        for i := 1; i < len(a); i++ {
          r += v ^ a[i]
          v = a[i]
        }
        return
      }

      // foo contains a [2500]uint32 array (10,000 bytes).
      // Expected Frame: ~10,000 bytes.
      func foo(n byte) uint16 {
        var a [2500]uint32
        for i := range a {
          a[i] = rand.Uint32()
        }
        x := bar(a[:1250], n)
        y := bar(a[1250:], ^n)
        return uint16(x)<<8 | uint16(y)
      }

      // duck contains a massive array in a branch that is effectively unreachable.
      // Expected Frame: > 16,000 bytes.
      var v interface{}
      func duck() {
        // Dummy type declaration to prevent the compiler from inlining this
        // function into main().
        type _ int

        if v != nil {
          // Even though v is nil and this block never runs,
          // the compiler MUST allocate the 16,000 bytes.
          v = [16000]byte{}
          panic("unreachable")
        }
      }

      func main() {
        x := foo(123)
        fmt.Println(x)
        duck()
      }

    ```


  -- Inspecting the Assembly (-gcflags=-S)
  --
    You can inspect the calculated frame sizes using:
      go build -gcflags=-S frame.go

      INTERPRETING THE ASM OUTPUT (Example for 64-bit):

      TEXT "".bar(SB) ... $5024-32
        -> $5024 is the Frame Size (5000 for array + 24 for bookkeeping).
        -> -32   is the size of the arguments/results (Slice header 24 + byte 1 + return 1 + padding).

      TEXT "".foo(SB) ... $10056-8
        -> $10056 is the Frame Size (10,000 for array + 56 for bookkeeping/locals).

      TEXT "".duck(SB) ... $16024-0
        -> $16024 is the Frame Size. This proves that even "dead" branches inflate
           the stack frame size.





-- Stack Growth and Shrinkage
--
  Unlike many languages that use fixed-size stacks (which can lead to easy
  stack overflows), Go uses a dynamic, "contiguous" stack model. Stacks grow
  and shrink at runtime to optimize memory usage and support millions of
  concurrent goroutines.


  -- The Stack Cursor and Frame Allocation
  --
    Internally, Go manages the stack using a simple cursor:
    1. Entering a function: The cursor moves forward by the size of the new
       function's stack frame.
    2. Exiting a function: The cursor moves backward, freeing the frame.


  Because the frame sizes are pre-calculated at compile time, this is an
  extremely fast O(1) operation.





-- Dynamic Sizing Logic
--
  -- Initial Size
  --
    - Starting size: 2KiB (default for most platforms; 8KiB on Windows).
    - Modern Go (1.19+): Uses an adaptive initial size for better performance.


  -- Growth (The 2^n Rule)
  --
    When a goroutine needs more space than currently available:
    1. Go allocates a new memory segment exactly double the size of the current one.
    2. The entire contents of the old stack are copied to the new segment.
    3. The old segment is released.
    4. Stacks are always powers of 2 (2KiB, 4KiB, 8KiB, etc.).


  -- Shrinkage (Conservative Cleaning)
  --
    To avoid "thrashing" (frequently growing and shrinking), Go is conservative:
    1. Timing: Shrinkage only happens during a Garbage Collection (GC) cycle.
    2. Condition: The stack is only halved if its used portion is less than
       1/4 of its total size.
    3. Exception: Shrinkage does not happen if the goroutine is currently
       active in a syscall or cgo call.


  -- Code Demonstration: Address Volatility
  --
    When a stack grows or shrinks, the data is moved to a new memory location.
    This means the memory addresses of stack-allocated variables CHANGE.
    Go's runtime automatically updates all pointers on that stack to point to
    the new locations—a feat nearly impossible in C/C++.


    example
      ```go
        //go:noinline
        func forceGrowth(i int) byte {
          // Allocate an 8KiB array on the stack.
          // This will force the initial 2KiB stack to grow.
          var a [1 << 13]byte
          return a[i]
        }

        func stackMovementDemo() {
          var x int

          // 1. Initial Address
          fmt.Printf("Initial Address:   %p\n", &x)

          // 2. Trigger Growth
          forceGrowth(1)

          // Address has likely changed due to stack copy
          fmt.Printf("After Growth:      %p\n", &x)

          // 3. Trigger Shrinkage
          // Manually starting GC cycles facilitates stack check/shrinkage.
          runtime.GC()
          fmt.Printf("After GC (Shrink): %p\n", &x)
        }

      ```


    -- REPRESENTATIVE OUTPUT (Addresses will vary):
    --
      Initial Address:   0xc00004e758
      After Growth:      0xc0000a2f58
      After GC (Shrink): 0xc00004cf58





-- Global Stack Limits
--
  To prevent a runaway goroutine from consuming all system RAM, Go enforces
  a hard maximum limit on stack size.


  -- Default Limits (Go 1.19+)
  --
    - 64-bit systems: 1 GB
    - 32-bit systems: 250 MB


  -- The "Real" Limit
  --
    Due to the implementation of the growth algorithm, the effective limit
    is roughly half the setting (e.g., 512 MiB on 64-bit).


  -- Configuration
  --
    Use `debug.SetMaxStack(bytes)` to modify this global limit.


  example
    ```go
      func stackOverflowDemo() {
        // 10MB array
        const N = 1024 * 1024 * 10

        // Recursive function with large stack frames
        var f func(v [N]byte, n int)
        f = func(v [N]byte, n int) {
          if n > 0 {
            f(v, n-1)
          }
        }

        var x [N]byte
        fmt.Println("Starting deep recursion...")
        // This will eventually exceed the 1GB limit and crash the program.
        f(x, 100)
      }

    ```

    -- CRASH OUTPUT (Simulated):
    --
      Starting deep recursion...
      runtime: goroutine stack exceeds 1000000000-byte limit
      fatal error: stack overflow






-- Stack Overhead & Advanced Escape Scenarios
--
  Even though Go's stack management is highly efficient, there are specific
  architectural limits and compiler behaviors that lead to either memory
  waste or unexpected heap escapes.


  -- Memory Waste in Stack Spaces
  --
    Every goroutine requires a stack. Because stacks grow in fixed increments
    (powers of 2), a goroutine usually has more memory allocated than it
    is currently using.


  -- The Cumulative Impact
  --
    1. Unused Space: A stack might grow to 16KiB to accommodate a single deep
       call, but then stay at 16KiB for a long time even if only 2KiB are used.
    2. Metadata Cost: Each goroutine requires additional memory for its
       internal descriptor (the 'G' struct in the runtime).
    3. Scaling: While 10,000 goroutines might only waste ~40MB, 1 million
       goroutines can waste several GiB of RAM, potentially leading to
       Out-of-Memory (OOM) errors on smaller systems.


  -- Escape Scenario: Loop Variables Referenced Externally
  --
    A variable declared inside a loop that is referenced by a pointer existing
    outside that loop will ALWAYS escape to the heap.


  -- The Technical "Why"
  --
    Function stack frames must be a fixed size calculated at compile time.
    However, a loop can run an arbitrary number of times. If a pointer outside
    the loop captures the address of a loop-local variable, the compiler
    cannot guarantee how many unique instances of that variable are "alive"
    simultaneously. To remain safe, it allocates them on the heap.


  example
    ```go
      func loopEscapeDemo() {
        var x *int

        // Even though this loop runs exactly once (break), the compiler
        // is conservative.
        for {
          var n = 1 // Moved to heap: n
          x = &n
          break
        }

        fmt.Println(*x)
      }

    ```
    -- COMPILER OUTPUT (-gcflags="-m"):
    --
      ./main.go:37:7: moved to heap: n





-- Escape Scenario: Interface Method Calls
--
  Pointers passed as arguments to interface methods often escape to the heap.


  -- The Technical "Why"
  --
    Interfaces use "dynamic dispatch." In many cases, the compiler doesn't
    know which concrete method will actually be called at runtime.
    Because it cannot inspect the code of the "unknown" target method, it
    cannot prove that the target method won't store the pointer in a global
    variable or send it to another goroutine.
    Therefore, it must assume the worst and force the data to the heap.


  example
    ```go
      type Submitter interface {
        Submit(*int)
      }


      type Work struct{}
      func (Work) Submit(p *int) { /* doesn't do much */ }


      func interfaceEscapeDemo() {
        var w Work
        var s Submitter = w

        val := 42

        // This call is devirtualized (direct call), so 'val' stays on stack.
        w.Submit(&val)

        // This is an interface call. Because 's' is a package-level variable
        // (or simply opaque to the analyzer), the argument escapes.
        y := 100 // Moved to heap: y
        s.Submit(&y)
      }

    ```


-- Optimization: Devirtualization
--
  Modern Go compilers attempt "Devirtualization." If the compiler can
  definitively prove which concrete type is stored in the interface at
  compile time, it transforms the interface call into a direct method call.
  If this happens, the escape may be avoided.


  example
    ```go
      func devirtualizationFix() {
        // 'i' is local and assigned 't' immediately.
        // The compiler can "see through" the interface.
        type T struct{}
        func (T) M(p *int) {}

        var t T
        var i interface{ M(*int) } = t

        localVal := 500
        i.M(&localVal) // ELIMINATED: Stays on stack due to devirtualization.
      }

    ```





-- Reflection, Results, and Inlining
--
  Escape analysis is the compiler's mechanism for deciding if a variable can
  live on the fast-access "Stack" or must be moved to the managed "Heap."
  Certain standard library calls and architectural patterns have significant
  impacts on these decisions.


  -- The Reflection and 'fmt' Escape Trap
  --
    Calls to `reflect.ValueOf()` almost always force their arguments to escape.
    This is because `reflect.ValueOf` takes an `interface{}` parameter. To "box"
    a value into an interface, the runtime requires a stable memory address,
    forcing a heap allocation.


    -- The 'fmt' Connection
    --
      Functions like `fmt.Println`, `fmt.Printf`, and `fmt.Sprint` use reflection
      internally to inspect types. Consequently, passing a variable's address
      to `fmt.Println` will trigger a heap escape for that variable.


    example
      ```go
        func reflectionEscapeDemo() {
          var n = 1
          // n escapes to the heap because ValueOf needs to wrap it in an interface.
          _ = reflect.ValueOf(&n)


          var x = 10
          // fmt.Println uses reflection internally. x escapes to the heap.
          fmt.Println("Address of x:", &x)


          var y = 20
          // Built-in 'println' does NOT use reflection. y remains on the stack.
          println("Address of y:", &y)
        }

      ```
      -- COMPILER OUTPUT (-gcflags="-m"):
      --
        ./main.go:28:6: moved to heap: n
        ./main.go:31:6: moved to heap: x
        ./main.go:35:6: y does not escape


  -- Escaping via Return Results
  --
    When a function returns a pointer to a local variable, that variable
    MUST escape to the heap. The stack frame of the function is destroyed
    upon return, so any pointers to that frame would become "dangling"
    (invalid). Go prevents this by moving the data to the heap.


    example
      ```go
        //go:noinline
        func createInt(val int) *int {
          n := val + 1 // n is moved to the heap
          return &n
        }


        func resultEscapeDemo() {
          p := createInt(100)
          fmt.Println(*p)
        }

      ```

      -- ANALYSIS:
      --
        The compiler is conservative. It is computationally expensive to trace
        exactly how every caller uses a returned pointer. Therefore, the rule is
        simple: if a pointer to a local "escapes" the function via return, it goes
        to the heap.


  -- The Impact of Function Inlining
  --
    Function inlining is a major optimization where the compiler replaces a
    function call with the actual code of the function. This has a profound
    effect on escape analysis.


  -- Why Inlining Prevents Escape
  --
    When a function is inlined, the variables that would have "escaped" via
    return are now effectively local variables within the CALLER's scope.
    Since the compiler can now see that the variables never leave the
    caller, it can safely keep them on the stack.


  example
    ```go
      func fastAdd(x *int) *int {
        n := *x + 1
        return &n // This looks like it should escape...
      }

      func inliningDemo() {
        t := 1
        // If 'fastAdd' is inlined, 'n' from fastAdd stays on the stack of main!
        p := fastAdd(&t)

        println(*p)
        // PROOF: Check the distance between addresses.
        // If addresses are very close (e.g., 8 bytes apart), they are on the stack.
        // If they are far apart, one is likely on the heap.
        println("t addr:", &t)
        println("p addr:", p)
      }

    ```
    -- REPRESENTATIVE OUTPUT (Function Inlined):
    --
      2
      t addr: 0xc000034760
      p addr: 0xc000034768  <-- Only 8 bytes away! Both are on the stack.


    -- REPRESENTATIVE OUTPUT (Inlining Disabled via //go:noinline):
    --
      2
      t addr: 0xc000034758
      p addr: 0xc0000140c0  <-- Huge jump! p is on the heap.





-- Advanced Escape Analysis & Manual Control
--
  Even with Go's powerful compiler optimizations, there are edge cases where
  function inlining fails to prevent heap escapes, and scenarios where a
  developer might need to manually force a value onto the heap.


  -- When Inlining Fails to Prevent Escape (Go v1.19 Defects)
  --
    Theoretically, inlining a function should allow the escape analyzer to see
    values as local to the caller. However, due to defects in constant
    propagation, passing a constant as a parameter to a function that calls
    make() often results in a heap allocation, even if the call is inlined.


  example
    ```go
      // createSlice is a simple wrapper around make.
      func createSlice(n int) []byte {
        // Logic: In Go v1.19, if 'n' is a parameter, the compiler is less
        // likely to keep this on the stack, even if 'n' is a constant in the caller.
        return make([]byte, n)
      }


      func inliningDefectDemo() {

        // -- Scenario A: Via Inlined Function --
        // -------------------------------------------------------------------------
        // Even if 'createSlice' is inlined here, the compiler (v1.19-) sees 'n'
        // as a variable within the inlined context and forces a heap allocation.
        x := createSlice(32)


        // -- Scenario B: Direct Allocation --
        // -------------------------------------------------------------------------
        // The compiler sees the constant '32' directly and keeps this on the stack.
        y := make([]byte, 32)

        _, _ = x, y
      }

    ```


-- COMPILER OUTPUT (-gcflags="-m"):
--
  ./main.go:26:21: inlining call to createSlice
  ./main.go:19:13: make([]byte, n) escapes to heap
  ./main.go:26:21: make([]byte, n) escapes to heap
  ./main.go:31:14: make([]byte, 32) does not escape





-- Manually Forcing Heap Allocation (The "Sink" Trick)
--
  Occasionally, for testing, performance tuning, or avoiding stack-size issues,
  you may want to ensure a local variable escapes to the heap.


  -- The Standard Library Pattern
  --
    The Go standard library uses a package-level "sink" variable. Assigning a
    local variable to a global one forces an escape because the global variable
    outlives the local stack frame.


  example
    ```go
      var sink interface{}

      //go:noinline
      func internalEscape(x interface{}) {
        sink = x    // Force the analyzer to see a global reference
        sink = nil  // Immediately clear it so the object can eventually be GC'd
      }

    ```


-- The User-Space Pattern (No-Inline Workaround)
--
  Since //go:noinline is not intended for standard user code, we can use a
  recursive logic trick. The Go compiler will not inline a function that
  might call itself.


  example
    ```go
      func userSpaceEscape(x interface{}) {
        if x == nil {
          // This recursive call (even if unreachable) prevents inlining.
          userSpaceEscape(&x)
          panic("x must not be nil")
        }
        sink = x
        sink = nil
      }


      func forceEscapeDemo() {
        var a = 1
        var b = true

        // These variables would normally stay on the stack.
        // By passing them to our escape function, we move them to the heap.
        userSpaceEscape(&a)
        userSpaceEscape(&b)

        fmt.Println(a, b)
      }

    ```
      -- COMPILER OUTPUT (-gcflags="-m"):
      --
        ./main.go:76:6: moved to heap: a
        ./main.go:77:6: moved to heap: b





-- Helping the Compiler with Explicit Copies
--
  Escape Analysis is the compiler's way of determining if a variable can stay
  on the fast "Stack" or must be moved to the managed "Heap."


  One of the most common reasons for a "Heap Escape" is when a closure
  (anonymous function) captures a variable by reference. However, we can
  often help the compiler keep data on the stack by providing explicit
  value copies.


  -- The Problem: Shared State via Closures
  --
    In the following example, the compiler conservatively moves 'a' and 'b'
    to the heap because they are shared and mutated across goroutine boundaries.


    example
      ```go
        func leakyExample() {
          var (
            a = 1               // moved to heap: a
            b = true            // moved to heap: b
            c = make(chan struct{})
          )


          go func() {
            // This closure captures 'a' and 'b' by REFERENCE.
            if b {
              a++
            }
            close(c)
          }()


          <-c
          // 'b' is mutated here in the main goroutine.
          // To ensure the child goroutine sees the correct memory,
          // Go forces 'b' to the heap.
          b = !b
          fmt.Println(a, b)
        }

      ```
        -- COMPILER OUTPUT (-gcflags="-m"):
        --
          ./main.go:21:3: moved to heap: a
          ./main.go:22:3: moved to heap: b
          ./main.go:26:5: func literal escapes to heap


  -- The Solution: Explicit Value Copying
  --
    We can avoid the heap escape by ensuring the goroutine never references
    the original variables. Instead, we provide it with its own private copies.


    example
      ```go
        func optimizedExample() {
          var (
            a = 1               // stays on stack
            b = true            // stays on stack
            c = make(chan int)
          )


          // TECHNIQUE 1: Explicit Local Copy
          // By creating 'b1' here, we "snapshot" the current value of 'b'.
          // The closure below will capture 'b1' instead of the mutable 'b'.
          b1 := b


          // TECHNIQUE 2: Function Parameter
          // We pass 'a' as a value argument to the goroutine function.
          // This creates a local copy 'aCopy' inside the goroutine's own stack.
          go func(aCopy int) {
            if b1 {
              aCopy++
              // We send the result back through the channel
              // rather than modifying the outer variable directly.
              c <- aCopy
            }
          }(a)


          // Sync and update the local 'a' with the calculated value.
          a = <-c


          // 'b' is mutated here, but since the goroutine only used 'b1',
          // 'b' never had to escape.
          b = !b


          fmt.Println(a, b)
        }

      ```
        -- COMPILER OUTPUT (-gcflags="-m"):
        --
          ./main.go:58:6: a does not escape
          ./main.go:59:6: b does not escape
          ./main.go:68:5: func literal does not escape  <-- The closure itself stays local!


  -- Why This Matters
  --
    -- Performance
    --
      Stack allocation is nearly instantaneous (pointer increment), whereas
      heap allocation requires a search through the memory allocator's buckets.


    -- GC Pressure
    --
      Values on the stack are "free." They are cleaned up automatically when
      the function returns. Values on the heap must be scanned and cleaned by
      the Garbage Collector, which consumes CPU cycles.


    -- Determinism
    --
      Passing values explicitly as arguments makes your concurrent logic
      significantly easier to reason about, as it eliminates "hidden" shared
      state between the parent and child goroutines.





-- Memory Size Thresholds for Stack vs. Heap Placement
--
  Even when the Go compiler's Escape Analysis proves that a value part is used
  only by a single goroutine (i.e., it "does not escape"), the compiler may
  still decide to allocate it on the HEAP.


  The primary reason is safety: Go goroutines have small, dynamic stacks.
  Allocating very large values on the stack increases the risk of a "Stack
  Overflow" crash. To prevent this, the compiler uses specific size thresholds.


  -- String and Byte Slice Conversions
  --
    Conversions between strings and byte slices often require a new memory block.

    -- Standard Conversion Threshold: 32 Bytes
    --
      If the result of `string(byteSlice)` or `[]byte(str)` is > 32 bytes, the
      underlying byte sequence is allocated on the heap.


    -- Constant String Optimization: 64 KB
    --
      If you convert a CONSTANT string to a byte slice, the threshold is relaxed.
      The data stays on the stack if the result is <= 64 KB (65,536 bytes).


    example
      ```go
      func conversionThresholds() {
          // 33-byte slice
          bs33 := make([]byte, 33)

          // 1. Result > 32 bytes -> Heap Allocation
          _ = string(bs33)

          // 2. Result <= 32 bytes -> Stack Allocation
          _ = string(bs33[:32])

          // 3. Constant string conversion (relaxed limit)
          const largeConst = "..." // assume length 1000
          _ = []byte(largeConst)   // Stays on STACK because 1000 < 64K
        }

      ```


  -- Pointer Allocations: `new(T)` and `&T{}`
  --
    When you create a value via `new` or a pointer-to-composite-literal, the
    size of the type `T` determines the location.


    -- Threshold: 64 KB
    --
      If `unsafe.Sizeof(T)` > 64 KB (65,536 bytes), the value is moved to the heap.


    example
      ```go
        type Small struct { data [65535]byte } // 64K - 1
        type Large struct { data [65537]byte } // 64K + 1


        func pointerThresholds() {
          // Allocated on STACK (below 64K)
          _ = new(Small)
          _ = &Small{}


          // Allocated on HEAP (above 64K)
          _ = new(Large)
          _ = &Large{}
        }

      ```





-- Slices via `make([]T, N)`
--
  The allocation of the backing array for a slice created with `make` depends
  on whether the size is known at compile time.


  -- Rule A: Constant Size (Threshold 64 KB)
  --
    If N is a constant and `Size(T) * N` > 64 KB, the backing array goes to the heap.


  -- Rule B: Variable Size (Always Heap)
  --
    If N is a variable (non-constant), the compiler cannot determine the size
    at compile time. For safety, the backing array is ALWAYS allocated on the heap,
    even if the runtime value of N is small (e.g., 1).


  example
    ```go
      func makeThresholds() {
        const N = 64 * 1024
        var v = 10

        // 1. Constant N, fits in 64K -> STACK
        _ = make([]byte, N-1)

        // 2. Constant N, exceeds 64K -> HEAP
        _ = make([]byte, N+1)

        // 3. Variable N -> HEAP (regardless of value)
        _ = make([]byte, v)
      }

    ```


  -- Local Variable Declarations
  --
    Standard local variables (not created via pointers or make) have a much
    higher threshold before they are moved to the heap.


    -- Threshold: 10 MB
    --
      If the direct part of a variable is > 10 MB (10,485,760 bytes), it is
      moved to the heap.


    example
      ```go
        func variableThresholds() {
          const TenMB = 10 * 1024 * 1024

          // 1. Exactly 10MB -> STACK
          var a [TenMB]byte

          // 2. 10MB + 1 byte -> HEAP
          var b [TenMB + 1]byte

          _, _ = a[0], b[0]
        }

      ```

-- Technical Note: The "Does Not Escape" Message
--
  When running `go build -gcflags="-m"`, you might see the message:
    "./main.go:XX:YY: make([]byte, 70000) does not escape"

  This is often confusing. It means the compiler successfully performed the
  logical analysis and found the value is not shared with other goroutines.

  HOWEVER, because 70,000 > 64K, the value is still moved to the heap for
  size reasons. The "escape" analysis only covers the logic of the code;
  the placement is determined by the logic + size thresholds.





-- Tuning and Bypassing Allocation Thresholds
--
  Go's allocation logic is governed by specific size thresholds. However, these
  thresholds are not entirely static; they can be modified by compiler flags
  or effectively "bypassed" using specific variable declaration patterns.


  -- The `-smallframes` Compiler Flag
  --
    The standard Go thresholds (64 KB for pointers/makes, 10 MB for local vars)
    are designed for modern systems. For memory-constrained environments, Go
    provides the `-smallframes` flag, which significantly lowers these limits.


  -- Threshold Shift
  --
    +-----------------------+------------------------+-----------------------------+
    | Operation Type        | Default Threshold      | -smallframes Threshold      |
    +-----------------------+------------------------+-----------------------------+
    | new(T), &T{}, make()  | 64 KB (65,536)         | 16 KB (16,384)              |
    | var x [N]T            | 10 MB (10,485,760)     | 128 KB (131,072)            |
    +-----------------------+------------------------+-----------------------------+


  example
    ```go
      // Constants reflecting the -smallframes limits
      const (
        SmallN = 16 * 1024  // 16KB
        SmallM = 128 * 1024 // 128KB
      )


      var globalX byte = 123


      // This function will allocate on the STACK normally,
      // but on the HEAP if -smallframes is enabled.
      func checkSmallframesAllocation() {
        // Size is slightly over 16KB
        a := make([]byte, SmallN+1)
        a[0] = globalX
      }

    ```
      -- VERIFICATION COMMANDS:
      --
        1. Standard Run:
           $ go run main.go
           (Allocations: 0)

        2. Small Frames Run:
           $ go run -gcflags="-smallframes" main.go
           (Allocations: 1)





-- Allocating Large Slices on the Stack
--
  As established, calling `make([]T, N)` with a constant N > 64 KB results
  in a heap allocation. However, you can bypass this 64 KB limit and reach
  the much higher 10 MB limit by changing your declaration pattern.


  -- The Technique
  --
    Instead of creating the slice directly, declare a local ARRAY first,
    then slice it.


  -- Why it works
  --
    The compiler applies a much more generous threshold (10 MB) to direct
    variable declarations (`var a [N]T`) than it does to dynamic header
    creations like `make()`.


  example
    ```go
      func stackLargeSliceDemo() byte {
        // 1. Standard approach: make([]byte, 1M) -> HEAP (because 1M > 64K)
        // 2. Optimized approach:

        const TenMB = 10 * 1024 * 1024

        // This array is 10MB. Because it is a direct 'var' declaration and
        // doesn't exceed 10MB, it stays on the STACK.
        var a [TenMB]byte

        // Derive a slice from the stack-allocated array.
        // The slice header is on the stack, and it points to the stack array.
        var s = a[:]

        // Perform work to ensure it's not optimized away
        s[500] = 1
        return s[500]
      }

    ```
    This is a powerful optimization for high-performance processing (like parsing
    large files or crypto buffers). It allows you to use massive temporary
    working memory with zero GC overhead.


  -- Safety Warnings
  --
    -- The Per-Goroutine Limit
    --
      Even if the 10MB hack works, remember that every goroutine has a hard
      stack limit (default 1GB on 64-bit). If you use this 10MB hack inside a
      recursive function, you will hit a Stack Overflow crash very quickly.


    -- Version Dependency 
    --
      These specific thresholds are implementation details of the standard 'gc'
      compiler. While they have been stable for several versions (v1.17 - v1.25),
      always verify with benchmarks and `-gcflags="-m"` when upgrading.





-- Advanced Stack Allocation Tricks & Pitfalls
--
  Go's Escape Analysis determines if a value lives on the Stack or Heap. 
  While standard thresholds (like 64KB for make) are usually enforced, 
  there are specific "corner cases" and compiler gaps (specifically in v1.19) 
  where the compiler fails to move massive values to the heap, allowing 
  for arbitrary-sized stack allocations.


  -- The Composite Literal Hack (Bypassing make() limits)
  --
    As established in previous notes, `make([]byte, N)` moves the backing array 
    to the heap if N > 64KB or if N is a variable. 

    
    -- The Gap (v1.19) 
    --
      The Go 1.19 compiler's escape analysis treats "Keyed Composite Literals" 
      differently. Using `[]byte{N: 0}` creates a slice of length N+1, but 
      the compiler often fails to apply the 64KB threshold here. If it can prove 
      the slice is used only in one goroutine, it stays on the stack—regardless 
      of size.


    example
      ```go
        // createMassiveStackSlice attempts to put 500MB on the stack.
        func createMassiveStackSlice() byte {
          const N = 500 * 1024 * 1024 // 500MB
          var v byte = 123

          // --- THE HINT ---
          // Using a composite literal with a high index key ({N: 0}).
          // Logic: In Go 1.19, this bypasses the standard heap-escape thresholds.
          var s = []byte{N: 0} 
          
          // Perform work to ensure the variable isn't optimized away.
          for i := 0; i < 10; i++ {
            s[i] = v
          }
          return s[0]
        }

      ```



-- BENCHMARK PROOF:
--
  ```go
    func main() {
        stat := testing.AllocsPerRun(1, func() {
            _ = createMassiveStackSlice()
        })
        fmt.Printf("Heap Allocations for 500MB slice: %v\n", stat)
    }

  ```

    -- EXPECTED OUTPUT (Go 1.19):
    --
      Heap Allocations for 500MB slice: 0


    -- WARNING: 
    --
      This is highly version-dependent. Future compilers will likely fix this 
      "leak" into the stack to prevent accidental stack overflows.
                                                                                
  



-- Arbitrary-Size Values on Stack: Corner Cases
--
  There are three other scenarios where Go may place massive arrays on the stack 
  instead of the heap, bypassing the usual 10MB variable threshold.


  example
    ```go
const MassiveSize = 1024 * 1024 * 100 // 100MB

      // -----------------------------------------------------------------------------
      // -- Case A: Boxing Large Values into Interfaces --
      // -----------------------------------------------------------------------------
      // While standard pointer-to-struct usually escapes, directly assigning a 
      // massive array to an interface can sometimes remain on the stack if the 
      // interface does not escape the function.
      func boxLargeValue() {
        type _ int // prevents inlining
        var x interface{} = [MassiveSize]byte{} 
        fmt.Println("Is interface nil?", x == nil)
      }

      // -----------------------------------------------------------------------------
      // -- Case B: Massive Value Parameters --
      // -----------------------------------------------------------------------------
      // Passing a massive array by value as a parameter. Because the parameter 
      // is part of the function's stack frame (pre-calculated), it stays on the stack.
      func largeParameter(x [MassiveSize]byte) {
        type _ int // prevents inlining
        _ = x[0]
      }

      // -----------------------------------------------------------------------------
      // -- Case C: Large Map Elements --
      // -----------------------------------------------------------------------------
      // Assigning a massive array as a value in a map. While the map structure 
      // itself is on the heap, the temporary value being assigned often lives 
      // on the stack before the copy into the map happens.
      func largeMapElement() {
        type _ int // prevents inlining
        var m = make(map[int][MassiveSize]byte)
        m[0] = [MassiveSize]byte{}
      }


    ```


  -- CRITICAL WARNING: Performance vs. Placement
  --
    Just because you CAN force a 100MB value onto the stack does not mean 
    you SHOULD. 


    -- Value Copy Costs 
    --
      As established in "Value Copy Costs," Go moves data by copying the direct part.
      - If you use a massive array as a function parameter (Case B), Go must 
        copy all 100MB onto the new function's stack frame. 
      - This is EXTREMELY slow compared to passing a 8-byte pointer.


    -- Stack Overflow Risk 
    --
      Stacks have a hard limit (default 1GB). If you allocate a 500MB slice 
      on the stack (Section 1) and then call a function that also needs 
      600MB of stack space, the program will crash immediately with a 
      "stack overflow," even if the system has 64GB of free RAM.


    -- Determinism 
    --
      These "tricks" rely on current compiler implementation details (v1.19). 
      Code that relies on these behaviors is fragile and likely to break (i.e., 
      move back to the heap) in future Go versions.





-- Optimizing Stack Growth Frequency
--
  In Go, goroutine stacks are dynamic. While this flexibility allows Go to 
  handle millions of goroutines, the process of growing a stack is not free. 
  

  -- The Cost of Growth 
  --
    A stack growth involves two distinct steps:
    1. Allocation: Finding a new, larger continuous memory segment.
    2. Migration: Copying the existing data from the old stack to the new one 
       and updating any pointers.
  

  The "Migration" step is the bottleneck. The larger the current stack is, 
  the more data must be copied during the next growth. If a goroutine grows 
  its stack 10 times, it performs 10 increasingly expensive memory copies.


  -- The Strategy: Pre-emptive Stack Warming
  --
    If you can predict that a goroutine will eventually require a large amount 
    of stack space (e.g., due to deep recursion or large local buffers), you 
    can "warm" the stack.

    
    -- The Trick 
    --
      Call a dummy function early in the goroutine's life that declares a massive 
      local variable (like a large array). This forces the Go runtime to jump 
      the stack size to a high "peak" value immediately.

    
    -- Why it is faster 
    --
      By forcing growth when the stack is nearly empty, the "Migration" step copies 
      almost nothing. Subsequent logic then has all the "room" it needs, 
      completely bypassing the overhead of multiple incremental reallocations.


  -- Code Demonstration: foo vs. bar
  --
    ```go
      // demo is a recursive function that consumes stack space incrementally.
      func demo(n int) byte {
        var a [8192]byte // Each frame is ~8KiB
        var b byte
        if n--; n > 0 {
          b = demo(n)
        }
        return a[n] + b
      }

      // foo represents the standard approach.
      // It will trigger many stack growths as the recursion goes deeper.
      func foo(c chan time.Duration) {
        start := time.Now()
        demo(8192)
        c <- time.Since(start)
      }

      // bar represents the optimized approach.
      // It uses a dummy anonymous function to force the stack to 128 MiB immediately.
      func bar(c chan time.Duration) {
        start := time.Now()

        // --- THE STACK WARMING TRICK ---
        // We call an anonymous function with a signature that prevents the 
        // compiler from optimizing it away easily.
        func(x *interface{}) {
          // Define a dummy type to prevent inlining.
          // Alternatively, one could use //go:noinline in a separate function.
          type _ int 
          
          if x != nil {
            // This branch is never taken (x is nil), but the compiler 
            // calculates the stack frame size based on the POSSIBILITY 
            // of this array existing. 
            // Frame size becomes ~64 MiB.
            *x = [1024 * 1024 * 64]byte{}
          }
        }(nil)

        // Now that the stack is pre-grown to >64 MiB, the deep recursion 
        // in demo() will not trigger any additional growths.
        demo(8192)
        c <- time.Since(start)
      }

      func main() {
        var c = make(chan time.Duration, 1)

        // Standard execution
        go foo(c)
        fmt.Println("Standard approach (foo): ", <-c)

        // Optimized execution
        go bar(c)
        fmt.Println("Pre-grown approach (bar):", <-c)
      }

    ```

  -- Performance Analysis
  --
    REPRESENTATIVE OUTPUT (Timing will vary based on hardware):
      Standard approach (foo):  42.051369ms
      Pre-grown approach (bar): 4.740523ms

  -- ANALYSIS:
  --
    - foo: Triggers ~10+ separate stack growths. Each time, it has to copy more 
      and more data as the recursion depth increases.
    - bar: Triggers exactly ONE stack growth at the very beginning. Because the 
      stack is empty, the copy cost is near-zero. 
    - RESULT: bar is ~10x more efficient in this specific recursive scenario.


  -- How the Compiler Views the Trick
  --
    In the `bar` example:
    1. The compiler sees a local array of 64 MiB in the anonymous function.
    2. It calculates the "Stack Frame Size" for that function as ~64 MiB.
    3. Upon entering that function, the runtime checks if the current stack 
       has 64 MiB available. It doesn't, so it allocates a new 128 MiB stack 
       (the next power of 2) and copies the current (minimal) state.
    4. The `demo` function then executes within that 128 MiB buffer.
