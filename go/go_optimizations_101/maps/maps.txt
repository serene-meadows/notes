-- Map Performance & Memory
--
  In Go, maps are implemented as hashtables. While they provide O(1) average 
  time complexity for lookups, their internal management of memory and 
  interaction with the Garbage Collector (GC) can have significant impacts 
  on application performance.





-- Growth and Persistence
--
  -- Theoretical vs. Physical Capacity 
  --
    Maps have "unlimited" capacity in theory (bounded only by RAM). Because of 
    their dynamic nature, the built-in `cap()` function is not applicable to maps.


  -- The Backing Array 
  --
    Internally, a map maintains an array of "buckets." 
    1. As entries are added, the "load factor" increases.
    2. When the load factor hits a threshold, Go allocates a NEW, larger backing 
       array (usually double the size).
    3. Existing entries are gradually moved to the new array (incremental evacuation).
    4. CRITICAL FACT: In the standard Go runtime, the backing array of a map 
       NEVER SHRINKS. Even if you delete every single entry, the map retains 
       the largest amount of memory it ever allocated.





-- Clearing Map Entries: Loop vs. Reallocation
--
  There are three main ways to clear a map. The choice depends on whether you 
  want to keep the allocated memory for reuse or release it to the OS.


  example
    ```go
      func clearingDemo() {
        m := make(map[int]int, 1000)

        // --- The Optimized Delete Loop 
        // Since Go 1.11, the compiler recognizes this specific pattern and 
        // optimizes it to be extremely fast.
        // NOTE: This does NOT release memory to the OS. It keeps the buckets 
        // allocated for future writes.
        for k := range m {
          delete(m, k)
        }


        // --- The `clear` Builtin (Go 1.21+) 
        // INFO GAIN: This is the modern, preferred way. It is even faster than 
        // the loop above and handles NaNs correctly.
        // NOTE: Like the loop, this clears data but keeps the allocated capacity.
        clear(m)


        // --- Releasing Memory 
        // To actually let the GC reclaim the memory block used by the buckets, 
        // you must lose all references to the old map.
        m = nil 
        // OR 
        m = make(map[int]int) 
      }

    ```





-- Efficient Mutation: Single vs. Double Hashing
--
  Every time you access a map via `m[key]`, Go must compute the hash of the key.
  Shorthand operators like `++` and `+=` are optimized to hash the key only ONCE.


  example
    ```go
      func incrementEfficiency() {
        m := map[int]int{99: 0}

        // --- FAST (Single Hash) ---
        m[99]++
        m[99] += 1

        // --- SLOW (Double Hash) ---
        // The runtime hashes 99 to find the value to read, then hashes 99 AGAIN 
        // to find the bucket to write to.
        m[99] = m[99] + 1
      }

    ```

      -- BENCHMARK RESULTS (Approximate):
      --
        m[k]++              11.31 ns/op
        m[k] += 1           11.21 ns/op
        m[k] = m[k] + 1     16.10 ns/op  (<-- ~40% Slower)
                                                                                
  


                                                                                
-- GC Optimization: Pointer-Free Maps
--
  The Garbage Collector (GC) "Mark" phase scans pointers to find reachable memory.


  THE OPTIMIZATION: If a map's Key and Value types contain NO pointers, 
  the GC marks the entire map as a "leaf" and skips scanning its entries 
  entirely. This can reduce GC pause times by orders of magnitude for large maps.


  -- Avoid Strings for Small, Fixed Keys 
  --
    Internally, a `string` contains a pointer to its byte array. Therefore, 
    any map with string keys (or values) MUST be scanned by the GC.


  -- Use Byte Arrays [N]byte instead 
  --
    If your keys have a predictable maximum length, use a fixed-size byte array. 
    Arrays are value types and, if they contain raw bytes, have no pointers.


  example
    ```go
      func gcOptimizationDemo() {
        // GC MUST scan every entry here (65,536 strings = 65,536 pointers).
        var mapA = make(map[string]int, 1<<16)

        // GC SKIPS scanning entries here. [32]byte contains no pointers.
        // This map is "GC Invisible."
        var mapB = make(map[[32]byte]int, 1<<16)

        _ = mapA
        _ = mapB
      }

    ```





-- Advanced Map Performance & Optimizations
--
  Maps are incredibly flexible, but high-frequency updates can trigger hidden 
  performance costs, especially when using byte-slice-to-string conversions. 
  This section explores how to bypass string allocation traps and optimize 
  map growth.


  -- The L-Value vs. R-Value Allocation Trap
  --
    As established in the "Strings" chapter:
    - R-Value (Retrieval): `v := m[string(byteSlice)]` is OPTIMIZED to be 0-alloc.
    - L-Value (Assignment): `m[string(byteSlice)] = v` triggers a HEAP ALLOCATION.


    Because `m[key]++` is logically an assignment (`m[key] = m[key] + 1`), it 
    counts as an L-value operation and triggers an allocation for the string 
    key every time it is called.





-- Case Study: High-Frequency Counter Patterns
--
  ```go
    // -----------------------------------------------------------------------------
    // -- Strategy A: The Naive Counter (Slowest) --
    // -----------------------------------------------------------------------------
    // Every increment triggers a string allocation because the conversion 
    // occurs in an assignment context.
    var wordCounterA = make(map[string]int)

    func IncA(w []byte) {
      wordCounterA[string(w)]++ 
    }

    // -----------------------------------------------------------------------------
    // -- Strategy B: The Pointer Counter (Faster) --
    // -----------------------------------------------------------------------------
    // We use a map of pointers: `map[string]*int`.
    // 1. We READ the map first: `p := m[string(w)]` (Optimized to 0-alloc).
    // 2. If it exists, we increment the integer at the pointer address.
    // 3. We only perform an L-value assignment (and thus an allocation) the 
    //    VERY FIRST time we see a new word.
    var wordCounterB = make(map[string]*int)

    func IncB(w []byte) {
      p := wordCounterB[string(w)] // Read (0-alloc)
      if p == nil {
        p = new(int)
        wordCounterB[string(w)] = p // Allocate & Write (Only happens once per unique key)
      }
      *p++ // Increment value at address (0-alloc)
    }

    // -----------------------------------------------------------------------------
    // -- Strategy C: The Index-Table Pattern (Peak Efficiency) --
    // -----------------------------------------------------------------------------
    // While Strategy B is fast, it creates millions of small pointers on the heap, 
    // which significantly slows down the Garbage Collector (GC scan).
    // 
    // Strategy C uses a map to store `string -> slice_index` and a separate slice 
    // to hold the actual counts.
    var wordIndexes = make(map[string]int)
    var wordCounters []int

    func IncC(w []byte) {
      if i, ok := wordIndexes[string(w)]; ok { // Read (0-alloc)
        wordCounters[i]++ // Update slice element (0-alloc)
      } else {
        // New entry: only allocate string key once
        wordIndexes[string(w)] = len(wordCounters)
        wordCounters = append(wordCounters, 1)
      }
    }

  ```

    -- BENCHMARK RESULTS (Approximate):
    --
      Benchmark_A:    11600 ns/op    2336 B/op    62 allocs/op
      Benchmark_B:     1543 ns/op       0 B/op     0 allocs/op
      Benchmark_C:     1609 ns/op       0 B/op     0 allocs/op


    -- ANALYSIS:
    --
      1. Strategy B and C are nearly 8x faster than the naive approach.
      2. Strategy C is superior for long-running apps because it creates almost 
         no pointers, keeping GC pause times low.
      3. Note: Benchmarks report "0 allocs" if the rate is < 1 per op, which 
         happens here because we only allocate once per unique key.
                                                                                
  


                                                                                
-- Optimization: Single-Step Map Growth
--
  Like slices, maps grow dynamically. When a map hits its load factor, 
  the runtime allocates a larger bucket array and re-hashes all existing items. 
  This is a very expensive operation.


  -- The Best Practice 
  --
    If you know the approximate number of entries in advance, provide a "Size Hint".
    This ensures the map starts with a large enough bucket array to avoid 
    mid-execution re-hashing.


    example
      ```go
        func mapGrowthOptimization() {
          const ExpectedEntries = 10000
          
          // --- BAD: Starts small, will resize many times ---
          m1 := make(map[string]int)

          // --- GOOD: One-step allocation ---
          m2 := make(map[string]int, ExpectedEntries)
          
          _ = m1
          _ = m2
        }

      ```





-- Index Tables vs. Maps for Small Key Sets
--
  In Go, it is common to want to eliminate verbose 'if-else' blocks to improve 
  code readability. While using a map with a boolean key (map[bool]T) is a 
  popular "clean" alternative, it carries a significant performance penalty. 


  For high-performance code, "Index Tables" (arrays) combined with a simple 
  conversion function provide the best of both worlds: clean syntax and 
  near-instant execution.


  -- The Comparison: Three Ways to Branch
  --
    Consider a scenario where we need to choose a function based on a boolean condition.


    ```go
      // Dummy functions for demonstration
      //go:noinline
      func f() { /* logic for true */ }

      //go:noinline
      func g() { /* logic for false */ }

      // -----------------------------------------------------------------------------
      // -- Strategy A: Standard If-Else (Fastest / Verbose) --
      // -----------------------------------------------------------------------------
      func IfElse(x bool) {
        if x {
          f()
        } else {
          g()
        }
      }

      // -----------------------------------------------------------------------------
      // -- Strategy B: Map Lookup (Clean / Slowest) --
      // -----------------------------------------------------------------------------
      // This reduces boilerplate but forces the runtime to calculate a hash for the 
      // boolean key and perform a map bucket lookup every time.
      var m = map[bool]func(){true: f, false: g}

      func MapSwitch(x bool) {
        m[x]()
      }

      // -----------------------------------------------------------------------------
      // -- Strategy C: Index Table (Fastest / Clean) --
      // -----------------------------------------------------------------------------
      // We use a small array (the index table). To access it, we convert the bool 
      // to an integer (0 or 1). This replaces a complex hash lookup with a 
      // simple memory offset calculation.

      func b2i(b bool) int {
        if b {
          return 1
        }
        return 0
      }

      // a[0] = false result, a[1] = true result
      var a = [2]func(){g, f}

      func IndexTable(x bool) {
        a[b2i(x)]()
      }

    ```

-- Benchmark Analysis (Go v1.19+, 64-bit)
--
  -- BENCHMARK RESULTS:
  --
    Benchmark_IfElse-4       4.155 ns/op    (Baseline)
    Benchmark_MapSwitch-4   47.460 ns/op    (~11x Slower!)
    Benchmark_IndexTable-4   4.135 ns/op    (Equivalent to If-Else)


  -- ANALYSIS:
  --
    The Map approach is an order of magnitude slower because maps are heavy 
    data structures designed for large, dynamic sets of keys. For a key type 
    with only two possible values (bool), the map overhead is strictly wasted.
                                                                                
  


                                                                                
-- Practical Pattern: Boolean to Integer Conversion
--
  Since Go does not allow direct casting of bool to int (like C does), 
  the `b2i` helper is essential. This pattern can be used for counters, 
  toggles, or function selection.


  ```go
    func counterExample(condition bool) {
      counter := 10
      
      // --- Instead of: ---
      // if condition { counter++ } else { counter-- }
      
      // --- Use an Index Table for the delta: ---
      deltas := [2]int{-1, 1}
      counter += deltas[b2i(condition)]
      
      fmt.Println("New Counter:", counter)
    }

  ```
