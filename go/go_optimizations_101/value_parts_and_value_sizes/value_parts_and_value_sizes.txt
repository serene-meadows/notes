-- Value Parts and Memory Layout
--
  Understanding how Go represents values in memory is fundamental to mastering
  performance and avoiding bugs related to shared state.


  In Go, every value is composed of one or more "parts" in memory.
  (Note: The terminology "Value Part" is a conceptual model used in Go 101
  for clarity, though not a strictly official specification term.)





-- The Core Mental Model: Direct vs. Indirect Parts
--
  Every value in Go has a "Direct Part". Some values also have "Indirect Parts".


  -- Direct Part
  --
    This is the memory segment directly associated with the variable identifier.
    It is stored "in-line" (e.g., on the stack or within a parent struct).


  -- Indirect Parts
  --
    These are separate memory segments referenced by the direct part via pointers.
    If a value has indirect parts, the direct part acts as a "header" or
    "descriptor" for that data.





-- Classification of Types
--
  -- Category A: Single-Part Types (Direct Part Only)
  --
    Values of these types are always hosted on a single memory block.

    - Boolean types
    - Numeric types (int8/16/32/64, uint8/16/32/64, float32/64, complex64/128, etc.)
    - Pointer types (Safe and Unsafe)
    - Struct types (Fields are part of the direct memory segment)
    - Array types (Elements are part of the direct memory segment)


  -- Category B: Multi-Part Types (Direct + Indirect Parts)
  --
    Values of these types consist of a direct header pointing to internal data.

    - Slice types      (Direct: pointer/len/cap | Indirect: backing array)
    - Map types        (Direct: pointer         | Indirect: hashtable internal)
    - Channel types    (Direct: pointer         | Indirect: circular buffer/wait queues)
    - Function types   (Direct: pointer         | Indirect: closure/function object)
    - Interface types  (Direct: type/data ptrs  | Indirect: dynamic value data)
    - String types     (Direct: pointer/len     | Indirect: underlying byte sequence)






-- Copy Behavior: The "Shallow Copy" Rule
--
  RULE: In Go, every assignment or value copy is a SHALLOW copy.
  Only the DIRECT PART of the value is duplicated.


  -- The Consequence
  --
    After a copy, the destination and source direct parts both reference the
    SAME indirect underlying parts. This is why modifying a slice element
    through a copy affects the original, whereas modifying a struct field
    through a copy does not.


  example
    ```go
      func copyDemo() {
        // --- Example 1: Single-Part Copy (Array)
        originalArr := [2]int{1, 2}
        copiedArr := originalArr // Full data copy
        copiedArr[0] = 99
        fmt.Println(originalArr[0]) // Output: 1 (Original is unchanged)


        // --- Example 2: Multi-Part Copy (Slice)
        originalSlc := []int{1, 2}
        copiedSlc := originalSlc // Only header (ptr, len, cap) is copied
        copiedSlc[0] = 99
        fmt.Println(originalSlc[0]) // Output: 99 (Original is modified!)
      }

    ```


-- Runtime Memory Association
--
  - Each value part is carried on one "Memory Block".
  - A single-part value (like a struct) occupies exactly one memory block.
  - A multi-part value (like a map) is distributed across at least two
    different memory blocks.


  -- MEMORY VISUALIZATION (Slice Example):
  --
     Variable 's' (Direct Part)         Backing Array (Indirect Part)
     +-----------------------+          +-----------+
     | Pointer -----------------------> | Element 0 |
     | Len: 2                |          | Element 1 |
     | Cap: 2                |          +-----------+
     +-----------------------+
     (Block #1: Stack/Heap)             (Block #2: Heap)





-- Value and Type Sizes
--
  In Go, "size" refers specifically to the number of bytes allocated for the
  "Direct Part" of a value at runtime. Understanding what contributes to this
  size is key to optimizing memory and understanding Go's copy semantics.


  -- The Core Definition: Size == Direct Part
  --
    The size of a value is exactly the size of its direct part.
    Indirect parts (the data referenced via pointers) DO NOT contribute to the
    size of the value.


  -- Why?
  --
    Since Go uses shallow copies (copying only the direct part), the size reflects
    exactly what is moved during an assignment. Because indirect parts can be
    shared by multiple values (e.g., two slices pointing to one array), it would
    be logically inconsistent to include them in the size.


  -- Type Size
  --
    In the standard Go compiler, all values of a specific type have the same size.
    Thus, "Value Size" and "Type Size" are interchangeable terms.


  -- Structs and Arrays (The "Inline" Containers)
  --
    Structs and Arrays are the only type kinds where size varies based on
    composition/length. They "hold" their data directly.


    -- Structs
    --
      A struct is composed of its fields. At runtime, all fields are allocated in
      the same memory block as the struct.
      Rule: Struct Size = Sum of Field Sizes (+ Padding bytes).


    -- Arrays
    --
      An array is composed of its elements. All elements live in the same memory
      block as the array header.
      Rule: Array Size = Element Size * Array Length.


    example
      ```go
        func aggregateSizeDemo() {
          type Sample struct {
            a int64 // 8 bytes
            b int64 // 8 bytes
          }

          var s Sample
          var arr [10]int64

          fmt.Printf("Struct size: %d bytes\n", unsafe.Sizeof(s))   // Output: 16
          fmt.Printf("Array size:  %d bytes\n", unsafe.Sizeof(arr)) // Output: 80
        }

      ```


  -- Headers and Pointers (The Constant Sized)
  --
    For these types, the size is constant for the "Kind," regardless of how much
    data they reference.


    -- Pointers (*T)
    --
      A pointer only holds a memory address. It does not hold the value it
      points to.
      - Size: 1 Word (8 bytes on 64-bit).
      - Fact: A nil pointer and a non-nil pointer have the exact same size.


    -- Slices ([]T)
    --
      A slice is a header containing: { pointer to data, length, capacity }.
      - Size: 3 Words (24 bytes on 64-bit).
      - Fact: A slice of 1 element and a slice of 1 million elements are the
        same size because the elements are stored in the indirect part.


    -- Strings
    --
      A string is a header containing: { pointer to data, length }.
      - Size: 2 Words (16 bytes on 64-bit).
      - Fact: In logic, we think strings "hold" bytes; in implementation, they
        only reference them.


    -- Interfaces
    --
      An interface is a header containing: { type info pointer, data pointer }.
      - Size: 2 Words (16 bytes on 64-bit).


  -- Internal Pointers (The "Hidden" Pointers)
  --
    In the standard Go compiler, Maps, Channels, and Functions are implemented
    directly as pointers to internal runtime structures.

    - Map: Internal `*hmap`. Size: 1 Word.
    - Channel: Internal `*hchan`. Size: 1 Word.
    - Function: Internal function pointer/closure. Size: 1 Word.



  example
    ```go
      func headerSizeDemo() {
        var ptr *int                    // 8 bytes
        var slc []int                   // 24 bytes
        var str string                  // 16 bytes
        var iface any                   // 16 bytes
        var m map[int]int               // 8 bytes
        var c chan int                  // 8 bytes

        fmt.Println("--- Header Sizes (64-bit) ---")
        fmt.Printf("Pointer:   %d\n", unsafe.Sizeof(ptr))
        fmt.Printf("Slice:     %d\n", unsafe.Sizeof(slc))
        fmt.Printf("String:    %d\n", unsafe.Sizeof(str))
        fmt.Printf("Interface: %d\n", unsafe.Sizeof(iface))
        fmt.Printf("Map:       %d\n", unsafe.Sizeof(m))
        fmt.Printf("Channel:   %d\n", unsafe.Sizeof(c))
      }

    ```





-- Detailed Type Sizes & Alignment
--
  This section provides a granular look at how many bytes different Go types
  occupy and the rules the compiler follows to "align" those bytes in memory
  for optimal CPU performance.


  -- Value Sizes (Standard Go Compiler / gc)
  --
    "One Word" = 4 bytes on 32-bit systems, 8 bytes on 64-bit systems.

    +--------------------------------+--------------+------------------------------+
    | Type Kind                      | Size (Bytes) | Go Specification Requirement |
    +--------------------------------+--------------+------------------------------+
    | int8, uint8 (byte), bool       | 1            | Exactly 1 (bool unspecified) |
    | int16, uint16                  | 2            | Exactly 2                    |
    | int32 (rune), uint32, float32  | 4            | Exactly 4                    |
    | int64, uint64, float64,        | 8            | Exactly 8                    |
    | complex64                      | 8            | Exactly 8                    |
    | complex128                     | 16           | Exactly 16                   |
    |--------------------------------|--------------|------------------------------|
    | int, uint                      | 1 Word       | Platform dependent           |
    | uintptr, pointer (safe/unsafe) | 1 Word       | Large enough for address     |
    | map, channel, function         | 1 Word       | Unspecified                  |
    |--------------------------------|--------------|------------------------------|
    | string                         | 2 Words      | Unspecified                  |
    | interface                      | 2 Words      | Unspecified                  |
    |--------------------------------|--------------|------------------------------|
    | slice                          | 3 Words      | Unspecified                  |
    |--------------------------------|--------------|------------------------------|
    | array                          | ElemSize * N | Zero if ElemSize is zero     |
    | struct                         | FieldSum +   | Zero if all fields are zero  |
    |                                | Padding      |                              |
    +--------------------------------+--------------+------------------------------+


  -- Memory Alignment Guarantees
  --
    To utilize CPU instructions efficiently, the starting address of a memory
    block must be a multiple of N. This N is the "Alignment Guarantee."

    -- Measuring Alignment
    --
      - unsafe.Alignof(v): General alignment (used for variables/elements).
      - unsafe.Alignof(s.f): Field alignment (used for fields within a struct).
      - reflect.TypeOf(v).Align(): Runtime general alignment.
      - reflect.TypeOf(v).FieldAlign(): Runtime field alignment.


    -- Alignment Table (Standard Compiler)
    --
      +----------------------------+-------------------------------------------------+
      | Type                       | Alignment Guarantee (Bytes)                     |
      +----------------------------+-------------------------------------------------+
      | bool, uint8, int8 (byte)   | 1                                               |
      | uint16, int16              | 2                                               |
      | uint32, int32 (rune)       | 4                                               |
      | float32, complex64         | 4                                               |
      | Arrays                     | Matches Element Type Alignment                  |
      | Structs                    | Matches Largest Alignment of its fields         |
      | Other Types                | Size of a Native Word (4 or 8)                  |
      +----------------------------+-------------------------------------------------+


  -- Formal Specification Rules
  --
    The Go spec defines two recursive rules for composite types:

    1. Structs: The alignment of a struct is the maximum alignment of any
       of its individual fields (but at least 1).
    2. Arrays: The alignment of an array is identical to the alignment of
       its element type.


  -- The Relationship Between Size and Alignment
  --
    The Rule: The total size of a type is ALWAYS a multiple of its
    alignment guarantee.

    If a struct contains an int64 (8-byte alignment), the entire struct's
    size will be padded until it is a multiple of 8. This ensures that
    in an array of that struct, every element starts at an 8-byte boundary.


  -- Code Demonstration
  --
    ```go
      func main() {
        type Example struct {
          a int8  // Size 1, Align 1
          b int64 // Size 8, Align 8
        }

        var e Example
        t := reflect.TypeOf(e)

        fmt.Println("--- Alignment Guarantees (64-bit) ---")

        // Predeclared rule: Struct alignment = max(field alignments)
        // int64 is 8, so Example is 8.
        fmt.Printf("Struct Alignment: %d\n", unsafe.Alignof(e))

        // Predeclared rule: Array alignment = element alignment
        var arr [10]int8
        fmt.Printf("Array [10]int8 Alignment: %d\n", unsafe.Alignof(arr)) // Output: 1

        // Size/Alignment Multiplicity Check
        // Size of Example is 16 (1 + 7 padding + 8).
        // 16 is a multiple of 8.
        fmt.Printf("Struct Size: %d, Alignment: %d\n", unsafe.Sizeof(e), unsafe.Alignof(e))

        // Measuring via Reflection
        fmt.Printf("Reflection General Align: %d\n", t.Align())
        fmt.Printf("Reflection Field Align:   %d\n", t.FieldAlign())
      }

    ```

    -- EXPECTED OUTPUT (64-bit Architecture):
    --
      --- Alignment Guarantees (64-bit) ---
      Struct Alignment: 8
      Array [10]int8 Alignment: 1
      Struct Size: 16, Alignment: 8
      Reflection General Align: 8
      Reflection Field Align:   8





-- Struct Padding & Memory Efficiency
--
  Go compilers ensure that fields within a struct are properly aligned to
  optimize CPU memory access. To do this, the compiler often inserts
  "Padding Bytes"—unused space between fields or at the end of a struct.


  -- The Mechanics of Padding
  --
    Padding occurs for two reasons:

    1. FIELD ALIGNMENT:
       Each field must start at an address that is a multiple of its own
       alignment guarantee. If the previous field ends at an "unaligned" address,
       the compiler adds padding.

    2. STRUCT ALIGNMENT (Trailing Padding):
       The total size of a struct must be a multiple of its own alignment
       guarantee (which is the largest alignment of its fields). This ensures
       that when structs are placed in an array, every element remains aligned.


  -- Case Study: Field Order Matters
  --
    ```go
      // -- Example T1: Unoptimized Layout (64-bit)

      // Alignment Guarantee: 8 bytes (due to int64)
      type T1 struct {
        a int8  // Size: 1. Offset: 0.
                // [7 bytes PADDING] to align 'b' to 8-byte boundary.
        b int64 // Size: 8. Offset: 8.
        c int16 // Size: 2. Offset: 16.
                // [6 bytes PADDING] to make total size multiple of 8 (24).
      }


      // -- Example T2: Optimized Layout (64-bit) --

      // Alignment Guarantee: 8 bytes (due to int64)
      type T2 struct {
        a int8  // Size: 1. Offset: 0.
                // [1 byte PADDING] to align 'c' to 2-byte boundary.
        c int16 // Size: 2. Offset: 2.
                // [4 bytes PADDING] to align 'b' to 8-byte boundary.
        b int64 // Size: 8. Offset: 8.
      }
      // Total size: 16. (16 is a multiple of 8, so no trailing padding needed).

      func main() {
        // Visualization of results on 64-bit architectures
        fmt.Printf("Size of T1 (Unoptimized): %d bytes\n", unsafe.Sizeof(T1{})) // Output: 24
        fmt.Printf("Size of T2 (Optimized):   %d bytes\n", unsafe.Sizeof(T2{})) // Output: 16

        // INFO GAIN: Breaking down the offsets
        t1 := T1{}
        fmt.Printf("T1 Offsets -> a:%d, b:%d, c:%d\n",
          unsafe.Offsetof(t1.a), unsafe.Offsetof(t1.b), unsafe.Offsetof(t1.c))

        t2 := T2{}
        fmt.Printf("T2 Offsets -> a:%d, c:%d, b:%d\n",
          unsafe.Offsetof(t2.a), unsafe.Offsetof(t2.c), unsafe.Offsetof(t2.b))
      }

    ```

      -- EXPECTED OUTPUT (64-bit):
      --
        Size of T1 (Unoptimized): 24 bytes
        Size of T2 (Optimized):   16 bytes
        T1 Offsets -> a:0, b:8, c:16
        T2 Offsets -> a:0, c:2, b:8


  -- Practical Optimization Strategy
  --
    While Go does not automatically reorder your fields (to respect the order
    you defined), you can manually optimize for memory by following this rule:

    Rule: Order fields from LARGEST alignment requirement to SMALLEST.

    1. 8-byte fields (int64, float64, pointers, maps, slices, channels)
    2. 4-byte fields (int32, float32)
    3. 2-byte fields (int16)
    4. 1-byte fields (int8, bool, byte)


    example
      ```go
        // --- WORST (24 bytes) ---
        type Bad struct {
           a bool
           b *int
           c bool
        }

        // --- BEST (16 bytes) ---
        type Good struct {
           b *int
           a bool
           c bool
        }

      ```


  -- The Trade-off: Performance vs. Readability
  --
    1. PERFORMANCE: Alignment is mandatory for CPU efficiency. Padding is the
       cost we pay for that speed.

    2. READABILITY: Often, it is better to group logically related fields
       together (e.g., firstName and lastName) even if it wastes a few bytes.

    3. WHEN TO OPTIMIZE: Only reorder fields when you are dealing with:
       - Structs created in massive quantities (millions of instances).
       - Memory-constrained environments (embedded systems).
       - Large structs where reordering saves significant percentages of space.





-- Value Copy Costs & Small-Size Types
--
  In Go, value copying is a frequent operation (assignments, parameter passing,
  etc.). While the cost is generally proportional to the size of the value,
  modern compilers and CPUs apply specific optimizations to "small-size" types
  that can make them significantly faster than expected.


  -- Defining "Small-Size" Types
  --
    There is no formal definition of a "small-size" type, as it depends on the
    CPU architecture and compiler version. However, we can categorize them:


    -- Inherently Small Types
    --
      These types are always small because their "direct part" is a fixed header:
      - Pointers, Booleans, Numeric types (int, float, etc.)
      - Strings (2 native words)
      - Interfaces (2 native words)
      - Slices (3 native words)
      - Maps, Channels, Functions (1 native word/pointer)


    -- Structs and Arrays (The Variable Factor)
    --
      Structs and arrays are "small" only if their total size is low.
      A practical rule of thumb:
      - Structs with <= 4 fields (of native word size)
      - Arrays with <= 4 elements (of native word size)


  -- The Performance "Leap" (Compiler Optimization Thresholds)
  --
    The Go compiler (specifically gc) uses CPU registers to move small data.
    Once the data exceeds a certain threshold, the compiler stops using registers
    and switches to a generic memory-copying function (like memmove), which
    results in a significant performance "leap" or regression.


    -- Benchmark Evidence: 9 vs 10 Elements (Go 1.19/1.20+) --
    --
      Benchmarks show a drastic increase in time-per-operation once a struct
      exceeds 9 fields or an array exceeds 9 elements (assuming native word size).

      -- BENCHMARK RESULTS (Approximate):
      --
        Benchmark_CopyArray_9_elements-4     3974 ns/op  <-- Optimized
        Benchmark_CopyArray_10_elements-4    8896 ns/op  <-- ~2.2x Slower

        Benchmark_CopyStruct_9_fields-4      2970 ns/op  <-- Optimized
        Benchmark_CopyStruct_10_fields-4     8471 ns/op  <-- ~2.8x Slower


  -- Performance in Function Calls (T4 vs T5)
  --
    This optimization also applies to how values are returned from functions.
    In the example below, adding just one field (from 4 to 5) causes a massive
    performance hit because the compiler can no longer fit the struct entirely
    into registers for the return operation.


    example
      ```go
        type T4 struct{ a, b, c, d float32 }
        type T5 struct{ a, b, c, d, e float32 }


        // //go:noinline prevents the compiler from removing the copy cost
        // via inlining, allowing us to measure the raw transfer cost.


        //go:noinline
        func Add4(x, y T4) T4 {
          return T4{x.a + y.a, x.b + y.b, x.c + y.c, x.d + y.d}
        }


        //go:noinline
        func Add5(x, y T5) T5 {
          return T5{x.a + y.a, x.b + y.b, x.c + y.c, x.d + y.d, x.e + y.e}
        }


        func main() {
          // The performance difference between these two calls is non-linear.
          // Add4 is often 7x to 10x faster than Add5 in benchmarks.
          res4 := Add4(T4{1, 1, 1, 1}, T4{2, 2, 2, 2})
          res5 := Add5(T5{1, 1, 1, 1, 1}, T5{2, 2, 2, 2, 2})

          fmt.Println("T4 Result:", res4)
          fmt.Println("T5 Result:", res5)
        }

      ```


    -- EXPECTED BENCHMARK OUTPUT (Standard gc compiler):
    --
      Benchmark_Add4-4    2.65 ns/op   (Highly Optimized via Registers)
      Benchmark_Add5-4   19.15 ns/op   (Unoptimized / Memory Move)


  -- Common Copy Scenarios
  --
    Value copying occurs in the following situations:
    1. Value Boxing: Converting a concrete value into an interface.
    2. Function Boundaries: Passing arguments to a function and returning results.
    3. Channel Operations: Sending a value to or receiving a value from a channel.
    4. Map Mutations: Inserting an entry into a map.
    5. Slice Growth: Using the built-in 'append' function.
    6. Iteration: Using 'for-range' loops (especially with the second variable).


  -- Case Study 1: The 'for-range' Overhead on Arrays
  --
    When ranging over a container, Go makes a copy of the container's direct part.
    For a slice, this is just a 24-byte header. For an array, this is the
    ENTIRE data set.


    example
      ```go
        const N = 1024

        // --- SLOWEST: Double Copy ---
        // 1. The array is copied when passed as a parameter.
        // 2. The array is copied AGAIN by the 'range' keyword.
        func Sum_RangeArray(a [N]int) (r int) {
          for _, v := range a {
            r += v
          }
          return
        }

        // --- FASTER: Single Copy ---
        // 1. Pointer is passed (cheap).
        // 2. But 'range *a' explicitly dereferences and COPIES the array.
        func Sum_RangeArrayPtr1(a *[N]int) (r int) {
          for _, v := range *a {
            r += v
          }
          return
        }

        // --- FASTEST: No Array Copy ---
        // 1. Pointer is passed (cheap).
        // 2. Go optimization: Ranging over a pointer to an array does NOT copy the array.
        func Sum_RangeArrayPtr2(a *[N]int) (r int) {
          for _, v := range a {
            r += v
          }
          return
        }

        // --- FASTEST: Slice Header ---
        // 1. Only the 24-byte slice header is copied.
        func Sum_RangeSlice(a []int) (r int) {
          for _, v := range a {
            r += v
          }
          return
        }

      ```

      -- REPRESENTATIVE BENCHMARK RESULTS (N=1024):
      --
        Benchmark_Sum_RangeArray-4      897.6 ns/op  (Double Copy)
        Benchmark_Sum_RangeArrayPtr1-4  799.3 ns/op  (Single Copy)
        Benchmark_Sum_RangeArrayPtr2-4  555.3 ns/op  (Optimized - No Copy)
        Benchmark_Sum_RangeSlice-4      561.7 ns/op  (Optimized - No Copy)


  -- Case Study 2: The Second Iteration Variable Trap
  --
    When you use the form 'for i, v := range container', Go copies the value
    of the current element into 'v' on every single iteration.
    If the elements are large (e.g., structs or arrays), this is very expensive.


    example
      ```go
        type Element [10]int64 // 80 bytes (not a "small-size" type)

        // --- FAST: Index-only ---
        func Sum_OneIterationVar(s []Element) (r int64) {
          for i := range s {
            // We access the element directly via index; no copy of the element is made.
            r += s[i][0]
          }
          return
        }


        // --- SLOW: Value Copy ---
        func Sum_UseSecondIterationVar(s []Element) (r int64) {
          for _, v := range s {
            // 'v' is a fresh 80-byte copy of the element in every iteration.
            r += v[0]
          }
          return
        }

      ```

      -- REPRESENTATIVE BENCHMARK RESULTS (1000 Elements):
      --
        Benchmark_OneIterationVar-4        929.3 ns/op
        Benchmark_UseSecondIterationVar-4  3753.0 ns/op  (~4x Slower!)





  -- Case Study 3: The "Small-Size" Struct Threshold
  --
    As established in previous notes, Go optimizes copies for structs with
    4 or fewer fields. Once you hit 5 fields (of native word size),
    performance drops if you use the second iteration variable.


    example
      ```go
        type S struct { a, b, c, d, e int } // 5 fields: Exceeds register optimization


        // --- SLOW: Copies 5 fields per iteration ---
        func sum_UseSecondIterationVar(s []S) int {
          var sum int
          for _, v := range s {
            sum += v.c + v.d + v.e
          }
          return sum
        }


        // --- FAST: Access via Index ---
        func sum_OneIterationVar_Index(s []S) int {
          var sum int
          for i := range s {
            sum += s[i].c + s[i].d + s[i].e
          }
          return sum
        }


        // --- FAST: Access via Pointer ---
        func sum_OneIterationVar_Ptr(s []S) int {
          var sum int
          for i := range s {
            v := &s[i] // Taking the address is cheap
            sum += v.c + v.d + v.e
          }
          return sum
        }

      ```
      -- REPRESENTATIVE BENCHMARK RESULTS (1000 Elements):
      --
        Benchmark_UseSecondIterationVar-4  2208 ns/op
        Benchmark_OneIterationVar_Index-4  1212 ns/op  (~2x Faster)
        Benchmark_OneIterationVar_Ptr-4    1182 ns/op  (~2x Faster)


      If we removed field 'e' (reducing to 4 fields), all three would perform
      identically because the compiler would optimize the copy via CPU registers.
