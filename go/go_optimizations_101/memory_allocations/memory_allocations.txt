-- Memory Allocations
--
  Go's memory management is designed to be automated and efficient. However,
  understanding the underlying mechanics of memory blocks and the distinction
  between Stack and Heap allocations is vital for writing high-performance code.


  -- Memory Blocks: The Fundamental Unit
  --
    A "Memory Block" is a continuous segment of memory. It serves as the physical
    container for "Value Parts" at runtime.

    -- Key Properties
    --
      1. One-to-Many: A single memory block can host multiple value parts
         (e.g., all fields of a struct or all elements of an array).
      2. Size Constraint: A memory block's size must be greater than or equal to
          the total size of all value parts it contains.
      3. Referencing: When a value part is stored in a block, that value part is
         said to be "referencing" the block.
      4. Performance Cost: Allocating a block requires the CPU to find a suitable
         segment. Reducing the number of allocations directly improves execution speed.


  -- Allocation Zones: Stack vs. Heap
  --
    When Go needs a memory block, it chooses between two distinct zones:

    -- The Stack (Goroutine Local)
    --
      - Scope: Each goroutine has its own stack.
      - Cost: Extremely cheap to allocate and deallocate.
      - Cleanup: No Garbage Collection (GC) required. The entire stack is reclaimed
        at once when the goroutine exits.
      - Visibility: Data on a stack is only accessible by that specific goroutine.


    -- The Heap (Program Shared)
    --
      - Scope: One global heap for the entire program.
      - Cost: Expensive to allocate compared to the stack.
      - Cleanup: Managed by the Garbage Collector. Blocks are only collected when
        no "alive" value parts reference them.
      - GC Pressure: The more blocks on the heap, the harder the GC has to work,
        consuming more CPU resources.


    example
      ```go
        func allocationDemonstration() {
          // --- Stack Allocation
          // 'x' is likely allocated on the stack because it doesn't leave
          // the scope of this function.
          x := 42
          fmt.Println(x)


          // --- Heap Allocation
          // 'y' might "escape" to the heap if it is shared outside this scope
          // or if the compiler cannot prove it stays local.
          y := new(int)
          *y = 100
        }

      ```


  -- Benchmarking and Metrics
  --
    In Go, when you run benchmarks (`go test -bench`), the "allocs/op" metric
    refers strictly to HEAP allocations.


    Stack allocations are so fast that they are generally ignored in high-level
    metrics, though they still carry a minor cost. Reducing heap allocations
    is the primary goal of memory optimization in Go.





-- Escape Analysis: The Decision Maker
--
  The Go compiler uses a process called "Escape Analysis" to decide where
  memory should be allocated.


  -- The Goal
  --
    The compiler wants to put as much as possible on the Stack.
    It moves a value to the Heap ONLY if:
    1. The value is referenced across goroutines.
    2. The value is returned from a function (escapes the stack frame).
    3. The value is too large for the stack.
    4. The compiler cannot definitively prove the value won't be accessed
       after the function returns.


  -- Code Pattern: Stack vs. Heap Escape
  --
    ```go
      type Data struct {
        Value int
      }

      // This function likely results in a stack allocation for 'd'.
      func stayOnStack() int {
        d := Data{Value: 10}
        return d.Value
      }

      // This function results in a heap allocation for 'd' because
      // the pointer to 'd' is returned, escaping the function's stack frame.
      func escapeToHeap() *Data {
        d := Data{Value: 20}
        return &d
      }

    ```





-- Memory Allocation Scenarios & Wasting
--
  Go's memory management is optimized for speed. One of the ways it achieves
  this is by allocating memory in fixed-size "buckets" rather than arbitrary
  byte counts. This leads to high-performance allocation but results in
  "Internal Fragmentation" (memory wasting).


  -- Standard Allocation Scenarios
  --
    Generally, each of the following operations triggers at least one allocation
    (usually on the heap, depending on escape analysis):


  -- Construction & Declarations
  --
    - Variable declarations (var x T).
    - Calling built-in 'new(T)' (1 block).
    - Calling built-in 'make(T, ...)' (Multiple blocks for headers and data).
    - Initializing slices or maps with composite literals (T{...}).


  -- Conversions & String Logic
  --
    - Converting integers to strings: string(65) -> "A".
    - String concatenation using '+': "a" + "b" (unless optimized by compiler).
    - string <-> []byte and string <-> []rune (requires a deep copy).


  -- Dynamic Data Growth
  --
    - Boxing values into interfaces: var i any = 10 (allocates to store the int).
    - slice append: If current capacity is exceeded, a new larger block is allocated.
    - map insertion: If the hash table needs to resize, new memory blocks are allocated.


  NOTE: The standard Go compiler (gc) performs "Escape Analysis" and specific
  optimizations to skip these allocations if it proves the data is short-lived
  or used in a way that doesn't require a fresh heap block.


  -- The "Size Class" Strategy (Internal Fragmentation)
  --
    Heap memory is not allocated in arbitrary sizes. Go uses two strategies based
    on the request size:


    -- Small Objects (<= 32,768 bytes)
    --
      Small objects are binned into "Size Classes."
      Common buckets: 8, 16, 24, 32, 48, 64, 80, 96, ... bytes.


      The Rule: If you request 33 bytes, Go cannot give you exactly 33.
      It gives you the next available bucket: 48 bytes.
      Result: 15 bytes of wasted space (Internal Fragmentation).


    -- Large Objects (> 32,768 bytes)
    --
      Large objects are allocated as a multiple of "Memory Pages."
      Page Size (Go 1.19+): 8,192 bytes (8 KiB).


      The Rule: If you request 32,769 bytes (32KB + 1), Go rounds up to
      the next page boundary.
      Calculation: 32,768 + 8,192 = 40,960 bytes.
      Result: 8,191 bytes wasted.


  -- Code Demonstration: Proving Memory Wasting
  --
    ```go
      func fragmentationDemo() {
        // --- Case A: Small Struct ---
        // [5]int64 is 5 * 8 = 40 bytes.
        var t *[5]int64


        // unsafe.Sizeof reports the logical size.
        logicalSize := unsafe.Sizeof(*t)


        // testing.Benchmark reports the actual bytes allocated on the heap.
        // The next Size Class bucket after 40 is 48.
        res := testing.Benchmark(func(b *testing.B) {
          for i := 0; i < b.N; i++ {
            t = &[5]int64{} // Heap allocation
          }
        })


        fmt.Printf("Logical size: %d, Actual allocated: %d (Waste: %d)\n",
          logicalSize, res.AllocedBytesPerOp(), res.AllocedBytesPerOp() - uint64(logicalSize))


        // --- Case B: Large Slice ---
        // Requesting 32,769 bytes (1 byte over 32KB).
        resLarge := testing.Benchmark(func(b *testing.B) {
          for i := 0; i < b.N; i++ {
            _ = make([]byte, 32769)
          }
        })
        // Expected actual: 40,960 (5 pages of 8192)
        fmt.Printf("Request: 32769, Actual allocated: %d (Waste: %d)\n",
          resLarge.AllocedBytesPerOp(), resLarge.AllocedBytesPerOp() - 32769)
      }

    ```
      -- EXPECTED OUTPUT (64-bit Go 1.19+):
      --
        Logical size: 40, Actual allocated: 48 (Waste: 8)
        Request: 32769, Actual allocated: 40960 (Waste: 8191)



  -- Deep Dive: String Concatenation Allocations
  --
    String operations are often "allocation-heavy." Let's analyze:
    `r = string(s) + string(s)` where `len(s)` is 33.


    example
      ```go
        func concatenationAnalysis() {
          s := make([]byte, 33) // 33 bytes

          // In the expression: string(s) + string(s)
          // 1. string(s) #1 -> 33 bytes logically, but 48 bytes allocated (Size Class).
          // 2. string(s) #2 -> 33 bytes logically, but 48 bytes allocated (Size Class).
          // 3. Concatenation -> 33 + 33 = 66 bytes logically.
          //    The next Size Class bucket after 66 is 80 bytes.

          // Total bytes allocated: 48 + 48 + 80 = 176 bytes.
          // Total bytes used: 33 + 33 + 66 = 132 bytes.
          // Wasted space during this statement: 44 bytes.

          res := testing.Benchmark(func(b *testing.B) {
            for i := 0; i < b.N; i++ {
              _ = string(s) + string(s)
            }
          })

          fmt.Printf("Concat 33+33 - Allocs: %d, Total Bytes: %d\n",
            res.AllocsPerOp(), res.AllocedBytesPerOp())
        }

      ```
        -- EXPECTED OUTPUT:
        --
          Concat 33+33 - Allocs: 3, Total Bytes: 176





-- Reducing Memory Allocations
--
  In Go, minimizing memory allocations is one of the most effective ways to
  improve application performance. Fewer allocations mean less CPU spent
  finding memory blocks and significantly less pressure on the Garbage
  Collector (GC).


  -- The Strategy: Pre-allocation
  --
    The most common source of unnecessary allocations is the built-in 'append'
    function. When 'append' is used on a slice that has reached its capacity,
    the runtime must:
    1. Allocate a brand new, larger memory block.
    2. Copy all existing elements from the old block to the new one.
    3. Discard the old block (which then becomes garbage for the GC).


    The Solution: If you know (or can calculate) the final size of a
    collection, allocate the required capacity upfront using 'make(T, len, cap)'.


  -- Case Study: Merging Slices
  --
    Consider two ways to merge multiple slices into one.


    example
      ```go
        // --- Strategy A: The Naive Loop (MergeWithOneLoop) ---
        // This version is simple but inefficient. It starts with a nil slice and
        // grows it repeatedly, triggering multiple reallocations and data copies.
        func MergeWithOneLoop(data ...[]int) []int {
          var r []int
          for _, s := range data {
            r = append(r, s...)
          }
          return r
        }


        // --- Strategy B: The Pre-allocation Loop (MergeWithTwoLoops) ---
        // This version uses an extra pass to calculate the total required size,
        // then performs exactly ONE allocation.
        func MergeWithTwoLoops(data ...[]int) []int {
          n := 0
          for _, s := range data {
            n += len(s)
          }

          // Pre-allocate the exact capacity needed.
          // Initial length is 0 so append works correctly.
          r := make([]int, 0, n)

          for _, s := range data {
            r = append(r, s...)
          }
          return r
        }

      ```

    -- BENCHMARK RESULTS (Go v1.25.n):
    --
      Benchmark_MergeWithOneLoop-4    636.6 ns/op    352 B/op    4 allocs/op
      Benchmark_MergeWithTwoLoops-4   268.4 ns/op    144 B/op    1 allocs/op


    -- ANALYSIS:
    --
      The pre-allocation strategy is ~2.3x faster, uses ~60% less memory per op,
      and reduces the allocation count from 4 down to 1.





-- Advanced: Safety and Zeroing
--
  -- Handling Integer Overflow
  --
    When summing lengths of slices, it is possible (though rare) to overflow
    the 'int' type on a 32-bit system. A professional implementation guards this.


  example
    ```go
      func MergeWithSafety(data ...[]int) []int {
        n := 0
        for _, s := range data {
          if k := n + len(s); k < n {
            panic("slice length overflows")
          } else {
            n = k
          }
        }
        return make([]int, 0, n)
      }

    ```


  -- The Cost of Zeroing Memory
  --
    Note: When you call 'make([]int, 100)', Go allocates memory AND zeros it out
    (fills it with 0).


    In our 'MergeWithTwoLoops', we use 'make([]int, 0, n)'. This allocates
    memory but does NOT zero the backing array because the length is 0.
    This makes the pre-allocation even more efficient than allocating with
    full length and then indexing.





--  Zero-Allocation Optimization
--
  In high-performance Go programming, the hierarchy of efficiency is:
  1. Reducing allocations (Pre-allocation).
  2. Eliminating allocations (Zero-allocation).


  Zero-allocation is often achieved through "In-Place" algorithms. These
  algorithms reuse the memory already allocated for the input data to store
  the results, avoiding the overhead of the heap allocator and the GC entirely.


  -- The Principle: Reusing the Backing Array
  --
    Since a slice is just a header {pointer, len, cap}, we can create a new
    slice header that points to the same underlying memory block as the input.


    By maintaining a separate index for "valid" elements, we can overwrite the
    existing array as we iterate through it.


  -- Case Study: Filtering a Slice
  --
    ```go
      // Helper to check if a value should be kept (e.g., even numbers)
      func check(v int) bool {
        return v%2 == 0
      }


      // -- Strategy A: Filter with One Allocation --
      // -----------------------------------------------------------------------------
      // This version creates a brand new slice. Even with pre-allocation (make),
      // it still requests a new memory block from the heap.
      func FilterOneAllocation(data []int) []int {
        // 1 Allocation happens here
        var r = make([]int, 0, len(data))
        for _, v := range data {
          if check(v) {
            r = append(r, v)
          }
        }
        return r
      }


      // -- Strategy B: Filter with Zero Allocations (In-Place) --
      // -----------------------------------------------------------------------------
      // This version modifies the 'data' slice's underlying array. It uses
      // a "write pointer" (k) to track where the next valid element should go.
      func FilterNoAllocations(data []int) []int {
        var k = 0
        for _, v := range data {
          if check(v) {
            // Overwrite the original array at index k
            data[k] = v
            k++
          }
        }
        // Return a sub-slice of the original backing array.
        // No new memory block is allocated.
        return data[:k]
      }

    ```


  -- Performance & Benchmark Analysis
  --
    -- BENCHMARK RESULTS (1024 elements):
    --
      Benchmark_FilterOneAllocation-4    7263 ns/op    8192 B/op    1 allocs/op
      Benchmark_FilterNoAllocations-4     903 ns/op       0 B/op    0 allocs/op


    ANALYSIS:
    1. Speed: The zero-allocation version is ~8x faster.
    2. Memory: The one-allocation version uses 8KB per operation; the in-place
       version uses 0 bytes.
    3. GC Impact: The zero-allocation version produces zero garbage, meaning
       the program can run longer without triggering a GC pause.


  -- Trade-offs and Constraints
  --
    -- Mutability vs. Immutability 
    --
      - Use "One Allocation" if the original input slice must remain unchanged
        for other parts of the program.
      - Use "No Allocation" if the input is temporary or if the caller explicitly
        allows the data to be modified (mutated).


    -- Memory Leak Warning (Info Gain)
    --
      Remember that `data[:k]` still references the original large backing array.
      If the original slice was 1GB and the filtered result is only 1KB, the
      1GB block will stay in memory until the filtered result is no longer reachable.
      
      In such specific "Shrinkage" cases, a single allocation to "shrink" the
      backing array might actually be better for long-term memory health.





-- Combining Memory Blocks for Efficiency
--
  In Go, every heap allocation has a fixed CPU cost to find space and a fixed 
  overhead in the Garbage Collector's (GC) accounting. One of the most effective 
  ways to optimize a program is to consolidate many small allocations into 
  a single large memory block.


  -- The Strategy: Batch Allocation
  --
    Instead of allocating a small memory block for every individual value part, 
    we can allocate one large block (usually an array or a slice of values) to 
    carry many parts at once.


    -- Benefits 
    --
      1. Reduced CPU Usage: Fewer calls to the runtime allocator.
      2. Relieved GC Pressure: The GC has fewer objects to track and scan.
      3. Cache Locality: Data is physically closer in RAM, improving CPU cache hits.
      4. Reduced Fragmentation: Avoids "Swiss cheese" memory layouts.



  -- Code Comparison: Individual vs. Batch Allocation
  --
    ```go
      type Book struct {
        Title  string // 16 bytes
        Author string // 16 bytes
        Pages  int    // 8 bytes
      } // Total: 40 bytes (on 64-bit)


      // --- Strategy A: Many Small Blocks (Inefficient) ---
      // This creates N pointers and N separate heap blocks for the structs.
      // Total Allocations: N + 1 (The slice header + N calls to new).
      //go:noinline
      func CreateOnManySmallBlocks(n int) []*Book {
        books := make([]*Book, n)
        for i := range books {
          books[i] = new(Book) 
        }
        return books
      }

      // --- Strategy B: One Large Block (Optimized) ---
      // This creates 1 block for all structs and 1 block for the pointers.
      // Total Allocations: 2.
      //go:noinline
      func CreateOnOneLargeBlock(n int) []*Book {
        // Allocate all structs in one contiguous block
        books := make([]Book, n)
        // Create the pointer slice
        pbooks := make([]*Book, n)
        for i := range pbooks {
          // Point into the large contiguous block
          pbooks[i] = &books[i]
        }
        return pbooks
      }

    ```


  -- Benchmark Analysis (N = 100)
  --
    -- BENCHMARK RESULTS:
    --
      Benchmark_CreateOnOneLargeBlock-4      4372 ns/op    4992 B/op      2 allocs/op
      Benchmark_CreateOnManySmallBlocks-4   18017 ns/op    5696 B/op    101 allocs/op


  -- Understanding Memory Wasting (The "Padding" Factor)
  --
    Why does the "Many Small Blocks" version use MORE memory for the same data?
    
    Recall that Go uses "Size Classes" for small objects. 
    - Our `Book` struct is 40 bytes.
    - The runtime's nearest bucket (Size Class) is 48 bytes.
    
    Result: Every individual `new(Book)` wastes 8 bytes. 
    For N=100, we waste 800 bytes purely in allocator padding.
    
    In the "One Large Block" version, the structs are packed tightly at exactly 
    40-byte intervals. Only the very end of the single large block might be 
    rounded up to a size class or page boundary.


  -- The Large Block Paradox (The 32KB Threshold)
  --
    Combining blocks is usually better, but not ALWAYS. There is an edge case 
    where a large block becomes less efficient than many small ones.


    -- The Scenario: N = 820 
    --
      1. Total Struct Size: 820 * 40 bytes = 32,800 bytes.
      2. The Threshold: The maximum size for a "small object" is 32,768 bytes (32KB).
      3. The Result: 32,800 is just over the limit. Go switches to "Large Object" 
         allocation, which uses 8KB Pages.
      4. Allocation: 32,800 needs 5 pages (8192 * 5) = 40,960 bytes.
      5. Wasted Space: 40,960 - 32,800 = 8,160 bytes wasted.

      BENCHMARK (N = 820):
      Large Block:      30491 ns/op    47744 B/op      2 allocs/op
      Many Small:      145204 ns/op    46144 B/op    821 allocs/op

      OBSERVATION: 
      Even though the Large Block version wastes more RAM here (~3.5% more), 
      it is still ~4.7x faster in terms of CPU execution.





-- Value Cache Pools (Object Reuse)
--
  In performance-critical applications (like RTS games or high-frequency 
  trading systems), allocating and discarding thousands of objects per second 
  can overwhelm the Garbage Collector (GC). 

  
  Even though Go manages memory automatically, "GC Pressure" causes CPU spikes 
  and latency "jitter." The solution is to reuse objects via a Cache Pool 
  (also known as a Free List).


  -- The Problem: High-Frequency Allocation
  --
    Imagine an RTS game spawning and destroying NPCs (Non-Player Characters) 
    constantly.
    

    example
      ```go
        type NPC struct {
          name       [64]byte
          nameLen    uint16
          blood      uint16
          properties uint32
          x, y       float64
        }

        // Naive implementation: Allocates a fresh block on the heap every time.
        // This eventually leads to a "Stop the World" GC pause or dropped FPS.
        func SpawnNPC_Naive(name string, x, y float64) *NPC {
          npc := &NPC{} // Triggered heap allocation
          npc.nameLen = uint16(copy(npc.name[:], name))
          npc.x, npc.y = x, y
          return npc
        }

      ```


  -- The Solution: Custom Cache Pool (Free List)
  --
    A Cache Pool maintains a list of already allocated but currently unused 
    objects. When you need a new object, you "check out" one from the pool. 
    When done, you "return" it.


    -- Benefits 
    --
      1. Zero Allocations: Reuses existing memory blocks.
      2. Predictable Performance: Stable FPS/Latency.
      3. Thread Safety: Can be protected by a Mutex if shared across goroutines.


    example
      ```go
        var npcPool = struct {
          sync.Mutex
          *list.List
        }{
          List: list.New(),
        }


        // newNPC attempts to recycle an NPC from the pool before allocating a new one.
        func newNPC() *NPC {
          npcPool.Lock()
          defer npcPool.Unlock()

          if npcPool.Len() == 0 {
            return &NPC{} // Pool empty, fallback to allocation
          }

          // Pull from the front of the list and type assert
          element := npcPool.Front()
          return npcPool.Remove(element).(*NPC)
        }

        // releaseNPC cleans and returns an NPC to the pool.
        func releaseNPC(npc *NPC) {
          npcPool.Lock()
          defer npcPool.Unlock()

          // CRITICAL: Clear the state so old data doesn't leak into the next use.
          // Assigning a zero-value struct is an efficient way to reset all fields.
          *npc = NPC{} 

          npcPool.PushBack(npc)
        }

      ```


  -- Standard Library Alternative: `sync.Pool`
  --
    Go provides `sync.Pool` for general-purpose object caching. It has 
    specific runtime characteristics:

    1. GC Integration: Objects in a `sync.Pool` are automatically cleared by 
       the GC if they aren't used for one or two GC cycles. This prevents 
       the pool from growing infinitely and causing a memory leak.
    2. Multi-Type Support: It stores `any` (interface{}), so you can put 
       anything in it.


    example
      ```go
        func syncPoolExample() {
          var stdNpcPool = sync.Pool{
            // New specifies a function to generate a value when Get returns nil.
            New: func() any {
              return &NPC{}
            },
          }

          // CHECK OUT
          npc := stdNpcPool.Get().(*NPC)
          
          // USE
          npc.blood = 100

          // RETURN
          *npc = NPC{} // Reset state first
          stdNpcPool.Put(npc)
        }

      ```



  -- Custom Pool vs. `sync.Pool`
  --
    While `sync.Pool` is standard, many developers prefer custom pools 
    for specific scenarios:


    -- Why Custom Pools? --
    1. Persistence: `sync.Pool` wipes itself during GC. If your workload requires 
       keeping a warm cache even when the GC runs, a custom pool is better.
    2. Predictability: You can set a "Max Size" on a custom pool to cap 
       memory usage. `sync.Pool` size is managed by the runtime pacer.
    3. Localization: If the pool is only used by ONE goroutine, you can remove 
       the Mutex entirely for massive speed gains. `sync.Pool` always involves 
       concurrency overhead.
