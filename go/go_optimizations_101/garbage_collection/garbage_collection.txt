-- Garbage Collection (GC)
--
  Go's Garbage Collector is a concurrent, tri-color, mark-and-sweep collector.
  Its goal is to automate memory management, allowing developers to focus on
  logic while the runtime handles the cleanup of unused heap memory.


  -- The Two Phases of a GC Cycle
  --
    -- Scan + Mark Phase
    --
      The "Stop The World" (STW) pauses in modern Go are sub-millisecond because
      most of this phase happens concurrently with the application.

      1. Root Discovery: The GC starts by looking at "roots" (globals and pointers
         on every goroutine's stack).
      2. Traversal: It follows every pointer in those roots to find more data.
         Any memory block it can reach is marked as "alive" (non-garbage).
      3. Workload: The more pointers your program uses, the more work the GC
         must do to "scan" the object graph.


  -- Sweep Phase
  --
    Once marking is finished, any memory blocks on the heap that were NOT marked
    are considered "garbage."
    - These blocks are collected and returned to the runtime's free list.
    - They may be reused for new allocations or eventually returned to the OS.


  -- The GC Pacer: When does GC start?
  --
    The "Pacer" determines the frequency of GC cycles. It uses three main
    strategies simultaneously:


    -- The New Heap Percentage Strategy (GOGC)
    --
      This is the primary trigger. After a GC cycle, the runtime calculates a
      "Target Heap Size" based on the GOGC variable (default: 100).

        Formula: Target = (Live Heap after last GC) * (1 + GOGC/100)
        Example: If 100MB is alive after GC and GOGC=100, the next GC starts at 200MB.


    -- The Periodic Strategy (2-Minute Rule)
    --
      If no GC has been triggered by heap growth for 2 minutes, Go forces a
      cycle. This ensures that:
      - Finalizers are eventually run.
      - Idle goroutine stacks are shrunk to save memory.


    -- The Memory Limit Strategy (GOMEMLIMIT)
    --
      Introduced in Go 1.19. This is a "Soft Cap" on the total memory the
      runtime uses. As memory usage approaches this limit, the pacer triggers
      GC cycles more aggressively to avoid an OOM (Out Of Memory) crash.





-- Reducing GC Pressure: Best Practices
--
  High GC pressure leads to CPU "theft" from your main logic. You can reduce
  this pressure by following two principles:

  -- Reduce Pointer Density
  --
    The GC must scan every pointer. A struct full of pointers requires more
    scanning than a struct full of raw values (ints, bools, etc.).


    example
      ```go
        type PointerHeavy struct {
          A, B, C, D *int // GC must scan these 4 pointers
        }

        type ValueHeavy struct {
          A, B, C, D int // GC skips these (they contain no pointers)
        }

      ```


  -- Reduce Heap Allocation Frequency
  --
    Every time a local variable "escapes" to the heap, Go creates an implicit
    pointer on the stack to track it. Reducing escapes reduces the pointer
    count and the mark-phase workload.





-- The Pointer Trade-off: Copy Cost vs. Scan Cost
--
  Using pointers can reduce "Copy Cost" (passing 8 bytes instead of 1KB),
  but it increases "Scan Cost" (GC must follow the pointer).
    - Small Structs: Pass by value. (Low copy cost, Zero GC scan cost).
    - Large Structs: Pass by pointer. (High copy cost avoided, but pays scan cost).





-- Memory Fragmentation
--
  Go manages memory blocks using a variation of `tcmalloc` (Thread-Caching Malloc).
  This approach is highly effective at preventing memory fragmentation by
  using predefined "size classes" for blocks. Generally, developers do not
  need to manually optimize for fragmentation.





-- Code Demonstration: Controlling the Pacer
--
  ```go
    func gcTuningDemo() {
      // --- Adjusting GOGC Programmatically
      // SetGCPercent returns the previous setting.
      // Setting to -1 disables GC (not recommended!).
      oldGOGC := debug.SetGCPercent(200)
      fmt.Printf("GOGC tuned from %d to 200\n", oldGOGC)


      // --- Adjusting GOMEMLIMIT (Go 1.19+)
      // Useful for containers (e.g. Docker/K8s) to prevent OOMs.
      // Sets a soft limit of 1GB.
      oldLimit := debug.SetMemoryLimit(1024 * 1024 * 1024)
      fmt.Printf("Memory limit tuned from %d to 1GB\n", oldLimit)


      // --- Monitoring GC Behavior
      // You can observe GC cycles in real-time by setting the environment variable:
      // export GODEBUG=gctrace=1
    }

    func main() {
      gcTuningDemo()
    }

  ```





-- Memory Retention and GC Pressure
--
  Go's Garbage Collector (GC) operates on "Memory Blocks." A memory block is
  the atomic unit of collection. Even if your program only needs a single byte
  from a 1GB block, that entire block must remain in RAM. This behavior can
  lead to significant memory wasting if not managed carefully.


  -- The Shared Memory Block Trap
  --
    When multiple "value parts" (like elements of a slice or fields of a struct)
    share a single heap-allocated memory block, the block stays alive until
    EVERY part within it is no longer referenced.


  -- Common Scenarios
  --
    1. Keeping a single element of a massive slice/array.
    2. Keeping a single field of a large struct.
    3. Substrings/subslices where the small result "pins" a large source.


  -- Code Demonstration: The "Pinning" Proof
  --
    We can use runtime.SetFinalizer to detect exactly when a memory block is
    collected. Note: Finalizers are used here for educational proof, not for
    production logic.


    example
      ```go
        func sharedBlockDemo(pinElement bool) {
          // Create a slice of 1000 integers. This is one contiguous memory block.
          var s = make([]int, 1000)


          for i := range s {
            s[i] = i
            // Set a finalizer on the FIRST element.
            // Since all elements share one block, the finalizer effectively
            // tracks the whole block.
            if i == 0 {
              runtime.SetFinalizer(&s[i], func(ptr *int) {
                fmt.Println(">> [GC] The large memory block has been collected.")
              })
            }
          }


          var p *int
          if pinElement {
            // --- THE TRAP ---
            // We keep a pointer to just ONE element (the last one).
            // This single reference "pins" the entire 1000-element block in RAM.
            p = &s[999]
            fmt.Printf("Keeping a pointer to element 999 (Value: %d)\n", *p)
          } else {
            // --- THE FIX ---
            // We copy the value we need into a new, tiny memory block.
            v := s[999]
            p = &v
            fmt.Printf("Keeping a pointer to a CLONE of element 999 (Value: %d)\n", *p)
          }

          // Trigger GC manually to see if the block is reclaimed
          runtime.GC()

          // Use the pointer to ensure it is considered "alive" by the compiler/GC
          _ = p

          // Finalizers run in a separate goroutine; give them a moment.
          time.Sleep(time.Millisecond * 100)
        }

      ```

    -- CODE OUTPUT:
    --
      Scenario 1: pinElement = true
      Output:
         Keeping a pointer to element 999 (Value: 999)
         (No GC message - memory is pinned!)

      Scenario 2: pinElement = false
      Output:
         Keeping a pointer to a CLONE of element 999 (Value: 999)
         >> [GC] The large memory block has been collected.





-- Standard Library Hazards (Strings and Bytes)
--
  Many popular functions in the `strings` and `bytes` packages return
  "views" (substrings or subslices) that share memory with the original data.


  -- Functions that Share Memory
  --
    - strings.Split, strings.Fields, strings.Trim
    - bytes.Split, bytes.Trim


  -- The "Clone" Strategy
  --
    If you process a 10MB log line and only keep a 10-byte "Error ID", you
    should duplicate those 10 bytes to release the 10MB source.


  example
    ```go
      func subsliceOptimization(largeSource string) string {
        // Logic: Get only the first word.
        parts := strings.Fields(largeSource) // Result shares memory with largeSource
        if len(parts) == 0 {
          return ""
        }

        // FIXED (Go 1.18+): strings.Clone ensures the result lives in its
        // own fresh, tiny memory block.
        return strings.Clone(parts[0])
      }

    ```





-- Controlling Garbage Production Frequency
--
  Most GC cycles are triggered by the sheer volume of short-lived "garbage"
  objects. Reducing temporary allocations lowers GC frequency and CPU theft.


  -- Top 3 Culprits
  --
    1. String Concatenation:
       - Avoid: `s := "a" + "b" + "c"` inside loops.
       - Use: `strings.Builder` to build strings with a single allocation.


    2. String/Byte Conversions:
       - Every `[]byte(str)` or `string(bytes)` call usually makes a deep copy.
       - Tip: Check if you can work with the data in its current form or use
         compiler-optimized patterns (like `m[string(key_bytes)]`).


    3. Interface Boxing:
       - Assigning non-pointer values to an interface (e.g., `any(123)`)
         almost always requires a heap allocation.
       - Tip: Use specific types where possible instead of `any`.





-- GOGC and the Pacer Strategy
--
  Go's Garbage Collector (GC) uses a "Pacer" to determine when to start the
  next collection cycle. The primary mechanism for this is the "New Heap
  Memory Percentage" strategy, controlled by the GOGC variable.


  -- Understanding GC Roots
  --
    Before calculating the next GC trigger, the runtime identifies "Roots."
    - Roots: Value parts known to be alive (stacks and global variables).
    - Root Memory Blocks: The physical memory hosting these roots.
    - Implementation Detail: The runtime treats all stack data as scannable
      roots, even if they don't contain pointers, to simplify the engine.


  -- The GOGC Formula (The Pacer Algorithm)
  --
    When the "Mark" phase of a GC cycle finishes, the runtime calculates the
    "Target Heap Size" for the next cycle.


    -- For Go 1.18 and later 
    --
      Target = Live Heap + (Live Heap + GC Roots) * GOGC / 100


    -- For versions before 1.18 
    --
      Target = Live Heap + (Live Heap) * GOGC / 100


    -- Minimum Target
    --
      The absolute minimum target is (GOGC * 4 / 100) MB.
      With GOGC=100, the first GC usually triggers at 4MB.


  -- Configuring GOGC
  --
    1. Environment Variable: export GOGC=200
    2. Programmatic: debug.SetGCPercent(200) (returns the previous value)
    3. Disabling GC:
       - Setting GOGC=off or -1 disables the percentage strategy AND the
         "2-minute periodic GC" strategy.
       - Setting GOGC=math.MaxInt64 effectively disables the percentage
         trigger while keeping the periodic 2-minute trigger active.


  -- Monitoring GC with `gctrace`
  --
    Setting the environment variable GODEBUG=gctrace=1 prints a summary line
    for every GC cycle.


  -- Line Format 
  --
    gc # @#s #%: ..., #->#-># MB, # MB goal, # MB stacks, # MB globals ...


  -- Key Field Breakdown 
  --
    #%: Percentage of total program time spent in GC.
    #->#-># MB: Heap size at (GC Start) -> (Scan End) -> (Sweep End).
      - The third number is the "Live Heap."
    # MB goal: The Target Heap Size calculated by the pacer.


  -- Code Example: Garbage Production & Tuning
  --
    ```go
      // x is a large global array (GC Root)
      var x [512][]*int

      func garbageProducer() {
        // Seed the random generator (Pre-Go 1.20 style)
        rand.Seed(time.Now().UnixNano())

        for {
          // Generate variable amounts of "junk" data
          n := 6 + rand.Intn(6)
          for j := range x {
            // Allocate a slice (Indirect part on heap)
            x[j] = make([]*int, 1<<n)
            for k := range x[j] {
              // Allocate many small ints on heap
              x[j][k] = new(int)
            }
          }
          // Throttle slightly
          time.Sleep(time.Millisecond)
        }
      }

      func main() {
        // In a real scenario, we might use runtime/debug.SetGCPercent
        // but for this demo, we assume configuration via Env Vars.
        garbageProducer()
      }

    ```
      -- CASE 1: Default GOGC=100
      --
        $ GODEBUG=gctrace=1 go run main.go

        gc 1 @0.017s 8%: ..., 3->4->4 MB, 4 MB goal
        gc 2 @0.037s 8%: ..., 7->8->4 MB, 8 MB goal
        ...
        gc 10 @0.269s 12%: ..., 8->10->10 MB, 9 MB goal


        -- ANALYSIS:
        --
          - GC is running very frequently.
          - 12% of CPU is being "stolen" by the GC.
          - The live heap is tiny (< 10MB), causing the pacer to trigger
            at very small thresholds.


      -- CASE 2: Tuned GOGC=1000 (Trade RAM for CPU)
      --
        $ GOGC=1000 GODEBUG=gctrace=1 go run main.go

        gc 1 @0.074s 2%: ..., 38->43->15 MB, 40 MB goal
        gc 2 @0.810s 1%: ..., 160->163->9 MB, 167 MB goal
        ...
        gc 6 @3.250s 1%: ..., 199->201->8 MB, 204 MB goal


        -- ANALYSIS:
        --
          - GC frequency dropped significantly.
          - CPU overhead dropped from ~10% to ~1%.
          - The heap is allowed to grow much larger (~200MB) before a sweep.





-- GC Roots and Pacer Behavior (Go 1.18+)
--
  Since Go 1.18, a fundamental change was made to the GC Pacer: the size of 
  "GC Roots" (stacks and global variables) is now included in the calculation 
  for the next GC trigger point.


  This change means that the larger your stacks or global variables are, the 
  higher the "Target Heap" becomes, which leads to longer intervals between 
  GC cycles and more stable memory behavior.


  -- The Updated Pacer Formula
  --
    Target heap size = Live heap + (Live heap + GC roots) * GOGC / 100

    Because 'GC roots' is now part of the additive factor, large roots act 
    as a "memory ballast." They effectively "pad" the heap target, meaning 
    the program must allocate more garbage before the next cycle is triggered.


  -- Case Study 1: Large Goroutine Stacks
  --
    In this example, we create a goroutine with a massive 150 MiB array 
    allocated on its stack. 


    example
      ```go
        var x [512][]*int

        // garbageProducer creates constant churn to trigger GC cycles.
        func garbageProducer() {
          for {
            n := 6 + rand.Intn(6)
            for j := range x {
              x[j] = make([]*int, 1<<n)
              for k := range x[j] {
                x[j][k] = new(int)
              }
            }
            time.Sleep(time.Millisecond)
          }
        }

        // bigStack forces a large stack allocation.
        func bigStack(c chan int, v byte) byte {
          defer func() {
            if c != nil { c <- 1 }
          }()
          // This array stays on the stack because it doesn't escape.
          var s = []byte{150 << 20: 0} 
          for i := range s { s[i] = v }
          return s[v]
        }

        func main() {
          // Start the big stack in the background.
          go bigStack(nil, 123)
          garbageProducer()
        }

      ```
      -- GCTRACE ANALYSIS (Go 1.19+):
      --
        $ GODEBUG=gctrace=1 go run bigstacks.go

        gc 1 @0.018s ... 3->3->1 MB, 4 MB goal, 256 MB stacks
        gc 2 @0.565s ... 148->150->16 MB, 152 MB goal, 256 MB stacks
        gc 3 @1.360s ... 178->180->17 MB, 183 MB goal, 256 MB stacks


      -- OBSERVATIONS:
      --
        1. Stability: Note how the "goal" (Target Heap) stays high and consistent 
           (~150-180 MB) despite the live heap being small (1-17 MB).
        2. Frequency: Because the goal is much higher than the actual data, the GC 
           runs much less frequently compared to a program with small stacks.


        -- VERSION DIFFERENCE:
        --
          - Go 1.18: Uses the full allocated stack segment size in calculations.
          - Go 1.19+: Uses the "real" used size of the stack, though the log output 
            still reports the segment size.
                                                                                
  



-- Case Study 2: Large Global Variables
--
  Large global (package-level) variables that contain at least one pointer 
  are considered "Scannable Roots." They contribute to the ballast effect.


  example
    ```go
      // --- bigglobals.go ---
      package main

      var bigGlobal struct {
        p *int             // A pointer makes the whole struct "scannable"
        _ [150 << 20]byte  // 150 MiB of data
      }

      func main() {
        garbageProducer() // Defined above
        _ = bigGlobal.p
      }

    ```


    -- GCTRACE ANALYSIS (Go 1.18+):
    --
      $ GODEBUG=gctrace=1 go run bigglobals.go

      gc 1 @0.266s 0%: ..., 150 MB goal, ..., 150 MB globals
      gc 2 @0.885s 0%: ..., 154 MB goal, ..., 150 MB globals
      gc 3 @1.616s 0%: ..., 168 MB goal, ..., 150 MB globals

    -- OBSERVATION:
    --
      Similar to the big stacks example, the 150 MB of global roots pushes the 
      GC goal up. The pacer now thinks: "I have 150MB of base data, so I shouldn't 
      worry about triggering until the heap grows significantly relative to that 
      total base."
                                                                                
  



-- The "Ballast" Pattern
--
  Before Go 1.19 introduced `GOMEMLIMIT`, developers used a "Memory Ballast" 
  trick to stabilize GC. They would allocate a massive slice at the start 
  of `main` and keep it alive for the duration of the program.

  In Go 1.18+, because of the updated formula, this ballast effectively 
  forces the GC to run at much larger (and thus fewer) intervals, saving 
  CPU cycles at the cost of higher baseline RAM usage.





-- Ballasts vs. GOMEMLIMIT
--
  In Go, frequent Garbage Collection (GC) cycles can cause performance jitter 
  and high CPU overhead. To stabilize the GC pacer, developers historically 
  used the "Memory Ballast" trick. Since Go 1.19, the "Memory Limit" strategy 
  provides an official, cleaner alternative.


-- The Memory Ballast Trick (Legacy / Manual)
--
  Before Go 1.19, the primary way to reduce GC frequency was to artificially 
  inflate the "Live Heap" size. 


  -- How it works 
  --
    By allocating a massive byte slice that is kept alive for the duration of 
    the program, you force the GC Target formula to stay high.

    Recall: Target = Live Heap * (1 + GOGC/100)
    If your app normally uses 10MB, but you add a 150MB ballast:
    Target = (10 + 150) * (1 + 100/100) = 320MB.
    The GC won't trigger until the heap reaches ~320MB, significantly reducing 
    the number of cycles.


    example
      ```go
        func ballastDemo() {
          // 1. Allocate 150 MiB ballast.
          // NOTE: On Linux, if you don't read/write these bytes, they stay in 
          // Virtual Memory and do not consume physical (Resident) RAM.
          ballast := make([]byte, 150<<20)

          // 2. Run your program...
          go producer()

          // 3. CRITICAL: Prevent the ballast from being garbage collected.
          runtime.KeepAlive(&ballast)
        }

      ```

      -- GCTRACE ANALYSIS (GOGC=100, Ballast=150MB):
      --
        gc 2 @0.333s 5%: ..., 255->261->171 MB, 300 MB goal
        gc 3 @0.989s 2%: ..., 293->296->161 MB, 344 MB goal


      -- OBSERVATION: 
      --
        The "goal" (Target) is now locked near 300-340MB. The GC runs at a much 
        more stable and less frequent pace.
                                                                                
  
  -- The GOMEMLIMIT Strategy (Modern / Go 1.19+)
  --
    Go 1.19 introduced `GOMEMLIMIT`, which sets a "Soft Limit" on the total 
    memory the Go runtime should use. This effectively replaces the ballast 
    trick with a formal runtime setting.

    -- Key Characteristics 
    --
      1. Soft Limit: The runtime will try its best to stay under this limit, 
         but it will not crash (OOM) the program if it is exceeded.
      2. Interaction: It works alongside GOGC. If GOGC=off, GOMEMLIMIT is 
         the sole trigger for GC.
      3. Flexibility: You can set it via Environment Variable or at runtime.


    example
      ```go
        func memoryLimitDemo() {
          // --- A. Programmatic Setting ---
          // Set the soft limit to 175 MiB.
          debug.SetMemoryLimit(175 << 20)

          // --- B. Environment Variable (Preferred for Containers) ---
          // $ GOMEMLIMIT=175MiB GOGC=off GODEBUG=gctrace=1 go run main.go
        }

      ```


      -- GCTRACE ANALYSIS (GOMEMLIMIT=175MiB, GOGC=off):
      --
        gc 1 @0.283s 1%: ..., 153->155->18 MB, 157 MB goal
        gc 2 @0.784s 1%: ..., 151->152->6 MB, 155 MB goal


      -- OBSERVATION: 
      --
        The GC now ignores the percentage growth and only triggers when the heap 
        approaches the ~175MiB limit. This results in very efficient CPU usage (~1%).
                                                                                
  



-- Ballast vs. GOMEMLIMIT: Which to choose?
--
  +-----------------+---------------------------+--------------------------------+
  | Feature         | Memory Ballast            | GOMEMLIMIT                     |
  +-----------------+---------------------------+--------------------------------+
  | Go Version      | All versions              | 1.19+                          |
  | Complexity      | Requires dummy code       | Simple Env Var / Config        |
  | OOM Protection  | Poor (Constant baseline)  | Excellent (Soft cap)           |
  | Best Use Case   | Legacy Go versions        | Modern Cloud/Container apps    |
  +-----------------+---------------------------+--------------------------------+

  -- The "Thrashing" Warning 
  --
    While GOMEMLIMIT is cleaner, setting it too low is dangerous. 
    If your program's "Live Heap" (data it MUST keep) is 100MB and you set 
    GOMEMLIMIT to 105MB, the GC will run constantly (thrashing) to try and 
    save those last 5MB. 

    In contrast, a Ballast never causes thrashing because it expands the 
    Target, it doesn't compress it.
